# BuildKit 캐시 마운트와 빌드 시간 단축

## 개요
Docker BuildKit의 캐시 마운트(`--mount=type=cache`) 기능을 활용하여 빌드 시간을 단축하는 방법을 학습한다.

## 상세 내용

### BuildKit이란
Docker 18.09에서 도입된 차세대 빌드 엔진으로, Docker Engine v23.0부터는 기본 빌드 도구로 채택되었다.

기존 빌드 엔진 대비 개선된 점은 다음과 같다.
- 독립적인 빌드 스테이지를 병렬로 실행
- 체크섬 기반의 새로운 캐싱 모델 (기존 휴리스틱 방식 대비 정확도 향상)
- 최종 이미지에 영향을 주지 않는 빌드 스테이지는 자동으로 제외
- `--mount=type=cache`, `--mount=type=secret` 등의 고급 기능 지원

**활성화 방법**

Docker Engine v23.0 이상에서는 기본 활성화되어 있다. 이전 버전에서는 아래 방법을 사용한다.

```bash
# 환경변수 방식
DOCKER_BUILDKIT=1 docker build .
```

```json
// /etc/docker/daemon.json
{
  "features": {
    "buildkit": true
  }
}
```

### 기존 Docker 빌드의 캐시 문제점
기존 레이어 캐시는 **전부 아니면 무(all-or-nothing)** 방식으로 동작한다.

**레이어 캐시 무효화가 발생하는 상황**
- `COPY package.json ./` 이후 package.json이 한 줄이라도 변경되면 그 아래 모든 레이어가 무효화된다.
- 레이어가 무효화되면 `npm install`, `pip install` 등의 의존성 설치가 처음부터 다시 실행된다.
- 멀티스테이지 빌드에서 `COPY --from`으로 아티팩트를 가져올 때 이전 스테이지의 캐시가 날아가면 연쇄적으로 캐시 손실이 발생한다.

**핵심 문제**: 레이어 캐시는 이미지 레이어 단위로 관리되므로, 패키지 매니저(npm, pip, apt 등)의 내부 캐시(패키지 저장소)는 보존되지 않는다.

### --mount=type=cache 캐시 마운트

캐시 마운트는 빌드 중 컨테이너에 영구적인 디렉토리를 주입하는 기능이다. 레이어 캐시와 달리 빌드 간에 부분적인 상태(패키지 저장소, 컴파일 캐시 등)를 유지할 수 있다.

**핵심 특징**
- 캐시 마운트의 내용은 최종 이미지 레이어에 포함되지 않는다. 이미지 크기는 그대로 유지하면서 빌드 속도를 높일 수 있다.
- 레이어 캐시가 무효화되어도 캐시 마운트는 그대로 유지된다. 이것이 레이어 캐시의 "두 번째 방어선"이다.
- 여러 빌드에 걸쳐 읽고 쓸 수 있는 누적 캐시다.

**기본 문법**

```dockerfile
# syntax=docker/dockerfile:1
RUN --mount=type=cache,target=<캐시_경로> <명령어>
```

`target`에 지정한 경로가 빌드 중에 마운트되어 패키지 매니저가 캐시 데이터를 읽고 쓸 수 있게 된다.

**패키지 매니저별 캐시 디렉토리**

| 도구 | target 경로 |
|------|------------|
| apt | `/var/cache/apt` |
| pip | `/root/.cache/pip` |
| npm | `/root/.npm` |
| pnpm | `/root/.local/share/pnpm/store` |
| yarn | `/root/.yarn/cache` |
| Go | `/go/pkg/mod`, `/root/.cache/go-build` |
| Maven | `/root/.m2/repository` |
| Gradle | `/root/.gradle` |
| Cargo (Rust) | `/usr/local/cargo/registry` |
| NuGet (.NET) | `/root/.nuget/packages` |

**패키지 매니저별 실제 적용 예시**

```dockerfile
# apt
RUN --mount=type=cache,target=/var/cache/apt,sharing=locked \
    --mount=type=cache,target=/var/lib/apt,sharing=locked \
    apt-get update && apt-get install -y curl

# pip
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# npm
RUN --mount=type=cache,target=/root/.npm \
    npm ci

# pnpm
COPY package.json pnpm-lock.yaml ./
RUN --mount=type=cache,target=/root/.local/share/pnpm/store \
    pnpm install --frozen-lockfile

# Gradle (Spring Boot)
RUN --mount=type=cache,target=/root/.gradle \
    ./gradlew build -x test
```

**sharing 옵션**

병렬 빌드에서 동일한 캐시 마운트에 여러 프로세스가 접근할 때의 동작을 제어한다.

| 옵션 | 동작 | 적합한 도구 |
|------|------|-----------|
| `shared` (기본값) | 여러 빌더가 동시에 접근 가능 | npm, pip, pnpm |
| `locked` | 한 번에 하나의 빌더만 접근 (나머지는 대기) | apt (파일 잠금 필요) |
| `private` | 각 빌더가 별도의 캐시 인스턴스를 사용 | 캐시 격리가 필요한 경우 |

**id 옵션**

모노레포 등에서 여러 앱이 같은 target을 사용할 때 캐시를 격리하기 위해 id를 명시한다.

```dockerfile
RUN --mount=type=cache,target=/root/.npm,id=app-a npm ci
```

### --mount=type=secret, --mount=type=ssh

빌드 시 민감 정보를 안전하게 전달하는 방법이다.

**기존 ARG/ENV 방식의 문제**

```dockerfile
# 위험: 이미지 히스토리에 값이 남는다
ARG NPM_TOKEN
RUN echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > ~/.npmrc
```

`docker history <이미지>` 명령으로 ARG 값이 노출될 수 있다.

**--mount=type=secret 방식**

```dockerfile
# Dockerfile
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm install
```

```bash
# 빌드 시 시크릿 전달
docker build --secret id=npmrc,src=$HOME/.npmrc .
```

시크릿 파일은 빌드 중에만 마운트되며, 최종 이미지에 포함되지 않는다.

**--mount=type=ssh**

SSH 에이전트 소켓을 빌드 컨테이너에 전달한다. 비공개 Git 저장소에서 의존성을 받아야 할 때 유용하다.

```dockerfile
RUN --mount=type=ssh \
    git clone git@github.com:private/repo.git
```

```bash
# 빌드 시 SSH 포워딩
docker build --ssh default .
```

### 빌드 시간 단축 실전 전략

**1. .dockerignore 최적화**

빌드 컨텍스트에 불필요한 파일이 포함되면 컨텍스트 전송 시간이 늘어나고, COPY 명령의 체크섬이 자주 바뀌어 캐시가 쉽게 무효화된다.

```
node_modules
.git
*.log
dist
build
.env
```

**2. 레이어 순서 배치 전략**

변경 빈도가 낮은 레이어를 상위에 배치해 캐시 무효화 범위를 최소화한다.

```dockerfile
# 변경 빈도: OS 설정 < 의존성 정의 파일 < 소스 코드
FROM openjdk:21-jdk-slim

# 의존성 정의 파일만 먼저 복사 (소스 변경 시에도 이 레이어는 캐시 유지)
COPY build.gradle settings.gradle gradlew ./
COPY gradle/ gradle/
RUN --mount=type=cache,target=/root/.gradle \
    ./gradlew dependencies --no-daemon

# 소스 코드는 마지막에 복사
COPY src/ src/
RUN --mount=type=cache,target=/root/.gradle \
    ./gradlew build -x test --no-daemon
```

**3. 멀티스테이지 빌드와 캐시 마운트 조합**

빌드 도구와 최종 런타임을 분리하여 이미지 크기를 최소화하면서, 캐시 마운트로 빌드 속도를 높인다.

```dockerfile
# syntax=docker/dockerfile:1
FROM gradle:8-jdk21 AS builder

WORKDIR /app
COPY build.gradle settings.gradle gradlew ./
COPY gradle/ gradle/
RUN --mount=type=cache,target=/root/.gradle \
    ./gradlew dependencies --no-daemon

COPY src/ src/
RUN --mount=type=cache,target=/root/.gradle \
    ./gradlew bootJar -x test --no-daemon

FROM eclipse-temurin:21-jre-jammy
WORKDIR /app
COPY --from=builder /app/build/libs/*.jar app.jar
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### BuildKit 인라인 캐시 vs 외부 캐시

**--cache-from / --cache-to 옵션**

빌드된 이미지의 레이어 캐시를 외부에 저장하고 재사용한다.

```bash
# 캐시를 레지스트리에 저장
docker buildx build \
  --cache-to type=registry,ref=my-registry/my-image:cache,mode=max \
  --cache-from type=registry,ref=my-registry/my-image:cache \
  -t my-registry/my-image:latest .
```

**캐시 백엔드 비교**

| 캐시 백엔드 | 특징 | 적합한 환경 |
|------------|------|-----------|
| `type=local` | 로컬 디렉토리에 캐시 저장 | 단일 머신 |
| `type=registry` | 컨테이너 레지스트리(ECR, GCR 등)에 캐시 저장 | 분산 CI/CD |
| `type=gha` | GitHub Actions 캐시 스토리지 활용 | GitHub Actions |
| `type=s3` | AWS S3에 캐시 저장 | AWS 환경 |

**GitHub Actions에서의 캐시 활용 예시**

```yaml
- name: Build and push
  uses: docker/build-push-action@v6
  with:
    context: .
    push: true
    tags: my-registry/my-image:latest
    cache-from: type=gha
    cache-to: type=gha,mode=max
```

`type=gha`는 캐시 마운트 내용까지 포함하여 GitHub Actions 캐시 스토리지에 저장하므로, 워크플로 실행 간에 의존성 캐시가 유지된다.

## 핵심 정리
- 캐시 마운트(`--mount=type=cache`)는 레이어 캐시와 달리 패키지 매니저의 내부 캐시를 빌드 간에 보존하며, 최종 이미지에는 포함되지 않는다.
- 레이어 순서는 변경 빈도 낮은 것부터(OS 설정 → 의존성 정의 → 소스 코드) 배치하여 캐시 무효화 범위를 최소화한다.
- apt처럼 파일 잠금이 필요한 도구는 `sharing=locked`을 사용하고, npm/pip처럼 동시 접근을 지원하는 도구는 기본값인 `sharing=shared`를 사용한다.
- 민감 정보는 ARG/ENV 대신 `--mount=type=secret`을 사용해야 이미지 히스토리에 값이 남지 않는다.
- CI/CD에서는 `--cache-from` / `--cache-to`로 외부 캐시 백엔드(레지스트리, GHA, S3)를 활용해 빌드 머신이 바뀌어도 캐시를 재사용할 수 있다.

## 키워드

### Docker
Docker는 컨테이너 기반 애플리케이션을 빌드, 배포, 실행하기 위한 플랫폼이다.

### BuildKit
Docker 18.09에서 도입된 차세대 빌드 엔진으로, 병렬 빌드, 개선된 캐싱, 고급 마운트 기능(`--mount=type=cache/secret/ssh`)을 제공한다. Docker Engine v23.0부터 기본 빌드 도구이다.

### 캐시 마운트
`RUN --mount=type=cache,target=<경로>` 문법으로 사용하는 BuildKit 기능으로, 빌드 간 패키지 매니저의 내부 캐시를 유지하면서 최종 이미지에는 포함시키지 않는다.

### --mount=type=cache
BuildKit RUN 명령의 캐시 마운트 플래그. `target`(캐시 경로), `sharing`(동시 접근 정책), `id`(캐시 네임스페이스) 옵션을 지원한다.

### 멀티스테이지 빌드
하나의 Dockerfile에서 여러 `FROM` 단계를 사용해 빌드 환경과 런타임 환경을 분리하는 기법이다. 빌드 도구가 최종 이미지에 포함되지 않아 이미지 크기가 줄어든다.

### CI/CD 캐시
GitHub Actions의 `type=gha`, 레지스트리의 `type=registry` 등 외부 캐시 백엔드를 사용해 빌드 머신이 달라져도 레이어 캐시와 캐시 마운트를 재사용하는 방법이다.

### Docker 레이어 캐시
Dockerfile의 각 명령어 실행 결과를 레이어 단위로 캐시하는 기능이다. 특정 레이어가 변경되면 그 이후 레이어는 모두 무효화된다.

### .dockerignore
빌드 컨텍스트에서 제외할 파일/디렉토리를 지정하는 파일이다. `node_modules`, `.git` 등을 제외해 컨텍스트 전송 시간을 줄이고 불필요한 캐시 무효화를 방지한다.

### 빌드 최적화
레이어 순서 배치, 캐시 마운트, 멀티스테이지 빌드, .dockerignore 관리, 외부 캐시 백엔드 활용 등을 조합해 Docker 빌드 시간과 이미지 크기를 줄이는 전략이다.

### buildx
Docker의 확장 빌드 CLI 플러그인으로, BuildKit 기능을 완전히 활용할 수 있다. 멀티 플랫폼 빌드, 외부 캐시 백엔드(`--cache-from`, `--cache-to`) 등의 고급 기능을 제공한다.
