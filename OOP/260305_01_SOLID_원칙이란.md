# SOLID 원칙이란

## 개요
객체지향 설계의 5가지 핵심 원칙인 SOLID에 대해 학습한다. 유지보수성과 확장성이 높은 소프트웨어를 설계하기 위한 기본 지침이다.

SOLID는 로버트 C. 마틴(Robert C. Martin, "Uncle Bob")이 2000년 논문 "Design Principles and Design Patterns"에서 제시한 설계 원칙들을 마이클 페더스(Michael Feathers)가 두문자어로 정리한 것이다.

| 약어 | 원칙명 | 핵심 요약 |
|------|--------|-----------|
| S | Single Responsibility Principle | 하나의 클래스는 하나의 변경 이유만 가진다 |
| O | Open/Closed Principle | 확장엔 열려 있고, 수정엔 닫혀 있어야 한다 |
| L | Liskov Substitution Principle | 하위 타입은 상위 타입을 완전히 대체할 수 있어야 한다 |
| I | Interface Segregation Principle | 사용하지 않는 메서드에 의존하도록 강요하지 않는다 |
| D | Dependency Inversion Principle | 고수준 모듈과 저수준 모듈 모두 추상화에 의존한다 |

---

## SRP - 단일 책임 원칙 (Single Responsibility Principle)

### 정의 및 목적
> "A class should have one, and only one, reason to change." - Robert C. Martin

하나의 클래스는 하나의 책임만 가져야 한다. 여기서 "책임"은 기능의 개수가 아니라 **변경의 이유**를 의미한다. 클래스가 여러 책임을 가지면 하나의 책임 변경이 다른 책임에 예기치 않은 영향을 줄 수 있다.

SRP를 지키면 각 클래스가 응집도(Cohesion)가 높아지고, 결합도(Coupling)가 낮아진다.

### 위반 사례

```java
// Before: 사용자 정보 처리, DB 저장, 이메일 발송을 하나의 클래스가 담당
public class UserService {

    public void registerUser(String name, String email) {
        // 1. 비즈니스 로직
        if (name == null || email == null) {
            throw new IllegalArgumentException("이름과 이메일은 필수입니다.");
        }

        // 2. DB 저장 로직
        String sql = "INSERT INTO users (name, email) VALUES (?, ?)";
        // ... JDBC 코드 ...

        // 3. 이메일 발송 로직
        String subject = "회원가입 완료";
        String body = name + "님, 환영합니다!";
        // ... SMTP 코드 ...
    }
}
```

위 코드는 비즈니스 로직, DB 처리, 이메일 발송이라는 세 가지 변경 이유를 가진다. DB를 JPA로 바꾸거나 이메일 템플릿을 수정할 때 모두 이 클래스를 수정해야 한다.

### 올바른 적용 예시

```java
// After: 각 책임을 별도 클래스로 분리

// 비즈니스 로직만 담당
public class UserService {

    private final UserRepository userRepository;
    private final EmailService emailService;

    public UserService(UserRepository userRepository, EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }

    public void registerUser(String name, String email) {
        if (name == null || email == null) {
            throw new IllegalArgumentException("이름과 이메일은 필수입니다.");
        }
        userRepository.save(new User(name, email));
        emailService.sendWelcomeEmail(name, email);
    }
}

// DB 저장만 담당
public class UserRepository {
    public void save(User user) {
        String sql = "INSERT INTO users (name, email) VALUES (?, ?)";
        // ... JDBC 코드 ...
    }
}

// 이메일 발송만 담당
public class EmailService {
    public void sendWelcomeEmail(String name, String email) {
        String subject = "회원가입 완료";
        String body = name + "님, 환영합니다!";
        // ... SMTP 코드 ...
    }
}
```

이제 각 클래스는 단 하나의 변경 이유만 가진다. DB 변경은 `UserRepository`, 이메일 로직 변경은 `EmailService`만 수정하면 된다.

## OCP - 개방-폐쇄 원칙 (Open/Closed Principle)

### 정의 및 목적
> "Software entities should be open for extension, but closed for modification." - Bertrand Meyer / Robert C. Martin

기존 코드를 수정하지 않고도 새로운 기능을 추가할 수 있어야 한다. 추상화(인터페이스/추상 클래스)를 통해 구현한다. 새로운 요구사항이 생길 때마다 기존 코드를 건드리면 버그가 발생하기 쉽고, 기존 테스트가 깨질 위험이 있다.

### 위반 사례

```java
// Before: 새로운 결제 수단이 추가될 때마다 PaymentService를 수정해야 한다
public class PaymentService {

    public void pay(String paymentType, int amount) {
        if (paymentType.equals("CREDIT_CARD")) {
            System.out.println("신용카드로 " + amount + "원 결제");
        } else if (paymentType.equals("KAKAO_PAY")) {
            System.out.println("카카오페이로 " + amount + "원 결제");
        }
        // 새로운 결제 수단 추가 시 이 클래스를 계속 수정해야 함
    }
}
```

### 올바른 적용 예시 (전략 패턴 활용)

```java
// After: 인터페이스를 통해 확장에 열려 있고, 수정에 닫혀 있는 구조

// 추상화 - 결제 전략 인터페이스
public interface PaymentStrategy {
    void pay(int amount);
}

// 구현체 1
public class CreditCardPayment implements PaymentStrategy {
    @Override
    public void pay(int amount) {
        System.out.println("신용카드로 " + amount + "원 결제");
    }
}

// 구현체 2
public class KakaoPayPayment implements PaymentStrategy {
    @Override
    public void pay(int amount) {
        System.out.println("카카오페이로 " + amount + "원 결제");
    }
}

// 새로운 결제 수단 추가 시 PaymentService 수정 없이 구현체만 추가
public class NaverPayPayment implements PaymentStrategy {
    @Override
    public void pay(int amount) {
        System.out.println("네이버페이로 " + amount + "원 결제");
    }
}

// 고수준 모듈 - 수정 없이 어떤 전략이든 받아서 처리
public class PaymentService {

    private final PaymentStrategy paymentStrategy;

    public PaymentService(PaymentStrategy paymentStrategy) {
        this.paymentStrategy = paymentStrategy;
    }

    public void processPayment(int amount) {
        paymentStrategy.pay(amount);
    }
}
```

새로운 결제 수단이 추가되어도 `PaymentService`는 전혀 수정하지 않고, `PaymentStrategy` 인터페이스를 구현하는 새 클래스만 추가하면 된다.

### 전략 패턴과의 관계
OCP는 **전략 패턴(Strategy Pattern)**, **템플릿 메서드 패턴(Template Method Pattern)**, **데코레이터 패턴(Decorator Pattern)** 등을 통해 자연스럽게 구현된다. 공통점은 모두 추상화를 통해 "변하는 것"과 "변하지 않는 것"을 분리한다는 점이다.

## LSP - 리스코프 치환 원칙 (Liskov Substitution Principle)

### 정의 및 목적
> "If S is a subtype of T, then objects of type T may be replaced with objects of type S without altering any of the desirable properties of the program." - Barbara Liskov (1987)

상위 타입(부모 클래스/인터페이스)을 사용하는 코드에서 하위 타입(자식 클래스)으로 교체해도 프로그램의 정확성이 깨지지 않아야 한다. 단순히 컴파일이 되는 것이 아니라 **행동적으로도** 일관성이 있어야 한다.

### 위반 사례: 직사각형-정사각형 문제

```java
// Before: 정사각형이 직사각형을 상속 - LSP 위반

public class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) {
        this.width = width;
    }

    public void setHeight(int height) {
        this.height = height;
    }

    public int getArea() {
        return width * height;
    }
}

public class Square extends Rectangle {

    @Override
    public void setWidth(int width) {
        // 정사각형은 가로와 세로가 같아야 하므로 둘 다 변경
        this.width = width;
        this.height = width;
    }

    @Override
    public void setHeight(int height) {
        this.width = height;
        this.height = height;
    }
}

// 클라이언트 코드 - Rectangle을 기대하지만 Square를 받으면 결과가 달라진다
public class ShapeTest {
    public static void testRectangle(Rectangle rect) {
        rect.setWidth(5);
        rect.setHeight(4);
        // Rectangle이라면 넓이가 20이어야 하지만
        // Square를 넣으면 16이 된다 -> LSP 위반!
        assert rect.getArea() == 20 : "넓이는 20이어야 합니다. 실제: " + rect.getArea();
    }
}
```

수학적으로 "정사각형은 직사각형이다"가 성립해도, 소프트웨어에서는 **행동(Behavior)의 계약**이 다르기 때문에 LSP를 위반한다.

### 올바른 적용 예시

```java
// After: 상속 대신 공통 추상화(인터페이스)를 사용

public interface Shape {
    int getArea();
}

public class Rectangle implements Shape {
    private final int width;
    private final int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }

    @Override
    public int getArea() {
        return width * height;
    }
}

public class Square implements Shape {
    private final int side;

    public Square(int side) {
        this.side = side;
    }

    @Override
    public int getArea() {
        return side * side;
    }
}

// 이제 Shape 인터페이스를 사용하는 코드는 Rectangle이든 Square든 안전하게 동작
public class AreaCalculator {
    public int calculateTotalArea(List<Shape> shapes) {
        return shapes.stream()
                     .mapToInt(Shape::getArea)
                     .sum();
    }
}
```

### 상속 시 계약(Contract)을 깨뜨리지 않는 설계 원칙
- 하위 클래스는 상위 클래스의 사전조건(Precondition)을 강화해서는 안 된다.
- 하위 클래스는 상위 클래스의 사후조건(Postcondition)을 약화해서는 안 된다.
- 하위 클래스는 상위 클래스의 불변식(Invariant)을 유지해야 한다.

## ISP - 인터페이스 분리 원칙 (Interface Segregation Principle)

### 정의 및 목적
> "Clients should not be forced to depend upon interfaces that they do not use." - Robert C. Martin

클라이언트가 자신이 사용하지 않는 메서드에 의존하도록 강제해서는 안 된다. 하나의 범용적인 인터페이스보다 여러 개의 구체적인 인터페이스가 낫다. 불필요한 의존성을 제거하여 변경의 영향 범위를 줄인다.

### 위반 사례

```java
// Before: 하나의 큰 인터페이스가 모든 기능을 포함
public interface Worker {
    void work();
    void eat();
    void sleep();
}

// 로봇은 eat()과 sleep()이 필요 없지만 구현을 강요받는다
public class Robot implements Worker {

    @Override
    public void work() {
        System.out.println("로봇이 일합니다.");
    }

    @Override
    public void eat() {
        // 로봇은 밥을 먹지 않으므로 불필요한 메서드
        throw new UnsupportedOperationException("로봇은 밥을 먹지 않습니다.");
    }

    @Override
    public void sleep() {
        // 로봇은 잠을 자지 않으므로 불필요한 메서드
        throw new UnsupportedOperationException("로봇은 잠을 자지 않습니다.");
    }
}
```

### 올바른 적용 예시

```java
// After: 인터페이스를 역할 단위로 분리

public interface Workable {
    void work();
}

public interface Eatable {
    void eat();
}

public interface Sleepable {
    void sleep();
}

// 사람은 세 가지 모두 구현
public class Human implements Workable, Eatable, Sleepable {

    @Override
    public void work() {
        System.out.println("사람이 일합니다.");
    }

    @Override
    public void eat() {
        System.out.println("사람이 밥을 먹습니다.");
    }

    @Override
    public void sleep() {
        System.out.println("사람이 잠을 잡니다.");
    }
}

// 로봇은 필요한 인터페이스만 구현
public class Robot implements Workable {

    @Override
    public void work() {
        System.out.println("로봇이 일합니다.");
    }
}

// 클라이언트는 필요한 인터페이스만 의존
public class WorkManager {
    // Workable 인터페이스만 알면 되므로 Robot이든 Human이든 상관없다
    public void manageWork(Workable worker) {
        worker.work();
    }
}
```

### 실무 예시: Spring Data Repository

Spring Data JPA의 `Repository` 계층 구조가 ISP의 좋은 예이다.

```java
// CrudRepository, PagingAndSortingRepository, JpaRepository로 세분화되어 있어
// 필요한 기능만 골라서 의존할 수 있다
public interface UserRepository extends CrudRepository<User, Long> {
    // 단순 CRUD만 필요하다면 JpaRepository 전체를 의존할 필요가 없다
}
```

## DIP - 의존 역전 원칙 (Dependency Inversion Principle)

### 정의 및 목적
> "High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions." - Robert C. Martin

고수준 모듈(비즈니스 로직)이 저수준 모듈(구체적인 구현)에 직접 의존하면, 저수준 모듈 변경 시 고수준 모듈도 수정해야 한다. 둘 다 추상화(인터페이스)에 의존하게 하면 이 문제가 해결된다.

### 위반 사례

```java
// Before: 고수준 모듈이 저수준 모듈(구체 클래스)에 직접 의존
public class MySQLUserRepository {
    public User findById(Long id) {
        // MySQL 전용 구현
        return new User(id, "Alice");
    }
}

// OrderService(고수준)가 MySQLUserRepository(저수준)에 직접 의존
// MySQL을 다른 DB로 교체하면 OrderService도 수정해야 한다
public class OrderService {
    private MySQLUserRepository userRepository = new MySQLUserRepository();

    public void placeOrder(Long userId, String productName) {
        User user = userRepository.findById(userId);
        System.out.println(user.getName() + "님이 " + productName + "을 주문했습니다.");
    }
}
```

### 올바른 적용 예시

```java
// After: 추상화에 의존하도록 구조 변경

// 추상화 - 인터페이스
public interface UserRepository {
    User findById(Long id);
}

// 저수준 모듈 - MySQL 구현체
public class MySQLUserRepository implements UserRepository {
    @Override
    public User findById(Long id) {
        // MySQL 전용 구현
        return new User(id, "Alice");
    }
}

// 저수준 모듈 - MongoDB 구현체 (DB 교체 시 이것만 추가)
public class MongoUserRepository implements UserRepository {
    @Override
    public User findById(Long id) {
        // MongoDB 전용 구현
        return new User(id, "Bob");
    }
}

// 고수준 모듈 - 추상화(인터페이스)에만 의존
public class OrderService {
    private final UserRepository userRepository; // 인터페이스에 의존

    // 생성자 주입 - 어떤 구현체가 들어오든 상관없다
    public OrderService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public void placeOrder(Long userId, String productName) {
        User user = userRepository.findById(userId);
        System.out.println(user.getName() + "님이 " + productName + "을 주문했습니다.");
    }
}
```

### Spring의 DI(Dependency Injection)와의 관계

Spring Framework는 DIP를 구현하는 가장 대표적인 프레임워크이다. `@Autowired`, `@Component`, `@Service` 등을 통해 Spring 컨테이너가 의존성을 주입한다.

```java
// Spring에서 DIP 적용 예시
@Service
public class OrderService {

    private final UserRepository userRepository;

    // Spring이 UserRepository 구현체를 자동으로 주입
    @Autowired
    public OrderService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}

@Repository
public class JpaUserRepository implements UserRepository {
    // JPA를 이용한 구현
}
```

`application.properties`나 `@Profile` 등을 통해 구현체를 교체해도 `OrderService` 코드는 전혀 바꾸지 않아도 된다.

### 계층형 아키텍처에서의 적용

```
[표현 계층] → [서비스 인터페이스] ← [서비스 구현체]
                      ↓
             [저장소 인터페이스] ← [저장소 구현체]
```

각 계층은 하위 계층의 인터페이스에만 의존하고, 구체적인 구현체는 런타임에 주입된다. 이 구조를 통해 각 계층을 독립적으로 테스트(Mocking)할 수 있다.

## SOLID 원칙 간의 관계

SOLID 원칙들은 독립적으로 존재하는 것이 아니라 서로 보완하며 협력한다.

```
SRP → 클래스를 잘 분리하면
OCP → 기존 코드 수정 없이 확장할 수 있게 되고
LSP → 올바른 상속 구조로 다형성을 안전하게 활용하게 되며
ISP → 인터페이스를 잘게 나눠 불필요한 의존을 없애면
DIP → 추상화에 의존하는 구조가 자연스럽게 완성된다
```

| 원칙 | 주요 해결 문제 | 핵심 도구 |
|------|---------------|-----------|
| SRP | 높은 결합도, 낮은 응집도 | 클래스 분리 |
| OCP | 변경에 취약한 구조 | 추상화, 다형성 |
| LSP | 잘못된 상속으로 인한 버그 | 계약 기반 설계 |
| ISP | 불필요한 의존성 | 인터페이스 분리 |
| DIP | 구체 구현에 대한 강결합 | 의존성 주입 |

### 실무에서 함께 적용하는 패턴

**전략 패턴(Strategy Pattern)**: OCP + DIP를 함께 구현. 인터페이스로 전략을 추상화하고(DIP), 새 전략 추가 시 기존 코드 수정 없이 구현체만 추가(OCP).

**팩토리 패턴(Factory Pattern)**: OCP + DIP를 함께 구현. 객체 생성 로직을 팩토리에 위임하여 클라이언트가 구체 클래스를 몰라도 되도록 한다.

**레이어드 아키텍처(Layered Architecture)**: SRP + DIP를 함께 구현. 각 계층이 단일 책임을 가지고(SRP), 계층 간 의존은 인터페이스를 통해 이루어진다(DIP).

## SOLID 위반 시 발생하는 문제

로버트 C. 마틴은 설계 원칙을 위반한 소프트웨어의 세 가지 주요 증상을 다음과 같이 정의했다.

### 코드 경직성 (Rigidity)

**정의**: 모듈 간의 결합도가 너무 높아서 한 곳의 변경이 연쇄적인 수정을 요구하는 구조.

**증상**: "이 기능 하나 바꾸려고 했는데, 수정할 파일이 10개가 넘어요."

**원인**: SRP/DIP 위반으로 하나의 클래스가 너무 많은 것을 알거나, 고수준이 저수준에 직접 의존하는 경우.

```
변경 요청 → 클래스 A 수정 → 클래스 B도 수정 → 클래스 C도 수정 → ...
```

### 코드 취약성 (Fragility)

**정의**: 한 부분을 수정했는데 전혀 관련 없어 보이는 다른 부분이 깨지는 현상.

**증상**: "결제 로직을 수정했는데 왜 알림 기능이 안 되죠?"

**원인**: 클래스 간 숨겨진 결합(Hidden Coupling)이 존재하여, 한 변경의 파급 효과가 예측 불가능한 경우.

```
결제 로직 수정 → 예상치 못한 알림 기능 오류 발생
```

### 부동성 (Immobility)

**정의**: 재사용하고 싶은 코드가 있지만 다른 모듈과 너무 얽혀 있어 분리할 수 없는 상태.

**증상**: "저 프로젝트에서 이 유틸 기능만 가져다 쓰고 싶은데, 떼어내면 반이 부서져요."

**원인**: 모듈이 특정 컨텍스트에 강하게 결합되어 있어 독립적으로 재사용할 수 없는 경우.

```
재사용하고 싶은 모듈 → DB 구현체 의존 → 이메일 모듈 의존 → ... (분리 불가)
```

### 세 가지 증상의 공통 원인

세 증상 모두 **높은 결합도(High Coupling)** 와 **낮은 응집도(Low Cohesion)** 에서 비롯된다. SOLID 원칙은 이를 해결하기 위한 가이드라인이다.

---

## 핵심 정리

- **SRP**: 클래스를 변경할 이유가 두 가지 이상이면 분리를 고려하라. 응집도를 높이고 결합도를 낮추는 첫 번째 단계다.
- **OCP**: 요구사항 변경 시 기존 코드를 수정하는 것이 맞는지 의심하라. 인터페이스/추상 클래스로 추상화하면 수정 없이 확장이 가능하다.
- **LSP**: "is-a" 관계가 성립한다고 해서 무조건 상속을 쓰지 마라. 행동적 계약(Behavioral Contract)이 유지되어야 진정한 is-a 관계다.
- **ISP**: 인터페이스를 통해 불필요한 의존을 만들지 마라. 인터페이스가 크면 사용하는 쪽에서 필요 없는 메서드까지 구현해야 한다.
- **DIP**: `new` 키워드로 구체 클래스를 직접 생성하는 코드가 많다면 DIP를 위반하고 있을 가능성이 높다. 의존성 주입(DI)을 통해 추상화에 의존하라.
- SOLID는 서로 보완적이다. 하나를 잘 지키면 다른 원칙들도 자연스럽게 따라온다.
- 원칙을 지키는 목적은 변경에 강하고, 테스트하기 쉽고, 재사용 가능한 코드를 만들기 위해서다.

## 키워드

`SOLID`, `SRP`, `OCP`, `LSP`, `ISP`, `DIP`, `객체지향 설계`, `추상화`, `다형성`, `의존성 주입`

- **SOLID**: Robert C. Martin이 정리한 객체지향 설계의 5가지 핵심 원칙의 두문자어
- **SRP (Single Responsibility Principle)**: 하나의 클래스는 하나의 변경 이유만 가져야 한다는 단일 책임 원칙
- **OCP (Open/Closed Principle)**: 확장에는 열려 있고 수정에는 닫혀 있어야 한다는 개방-폐쇄 원칙
- **LSP (Liskov Substitution Principle)**: 하위 타입은 상위 타입을 완전히 대체해도 프로그램이 올바르게 동작해야 한다는 리스코프 치환 원칙
- **ISP (Interface Segregation Principle)**: 클라이언트가 사용하지 않는 메서드에 의존하도록 강요해서는 안 된다는 인터페이스 분리 원칙
- **DIP (Dependency Inversion Principle)**: 고수준 모듈과 저수준 모듈 모두 추상화에 의존해야 한다는 의존 역전 원칙
- **객체지향 설계**: 현실 세계의 개념을 객체로 모델링하여 소프트웨어를 설계하는 패러다임
- **추상화**: 복잡한 구현 세부사항을 숨기고 공통적인 인터페이스나 개념만 드러내는 것
- **다형성**: 동일한 인터페이스(메서드 시그니처)에 대해 서로 다른 구현체가 각자의 방식으로 동작하는 성질
- **의존성 주입 (DI, Dependency Injection)**: 객체가 필요로 하는 의존성을 직접 생성하지 않고 외부에서 주입받는 패턴으로 DIP를 구현하는 대표적인 기법
