# SQS와 Kafka의 차이

## 개요
메시지 큐(SQS)와 이벤트 스트리밍 플랫폼(Kafka)은 비동기 메시징 시스템이지만, 설계 철학과 사용 목적이 근본적으로 다르다. 두 시스템의 아키텍처, 메시지 처리 방식, 적합한 사용 사례를 비교하여 올바른 선택 기준을 학습한다.

## 상세 내용

### 1. 기본 개념과 설계 철학

**Amazon SQS (Simple Queue Service)**
SQS는 AWS에서 제공하는 완전 관리형 메시지 큐 서비스다. 핵심 설계 철학은 "작업(Task)의 분산 처리"에 있다. 메시지는 생산자(Producer)가 큐에 넣고, 소비자(Consumer) 중 하나가 꺼내 처리한 뒤 삭제하는 Point-to-Point 모델로 동작한다. 각 메시지는 단 하나의 Consumer에 의해 처리되는 것을 기본으로 하며, AWS 인프라가 모든 운영을 자동으로 관리한다.

**Apache Kafka**
Kafka는 LinkedIn에서 탄생한 분산 이벤트 스트리밍 플랫폼이다. 핵심 설계 철학은 "이벤트 로그의 영속적 보존과 다중 소비"에 있다. 메시지(이벤트)는 토픽(Topic)에 순서대로 추가되며, 소비자는 자신의 위치(Offset)를 기억하고 독립적으로 이벤트를 읽는다. 동일한 이벤트를 여러 Consumer Group이 각자의 목적으로 소비할 수 있는 Pub/Sub 모델을 지원한다.

| 항목      | SQS                    | Kafka                        |
| --------- | ---------------------- | ---------------------------- |
| 분류      | 메시지 큐              | 이벤트 스트리밍 플랫폼       |
| 모델      | Point-to-Point         | Pub/Sub + 분산 로그          |
| 관리      | 완전 관리형 (서버리스) | 클러스터 직접 운영 또는 MSK  |
| 핵심 목적 | 작업의 비동기 분산     | 이벤트의 영속적 스트리밍     |

### 2. 메시지 소비 모델 차이

**SQS의 소비 모델**
SQS는 Pull 기반으로 동작한다. Consumer가 직접 큐에 메시지를 요청하고, 메시지를 성공적으로 처리한 뒤 명시적으로 삭제(Delete)해야 한다. 메시지를 꺼낸 순간부터 Visibility Timeout이 시작되며, 이 시간 안에 삭제되지 않으면 메시지는 다시 큐에 노출되어 다른 Consumer가 처리할 수 있게 된다.

핵심은 "소비 후 삭제"라는 단방향 흐름이다. 한번 소비된 메시지는 복구가 불가능하고, 동일 메시지를 두 Consumer가 동시에 처리할 수 없다.

**Kafka의 소비 모델**
Kafka 역시 Pull 기반이지만, SQS와 결정적인 차이가 있다. 메시지는 소비해도 삭제되지 않고 Retention Period 동안 파티션 로그에 그대로 남는다. 각 Consumer Group은 자신이 어디까지 읽었는지를 Offset이라는 값으로 관리한다.

이 구조 덕분에 여러 Consumer Group이 동일 토픽을 독립적으로 소비할 수 있고, 문제가 생기면 Offset을 이전 시점으로 되돌려 재처리(Reprocessing)할 수 있다.

### 3. 메시지 순서 보장

**SQS의 순서 보장**
SQS는 큐 타입에 따라 순서 보장 수준이 다르다.

- **Standard Queue**: 높은 처리량을 우선시하며, 최선(Best-effort) 순서만 보장한다. 동일 메시지가 중복 전달될 수 있고 순서가 뒤바뀔 수 있다 (At-least-once 전달).
- **FIFO Queue**: 선입선출 순서를 엄격하게 보장하며 중복을 제거한다 (Exactly-once 처리). 단, 처리량이 초당 최대 300 TPS(배치 시 3,000 TPS)로 제한된다. Message Group ID 단위로 순서를 관리하므로, 서로 다른 그룹 간에는 병렬 처리가 가능하다.

**Kafka의 순서 보장**
Kafka는 파티션(Partition) 단위로 순서를 엄격하게 보장한다. 동일 파티션에 기록된 메시지는 항상 쓰인 순서대로 읽힌다. 단, 여러 파티션에 걸친 전체 토픽 수준의 순서는 보장되지 않는다.

따라서 순서가 중요한 데이터(예: 동일 사용자의 이벤트 시퀀스)는 동일 Partition Key를 지정하여 같은 파티션으로 라우팅해야 한다. 파티션 수가 많을수록 병렬 처리 능력이 높아지지만, 전체 순서 보장은 어려워지는 트레이드오프가 존재한다.

### 4. 메시지 보존과 재처리

**SQS의 메시지 보존**
SQS는 메시지를 최대 14일까지 보존할 수 있으며, 기본값은 4일이다. Consumer가 메시지를 성공적으로 처리하면 반드시 삭제 API를 호출해야 한다. 삭제하지 않으면 Visibility Timeout 이후 큐에 재노출된다.

재처리가 필요한 경우 DLQ(Dead Letter Queue)를 통해 실패 메시지를 별도로 보관하고, DLQ Redrive 기능으로 원래 큐로 돌려보낼 수 있다. 하지만 구조적으로 정상 처리된 메시지를 특정 시점으로 돌아가 전체 재처리하는 것은 불가능하다.

**Kafka의 메시지 보존**
Kafka의 가장 강력한 특징 중 하나는 메시지의 독립적 보존이다. Consumer가 메시지를 읽어도 토픽에서 삭제되지 않으며, 설정된 Retention Period(기본 7일, 용량 기반 또는 무기한 설정 가능) 동안 유지된다.

재처리가 필요하다면 `--reset-offsets` 명령으로 Consumer Group의 Offset을 특정 시점(earliest, latest, 특정 timestamp 등)으로 되돌릴 수 있다. 이는 버그 수정 후 특정 기간의 이벤트를 전체 재처리하거나, 새로운 Consumer Group을 붙여 과거 데이터를 처음부터 처리하는 시나리오에서 매우 강력하다.

**Kafka 메시지 보존 기능의 실무 활용 사례**

1. **장애 복구 및 버그 수정 후 재처리**: 결제 처리 Consumer에 버그가 있어 일부 결제가 누락되었다면, 버그를 수정한 뒤 Offset을 장애 발생 시점으로 되돌려 해당 기간의 결제 이벤트를 전체 재처리할 수 있다. SQS였다면 이미 삭제된 메시지를 복구할 방법이 없다.

2. **새로운 서비스 투입 시 과거 데이터 소급 처리**: 기존에 주문 이벤트를 처리하던 시스템에 "추천 엔진"을 새로 추가한다고 가정하자. 새 Consumer Group을 만들어 Offset을 earliest로 설정하면, 과거 주문 데이터를 처음부터 읽어 추천 모델의 초기 학습 데이터로 활용할 수 있다.

3. **감사 로그(Audit Log) 및 규정 준수**: 금융, 의료 등 규제 산업에서는 모든 트랜잭션 이벤트를 일정 기간 보존해야 한다. Kafka의 Retention을 장기(수개월~무기한)로 설정하면 이벤트 로그 자체가 감사 추적(Audit Trail) 역할을 한다.

4. **A/B 테스트 및 데이터 파이프라인 검증**: 새로운 데이터 처리 로직을 배포하기 전, 별도 Consumer Group으로 과거 이벤트를 흘려보내 기존 결과와 비교(Shadow Testing)할 수 있다. 실제 트래픽 없이도 프로덕션 데이터로 검증이 가능하다.

5. **이벤트 소싱(Event Sourcing) 저장소**: 애플리케이션 상태를 이벤트의 누적으로 관리하는 패턴에서 Kafka 토픽이 이벤트 저장소 역할을 한다. 어떤 시점의 상태든 이벤트를 처음부터 재생(Replay)하여 복원할 수 있다.

### 5. 확장성과 처리량

**SQS의 확장성**
SQS는 AWS가 자동으로 처리량을 관리하는 서버리스 서비스다. Standard Queue는 이론상 무제한에 가까운 TPS를 지원하며, 사용자가 별도의 스케일링 작업 없이 트래픽 급증을 흡수할 수 있다. 메시지 크기는 최대 256KB이며, 더 큰 데이터는 S3에 저장하고 SQS에는 참조 링크만 전달하는 패턴을 사용한다.

단점은 처리량이 Consumer 수에 비례하므로, 처리 속도를 높이려면 Consumer 인스턴스를 수평 확장해야 한다.

**Kafka의 확장성**
Kafka의 처리량은 파티션(Partition) 수에 의해 결정된다. 파티션 수를 늘리면 그에 비례하여 병렬 처리가 가능해진다. Kafka는 초당 수백만 건의 메시지를 처리할 수 있는 수준의 극고(high) 처리량을 제공하며, 대용량 데이터 파이프라인에 적합하다.

AWS의 Amazon MSK(Managed Streaming for Apache Kafka)를 사용하면 클러스터 운영 부담을 줄일 수 있다. MSK Express 브로커는 표준 Kafka 브로커 대비 최대 3배의 처리량을 제공하며, MSK Serverless는 클러스터 용량 관리 없이 사용량 기반으로 자동 확장된다.

### 6. Consumer Group과 메시지 분배

**SQS의 메시지 분배**
SQS에는 Consumer Group 개념이 없다. 동일 큐를 구독하는 여러 Consumer는 서로 메시지를 경쟁적으로 가져간다(Competing Consumers 패턴). 하나의 메시지는 반드시 하나의 Consumer만 처리하며, 이는 수평 확장을 통한 처리 속도 향상에 유리하다.

만약 동일 메시지를 여러 시스템이 각각 처리해야 한다면(예: 주문 완료 이벤트를 재고 시스템과 알림 시스템이 모두 받아야 하는 경우), SQS 앞에 SNS(Simple Notification Service)를 두고 Fan-out 구조를 별도로 구성해야 한다.

**Kafka의 Consumer Group**
Kafka는 Consumer Group이라는 개념을 통해 유연한 메시지 분배를 기본으로 지원한다.

- **그룹 내 분배**: 동일 Consumer Group에 속한 Consumer들은 파티션을 나눠 맡는다. Consumer가 3개이고 파티션이 6개라면 각 Consumer가 2개 파티션을 담당한다. 이를 통해 수평 확장 시 자동으로 부하가 분산된다.
- **그룹 간 독립 소비**: 서로 다른 Consumer Group은 완전히 독립적으로 동일 토픽을 소비한다. 재고 시스템과 알림 시스템이 각자의 Consumer Group으로 동일 토픽을 구독하면, 별도의 구성 없이 두 시스템이 모든 이벤트를 각자 처리할 수 있다.

Consumer가 추가되거나 장애가 발생하면 Kafka는 자동으로 파티션 재할당(Rebalancing)을 수행한다.

### 7. Dead Letter Queue (DLQ) vs Error Handling

**SQS의 DLQ**
SQS는 처리에 실패한 메시지를 관리하기 위한 DLQ(Dead Letter Queue) 기능을 기본으로 제공한다. 소스 큐에서 메시지 수신 횟수가 설정된 maxReceiveCount를 초과하면, 해당 메시지는 자동으로 DLQ로 이동한다.

주요 특징은 다음과 같다.
- Standard Queue의 DLQ는 Standard Queue여야 하고, FIFO Queue의 DLQ는 FIFO Queue여야 한다.
- FIFO Queue에서 DLQ를 사용하면 메시지 순서가 깨질 수 있으므로 주의가 필요하다.
- DLQ Redrive 기능으로 DLQ에 있는 메시지를 소스 큐로 되돌려 재처리할 수 있다.
- DLQ에 쌓인 메시지를 분석하여 반복 실패 원인을 파악할 수 있다.

**Kafka의 에러 처리**
Kafka에는 SQS처럼 내장된 DLQ 기능이 없다. 대신 애플리케이션 레벨에서 에러 처리 전략을 구현한다.

일반적인 패턴은 Dead Letter Topic을 별도로 생성하는 것이다. 처리에 실패한 메시지를 전용 에러 토픽으로 발행하고, 모니터링 시스템이나 별도의 Consumer가 해당 토픽을 구독하여 실패 메시지를 분석하고 재처리한다. Spring Kafka와 같은 프레임워크는 이 패턴을 지원하는 내장 기능을 제공한다.

Kafka의 가장 강력한 에러 복구 방법은 Offset Reset이다. 데이터 파이프라인 자체의 버그를 수정한 뒤 Offset을 되돌려 문제 발생 시점부터 전체를 재처리하는 것이 가능하다.

### 8. 운영 복잡도와 비용

**SQS**
- **운영 복잡도**: 서버리스 서비스로, 클러스터, 브로커, 파티션 등 인프라 관리가 전혀 필요 없다. AWS 콘솔이나 CLI로 간단히 설정할 수 있으며, 운영 부담이 극히 낮다.
- **비용 구조**: 전송된 메시지 요청 수 단위로 과금된다. 트래픽이 적을 때는 비용이 매우 저렴하지만, 고처리량 환경에서는 비용이 누적될 수 있다.

**Kafka (자체 운영)**
- **운영 복잡도**: Zookeeper(또는 KRaft) 기반 클러스터 운영, 파티션 수 설계, 브로커 모니터링, 토픽 Retention 정책 관리 등 상당한 운영 지식과 인력이 필요하다.
- **비용 구조**: 서버 인프라 비용이 고정으로 발생하므로, 트래픽이 적을 때는 비효율적이지만 고처리량 환경에서는 SQS 대비 비용 효율이 높다.

**Amazon MSK (Managed Streaming for Apache Kafka)**
AWS에서 제공하는 Kafka 완전 관리형 서비스로, 중간 지점에 해당한다. 클러스터 프로비저닝과 모니터링을 AWS가 담당하므로 운영 부담이 줄어드나, 직접 운영 대비 비용이 높다. MSK Serverless 옵션으로 사용량 기반 과금도 가능하다.

### 9. 실무 선택 기준

**SQS를 선택해야 할 때**
- 백그라운드 작업 큐(이메일 발송, 이미지 리사이징, 알림 전송 등)처럼 각 메시지가 독립적으로 처리되는 경우
- 소비자가 단순히 작업을 나눠 처리하는 Competing Consumers 패턴이 적합한 경우
- 빠른 개발과 낮은 운영 부담이 중요한 경우
- AWS 생태계 내에서 Lambda, ECS 등과 간단히 연동하고 싶을 때
- 메시지를 재처리하거나 과거 이벤트를 다시 볼 필요가 없을 때

**Kafka를 선택해야 할 때**
- 이벤트 소싱(Event Sourcing) 또는 CQRS 패턴을 구현하는 경우
- 동일 이벤트를 여러 시스템(Consumer Group)이 각자의 목적으로 독립 소비해야 하는 경우
- 실시간 데이터 파이프라인, 로그 수집, 스트림 처리가 필요한 경우
- 이벤트를 특정 시점으로 되돌려 재처리해야 하는 경우 (디버깅, 배치 재처리 등)
- 초고처리량(수십만 TPS 이상)이 요구되는 경우

| 이메일/알림 발송 서비스  | 회원가입, 주문 완료 등의 이벤트 발생 시 이메일이나 푸시 알림을 비동기로 발송한다. 각 메시지가 독립적이고, 한 Worker가 처리하면 끝나는 전형적인 작업 큐 패턴이다. |
| 이미지/동영상 인코딩     | 사용자가 업로드한 미디어를 썸네일 생성, 해상도 변환 등으로 후처리한다. 처리 시간이 긴 작업을 API 응답과 분리하여 사용자 경험을 개선한다.               |
| 주문 처리 파이프라인     | 주문 접수 → 재고 확인 → 결제 → 배송 준비 등의 단계를 각 SQS 큐로 연결하여 단계별 비동기 처리를 구성한다. 각 단계의 실패가 전체에 영향을 주지 않는다.   |
| Lambda 트리거 연동       | SQS → Lambda 조합으로 서버 없이 이벤트 기반 처리를 구현한다. 트래픽이 불규칙한 배치 작업(리포트 생성, 데이터 정제 등)에 적합하다.                     |

**Kafka 적용 서비스 예시**

| 서비스                          | 설명                                                                                                     |
| ------------------------------- | -------------------------------------------------------------------------------------------------------- |
| 실시간 로그 수집 및 분석        | 수백 대 서버의 애플리케이션 로그를 Kafka로 수집하고, Elasticsearch Consumer Group은 검색 인덱싱, Spark Consumer Group은 실시간 이상 탐지, S3 Consumer Group은 장기 아카이빙을 각각 독립 수행한다. |
| 이커머스 주문 이벤트 스트리밍   | 주문 생성 이벤트를 하나의 토픽에 발행하면, 재고 시스템·결제 시스템·추천 엔진·분석 시스템이 각자의 Consumer Group으로 동일 이벤트를 독립 소비한다.        |
| 실시간 사용자 활동 추적         | 클릭, 페이지뷰, 검색 등 사용자 행동 이벤트를 Kafka에 스트리밍하고, 실시간 대시보드와 개인화 추천 엔진이 동시에 소비한다.                               |
| CDC(Change Data Capture)        | Debezium 등을 통해 DB 변경 사항을 Kafka 토픽으로 캡처하고, 검색 엔진 동기화·캐시 무효화·데이터 웨어하우스 적재 등 다양한 후속 처리를 연결한다.           |
| 금융 거래 실시간 처리           | 초당 수십만 건의 거래 이벤트를 처리하며, 사기 탐지 시스템과 정산 시스템이 동일 이벤트를 각자 소비한다. 장애 시 Offset 재설정으로 거래 재검증이 가능하다.   |

**하이브리드 아키텍처**
두 시스템을 함께 사용하는 패턴도 실무에서 자주 활용된다. 예를 들어 Kafka를 이벤트 스트리밍 백본으로 사용하고, Kafka의 이벤트를 트리거로 SQS에 특정 작업 메시지를 전달하여 Lambda로 처리하는 구조가 있다. 각 시스템의 강점을 조합해 복잡한 요구사항을 해결할 수 있다.

## 핵심 정리
- SQS는 "작업 처리"를, Kafka는 "이벤트 스트리밍"을 위해 설계되었다. 각 메시지를 하나의 Consumer만 처리하는 작업 분산에는 SQS가, 다수의 Consumer가 동일 이벤트를 독립적으로 소비하는 시나리오에는 Kafka가 적합하다.
- 메시지 재처리 가능 여부가 핵심 선택 기준이다. Kafka는 Offset Reset으로 과거 이벤트를 언제든 재처리할 수 있지만, SQS는 소비된 메시지를 되돌릴 수 없다.
- SQS FIFO는 순서와 중복 제거를 보장하지만 처리량이 제한되고, Kafka는 파티션 단위 순서 보장과 극고 처리량을 동시에 달성한다.
- SQS는 운영 부담이 없는 서버리스 서비스이고, Kafka는 높은 운영 복잡도 대신 극한의 성능과 유연성을 제공한다. AWS에서는 MSK로 운영 부담을 완화할 수 있다.
- "이 메시지를 여러 시스템이 각자 처리해야 하는가?"라는 질문이 가장 빠른 선택 기준이 된다. 그렇다면 Kafka, 단순히 하나의 Worker가 처리하면 된다면 SQS를 선택하라.

## 키워드
- `SQS`: AWS에서 제공하는 완전 관리형 메시지 큐 서비스로, 분산 시스템 간 비동기 통신을 담당한다.
- `Kafka`: LinkedIn이 개발한 분산 이벤트 스트리밍 플랫폼으로, 높은 처리량과 내구성 있는 이벤트 로그를 제공한다.
- `메시지 큐`: 생산자와 소비자 사이에서 메시지를 임시 저장하고 전달하는 중간 버퍼 역할을 하는 시스템이다.
- `이벤트 스트리밍`: 이벤트를 실시간으로 생성, 전달, 처리하고 영속적으로 보존하는 데이터 처리 패러다임이다.
- `Consumer Group`: Kafka에서 동일 토픽을 소비하는 Consumer의 논리적 그룹으로, 그룹 내에서는 파티션을 분담하고 그룹 간에는 독립 소비한다.
- `Partition`: Kafka 토픽을 분할한 단위로, 파티션 수만큼 병렬 처리가 가능하며 파티션 내에서 메시지 순서가 보장된다.
- `Offset`: Kafka Consumer가 특정 파티션에서 어디까지 메시지를 읽었는지를 나타내는 위치 값이다.
- `Dead Letter Queue`: 처리에 반복 실패한 메시지를 격리하여 보관하는 특수 큐로, 장애 원인 분석과 재처리에 활용된다.
- `At-least-once`: 메시지가 최소 한 번 이상 전달됨을 보장하는 전달 의미론으로, 중복 처리 가능성이 있어 Consumer의 멱등성(Idempotency) 설계가 필요하다.
- `Pub/Sub`: 발행자(Publisher)와 구독자(Subscriber)를 분리하는 메시징 패턴으로, 구독자가 여럿인 경우 각자 독립적으로 메시지를 수신한다.
