# TCP 길이 프리픽스 프로토콜

## 개요
TCP 스트림에서 메시지 경계를 구분하기 위해 메시지 앞에 길이 정보를 붙이는 프레이밍(framing) 기법

---

## 상세 내용

### TCP 스트림의 메시지 경계 문제

TCP는 **바이트 스트림(Byte Stream) 프로토콜**이다. 즉, TCP는 데이터를 연속된 바이트의 흐름으로 취급하며, 애플리케이션이 보낸 "메시지"라는 개념을 알지 못한다.

**메시지 경계가 보장되지 않는 이유**

애플리케이션이 `send("Hello")`, `send("World")` 를 두 번 호출하더라도, 수신 측에서는 `recv()` 한 번에 `"HelloWorld"` 를 받을 수도 있고, `"He"` 와 `"lloWorld"` 처럼 전혀 다른 단위로 받을 수도 있다.

이는 TCP가 다음과 같은 이유로 메시지 경계를 보존하지 않기 때문이다.

- TCP 스택은 내부 버퍼에 데이터를 쌓은 뒤, 네트워크 효율에 따라 세그먼트를 재구성하여 전송한다.
- IP 계층에서 패킷이 분할(Fragmentation)되고 재조합될 수 있다.
- 수신 측 TCP 버퍼에 이미 도착한 여러 패킷의 데이터가 합쳐질 수 있다.

**Nagle 알고리즘과 TCP 세그먼트 병합**

Nagle 알고리즘(RFC 896)은 John Nagle이 제안한 알고리즘으로, 작은 패킷이 네트워크에 범람하는 "소형 패킷 문제(Small-Packet Problem)"를 해결하기 위해 설계되었다.

동작 원리는 다음과 같다.

> 아직 ACK를 받지 못한 전송 중인 데이터가 있다면, 새 데이터는 버퍼에 쌓아두고, ACK를 받거나 MSS(Maximum Segment Size, 보통 1,460 바이트) 만큼 쌓일 때까지 기다렸다가 한꺼번에 전송한다.

이 알고리즘은 네트워크 대역폭 효율은 높이지만, 애플리케이션이 연속으로 `send()` 를 호출하면 여러 메시지가 하나의 TCP 세그먼트로 합쳐질 수 있다. 이것이 TCP 레벨에서 메시지 경계가 사라지는 핵심 원인 중 하나다.

### 메시지 프레이밍 방식 비교

TCP 스트림에서 메시지 경계를 구분하는 방법은 크게 세 가지로 나뉜다.

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **길이 프리픽스 (Length-Prefix)** | 메시지 앞에 길이 값을 명시 | 이진 데이터 지원, 파싱 효율적 | 구현 시 버퍼 관리 필요 |
| **구분자 기반 (Delimiter-based)** | `\r\n`, `\0` 등으로 메시지 끝을 표시 | 구현 단순, 사람이 읽기 쉬움 | 구분자가 데이터에 포함될 경우 처리 복잡, 이진 데이터 부적합 |
| **고정 길이 (Fixed-Length)** | 모든 메시지가 동일한 크기 | 파싱 가장 단순 | 가변 길이 데이터 비효율, 패딩 낭비 |

**각 방식의 사용 예**

- 길이 프리픽스: HTTP/2, gRPC, MySQL 클라이언트 프로토콜
- 구분자 기반: HTTP/1.1 헤더(`\r\n\r\n`), SMTP, Redis RESP(부분적으로)
- 고정 길이: DNS 메시지(UDP), 일부 하드웨어 통신 프로토콜

### 길이 프리픽스 프로토콜 구조

기본 구조는 다음과 같다.

```
+------------------+-------------------------------+
|  Length (N bytes) |      Payload (Length bytes)   |
+------------------+-------------------------------+
```

**헤더(길이 필드) + 페이로드 구조**

수신 측은 다음 두 단계로 메시지를 읽는다.
1. 먼저 고정 크기의 길이 필드를 읽어 페이로드 크기를 확인한다.
2. 길이 필드가 알려준 바이트 수만큼 페이로드를 읽는다.

**길이 필드 크기 선택**

| 크기 | 표현 가능한 최대 메시지 크기 | 사용 사례 |
|------|--------------------------|-----------|
| 1바이트 | 255 bytes | 매우 작은 메시지 전용 |
| 2바이트 (uint16) | 65,535 bytes (~64KB) | 소형 프로토콜 |
| 3바이트 | 16,777,215 bytes (~16MB) | MySQL 패킷 |
| 4바이트 (uint32) | 4,294,967,295 bytes (~4GB) | 일반적인 선택, gRPC 등 |

대부분의 범용 프로토콜에서는 **4바이트(uint32)** 를 사용하며, 충분한 크기를 확보하면서도 오버헤드가 크지 않다.

**바이트 오더(Endianness) 고려 - Big Endian vs Little Endian**

길이 필드가 2바이트 이상이면 바이트 순서(Endianness)를 명확히 약속해야 한다.

- **Big Endian (Network Byte Order)**: 최상위 바이트(MSB)가 낮은 주소에 위치. 네트워크 표준. `0x00000005` → `[00][00][00][05]`
- **Little Endian**: 최하위 바이트(LSB)가 낮은 주소에 위치. x86 CPU 기본 방식. `0x00000005` → `[05][00][00][00]`

네트워크 프로토콜에서는 일반적으로 **Big Endian**을 사용한다. Java의 `ByteBuffer`는 기본적으로 Big Endian이며, `DataOutputStream`도 Big Endian으로 값을 기록한다.

```java
// Java에서 Big Endian으로 길이 프리픽스 메시지 전송 예시
byte[] payload = message.getBytes(StandardCharsets.UTF_8);
ByteBuffer buffer = ByteBuffer.allocate(4 + payload.length);
buffer.order(ByteOrder.BIG_ENDIAN);  // 기본값이지만 명시
buffer.putInt(payload.length);       // 4바이트 길이 필드
buffer.put(payload);                 // 페이로드
outputStream.write(buffer.array());
```

**길이 필드가 포함하는 범위 (헤더 포함 여부)**

프로토콜 설계 시 길이 필드 값이 "페이로드만의 크기"인지, "헤더를 포함한 전체 크기"인지를 명확히 정의해야 한다.

- 예: 페이로드 100바이트, 헤더 4바이트일 때
  - 페이로드만 포함: 길이 필드 = `100`
  - 전체 포함: 길이 필드 = `104`

### 구현 시 고려사항

**부분 수신(Partial Read) 처리**

TCP는 요청한 바이트 수를 한 번에 모두 전달한다고 보장하지 않는다. `read()` 또는 `recv()` 는 요청보다 적은 바이트를 반환할 수 있다. 이 경우를 **부분 수신(Partial Read)** 이라고 하며, 반드시 루프를 사용해 처리해야 한다.

```java
// 정확히 N바이트를 읽을 때까지 반복하는 유틸리티
public static void readFully(InputStream in, byte[] buf, int offset, int length)
        throws IOException {
    int totalRead = 0;
    while (totalRead < length) {
        int read = in.read(buf, offset + totalRead, length - totalRead);
        if (read == -1) {
            throw new EOFException("Connection closed before message was complete");
        }
        totalRead += read;
    }
}

// 사용 예
byte[] headerBuf = new byte[4];
readFully(inputStream, headerBuf, 0, 4);
int payloadLength = ByteBuffer.wrap(headerBuf).getInt();

byte[] payloadBuf = new byte[payloadLength];
readFully(inputStream, payloadBuf, 0, payloadLength);
```

**버퍼 관리 전략**

| 전략 | 설명 |
|------|------|
| 동적 할당 | 매번 페이로드 크기만큼 `new byte[length]` 할당. 구현 단순하나 GC 부담 증가 |
| 풀링(Pool) | 미리 할당된 버퍼 풀에서 가져다 쓰고 반납. 성능 최적화 가능 |
| 링 버퍼(Ring Buffer) | 고정 크기 링 버퍼에 수신 데이터를 계속 덮어쓰며 처리. Netty 등의 프레임워크에서 활용 |

**최대 메시지 크기 제한**

서버가 길이 필드를 읽은 직후 그 크기만큼 버퍼를 할당한다면, 악의적인 클라이언트가 `length = 2GB` 같은 거대한 값을 전송하여 서버 메모리를 고갈시킬 수 있다. 반드시 허용 가능한 최대 크기를 설정하고, 초과 시 연결을 끊어야 한다.

```java
int MAX_MESSAGE_SIZE = 4 * 1024 * 1024; // 4MB 제한

int payloadLength = ByteBuffer.wrap(headerBuf).getInt();
if (payloadLength < 0 || payloadLength > MAX_MESSAGE_SIZE) {
    throw new IllegalArgumentException("Invalid message length: " + payloadLength);
}
```

**잘못된 길이 값에 대한 방어 코드**

- 음수 길이: 부호 있는(signed) 정수를 쓸 경우 발생 가능 → 음수 체크
- 0 길이: 빈 메시지 허용 여부를 명확히 정의
- 상한 초과: 위의 최대 크기 제한으로 처리

### 실제 사용 사례

**HTTP/2 프레임 구조**

HTTP/2는 데이터를 "프레임(Frame)" 단위로 주고받으며, 각 프레임은 다음 9바이트의 고정 헤더로 시작한다.

```
+-----------------------------------------------+
|                 Length (24)                   |
+---------------+---------------+---------------+
|   Type (8)    |   Flags (8)   |
+-+-------------+---------------+-------------------------------+
|R|                 Stream Identifier (31)                      |
+=+=============================================================+
|                   Frame Payload (0...)                      ...
+---------------------------------------------------------------+
```

- **Length(3바이트)**: 프레임 페이로드의 크기 (최대 16,383바이트, 협상을 통해 최대 16MB까지 확장 가능)
- **Type(1바이트)**: DATA, HEADERS, SETTINGS 등 프레임 종류
- **Flags(1바이트)**: END_STREAM 등의 플래그
- **Stream Identifier(4바이트)**: 멀티플렉싱을 위한 스트림 ID

**Protocol Buffers (gRPC)의 Length-Prefixed Message**

gRPC는 HTTP/2 위에서 동작하며, HTTP/2 DATA 프레임 내에 gRPC 전용 5바이트 헤더를 추가한다.

```
+------------------+---------------------------+
| Compressed (1B)  |     Message Length (4B)   |  Protobuf Payload
+------------------+---------------------------+------------------+
```

- **Compressed Flag(1바이트)**: `0` = 비압축, `1` = 압축(gzip 등)
- **Message Length(4바이트, Big Endian)**: 뒤따르는 Protobuf 직렬화 데이터의 바이트 수

주목할 점은 HTTP/2 DATA 프레임 경계와 gRPC 메시지 경계가 일치하지 않을 수 있다는 것이다. 하나의 gRPC 메시지가 여러 DATA 프레임에 걸쳐 있거나, 하나의 DATA 프레임에 여러 gRPC 메시지가 담길 수 있다.

**Redis RESP 프로토콜**

Redis의 RESP(REdis Serialization Protocol)는 텍스트 기반이지만 Bulk String에서 길이 프리픽스 개념을 사용한다.

```
# Bulk String 형식
$<length>\r\n
<data>\r\n

# 예: "hello" 전송
$5\r\n
hello\r\n
```

`$5` 가 뒤따르는 데이터의 바이트 길이를 의미하는 길이 프리픽스다. 이를 통해 이진 데이터 안에 `\r\n`이 포함되어 있어도 안전하게 처리할 수 있다.

**MySQL 클라이언트/서버 프로토콜**

MySQL은 4바이트 패킷 헤더를 사용한다.

```
+------------------+------------------+------------------+
|  Payload Length  |  Payload Length  |  Payload Length  |  Sequence ID  |  Payload
|     (1바이트)    |     (1바이트)    |     (1바이트)    |    (1바이트)  |
+------------------+------------------+------------------+
      LSB                                    MSB
      <- 3바이트 Little Endian 길이 필드 ->
```

- **Payload Length(3바이트, Little Endian)**: 페이로드 크기, 최대 16MB-1 (0xFFFFFF)
- **Sequence ID(1바이트)**: 동일 쿼리 처리 내에서의 패킷 순서 번호

MySQL은 Big Endian이 아닌 **Little Endian**을 사용한다는 점이 특징적이다. 16MB를 초과하는 데이터는 0xFFFFFF 크기의 패킷으로 분할하여 전송한다.

### TCP 점착(Sticky) 패킷과 반패킷 문제

**점착 패킷(Sticky Packet) 현상**

여러 개의 메시지가 하나의 TCP 세그먼트에 합쳐져 도착하는 현상이다.

```
[송신 측]  send("MSG1") + send("MSG2")
[수신 측]  recv() → "MSG1MSG2"  (두 메시지가 합쳐짐)
```

Nagle 알고리즘, TCP 송신 버퍼 병합, 소켓 수신 버퍼에 쌓인 데이터가 원인이 될 수 있다.

**반패킷(Half Packet) 현상**

하나의 메시지가 여러 TCP 세그먼트로 쪼개져 도착하는 현상이다.

```
[송신 측]  send("LONG_MESSAGE_DATA")
[수신 측]  recv() → "LONG_MES"     (앞부분만 도착)
           recv() → "SAGE_DATA"    (뒷부분 나중에 도착)
```

메시지가 MTU(Maximum Transmission Unit)를 초과하거나 네트워크 중간에서 IP 단편화가 발생할 때 나타날 수 있다.

**길이 프리픽스로 해결하는 방법**

길이 프리픽스 방식은 위 두 현상을 모두 우아하게 처리한다.

```
[수신 루프]

1. 헤더(4바이트)를 readFully()로 읽는다 → 점착/반패킷 무관하게 정확히 4바이트 수신
2. 헤더에서 payloadLength를 파싱한다
3. payload를 readFully(payloadLength)로 읽는다 → 정확히 payloadLength 바이트 수신
4. 완전한 메시지 처리 후 1로 돌아간다
```

수신 루프가 항상 "정확한 바이트 수"를 읽도록 강제하기 때문에, TCP가 데이터를 어떤 단위로 전달하든 관계없이 메시지 경계를 정확히 복원할 수 있다.

## 핵심 정리

- TCP는 바이트 스트림 프로토콜이므로 메시지 경계를 보존하지 않는다. Nagle 알고리즘 등으로 인해 점착 패킷과 반패킷 현상이 발생한다.
- 길이 프리픽스는 "먼저 길이를 읽고, 그 길이만큼 페이로드를 읽는다"는 두 단계로 메시지 경계를 복원하는 가장 일반적이고 강력한 프레이밍 기법이다.
- 길이 필드는 보통 4바이트 uint32, Big Endian(Network Byte Order)을 사용한다. MySQL처럼 Little Endian을 쓰는 경우도 있으므로 프로토콜 스펙을 항상 확인해야 한다.
- `read()` / `recv()` 는 부분 수신을 할 수 있으므로, `readFully()` 패턴으로 원하는 바이트 수가 모두 도착할 때까지 반복해서 읽어야 한다.
- 보안을 위해 반드시 최대 메시지 크기를 제한해야 한다. 길이 필드를 신뢰 없이 그대로 버퍼 할당에 사용하면 메모리 고갈 공격(OOM Attack)에 취약해진다.
- HTTP/2(3바이트 length), gRPC(4바이트 length + 1바이트 compressed flag), MySQL(3바이트 length + 1바이트 sequence id) 등 실무 프로토콜마다 길이 프리픽스의 구체적인 형태가 다르다.

## 키워드

- `TCP 스트림`: TCP가 데이터를 메시지 단위가 아닌 연속된 바이트의 흐름으로 취급하는 특성. 수신 측에서 메시지 경계를 별도로 처리해야 한다.
- `메시지 프레이밍`: TCP 스트림 위에서 애플리케이션 메시지의 시작과 끝을 구분하는 기법. 길이 프리픽스, 구분자, 고정 길이 방식이 있다.
- `길이 프리픽스`: 메시지 페이로드 앞에 해당 메시지의 바이트 수를 명시하는 헤더를 붙이는 프레이밍 방식.
- `Length-Prefix`: 길이 프리픽스의 영문 표기. gRPC, HTTP/2, MySQL 등 다양한 프로토콜에서 활용된다.
- `바이트 오더`: 다중 바이트 정수를 메모리 또는 네트워크에 저장할 때의 바이트 순서. Big Endian과 Little Endian이 있다.
- `Endianness`: 바이트 오더와 동일한 개념. 네트워크 표준은 Big Endian(Network Byte Order)이다.
- `부분 수신`: `read()` 호출 한 번에 요청한 바이트 수보다 적은 양이 반환되는 TCP의 정상적인 동작. `readFully()` 패턴으로 반드시 처리해야 한다.
- `Sticky Packet`: 점착 패킷. 여러 메시지가 하나의 TCP 세그먼트로 합쳐져 수신되는 현상.
- `Half Packet`: 반패킷. 하나의 메시지가 여러 TCP 세그먼트로 분할되어 수신되는 현상.
- `프로토콜 버퍼`: Google이 개발한 이진 직렬화 포맷. gRPC에서 길이 프리픽스와 함께 사용하여 메시지를 효율적으로 전송한다.
