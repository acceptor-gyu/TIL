# CLAUDE 서브 에이전트를 활용한 자동화 파이프라인

## 개요
Claude Code의 서브 에이전트(SubAgent) 기능을 활용하여 반복적인 작업을 자동화하는 파이프라인을 구축하는 방법을 정리합니다.

## 상세 내용

### 1. 서브 에이전트란

**서브 에이전트**는 특정 작업을 위해 Claude가 위임할 수 있는 전문화된 AI 에이전트입니다. 각 서브 에이전트는 자체 컨텍스트 윈도우에서 독립적으로 실행되며, 커스텀 시스템 프롬프트, 제한된 도구 접근, 독립적인 권한을 가집니다.

Claude는 서브 에이전트의 `description` 필드를 바탕으로 어떤 작업을 위임할지 자동으로 판단합니다.

#### 서브 에이전트가 제공하는 이점

| 이점 | 설명 |
|------|------|
| **컨텍스트 보존** | 탐색·구현 작업을 메인 대화에서 분리 |
| **제약 강제** | 서브 에이전트가 사용할 수 있는 도구를 제한 |
| **설정 재사용** | User 레벨 서브 에이전트로 모든 프로젝트에서 활용 |
| **비용 절감** | 단순 검색은 Haiku, 복잡한 분석은 Opus 등 작업별 모델 선택 |
| **특화 행동** | 특정 도메인에 집중된 시스템 프롬프트 |

---

### 2. 내장 서브 에이전트

Claude Code는 기본 제공 서브 에이전트를 자동으로 활용합니다.

| 에이전트 | 모델 | 도구 | 용도 |
|----------|------|------|------|
| **Explore** | Haiku (빠름) | 읽기 전용 | 파일 탐색, 코드 검색, 코드베이스 분석 |
| **Plan** | Main 상속 | 읽기 전용 | Plan Mode에서 코드베이스 리서치 |
| **General-purpose** | Main 상속 | 전체 도구 | 탐색 + 수정이 모두 필요한 복잡한 작업 |
| **Bash** | Main 상속 | Bash 특화 | 별도 컨텍스트에서 터미널 명령 실행 |

**Explore** 에이전트는 Haiku 모델로 저지연 검색에 최적화되어 있어 코드베이스 탐색 시 컨텍스트를 메인 대화에 쌓지 않고 결과만 요약해 반환합니다.

### 3. 서브 에이전트 정의 방법

서브 에이전트는 YAML frontmatter와 Markdown 시스템 프롬프트로 구성된 `.md` 파일로 정의합니다.

#### 파일 위치와 우선순위

```
--agents CLI 플래그 (1순위, 세션 범위)
    ↓
.claude/agents/          (2순위, 프로젝트 범위)
    ↓
~/.claude/agents/        (3순위, 사용자 전역)
    ↓
플러그인 agents/ 디렉토리 (4순위, 최저)
```

같은 이름의 서브 에이전트가 여러 위치에 있으면 높은 우선순위가 낮은 우선순위를 덮어씁니다.

#### 서브 에이전트 파일 형식

```markdown
---
name: code-reviewer
description: 코드 품질, 보안, 유지보수성을 검토합니다. 코드 수정 후 자동으로 사용하세요.
tools: Read, Grep, Glob, Bash
model: sonnet
---

당신은 시니어 코드 리뷰어입니다.

호출 시:
1. git diff로 최근 변경 사항 확인
2. 수정된 파일에 집중
3. 즉시 리뷰 시작

리뷰 체크리스트:
- 코드 가독성 및 네이밍
- 중복 코드 제거
- 에러 처리 적절성
- 보안 취약점 (시크릿, 입력 검증)
- 테스트 커버리지
- 성능 고려사항
```

#### 주요 frontmatter 필드

| 필드 | 필수 | 설명 |
|------|------|------|
| `name` | Yes | 소문자 + 하이픈 조합의 고유 식별자 |
| `description` | Yes | Claude가 언제 이 서브 에이전트를 위임할지 결정하는 설명 |
| `tools` | No | 허용 도구 목록 (생략 시 전체 상속) |
| `disallowedTools` | No | 차단 도구 목록 (상속받은 도구에서 제외) |
| `model` | No | `sonnet`, `opus`, `haiku`, 또는 `inherit` (기본값) |
| `permissionMode` | No | `default`, `acceptEdits`, `dontAsk`, `bypassPermissions`, `plan` |
| `maxTurns` | No | 서브 에이전트가 실행할 최대 턴 수 |
| `memory` | No | 영구 메모리 범위: `user`, `project`, `local` |
| `skills` | No | 시작 시 컨텍스트에 주입할 Skills 목록 |
| `hooks` | No | 서브 에이전트 생명주기 Hooks |
| `background` | No | `true`면 항상 백그라운드로 실행 |
| `isolation` | No | `worktree`로 설정하면 임시 git worktree에서 격리 실행 |

### 4. 자동화 파이프라인 설계 패턴

#### 패턴 1: 순차 체이닝 (Sequential Chaining)

서브 에이전트를 순서대로 실행하여 이전 결과를 다음 단계에 전달합니다.

```
메인 에이전트
    │
    ├─ 1단계: code-reviewer → 성능 문제 목록 반환
    │
    └─ 2단계: optimizer → 목록 받아 수정 적용
```

```text
code-reviewer 서브 에이전트로 성능 문제를 찾고,
이후 optimizer 서브 에이전트로 수정해줘
```

#### 패턴 2: 병렬 리서치 (Parallel Research)

독립적인 조사 작업을 여러 서브 에이전트가 동시에 수행합니다.

```
메인 에이전트
    ├─ auth-analyzer    (병렬)
    ├─ db-analyzer      (병렬)
    └─ api-analyzer     (병렬)
         ↓ 결과 요약
메인 에이전트가 종합
```

```text
auth, database, API 모듈을 각각 별도의 서브 에이전트로 병렬 분석해줘
```

#### 패턴 3: 컨텍스트 격리 (Context Isolation)

대량 출력을 생성하는 작업을 메인 컨텍스트에서 분리합니다.

```text
서브 에이전트를 사용해서 테스트 스위트를 실행하고,
실패한 테스트와 에러 메시지만 요약해서 보고해줘
```

테스트 로그, 문서 탐색, 대용량 데이터 분석처럼 verbose한 출력이 발생하는 작업은 서브 에이전트에서 격리해 메인 컨텍스트의 낭비를 방지합니다.

### 5. TIL 자동화 파이프라인 실전 예시

이 TIL 저장소처럼 반복적인 문서 작업도 서브 에이전트 체이닝으로 자동화할 수 있습니다.

#### 구조

```
.claude/
├── agents/
│   ├── til-template-creator.md   # 1단계: 템플릿 파일 생성
│   └── til-content-writer.md     # 2단계: 내용 작성 + README 업데이트
└── skills/
    └── add-til/
        └── SKILL.md              # 사용자 진입점 슬래시 커맨드
```

#### til-template-creator.md

```markdown
---
name: til-template-creator
description: TIL 파일명과 카테고리를 받아 템플릿 파일을 생성합니다
tools: Read, Write, Bash
model: haiku
---

오늘 날짜와 카테고리를 받아 올바른 디렉토리에 TIL 템플릿 파일을 생성합니다.
파일명 형식: YYMMDD_NN_제목.md
```

#### til-content-writer.md

```markdown
---
name: til-content-writer
description: 생성된 TIL 템플릿에 내용을 작성하고 README를 업데이트합니다
tools: Read, Write, Edit, WebSearch
model: sonnet
---

공식 문서와 최신 정보를 참고하여 TIL 내용을 충실히 작성하고,
README.md를 날짜 역순으로 업데이트합니다.
```

### 6. 서브 에이전트 호출 방식

#### 자연어로 호출

```text
code-reviewer 서브 에이전트를 사용해서 최근 변경 사항을 검토해줘
```

Claude가 판단하여 위임 여부를 결정합니다.

#### @-멘션으로 명시적 호출

```text
@"code-reviewer (agent)" auth 변경 사항 확인해줘
```

`@` 입력 후 typeahead에서 에이전트를 선택하면 해당 서브 에이전트가 반드시 실행됩니다.

#### CLI 플래그로 세션 전체 적용

```bash
# 세션 전체를 해당 서브 에이전트의 시스템 프롬프트로 실행
claude --agent code-reviewer

# 프로젝트 기본 에이전트 설정 (.claude/settings.json)
{
  "agent": "code-reviewer"
}
```

#### 포그라운드 vs 백그라운드

- **포그라운드**: 서브 에이전트가 완료될 때까지 메인 대화가 블로킹
- **백그라운드**: 서브 에이전트와 동시에 작업 가능 (Ctrl+B로 현재 작업을 백그라운드로 전환)

### 7. 고급 기능

#### PreToolUse Hook으로 도구 조건부 제한

단순한 도구 허용/차단 목록보다 세밀한 제어가 필요할 때 Hook을 활용합니다.

```yaml
---
name: db-reader
description: 읽기 전용 DB 쿼리만 실행
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly-query.sh"
---
```

```bash
#!/bin/bash
# validate-readonly-query.sh
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

if echo "$COMMAND" | grep -iE '\b(INSERT|UPDATE|DELETE|DROP|CREATE|ALTER|TRUNCATE)\b' > /dev/null; then
  echo "Blocked: SELECT 쿼리만 허용됩니다" >&2
  exit 2  # exit 2 = 실행 차단 + Claude에게 에러 메시지 전달
fi
exit 0
```

#### 영구 메모리 (Persistent Memory)

서브 에이전트가 대화 간 학습 내용을 축적할 수 있습니다.

```yaml
---
name: code-reviewer
description: 코드 품질 리뷰 전문가
memory: project   # .claude/agent-memory/code-reviewer/ 에 저장
---

리뷰할 때마다 발견한 패턴, 컨벤션 위반 사례, 아키텍처 결정사항을
메모리에 업데이트하여 프로젝트 지식을 누적합니다.
```

| 범위 | 저장 위치 | 용도 |
|------|----------|------|
| `user` | `~/.claude/agent-memory/<name>/` | 모든 프로젝트에서 공유 |
| `project` | `.claude/agent-memory/<name>/` | 프로젝트 전용, 버전 관리 가능 |
| `local` | `.claude/agent-memory-local/<name>/` | 프로젝트 전용, 버전 관리 제외 |

#### MCP 서버 서브 에이전트 전용 연결

메인 대화에는 노출하지 않고 특정 서브 에이전트에만 MCP 서버를 연결할 수 있습니다. 이를 통해 메인 컨텍스트에 불필요한 도구 설명이 쌓이는 것을 방지합니다.

```yaml
---
name: browser-tester
description: Playwright로 브라우저 테스트를 실행합니다
mcpServers:
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@playwright/mcp@latest"]
---
```

### 8. 서브 에이전트 vs Agent Teams 선택 기준

Claude Code v2.1.32 이후 실험적 기능인 **Agent Teams**도 제공됩니다. 두 개념의 차이를 이해하고 올바른 패턴을 선택해야 합니다.

| 항목 | 서브 에이전트 | Agent Teams |
|------|-------------|-------------|
| **통신 방식** | 허브-스포크 (결과만 메인에 반환) | 피어-투-피어 (에이전트 간 직접 통신) |
| **컨텍스트** | 단일 세션 내 | 각자 독립 세션 |
| **토큰 비용** | 상대적으로 낮음 | 3~7배 높음 |
| **협업** | 서브 에이전트 간 직접 소통 불가 | 팀원 간 발견 공유 가능 |
| **적합한 작업** | 독립적 병렬 작업, 컨텍스트 격리 | 상호 의존적 작업 (프론트엔드 ↔ 백엔드 ↔ 테스트) |

#### 선택 가이드

```
작업이 독립적인가? (각 서브작업이 다른 결과에 의존하지 않음)
    │
    ├─ Yes → 서브 에이전트 (병렬 실행)
    │
    └─ No → 서브작업 간 실시간 소통이 필요한가?
                │
                ├─ Yes → Agent Teams
                │
                └─ No → 순차 체이닝 서브 에이전트
```

### 9. 파이프라인 설계 시 주의사항

#### 컨텍스트 비용 관리

- 여러 서브 에이전트가 상세한 결과를 반환하면 메인 컨텍스트가 빠르게 소모됨
- 서브 에이전트 시스템 프롬프트에 "핵심 요약만 반환"을 명시
- 자동 컴팩션은 약 95% 용량에서 트리거 (환경변수 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`로 조정 가능)

#### 서브 에이전트는 다른 서브 에이전트를 호출할 수 없음

서브 에이전트는 중첩 위임이 불가합니다. 다단계 파이프라인은 메인 에이전트에서 순차적으로 서브 에이전트를 체이닝하거나 Skills를 활용해야 합니다.

#### 에러 핸들링

백그라운드 서브 에이전트가 권한 부족으로 실패한 경우, 동일한 작업을 포그라운드 서브 에이전트로 재시도하면 인터랙티브 권한 프롬프트를 통해 해결할 수 있습니다.

## 핵심 정리

- 서브 에이전트는 독립된 컨텍스트에서 실행되어 메인 대화의 컨텍스트를 보존하고 비용을 절감한다
- `description` 필드를 명확하게 작성하면 Claude가 적절한 시점에 자동으로 위임한다
- `tools` 필드로 도구 접근을 제한하여 의도치 않은 파일 수정이나 명령 실행을 방지한다
- 순차 체이닝, 병렬 리서치, 컨텍스트 격리 세 가지 패턴을 상황에 맞게 조합하면 복잡한 워크플로우를 효율적으로 자동화할 수 있다
- 서브 에이전트 간 실시간 협업이 필요한 경우에만 Agent Teams를 고려한다 (높은 토큰 비용 수반)
