# Claude Code를 잘 활용하기 위해 알아야 하는 상식

## 개요

Claude Code(CLI 기반 AI 코딩 에이전트)를 효과적으로 사용하기 위해 알아야 할 핵심 개념과 실전 활용 팁을 정리한다. 공식 문서에서 제시하는 best practice와 실전에서 검증된 패턴을 중심으로 다룬다.

## 상세 내용

### 1. Claude Code의 동작 원리

Claude Code는 질문에 답하고 기다리는 챗봇이 아닌, **에이전틱 코딩 환경**이다. 파일을 읽고, 명령을 실행하고, 변경 사항을 만들며, 사용자가 지켜보거나 방향을 수정하거나 자리를 비운 상태에서도 자율적으로 문제를 해결한다.

#### 컨텍스트 윈도우가 핵심 자원이다

대부분의 best practice는 하나의 제약에서 비롯된다. **컨텍스트 윈도우가 빠르게 채워지고, 채워질수록 성능이 저하된다.**

컨텍스트 윈도우에는 다음이 모두 포함된다:
- 전체 대화 내용 (모든 메시지)
- Claude가 읽은 모든 파일 내용
- 실행한 명령의 모든 출력

단 하나의 디버깅 세션이나 코드베이스 탐색만으로도 수만 토큰이 소모될 수 있다. 컨텍스트 윈도우가 가득 차면 Claude는 초기 지시를 "잊거나" 더 많은 실수를 저지르기 시작한다.

#### 도구 호출 방식

Claude Code는 아래 도구들을 통해 파일 시스템과 상호작용한다.

| 도구 | 역할 |
|------|------|
| `Read` | 파일 내용 읽기 |
| `Write` | 새 파일 생성 또는 전체 덮어쓰기 |
| `Edit` | 기존 파일의 일부 수정 (diff만 전송하므로 효율적) |
| `Glob` | 파일명 패턴으로 파일 탐색 |
| `Grep` | 파일 내용 검색 |
| `Bash` | 쉘 명령 실행 |

### 2. 검증 수단을 함께 제공하라

Claude Code 활용에서 단일 최고 레버리지 행동은 **Claude가 스스로 결과를 검증할 수 있게 하는 것**이다.

검증 수단 없이는 그럴듯해 보이지만 실제로는 동작하지 않는 결과물이 나올 수 있다. 사용자가 유일한 피드백 루프가 되고, 모든 실수를 직접 발견해야 한다.

| 전략 | 개선 전 | 개선 후 |
|------|---------|---------|
| 검증 기준 제공 | "이메일 유효성 검사 함수 구현해줘" | "validateEmail 함수를 작성하고, user@example.com은 true, invalid는 false가 나오는지 테스트 실행해줘" |
| UI 변경 시각 검증 | "대시보드 더 보기 좋게 만들어줘" | "[스크린샷 첨부] 이 디자인으로 구현하고, 결과 스크린샷을 원본과 비교해서 차이점 수정해줘" |
| 근본 원인 해결 | "빌드가 실패해" | "이 에러로 빌드가 실패해: [에러 붙여넣기]. 에러를 억제하지 말고 근본 원인을 해결하고 빌드가 성공하는지 확인해줘" |

테스트 스위트, 린터, 예상 출력을 확인하는 Bash 명령 등 무엇이든 검증 수단으로 활용할 수 있다.

### 3. 탐색 → 계획 → 구현을 분리하라

Claude를 바로 코딩으로 뛰어들게 하면 잘못된 문제를 해결하는 코드가 나올 수 있다. **Plan Mode**를 사용해 탐색과 실행을 분리하면 훨씬 좋은 결과를 얻는다.

#### 권장 4단계 워크플로우

```
1. 탐색 (Plan Mode)
   Claude가 파일을 읽고 질문에 답하되 변경은 하지 않음
   예) "src/auth를 읽고 세션과 로그인 처리 방식을 파악해줘"

2. 계획 (Plan Mode)
   상세한 구현 계획 수립
   예) "Google OAuth를 추가하려면 어떤 파일을 변경해야 해? 계획 세워줘"
   Ctrl+G로 계획을 텍스트 에디터에서 직접 수정 가능

3. 구현 (Normal Mode)
   계획을 기반으로 코딩, 검증
   예) "계획대로 OAuth 플로우를 구현하고 테스트 작성 후 실행해줘"

4. 커밋
   예) "설명적인 메시지로 커밋하고 PR 열어줘"
```

단, 범위가 명확하고 수정이 작은 경우(오타 수정, 변수 이름 변경 등)는 바로 지시하는 것이 효율적이다. 변경 사항을 한 문장으로 설명할 수 있다면 계획 단계는 생략해도 된다.

### 4. 프롬프트에 구체적인 컨텍스트를 담아라

Claude는 의도를 추론할 수 있지만 마음을 읽을 수는 없다. 특정 파일을 참조하고, 제약 사항을 명시하고, 예시 패턴을 지적해야 한다.

| 전략 | 개선 전 | 개선 후 |
|------|---------|---------|
| 작업 범위 지정 | "foo.py 테스트 추가해줘" | "로그아웃 상태 엣지 케이스를 커버하는 foo.py 테스트를 써줘. 목(mock)은 쓰지 마" |
| 소스 지정 | "ExecutionFactory API가 왜 이렇게 이상해?" | "ExecutionFactory의 git 히스토리를 살펴보고 API가 이렇게 된 경위를 요약해줘" |
| 기존 패턴 참조 | "캘린더 위젯 추가해줘" | "홈페이지의 기존 위젯 구현 방식을 봐줘. HotDogWidget.php가 좋은 예시야. 그 패턴을 따라 월 선택과 연도 페이지네이션이 가능한 캘린더 위젯을 만들어줘" |

#### 풍부한 컨텍스트 제공 방법

- **`@` 파일 참조**: 코드 위치를 설명하는 대신 `@파일명`으로 직접 참조. Claude가 응답 전에 파일을 읽음
- **이미지 붙여넣기**: 스크린샷이나 디자인 이미지를 복사해서 바로 붙여넣기
- **URL 제공**: 문서나 API 레퍼런스 URL 제공. `/permissions`로 자주 쓰는 도메인을 허용 목록에 추가
- **파이프로 데이터 전달**: `cat error.log | claude`로 파일 내용을 직접 전달
- **Claude가 스스로 가져오게**: Bash 명령, MCP 도구, 파일 읽기로 컨텍스트를 직접 수집하게 지시

### 5. CLAUDE.md 설정 파일 활용

CLAUDE.md는 **매 대화 시작 시 Claude가 읽는 특수 파일**이다. Bash 명령, 코드 스타일, 워크플로우 규칙을 포함시켜 코드만으로는 추론할 수 없는 영속적인 컨텍스트를 제공한다.

#### 시작하는 방법

```bash
/init
```

`/init` 명령이 현재 프로젝트 구조를 분석해 빌드 시스템, 테스트 프레임워크, 코드 패턴을 감지하여 기초 파일을 생성해준다.

#### 좋은 CLAUDE.md 예시

```markdown
# 코드 스타일
- ES 모듈 문법 사용 (CommonJS require 대신 import/export)
- 가능하면 구조 분해 import 사용 (예: import { foo } from 'bar')

# 워크플로우
- 코드 변경 완료 후 반드시 타입 체크 실행
- 전체 테스트 스위트 대신 단일 테스트만 실행 (성능을 위해)
```

#### 포함 vs 제외 기준

| 포함해야 할 것 | 제외해야 할 것 |
|--------------|--------------|
| Claude가 추측할 수 없는 Bash 명령 | 코드를 읽으면 알 수 있는 내용 |
| 기본값과 다른 코드 스타일 규칙 | Claude가 이미 아는 언어 표준 규칙 |
| 테스트 지침과 선호 테스트 러너 | 상세한 API 문서 (링크로 대체) |
| 브랜치 이름, PR 컨벤션 등 저장소 에티켓 | 자주 변경되는 정보 |
| 프로젝트 고유 아키텍처 결정 사항 | 코드베이스 파일별 설명 |
| 개발 환경 특이사항, 필수 환경변수 | "클린 코드를 작성하라" 같은 자명한 규칙 |

CLAUDE.md가 너무 길면 Claude가 중요한 규칙을 놓친다. 각 줄마다 "이걸 없애면 Claude가 실수를 할까?"라고 물어봐야 한다. 아니라면 삭제하거나 Hook으로 전환하라.

#### 파일 위치 계층

- `~/.claude/CLAUDE.md`: 모든 Claude 세션에 적용
- `./CLAUDE.md`: 프로젝트 루트, git에 커밋해서 팀과 공유
- 부모 디렉토리: 모노레포에서 `root/CLAUDE.md`와 `root/foo/CLAUDE.md` 모두 자동으로 로드
- 자식 디렉토리: 해당 디렉토리의 파일 작업 시 Claude가 필요에 따라 로드

CLAUDE.md는 `@path/to/import` 문법으로 다른 파일을 임포트할 수도 있다.

### 6. 컨텍스트를 적극적으로 관리하라

#### 핵심 명령어

| 명령어 | 설명 |
|--------|------|
| `Esc` | Claude 중간 멈추기. 컨텍스트는 보존되므로 방향 전환 가능 |
| `Esc + Esc` 또는 `/rewind` | 이전 대화 및 코드 상태 복원 메뉴 |
| `/clear` | 컨텍스트 완전 초기화 (관련 없는 작업 전환 시) |
| `/compact` | 컨텍스트 압축 (예: `/compact API 변경 사항에 집중`) |
| `/btw` | 맥락에 남기지 않고 빠르게 질문. 답변이 오버레이로 표시되고 컨텍스트에 추가되지 않음 |

#### 컨텍스트 관리 원칙

- 관련 없는 작업 사이에는 `/clear`를 자주 사용
- 같은 문제로 두 번 이상 수정을 반복했다면 컨텍스트가 오염된 것이다. `/clear`하고 더 구체적인 프롬프트로 새로 시작
- Claude Code는 컨텍스트 한계에 가까워지면 자동으로 중요한 코드와 결정사항을 보존하면서 대화 히스토리를 압축한다 (약 95% 용량에서 트리거)
- CLAUDE.md에 `"압축 시 수정된 파일 목록과 테스트 명령을 항상 보존해줘"` 같은 지시를 추가해 압축 동작을 커스터마이즈할 수 있다

### 7. 체크포인트와 세션 재개 활용

#### 체크포인트

Claude가 변경을 만들기 전에 자동으로 체크포인트를 생성한다. `Esc` 두 번 또는 `/rewind`로 복원 메뉴를 열어 다음을 선택할 수 있다:
- 대화만 복원
- 코드만 복원
- 대화와 코드 모두 복원
- 선택한 메시지부터 요약

체크포인트는 세션을 닫아도 유지되므로, 터미널을 종료한 후에도 되돌릴 수 있다. 단, Claude가 만든 변경만 추적하며 외부 프로세스의 변경은 추적하지 않는다. git을 대체하는 것이 아니다.

#### 세션 재개

```bash
claude --continue    # 가장 최근 대화 재개
claude --resume      # 최근 대화 목록에서 선택
```

`/rename`으로 세션에 `"oauth-migration"` 같은 이름을 붙여 나중에 찾기 쉽게 할 수 있다. 여러 작업 흐름에 대해 별도의 영속 컨텍스트를 유지하는 식으로 브랜치처럼 활용하라.

### 8. 자동화와 규모 확장

#### 비대화형(Non-interactive) 모드

```bash
# 단발성 쿼리
claude -p "이 프로젝트가 뭘 하는지 설명해줘"

# 스크립트를 위한 구조화된 출력
claude -p "모든 API 엔드포인트 목록화" --output-format json

# 실시간 처리를 위한 스트리밍
claude -p "이 로그 파일 분석해줘" --output-format stream-json
```

CI 파이프라인, pre-commit hook, 자동화 워크플로우에 통합할 때 활용한다.

#### 대규모 마이그레이션 (파일별 병렬 실행)

```bash
for file in $(cat files.txt); do
  claude -p "$file를 React에서 Vue로 마이그레이션해줘. OK 또는 FAIL만 반환해줘" \
    --allowedTools "Edit,Bash(git commit *)"
done
```

`--allowedTools` 플래그로 무인 실행 시 Claude가 할 수 있는 작업을 제한한다. 2~3개 파일로 먼저 테스트하고 프롬프트를 다듬은 뒤 전체 실행하라.

#### Auto 모드

```bash
claude --permission-mode auto -p "모든 린트 오류 수정해줘"
```

별도 분류 모델이 명령을 실행 전에 검토해 범위 확장, 미확인 인프라, 악의적 콘텐츠 기반 동작만 차단하고 일상적인 작업은 프롬프트 없이 진행한다.

### 9. 흔한 실수와 안티패턴

| 안티패턴 | 증상 | 해결책 |
|---------|------|--------|
| **주방 싱크 세션** | 하나의 대화에서 서로 무관한 여러 작업 수행 | 관련 없는 작업 사이에 `/clear` |
| **반복 수정 루프** | 같은 문제로 두 번 이상 수정 요청 | `/clear` 후 실패에서 배운 내용을 담은 더 나은 프롬프트로 재시작 |
| **비대해진 CLAUDE.md** | Claude가 중요한 규칙을 무시 | 불필요한 줄 제거. Claude가 이미 올바르게 하는 것은 삭제 |
| **검증 없는 신뢰** | 그럴듯하지만 엣지 케이스를 처리하지 못하는 구현 | 항상 검증 수단(테스트, 스크립트, 스크린샷) 제공 |
| **무한 탐색** | 범위 없이 "조사해줘" 요청으로 파일 수백 개 읽음 | 탐색 범위를 좁히거나 서브 에이전트에 위임해 메인 컨텍스트 보호 |

### 10. 권한 관리

기본적으로 Claude Code는 시스템을 수정할 수 있는 작업(파일 쓰기, Bash 명령, MCP 도구 등)에 대해 권한을 요청한다. 이는 안전하지만 번거롭다. 세 가지 방법으로 인터럽션을 줄일 수 있다.

- **Auto 모드**: 별도 분류 모델이 승인을 처리. 작업의 전반적인 방향은 신뢰하지만 모든 단계를 확인하고 싶지 않을 때 적합
- **권한 허용 목록**: `npm run lint`, `git commit` 같이 안전한 특정 도구 허용
- **샌드박스**: OS 수준의 격리로 파일 시스템과 네트워크 접근을 제한하면서 Claude가 자유롭게 작동하도록 허용

### 11. CLI 도구와 MCP 서버 연동

CLI 도구는 외부 서비스와 상호작용하는 가장 컨텍스트 효율적인 방법이다.

- **GitHub**: `gh` CLI 설치 시 이슈 생성, PR 오픈, 코멘트 읽기 자동화 가능
- **처음 보는 CLI 도구**: `"foo-cli-tool --help로 foo 도구를 배우고 A, B, C를 해결해줘"` 같은 프롬프트로 Claude가 스스로 배우게 할 수 있음
- **MCP 서버**: `claude mcp add`로 Notion, Figma, 데이터베이스 등 외부 도구 연결

---

## 핵심 정리

- 컨텍스트 윈도우 관리가 Claude Code 활용에서 가장 중요한 변수이며, 성능과 비용 모두에 직접적인 영향을 미친다
- Claude가 스스로 결과를 검증할 수 있도록 테스트, 명령, 스크린샷 등의 검증 수단을 함께 제공하면 품질이 극적으로 향상된다
- 탐색(Plan Mode) → 계획 → 구현으로 단계를 분리하면 잘못된 방향으로 구현되는 것을 방지한다
- CLAUDE.md는 짧게 유지해야 하며, Claude가 이미 올바르게 하는 내용은 포함하지 않는다
- 같은 문제로 두 번 이상 수정하고 있다면 `/clear`하고 더 나은 프롬프트로 새로 시작하는 것이 낫다
