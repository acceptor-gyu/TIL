# 기존 서비스에 다국어 제공 - 현재는 한국어만, 새로 추가되는 언어는 영어부터

## 개요
이미 한국어로 운영 중인 서비스에 다국어(i18n) 제공 요청을 받아 이에 해당하는 전략과 고려사항을 정리한다. 첫 번째 추가 언어는 영어로 시작한다.

---

## 상세 내용

### 1. 다국어 도입 전 현황 분석

다국어를 도입하기 전에 현재 코드베이스의 상태를 정확히 파악해야 한다. 이 작업을 건너뛰면 마이그레이션 도중 예상치 못한 곳에서 누락된 텍스트가 발생한다.

#### 하드코딩된 문자열 파악

프론트엔드와 백엔드 모두를 대상으로 하드코딩된 한국어 문자열을 스캔한다.

- **프론트엔드**: JSX/TSX 내 리터럴 텍스트, `alert()`, `confirm()`, `placeholder`, `aria-label` 등
- **백엔드**: Exception 메시지, 이메일 템플릿, 알림 메시지, 검증 오류 메시지 등
- **grep 또는 정적 분석 도구**로 한글 유니코드 범위(`[\uAC00-\uD7A3]`)를 스캔하여 목록화

```bash
# 한글 포함 파일 일괄 스캔
grep -rn --include="*.java" "[가-힣]" ./src
grep -rn --include="*.tsx" "[가-힣]" ./src
```

#### 날짜, 숫자, 통화 포맷 범위 확인

단순 텍스트 번역 외에 **로케일별 포맷** 차이도 반드시 현황 파악이 필요하다.

| 항목 | 한국 | 영미권 |
|------|------|--------|
| 날짜 | 2026년 3월 17일 | March 17, 2026 |
| 통화 | ₩1,000,000 | $1,000.00 |
| 숫자 | 10,000 | 10,000 (소수점 표기 차이 존재) |
| 시간 | 오후 2:30 | 2:30 PM |

#### DB 저장 콘텐츠 구분

DB에 저장된 텍스트는 두 종류로 나뉘며, 처리 전략이 다르다.

- **시스템 메시지/코드성 데이터**: 카테고리명, 상태값, 약관 등 → 번역 파일 또는 번역 테이블로 관리
- **사용자 생성 콘텐츠(UGC)**: 게시글, 댓글, 리뷰 등 → 다국어 지원 여부를 별도로 결정 (초기에는 번역하지 않는 것이 일반적)

### 2. i18n 아키텍처 설계

#### 리소스 번들 / 메시지 키 기반 구조

모든 사용자 노출 텍스트는 코드에서 분리하여 언어별 파일로 관리한다.

**Spring Boot 백엔드 예시:**

```
src/main/resources/
  i18n/
    messages.properties       # 기본값 (한국어 Fallback)
    messages_ko.properties    # 한국어
    messages_en.properties    # 영어
```

```properties
# messages_ko.properties
user.login.success=로그인이 완료되었습니다.
user.login.fail=이메일 또는 비밀번호가 올바르지 않습니다.
error.not.found=요청한 리소스를 찾을 수 없습니다.
```

```properties
# messages_en.properties
user.login.success=You have successfully logged in.
user.login.fail=Invalid email or password.
error.not.found=The requested resource could not be found.
```

`application.properties` 에서 메시지 설정:

```properties
spring.messages.basename=i18n/messages
spring.messages.encoding=UTF-8
spring.messages.fallback-to-system-locale=false
spring.messages.use-code-as-default-message=false
```

#### 백엔드 LocaleResolver 전략

Spring Boot는 세 가지 LocaleResolver를 제공하며, 서비스 특성에 맞게 선택한다.

| LocaleResolver | 로케일 저장 위치 | 특징 |
|----------------|-----------------|------|
| `AcceptHeaderLocaleResolver` | 브라우저 Accept-Language 헤더 | 기본값, 별도 설정 불필요 |
| `CookieLocaleResolver` | 쿠키 | 비로그인 사용자에게도 설정 유지 |
| `SessionLocaleResolver` | 세션 | 로그인 세션 동안만 유지 |

**추천 전략 (로그인 서비스):**

```java
@Configuration
public class I18nConfig implements WebMvcConfigurer {

    // 로그인 유저: DB에서 로케일 로드
    // 비로그인 유저: CookieLocaleResolver로 Fallback
    @Bean
    public LocaleResolver localeResolver() {
        CookieLocaleResolver resolver = new CookieLocaleResolver("LOCALE");
        resolver.setDefaultLocale(Locale.KOREAN);
        resolver.setCookieMaxAge(Duration.ofDays(365));
        return resolver;
    }

    // ?lang=en 파라미터로 언어 변경 허용
    @Bean
    public LocaleChangeInterceptor localeChangeInterceptor() {
        LocaleChangeInterceptor interceptor = new LocaleChangeInterceptor();
        interceptor.setParamName("lang");
        return interceptor;
    }

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(localeChangeInterceptor());
    }
}
```

**MessageSource 사용:**

```java
@Service
@RequiredArgsConstructor
public class NotificationService {

    private final MessageSource messageSource;

    public String getLocalizedMessage(String code, Locale locale) {
        return messageSource.getMessage(code, null, locale);
    }
}
```

#### 프론트엔드 i18n 라이브러리 선택

| 라이브러리 | 적합한 환경 | 특징 |
|-----------|------------|------|
| `i18next` + `react-i18next` | React SPA | 생태계가 가장 넓음, 플러그인 풍부 |
| `next-intl` | Next.js App Router | App Router와 Server Component 완벽 지원 |
| `react-intl` (FormatJS) | React 전반 | ICU MessageFormat 기반, 표준 준수 |

**next-intl 기본 구조 예시 (Next.js App Router):**

```
messages/
  ko.json
  en.json
```

```jsonc
// messages/ko.json
{
  "auth": {
    "login": "로그인",
    "logout": "로그아웃",
    "welcome": "{name}님, 환영합니다!"
  }
}
```

```tsx
// 서버 컴포넌트에서 사용
import { useTranslations } from 'next-intl';

export default function LoginButton() {
  const t = useTranslations('auth');
  return <button>{t('login')}</button>;
}
```

### 3. 언어 전환 UX 설계

#### 언어 선택 UI 위치와 형태

언어 전환 버튼의 위치는 서비스 성격과 대상 사용자에 따라 결정한다.

| 위치 | 적합한 케이스 |
|------|--------------|
| **헤더 우측** | 다국어 사용자가 많고, 언어 전환이 빈번한 서비스 |
| **푸터** | 다국어 전환 빈도가 낮고, 콘텐츠 밀도가 높은 서비스 |
| **설정 페이지** | 로그인 사용자 대상, 언어를 프로필에 저장하는 경우 |
| **온보딩 화면** | 첫 방문 시 언어 선택 유도 |

**UI 설계 원칙:**
- 지구본 아이콘(🌐)을 함께 사용하여 언어 선택 UI임을 직관적으로 인지시킨다
- 언어 이름은 **해당 언어 그 자체로 표기**한다 (예: `한국어`, `English` → `KO`, `EN` 약어는 지양)
- 드롭다운 내 목록은 언어 이름을 해당 언어로 표기한다 (영어권 사용자는 `Korean`이 아닌 `한국어`를 봐야 선택 가능)

#### 사용자 언어 설정 저장 방식

```
요청 처리 우선순위:
1. 로그인 사용자의 DB 프로필 언어 설정
2. 쿠키에 저장된 언어 설정 (비로그인 또는 로그인 전 선택)
3. Accept-Language 헤더 (브라우저 기본 언어)
4. 서비스 기본 언어 (한국어)
```

| 저장 방식 | 장점 | 단점 |
|-----------|------|------|
| **DB 프로필** | 기기/브라우저 무관하게 유지 | 로그인 사용자에게만 적용 |
| **쿠키** | 비로그인 사용자에게도 적용 | 쿠키 삭제 시 초기화 |
| **브라우저 기본값** | 설정 불필요 | 사용자가 직접 변경 불가 |

#### 언어 변경 방식: 페이지 리로드 vs 실시간 전환

- **페이지 리로드**: 구현이 단순하고, 서버 렌더링 서비스(SSR/MVC)에 적합. 상태 초기화 위험이 있으므로 입력 중인 폼에서 변경 시 경고 필요
- **실시간(클라이언트 사이드) 전환**: SPA에서 자연스러운 경험 제공. 서버에서 가져와야 하는 콘텐츠가 있으면 복잡해짐

> 초기 도입 시에는 **페이지 리로드 방식**이 구현 복잡도가 낮아 리스크가 적다.

### 4. 번역 워크플로우

#### 번역 키 네이밍 컨벤션

계층 구조(`page.section.element`) 패턴이 가장 널리 쓰인다. 키는 **내용이 아닌 의미/컨텍스트**를 기준으로 명명한다.

```
// 좋은 예
auth.login.submit_button        // 로그인 페이지 > 제출 버튼
auth.login.email_placeholder    // 로그인 페이지 > 이메일 입력 placeholder
product.card.add_to_cart        // 상품 카드 > 장바구니 추가 버튼
error.validation.required_field // 검증 에러 > 필수 입력 항목

// 나쁜 예
login_btn                       // 컨텍스트 부족
button_text_1                   // 의미 없는 번호
submit                          // 재사용 유혹 → 문맥에 따라 번역이 달라질 수 있음
```

**동일 텍스트라도 다른 키를 사용하는 이유:**
> 영어로는 "Submit"이지만, 로그인 폼에서는 "로그인", 검색 폼에서는 "검색"으로 번역되어야 한다. 키를 재사용하면 번역 맥락이 사라진다.

#### 번역 누락 시 Fallback 전략

```
요청 언어 (en) 메시지 없음
    ↓
기본 언어 (ko) 메시지로 Fallback
    ↓
기본 언어도 없음
    ↓
키 자체를 노출 (개발 환경) / 빈 문자열 (프로덕션)
```

Spring Boot에서는 `messages.properties`를 기본 Fallback으로 설정한다:

```properties
# application.properties
spring.messages.fallback-to-system-locale=false
# 시스템 로케일이 아닌 명시적 기본값(messages.properties)으로 Fallback
```

#### 번역 관리 도구(TMS) 활용

코드베이스가 커질수록 스프레드시트로 번역을 관리하면 누락, 중복, 버전 불일치 문제가 생긴다. TMS를 도입하면 번역 메모리, 용어집, 담당자 워크플로우를 체계적으로 관리할 수 있다.

| 도구 | 특징 | 비용 |
|------|------|------|
| **Crowdin** | GitHub/GitLab 연동, 스크린샷 컨텍스트 제공 | 유료 (소규모 무료) |
| **Phrase (Lokalise)** | 엔터프라이즈 기능 풍부, API 지원 | 유료 |
| **Tolgee** | 오픈소스, 셀프 호스팅 가능 | 무료/유료 |
| **스프레드시트** | 초기 소규모에 적합 | 무료 |

### 5. DB 다국어 데이터 처리

DB에 저장되는 데이터의 다국어 처리는 크게 세 가지 패턴이 있다.

#### 패턴 1: 컬럼 분리 방식

지원 언어별로 별도 컬럼을 추가한다.

```sql
CREATE TABLE category (
    id         BIGINT PRIMARY KEY,
    name_ko    VARCHAR(100) NOT NULL,  -- 한국어
    name_en    VARCHAR(100),           -- 영어
    created_at TIMESTAMP
);
```

- **장점**: 쿼리가 단순하고 JOIN 불필요
- **단점**: 언어 추가 시마다 ALTER TABLE 필요, 컬럼 수가 늘어남
- **적합한 경우**: 지원 언어가 2~3개로 고정되고 번역 대상 컬럼이 적은 경우

#### 패턴 2: 별도 번역 테이블 (권장)

메인 테이블과 분리된 번역 전용 테이블을 만든다.

```sql
CREATE TABLE product (
    id         BIGINT PRIMARY KEY,
    price      DECIMAL(10, 2),
    created_at TIMESTAMP
);

CREATE TABLE product_translation (
    id          BIGINT PRIMARY KEY,
    product_id  BIGINT NOT NULL REFERENCES product(id),
    locale      VARCHAR(10) NOT NULL,  -- 'ko', 'en', 'ja'
    name        VARCHAR(200) NOT NULL,
    description TEXT,
    UNIQUE (product_id, locale)
);
```

```sql
-- 조회 예시
SELECT p.*, pt.name, pt.description
FROM product p
JOIN product_translation pt ON pt.product_id = p.id AND pt.locale = 'en'
WHERE p.id = 1;
```

- **장점**: 언어 추가 시 테이블 구조 변경 불필요, 번역 누락 현황 파악 쉬움
- **단점**: JOIN 필요, 쿼리 복잡도 증가
- **적합한 경우**: 다국어 확장 가능성이 있고, 번역 대상이 많은 경우

#### 패턴 3: JSON 컬럼 방식

MySQL 5.7+ / PostgreSQL의 JSON/JSONB 컬럼을 활용한다.

```sql
CREATE TABLE product (
    id         BIGINT PRIMARY KEY,
    name       JSON NOT NULL,  -- {"ko": "상품명", "en": "Product Name"}
    price      DECIMAL(10, 2),
    created_at TIMESTAMP
);
```

```sql
-- MySQL JSON 경로 조회
SELECT JSON_UNQUOTE(JSON_EXTRACT(name, '$.en')) AS name FROM product WHERE id = 1;
```

- **장점**: 스키마 변경 없이 언어 추가 가능, 유연한 구조
- **단점**: 인덱스 적용 어려움, JSON 함수를 통한 쿼리로 가독성 저하, ORM 연동 복잡
- **적합한 경우**: 프로토타입 단계, 번역 데이터가 자주 검색되지 않는 경우

#### 정적 코드성 데이터 처리 전략

DB 번역 테이블 대신 **번역 파일로 관리하는 방법**도 있다. 카테고리 코드, 상태값 같은 코드성 데이터는 DB에 코드값만 저장하고 화면 표시는 번역 파일에서 가져온다.

```properties
# messages_en.properties
category.ELECTRONICS=Electronics
category.FASHION=Fashion
status.ACTIVE=Active
status.INACTIVE=Inactive
```

### 6. 점진적 마이그레이션 전략

기존 서비스 전체를 한 번에 다국어 전환하려 하면 배포 리스크가 크다. **페이지/기능 단위 단계적 적용**이 안전하다.

#### 우선순위 결정 기준

```
1단계: 공통 UI (헤더, 푸터, 공통 에러 메시지, 유효성 검증 메시지)
2단계: 핵심 전환 흐름 (로그인, 회원가입, 결제)
3단계: 주요 페이지 (메인, 상품 목록, 상품 상세)
4단계: 부가 기능 (마이페이지, 고객센터, 약관)
5단계: 관리자 페이지 (내부 사용자는 한국어만으로도 가능)
```

#### 하드코딩 → 메시지 키 치환 작업 흐름

1. 대상 파일에서 한글 문자열 추출
2. 번역 키 설계 및 번역 파일에 추가
3. 코드에서 하드코딩 제거 후 번역 함수 호출로 교체
4. 번역가에게 영어 번역 요청
5. 영어 번역 완료 후 해당 페이지를 영어로 제공

#### 번역 커버리지 모니터링

```bash
# 키 개수 비교로 누락 번역 확인
wc -l messages_ko.properties   # 한국어 키 수
wc -l messages_en.properties   # 영어 키 수
# 차이가 있으면 누락 번역 존재
```

TMS를 사용하는 경우 대시보드에서 언어별 번역 완성도(%)를 실시간으로 확인할 수 있다.

### 7. 테스트 및 QA

#### 영어 텍스트 길이 차이로 인한 UI 깨짐 확인

영어는 한국어보다 평균적으로 1.3~2배 길어진다. 특히 버튼, 탭, 테이블 헤더, 카드 컴포넌트를 집중적으로 확인한다.

```css
/* 텍스트 넘침 방지 - 번역 도입 전에 미리 적용 */
.button {
    white-space: nowrap;
    overflow: hidden;
    text-overflow: ellipsis;
    min-width: fit-content;
}
```

**Pseudo-Localization으로 레이아웃 사전 검증:**

실제 번역 전에 텍스트를 인위적으로 늘려서 UI 깨짐을 사전에 검증하는 기법이다.

```
원문: "로그인"
pseudo-locale: "[~LOGIN~]" (특수문자와 길이 추가)
```

`pseudo-l10n` npm 패키지를 CI에 통합하면 번역 배포 전에 레이아웃 문제를 자동으로 감지할 수 있다.

#### RTL(Right-to-Left) 언어 향후 확장 고려

영어 추가 단계에서는 RTL 대응이 필요하지 않지만, 아랍어/히브리어 확장 가능성이 있다면 CSS에서 `direction` 속성과 `logical properties`를 미리 활용해두면 나중에 수정 범위가 줄어든다.

```css
/* 물리 속성 대신 논리 속성 사용 */
/* 변경 전 */
.card { padding-left: 16px; margin-right: 8px; }

/* 변경 후 - RTL 대응 용이 */
.card { padding-inline-start: 16px; margin-inline-end: 8px; }
```

#### 번역 누락 자동 감지 CI 파이프라인

`i18n-check` 같은 도구를 CI에 통합하여 번역 키 누락, 불일치를 자동으로 검출한다.

```yaml
# .github/workflows/i18n-check.yml
- name: Check translation completeness
  run: npx i18n-check --source messages/ko.json --targets messages/en.json
```

빌드 단계에서 누락 번역이 있으면 배포를 차단하거나 경고 알림을 발송한다.

## 핵심 정리

- **분석 먼저**: 무작정 라이브러리를 도입하기 전에 하드코딩 범위와 DB 콘텐츠 종류를 파악해야 마이그레이션 공수 산정이 가능하다
- **우선순위 기반 점진 적용**: 전체를 한 번에 바꾸지 말고 공통 UI → 핵심 전환 흐름 → 주요 페이지 순으로 단계적으로 적용한다
- **LocaleResolver 전략**: 로그인 사용자는 DB 프로필, 비로그인 사용자는 쿠키, 최후는 Accept-Language 헤더 순으로 우선순위를 두는 것이 일반적이다
- **번역 키는 의미 기반으로**: 텍스트 내용이 아닌 컨텍스트를 키 이름에 반영해야 번역 품질이 올라간다. `submit`보다 `auth.login.submit_button`이 번역가에게 맥락을 제공한다
- **같은 텍스트도 키를 분리**: 영어로 같은 문자열이라도 문맥이 다르면 번역이 달라질 수 있으므로 키를 재사용하지 않는다
- **DB 패턴 선택**: 2~3개 언어 고정이면 컬럼 분리, 확장 가능성이 있으면 번역 테이블이 유리하다
- **레이아웃은 미리 유연하게**: 영어 텍스트가 길어지는 것을 고려해 버튼, 카드 등 컴포넌트에 `min-width`, `text-overflow` 등을 미리 적용해둔다
- **UGC는 별도 판단**: 사용자가 생성한 콘텐츠(게시글 등)는 기계 번역 API 연동이나 미번역 그대로 노출 중 서비스 특성에 맞게 결정한다

## 키워드

| 키워드 | 설명 |
|--------|------|
| `i18n` | Internationalization의 약어 (i와 n 사이에 18글자). 소프트웨어가 다양한 언어와 지역을 지원할 수 있도록 설계하는 과정 |
| `L10n` | Localization의 약어 (L과 n 사이에 10글자). i18n으로 만들어진 기반 위에 특정 언어/지역에 맞게 실제 번역 및 포맷 적용 |
| `다국어` | 하나의 서비스에서 여러 언어를 지원하는 것 |
| `국제화` | i18n과 동의어. 특정 언어에 종속되지 않도록 소프트웨어 구조를 설계하는 것 |
| `현지화` | L10n과 동의어. 특정 국가/문화에 맞게 언어, 날짜, 통화, 이미지 등을 조정하는 것 |
| `LocaleResolver` | Spring MVC에서 HTTP 요청으로부터 사용자의 Locale(언어/지역)을 결정하는 인터페이스. 구현체로 AcceptHeaderLocaleResolver, CookieLocaleResolver, SessionLocaleResolver가 있음 |
| `리소스 번들` | 언어별로 분리된 번역 파일의 집합. Java에서는 `.properties` 파일, JavaScript에서는 JSON 파일로 관리 |
| `Fallback 언어` | 요청한 언어로 번역 메시지를 찾지 못했을 때 대신 사용하는 기본 언어. 보통 서비스의 주 언어(한국어)로 설정 |
| `번역 키` | 실제 텍스트 대신 코드에서 사용하는 식별자. `auth.login.submit_button`과 같이 계층 구조로 명명하며, TMS에서 번역을 관리하는 기준이 됨 |
| `언어 전환 UX` | 사용자가 서비스 언어를 변경할 때의 경험 설계. 전환 UI의 위치, 설정 저장 방식, 전환 시 페이지 동작 등을 포함 |
