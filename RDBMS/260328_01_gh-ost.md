# gh-ost (GitHub Online Schema Transmogrifier)

## 개요
gh-ost는 GitHub에서 개발한 MySQL 온라인 스키마 마이그레이션 도구로, 트리거를 사용하지 않고 binlog 스트리밍 방식으로 테이블 스키마를 변경한다. 운영 중인 서비스에 락이나 부하 없이 안전하게 DDL을 수행할 수 있다.

## 상세 내용

### 1. 기존 Online DDL의 한계

#### MySQL 기본 `ALTER TABLE`의 문제점
MySQL 기본 `ALTER TABLE`은 스키마 변경 중 해당 테이블에 메타데이터 락(MDL)을 걸어 모든 읽기/쓰기를 차단한다. 대용량 테이블의 경우 수십 분~수 시간 동안 서비스 장애가 발생할 수 있다. MySQL 5.6 이후 Online DDL 기능이 추가되었으나, 여전히 일부 DDL은 테이블 전체 복사가 필요하고 복제 지연이 심각하게 발생한다.

#### pt-online-schema-change (pt-osc)의 트리거 기반 방식과 한계
Percona Toolkit의 `pt-online-schema-change`는 트리거를 활용해 온라인 마이그레이션을 지원했다. 동작 방식은 다음과 같다.

1. 새로운 구조의 ghost 테이블 생성
2. 원본 테이블에 INSERT/UPDATE/DELETE 트리거 설치
3. 원본 테이블 데이터를 ghost 테이블로 청크 단위 복사
4. 트리거가 복사 중 발생하는 변경사항을 ghost 테이블에 동기 반영
5. `RENAME TABLE`로 테이블 교체

이 방식은 트리거가 모든 DML 쿼리에 동기적으로 개입하기 때문에 다음 문제가 있다.

- **파서/인터프리터 오버헤드**: 트리거는 해석되는 코드이므로 모든 DML에 추가 CPU 비용이 발생
- **락 경합**: 트리거 실행과 row copy가 동일한 트랜잭션 안에서 경쟁하며, 쓰기 동시성이 높을수록 데드락 위험 증가
- **일시 중지 불가**: 마스터 부하가 급증해도 트리거는 비활성화할 수 없으므로 마이그레이션을 즉시 멈출 방법이 없음
- **Replica 반영 지연**: 마스터에서 발생하는 트리거 부하가 replication lag를 유발

### 2. gh-ost의 동작 원리

gh-ost는 트리거 없이 binlog를 직접 파싱하는 비동기 방식을 채택했다.

#### 전체 마이그레이션 흐름

```
[원본 테이블] ──청크 복사──> [Ghost 테이블]
      │                          │
      └──DML 발생──> [binlog] ──┘ (비동기 적용)
                         │
                    gh-ost가 replica처럼 binlog 구독
```

1. **Ghost 테이블 생성**: `_tablename_gho`라는 이름으로 변경된 스키마의 임시 테이블 생성
2. **Changelog 테이블 생성**: `_tablename_ghc`라는 하트비트/진행 상태 추적용 테이블 생성
3. **Row Copy**: 원본 테이블 데이터를 `chunk-size` 단위로 ghost 테이블에 복사
4. **binlog 스트리밍**: gh-ost가 MySQL replica처럼 binlog를 구독하여 원본 테이블에 발생하는 INSERT/UPDATE/DELETE 이벤트를 ghost 테이블에 비동기 적용
5. **Cut-over**: row copy 완료 후 원본 테이블과 ghost 테이블을 원자적으로 교체

#### Row-based Replication (RBR) binlog 파싱
gh-ost는 반드시 `binlog_format=ROW`로 설정된 binlog를 파싱한다. Statement-based binlog는 실행 컨텍스트에 따라 결과가 달라질 수 있어 지원하지 않는다. RBR binlog에는 변경 전/후 row 이미지가 모두 포함되어 있으므로, gh-ost는 이를 파싱해 ghost 테이블에 정확히 반영한다.

#### Cut-over (테이블 교체) 메커니즘
Cut-over는 원본 테이블을 ghost 테이블로 원자적으로 교체하는 마지막 단계다. MySQL은 연결이 락을 유지한 상태에서 `RENAME TABLE`을 허용하지 않기 때문에, gh-ost는 다음과 같은 2단계 블로킹 스왑을 구현한다.

```sql
-- Connection 1 (lock holder)
LOCK TABLE original_table WRITE;

-- Connection 2 (rename waiter) - 락이 해제되는 순간 원자적으로 실행
RENAME TABLE original_table TO _original_table_del, ghost_table TO original_table;

-- Connection 1
UNLOCK TABLES;
```

추가로 sentry 테이블(`_tablename_del`)을 활용해 RENAME이 조기에 실행되지 않도록 보호한다. 이 과정에서 클라이언트 연결은 락이 걸린 짧은 시간 동안만 대기하며, 락이 해제되면 즉시 교체된 테이블에서 정상 처리된다.

- **성공 케이스**: 테이블이 원자적으로 스왑되고 클라이언트는 잠깐의 대기 후 마이그레이션된 테이블에서 정상 동작
- **실패 케이스**: 타임아웃이나 연결 오류 발생 시 원본 테이블이 그대로 유지되며, gh-ost는 재시도 가능

### 3. gh-ost의 아키텍처

#### 핵심 컴포넌트

| 컴포넌트 | 역할 |
|---|---|
| **Migrator** | 전체 마이그레이션 흐름 오케스트레이션. 각 컴포넌트를 조율하고 상태를 관리 |
| **Inspector** | MySQL 서버(주로 replica)에 연결해 테이블 구조, 복제 상태, 서버 파라미터 등을 검증 및 수집 |
| **Applier** | Ghost 테이블 생성, row copy, binlog 이벤트의 ghost 테이블 반영, cut-over 실행 담당 |
| **EventsStreamer** | MySQL binlog를 구독하여 원본 테이블에 발생하는 변경 이벤트를 캡처하고 Applier에 전달 |
| **Throttler** | 서버 부하(Threads_running), 복제 지연(lag), 플래그 파일 등을 모니터링하여 마이그레이션 속도 제어 |
| **Server** | Unix 소켓 또는 TCP를 통해 마이그레이션 상태를 조회하고 동적으로 제어하는 인터페이스 제공 |

#### Replica 서버에서 binlog를 읽는 구조

gh-ost의 기본 권장 동작 방식은 replica에 연결해 binlog를 읽는 것이다.

```
[Master] ──replication──> [Replica]
                               │
                          gh-ost (replica처럼 binlog 구독)
                               │
                          [Master]에 ghost 테이블 쓰기
```

- gh-ost는 replica의 binlog를 읽어 master에서 발생하는 DML 이벤트를 추적
- 실제 ghost 테이블로의 쓰기는 master에서 수행
- 이 구조 덕분에 master는 단일 연결에서 순차적으로 ghost 테이블에 쓰는 부하만 받음
- replica에서 binlog를 읽으므로 master I/O가 추가로 발생하지 않음

#### Throttle 메커니즘을 통한 부하 제어

gh-ost는 다음 조건에서 row copy와 binlog 이벤트 처리를 모두 일시 중지한다.

- `--max-load`에 설정한 MySQL 상태 변수 임계치 초과 (예: `Threads_running=25`)
- `--max-lag-millis`에 설정한 복제 지연 임계치 초과
- `--throttle-flag-file`로 지정한 파일이 존재할 때
- `--throttle-additional-flag-file`로 지정한 공유 파일이 존재할 때 (여러 마이그레이션을 동시에 멈출 때 유용)

throttle이 활성화되면 master에 대한 모든 쓰기가 완전히 멈추기 때문에, pt-osc의 트리거와 달리 서비스 영향 없이 일시 정지가 가능하다.

### 4. 주요 옵션과 설정

#### 기본 실행 예시

```bash
# replica 연결, master에서 마이그레이션 실행
gh-ost \
  --user="gh-ost" \
  --password="secret" \
  --host=replica.db.internal \
  --database="my_schema" \
  --table="users" \
  --alter="ADD COLUMN last_login DATETIME, ADD INDEX idx_last_login(last_login)" \
  --max-load=Threads_running=25 \
  --critical-load=Threads_running=1000 \
  --chunk-size=1000 \
  --max-lag-millis=1500 \
  --execute
```

#### `--allow-on-master` vs Replica 기반 마이그레이션

| 옵션 | 설명 | 권장 여부 |
|---|---|---|
| 기본 (replica 연결) | replica에서 binlog 읽기, master에 ghost 테이블 쓰기 | 권장 |
| `--allow-on-master` | master에 직접 연결하여 binlog 읽기 | Statement-based 레플리케이션 환경에서만 사용 |
| `--migrate-on-replica` | replica에서만 마이그레이션 수행 | replica 독립 검증 시 |

#### `--chunk-size`, `--max-load`, `--critical-load`

```bash
--chunk-size=1000          # 한 번의 row copy 트랜잭션에서 처리할 행 수 (기본 1000)
--max-load=Threads_running=25      # 이 값 초과 시 throttle 진입
--critical-load=Threads_running=1000  # 이 값 초과 시 마이그레이션 즉시 중단
--max-lag-millis=1500      # replica lag가 1500ms 초과 시 throttle 진입
```

`--chunk-size`는 마이그레이션 속도와 부하 사이의 트레이드오프를 조정한다. 값을 크게 하면 빠르지만 트랜잭션이 무거워지고, 작게 하면 부하는 줄지만 마이그레이션 시간이 길어진다.

#### `--postpone-cut-over-flag-file`을 활용한 수동 전환

```bash
gh-ost \
  --postpone-cut-over-flag-file=/tmp/ghost.postpone.flag \
  --execute
```

이 파일이 존재하는 동안 gh-ost는 row copy를 완료하고 binlog 동기화를 유지하지만 cut-over를 실행하지 않는다. 운영자가 적절한 시점(트래픽이 낮은 새벽 등)에 파일을 삭제하면 자동으로 cut-over가 진행된다. 이를 통해 마이그레이션 완료 시점을 비즈니스 일정에 맞게 제어할 수 있다.

#### `--test-on-replica` 옵션으로 사전 검증

```bash
gh-ost \
  --host=replica.db.internal \
  --test-on-replica \
  --database="my_schema" \
  --table="users" \
  --alter="ADD COLUMN last_login DATETIME" \
  --execute
```

replica에서 전체 마이그레이션을 수행한 뒤 cut-over 직전에 멈추고, 원본 테이블과 ghost 테이블의 데이터를 비교 검증한다. 이후 테이블을 원래 상태로 복원한다. 실제 master에는 영향을 주지 않으므로 프로덕션 적용 전 안전하게 검증할 수 있다.

#### 인터랙티브 제어 (Unix Socket)

마이그레이션 실행 중 소켓을 통해 동적으로 상태를 조회하거나 설정을 변경할 수 있다.

```bash
# 현재 마이그레이션 상태 조회
echo "status" | nc -U /tmp/gh-ost.my_schema.users.sock

# throttle 활성화 (즉시 일시 중지)
echo "throttle" | nc -U /tmp/gh-ost.my_schema.users.sock

# throttle 해제
echo "no-throttle" | nc -U /tmp/gh-ost.my_schema.users.sock

# chunk-size 동적 변경
echo "chunk-size=500" | nc -U /tmp/gh-ost.my_schema.users.sock
```

### 5. pt-osc vs gh-ost 비교

#### 트리거 방식 vs binlog 방식의 근본적 차이

| 항목 | pt-osc (트리거 방식) | gh-ost (binlog 방식) |
|---|---|---|
| 변경 감지 방법 | 원본 테이블에 트리거 설치 | binlog 스트리밍 |
| DML 반영 타이밍 | 동기 (트리거 내에서 즉시) | 비동기 (binlog 파싱 후 적용) |
| Master 부하 | 모든 DML에 트리거 오버헤드 추가 | ghost 테이블 단일 연결 쓰기만 추가 |
| 일시 중지 | 불가능 (트리거는 항상 실행) | 가능 (row copy + binlog 처리 모두 중단) |
| 데드락 위험 | 높음 (트리거와 row copy가 경합) | 낮음 (비동기 처리로 경합 없음) |
| Foreign Key 지원 | 지원 | 미지원 |
| Trigger 있는 테이블 | 지원 (MySQL 5.7+ 중첩 트리거 필요) | 미지원 |
| 최소 MySQL 버전 | 5.0+ | 5.7+ |
| binlog 요건 | Statement 또는 Row | Row-based 필수 |

#### 성능, 안정성, 제어 가능성 비교

**성능 측면**: gh-ost는 replica binlog를 읽기 때문에 master I/O를 추가로 유발하지 않는다. 반면 pt-osc는 트리거 실행이 master의 모든 DML에 영향을 미쳐 쓰기 집약적인 환경에서 성능 저하가 두드러진다.

**안정성 측면**: pt-osc의 트리거는 원본 테이블과 ghost 테이블이 동일한 트랜잭션 안에서 경합하므로 데드락 가능성이 있다. gh-ost는 비동기 처리이므로 이런 경합이 없다. 다만 gh-ost는 비동기 특성상 cut-over 시점에 binlog 동기화가 완료되어야 한다.

**제어 가능성 측면**: gh-ost는 마이그레이션 도중에도 Unix 소켓으로 동적 제어가 가능하며, throttle을 통한 즉각적인 일시 중지/재개를 지원한다. pt-osc는 프로세스를 종료하지 않고 세밀하게 제어하기 어렵다.

#### 각각이 더 적합한 상황

**gh-ost가 적합한 경우**
- 쓰기 부하가 높은 프로덕션 환경
- 마이그레이션 일시 중지/재개가 필요한 경우
- 세밀한 throttle 제어가 필요한 경우
- cut-over 시점을 수동으로 제어해야 하는 경우

**pt-osc가 적합한 경우**
- Foreign Key가 있는 테이블을 마이그레이션해야 하는 경우
- 트리거가 설정된 테이블을 마이그레이션해야 하는 경우
- MySQL 5.6 이하 버전 환경
- Row-based replication을 사용하지 않는 환경

### 6. 운영 환경에서의 고려사항

#### binlog_format=ROW 필수 조건

gh-ost는 반드시 Row-Based Replication (RBR) 형식의 binlog가 필요하다. Statement-based binlog는 실행 컨텍스트에 따라 결과가 달라질 수 있으므로 지원하지 않는다. 또한 binlog_row_image가 `FULL`이어야 변경 전/후 row 이미지를 모두 파싱할 수 있다.

```sql
-- 현재 설정 확인
SHOW VARIABLES LIKE 'binlog_format';
SHOW VARIABLES LIKE 'binlog_row_image';

-- 필요한 설정
binlog_format = ROW
binlog_row_image = FULL
```

master가 Statement-based replication을 사용하더라도, 최소 하나의 replica가 RBR로 설정되어 있으면 gh-ost를 그 replica에 연결하여 사용할 수 있다.

#### Foreign Key 제약 시 주의점

gh-ost는 현재 Foreign Key를 지원하지 않는다. FK가 있는 테이블을 마이그레이션하려 하면 오류가 발생한다. FK가 존재하는 경우 다음 중 하나를 선택해야 한다.

- pt-osc 사용 (FK 지원)
- FK를 일시적으로 제거 후 마이그레이션 후 재설정 (애플리케이션 레벨에서 무결성 보장 필요)

마찬가지로 원본 테이블에 트리거가 있는 경우에도 gh-ost를 사용할 수 없다.

#### 대용량 테이블 마이그레이션 시 소요 시간 예측

마이그레이션 소요 시간은 다음 요소에 따라 달라진다.

- **테이블 크기**: 행 수와 데이터 크기에 비례
- **chunk-size**: 작을수록 안전하지만 시간이 늘어남
- **서버 부하**: throttle이 자주 걸릴수록 시간이 늘어남
- **replica lag**: lag가 크면 throttle이 빈번하게 발생

gh-ost는 실행 중 예상 완료 시간(ETA)을 제공한다. `--exact-rowcount` 옵션을 사용하면 정확한 행 수를 COUNT(*)로 집계하여 더 정밀한 ETA를 제공하지만, 대용량 테이블에서는 이 집계 자체가 부하를 유발할 수 있으므로 주의한다.

#### 롤백 전략

gh-ost는 마이그레이션 도중 언제든 중단할 수 있으며, 원본 테이블은 항상 정상 상태로 유지된다.

- **중단**: 프로세스를 종료하면 ghost 테이블과 changelog 테이블이 남는다. `--initially-drop-ghost-table` 옵션으로 재시작 시 이전 ghost 테이블을 삭제하고 처음부터 다시 시작 가능
- **cut-over 이후 롤백**: cut-over가 완료된 후에는 old 테이블(`_tablename_del`)이 남아있다. 이를 통해 긴급 상황에서 수동으로 RENAME하여 원복 가능 (단, cut-over 이후 발생한 변경사항은 반영되지 않음)
- **v1.1.8 이후 체크포인트**: 마이그레이션 체크포인트 기능이 추가되어, 중단된 마이그레이션을 마지막 체크포인트부터 재개할 수 있다 (관련 binlog가 아직 유지되는 경우)

## 핵심 정리
- gh-ost는 트리거 없이 binlog 기반으로 온라인 스키마 변경을 수행하는 도구
- 운영 중 부하 제어(throttle)와 일시 중지/재개가 가능
- pt-osc 대비 락 경합과 데드락 위험이 낮음
- binlog_format=ROW가 필수이며, FK와 트리거가 있는 테이블은 지원하지 않음
- `--postpone-cut-over-flag-file`로 cut-over 시점을 비즈니스 일정에 맞게 조율 가능
- Unix 소켓 기반 인터랙티브 제어로 마이그레이션 도중에도 동적 설정 변경 가능

## 키워드
- `gh-ost`: GitHub이 개발한 MySQL 온라인 스키마 마이그레이션 도구로, 트리거 없이 binlog 기반으로 동작
- `Online DDL`: 서비스 중단 없이 테이블 스키마를 변경하는 기법
- `Schema Migration`: 데이터베이스의 스키마(구조)를 변경하는 작업
- `binlog`: MySQL 서버에서 발생하는 모든 데이터 변경 이벤트를 기록하는 바이너리 로그
- `pt-online-schema-change`: Percona Toolkit의 트리거 기반 온라인 스키마 변경 도구
- `MySQL`: 오픈소스 관계형 데이터베이스 관리 시스템
- `Cut-over`: 마이그레이션의 최종 단계로, ghost 테이블을 원본 테이블과 원자적으로 교체하는 과정
- `Trigger-less Migration`: 데이터베이스 트리거를 사용하지 않고 binlog 스트리밍으로 변경사항을 동기화하는 마이그레이션 방식
- `Throttle`: 마이그레이션 중 서버 부하나 복제 지연이 임계치를 초과할 때 작업을 일시 중지하는 부하 제어 메커니즘
