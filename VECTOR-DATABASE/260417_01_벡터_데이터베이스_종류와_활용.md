# 벡터 데이터베이스 종류와 활용

## 개요
벡터 데이터베이스는 고차원 벡터(임베딩)를 저장하고 유사도 검색을 수행하는 데 특화된 데이터베이스이다. AI/ML 시대에 텍스트, 이미지, 오디오 등 비정형 데이터를 의미 기반으로 검색하기 위해 핵심적으로 활용된다.

## 상세 내용

### 벡터 데이터베이스란
비정형 데이터(텍스트, 이미지, 오디오 등)는 딥러닝 모델을 통해 수백~수천 차원의 부동소수점 배열인 **임베딩(Embedding)**으로 변환된다. 임베딩은 데이터의 의미적 특성을 고차원 공간 상의 좌표로 표현하며, 의미가 유사한 데이터일수록 공간상에서 가까운 위치에 놓인다.

전통적인 RDBMS는 `WHERE name = 'apple'` 과 같은 정확 일치(Exact Match) 기반 검색에 최적화되어 있다. 반면 벡터 유사도 검색은 쿼리 벡터와 저장된 벡터들 사이의 거리 혹은 유사도를 계산하여 가장 가까운 데이터를 반환한다.

이 때문에 "사과와 의미적으로 유사한 문서를 찾아라" 같은 시맨틱 검색은 RDBMS로는 구현이 어렵고, 벡터 데이터베이스가 필요하다.

임베딩 생성 과정은 다음과 같다.
```
원문 데이터 → 임베딩 모델(OpenAI text-embedding, BERT 등) → 벡터(float[] 배열) → 벡터 DB에 저장
```

### 유사도 검색 알고리즘
벡터 간의 유사도를 측정하는 방법에는 크게 세 가지가 있다.

**코사인 유사도 (Cosine Similarity)**
두 벡터가 이루는 각도의 코사인 값으로 유사도를 측정한다. 벡터의 크기(magnitude)는 무시하고 방향만을 비교하기 때문에 문서의 길이 차이에 관계없이 내용의 방향적 유사성을 측정하는 데 적합하다.
- 값 범위: -1 ~ 1 (1에 가까울수록 유사)
- 적합한 사용 사례: NLP 텍스트 임베딩 비교, 문서 유사도 검색

**유클리드 거리 (Euclidean Distance / L2)**
두 벡터 사이의 직선 거리를 측정한다. 벡터의 크기와 방향 모두를 고려하기 때문에 임베딩에 크기 정보가 의미를 가질 때 유용하다.
- 값 범위: 0 이상 (0에 가까울수록 유사)
- 적합한 사용 사례: 이미지 유사도, 공간적 데이터

**내적 (Inner Product / Dot Product)**
두 벡터의 요소를 곱하여 합산한다. 벡터가 정규화(크기 = 1)된 경우 코사인 유사도와 동일하며, 코사인 유사도보다 계산이 빠르다.
- 적합한 사용 사례: 임베딩 모델이 내적 기반으로 학습된 경우, 추천 시스템

> 핵심 원칙: 임베딩 모델이 학습에 사용한 유사도 지표와 검색 시 사용하는 지표를 일치시켜야 한다. 예를 들어 `all-MiniLM-L6-v2` 모델은 코사인 유사도 기반으로 학습되었다.

### 인덱싱 기법
수백만~수십억 개의 벡터에 대해 완전 탐색(Brute-force)을 수행하면 응답 시간이 너무 길어진다. 이를 해결하기 위해 ANN(Approximate Nearest Neighbor) 알고리즘 기반의 인덱싱 기법을 사용한다. 완벽한 정확도를 약간 포기하는 대신, 100~1000배의 속도 향상을 얻으면서 95~99%의 정확도를 유지한다.

**HNSW (Hierarchical Navigable Small World)**
현재 대부분의 벡터 데이터베이스에서 기본 인덱스로 채택하고 있는 그래프 기반 ANN 알고리즘이다. 계층적 그래프 구조를 통해 검색 시 상위 레이어에서 Greedy하게 탐색 방향을 좁혀나가다가 최하위 레이어에서 정밀 탐색을 수행한다.

- 구조: 다층 그래프. 최상위 레이어는 노드가 적고 희소하며, 최하위 레이어는 모든 노드를 포함
- 검색: 최상위 엔트리 포인트에서 시작 → 각 레이어에서 그리디 탐색 → 레이어 하강 → 최하위 레이어에서 빔 서치(Beam Search)
- 장점: 높은 Recall, 밀리초 수준의 레이턴시
- 단점: 메모리 사용량이 많음 (그래프 구조를 메모리에 유지)
- 지원: Pinecone, Weaviate, Qdrant, Milvus, pgvector 등 대부분의 벡터 DB

**IVF (Inverted File Index)**
벡터 공간을 클러스터로 분할하고, 쿼리 벡터와 가장 가까운 클러스터(들)만 탐색한다. k-means 클러스터링으로 센트로이드를 생성하고, 각 벡터를 가장 가까운 센트로이드에 할당하는 방식이다.
- 장점: 메모리 효율적
- 단점: 쿼리와 관련 벡터가 다른 클러스터에 속할 경우 정확도 저하 가능

**PQ (Product Quantization)**
고차원 벡터를 여러 서브벡터로 분해하고 각 서브벡터를 양자화(Quantization)하여 압축 저장한다. 메모리 사용량을 대폭 줄일 수 있어 IVF와 결합하여 `IVF-PQ` 형태로 자주 사용된다.
- 장점: 높은 압축률, 대규모 데이터에 유리
- 단점: 압축 과정에서 정보 손실로 인한 정확도 저하

| 기법 | 메모리 | 정확도 | 적합 규모 |
|------|--------|--------|-----------|
| HNSW | 높음 | 높음 | 중~대규모 |
| IVF | 중간 | 중간 | 대규모 |
| IVF-PQ | 낮음 | 낮음~중간 | 초대규모 |

### 주요 벡터 데이터베이스 종류

**Pinecone**
완전 관리형(Fully Managed) 서버리스 벡터 데이터베이스이다. 인프라 관리 없이 빠르게 도입할 수 있으며, 멀티 리전 지원과 안정적인 성능이 강점이다. 수십억 개의 벡터 규모를 지원하며, 운영 부담을 최소화하려는 팀에 적합하다.
- 유형: 완전 관리형 클라우드 서비스
- 인덱스: HNSW 기반
- 특징: 서버리스, 메타데이터 필터링, 네임스페이스(멀티 테넌시)

**Milvus**
대규모 분산 처리에 강점을 가진 오픈소스 벡터 데이터베이스이다. 수십억 개 벡터 처리 경험이 풍부하며, Zilliz Cloud를 통해 관리형 서비스도 제공한다. 데이터 엔지니어링 역량이 있는 팀에서 전체 제어권을 원할 때 적합하다.
- 유형: 오픈소스 (관리형: Zilliz Cloud)
- 인덱스: HNSW, IVF, IVF-PQ 등 다양하게 지원
- 특징: 분산 아키텍처, 스트리밍 삽입 지원

**Weaviate**
모듈식 아키텍처와 내장 벡터화 기능이 특징인 오픈소스 벡터 데이터베이스이다. OpenAI, Cohere 등 임베딩 모델을 모듈로 연결해 데이터 삽입 시 자동으로 임베딩을 생성할 수 있다. 하이브리드 검색(키워드 + 벡터)을 기본 지원한다.
- 유형: 오픈소스 (관리형 옵션 제공)
- 특징: 내장 벡터화 모듈, 하이브리드 검색, GraphQL API

**Qdrant**
Rust로 구현된 고성능 오픈소스 벡터 데이터베이스이다. 메모리 효율이 뛰어나고 강력한 메타데이터 페이로드 필터링을 제공한다. 비용에 민감한 워크로드나 엣지 환경에 적합하다.
- 유형: 오픈소스 (관리형 옵션 제공)
- 특징: Rust 기반 고성능, 정밀한 페이로드 필터링, 양자화 지원

**Chroma**
개발자 친화적인 경량 오픈소스 벡터 데이터베이스이다. 로컬 환경에서 쉽게 실행할 수 있어 프로토타이핑과 소규모 앱에 적합하다. 수십억 개 벡터나 엔터프라이즈급 요구사항에는 적합하지 않다.
- 유형: 오픈소스
- 특징: 경량, 인메모리/영구 저장 모두 지원, Python 친화적

**pgvector**
PostgreSQL 확장으로, 기존 PostgreSQL 데이터베이스에 벡터 검색 기능을 추가한다. 이미 PostgreSQL을 사용 중인 경우 별도의 인프라 없이 벡터 검색을 통합할 수 있다. 현실적으로는 1000~1억 개 수준의 벡터까지 적합하며, 그 이상에서는 성능이 저하된다.
- 유형: PostgreSQL 확장 (오픈소스)
- 특징: 기존 RDBMS와 통합 용이, SQL 친화적, HNSW/IVFFlat 인덱스 지원

### 기존 DB의 벡터 검색 지원
전용 벡터 데이터베이스 외에도 기존 데이터베이스에서 벡터 검색 기능을 확장으로 지원하고 있다.

- **PostgreSQL + pgvector**: `CREATE EXTENSION vector;` 명령으로 활성화. HNSW 및 IVFFlat 인덱스를 지원하며, 복잡한 SQL 쿼리와 벡터 검색을 조합할 수 있다.
- **Elasticsearch kNN 검색**: `knn` 쿼리 파라미터로 벡터 유사도 검색 수행. 기존 BM25 전문 검색과 벡터 검색을 하이브리드로 사용 가능하다.
- **Redis Vector Search**: Redis Stack의 RediSearch 모듈을 통해 지원. 인메모리 특성으로 초저레이턴시가 필요한 경우 유리하다.

전용 벡터 DB vs 확장 기능 비교:
| 구분 | 전용 벡터 DB | 확장 기능 (pgvector 등) |
|------|-------------|------------------------|
| 최적화 | 벡터 검색에 특화 | 범용 DB에 추가 |
| 운영 복잡도 | 별도 인프라 필요 | 기존 인프라 재사용 |
| 대규모 확장성 | 높음 | 제한적 |
| 기존 데이터 통합 | 별도 동기화 필요 | 동일 DB에서 처리 가능 |

### 실전 활용 사례

**RAG (Retrieval-Augmented Generation) 파이프라인**
LLM의 한계(학습 데이터 컷오프, 환각)를 극복하기 위해 외부 문서를 실시간으로 검색해 컨텍스트로 제공하는 아키텍처이다. 2025년 기준 기업 AI 시스템의 51%가 RAG를 채택하고 있다.

```
사용자 질문 → 임베딩 변환 → 벡터 DB에서 유사 문서 검색
        → 검색된 문서를 컨텍스트로 LLM에 전달 → 응답 생성
```

**시맨틱 검색 엔진**
키워드가 아닌 의미 기반으로 검색하는 엔진이다. "강아지 사료"를 검색했을 때 "반려견 먹이"도 결과에 포함될 수 있다.

**추천 시스템**
사용자 행동을 임베딩으로 표현하고, 유사한 임베딩을 가진 아이템이나 사용자를 추천한다. 협업 필터링과 결합해 사용하기도 한다.

**이미지/오디오 유사도 검색**
CNN, CLIP 등의 모델로 생성한 이미지 임베딩을 저장하고, 유사한 이미지를 검색한다. 역방향 이미지 검색, 음악 유사도 검색 등에 활용된다.

**이상 탐지 (Anomaly Detection)**
정상 데이터의 임베딩 분포를 학습한 후, 분포에서 크게 벗어난 벡터를 이상치로 탐지한다. 금융 사기 탐지, 시스템 로그 이상 감지 등에 활용된다.

### 벡터 데이터베이스 선택 기준

| 상황 | 추천 |
|------|------|
| 빠른 도입, 운영 부담 최소화 | Pinecone |
| 수십억 개 벡터, 완전 제어 | Milvus |
| 하이브리드 검색 + 내장 벡터화 | Weaviate |
| 비용 효율 + 고성능 | Qdrant |
| 프로토타이핑, 소규모 앱 | Chroma |
| 기존 PostgreSQL 통합 | pgvector |

추가로 고려할 요소:
- **실시간 업데이트**: HNSW는 동적 삽입을 지원하지만, IVF 계열은 재인덱싱 비용이 크다.
- **메타데이터 필터링**: 벡터 유사도 검색 결과에 추가 조건(예: 날짜, 카테고리)을 적용해야 하는 경우 필터링 성능이 중요하다.
- **멀티 테넌시**: 여러 고객 데이터를 격리해야 하는 경우 네임스페이스 또는 컬렉션 분리를 지원하는지 확인한다.

## 핵심 정리
- 벡터 데이터베이스는 임베딩 기반의 고차원 벡터를 저장하고, ANN 알고리즘으로 빠른 유사도 검색을 제공하는 데이터베이스이다.
- HNSW는 현재 가장 널리 쓰이는 ANN 인덱스로, 높은 정확도와 밀리초 응답 속도를 제공하지만 메모리 사용량이 많다.
- 유사도 지표(코사인, 유클리드, 내적)는 임베딩 모델이 학습에 사용한 지표와 일치시켜야 한다.
- 도입 규모와 운영 역량에 따라 전용 벡터 DB(Pinecone, Milvus, Qdrant 등)와 확장 기능(pgvector)을 선택한다.
- RAG 파이프라인에서 벡터 데이터베이스는 LLM에 외부 지식을 실시간으로 제공하는 핵심 컴포넌트로 활용된다.

## 키워드

- `벡터 데이터베이스 (Vector Database)`: 고차원 벡터(임베딩)를 저장하고 유사도 기반 검색을 수행하는 데 특화된 데이터베이스. 비정형 데이터의 의미 기반 검색을 가능하게 한다.
- `임베딩 (Embedding)`: 텍스트, 이미지 등 비정형 데이터를 고차원 실수 벡터로 변환한 표현. 의미적으로 유사한 데이터는 벡터 공간에서 가깝게 위치한다.
- `유사도 검색 (Similarity Search)`: 쿼리 벡터와 저장된 벡터들 사이의 거리 또는 유사도를 계산하여 가장 유사한 데이터를 반환하는 검색 방식.
- `HNSW (Hierarchical Navigable Small World)`: 다층 그래프 구조 기반의 ANN 인덱싱 알고리즘. 상위 레이어에서 그리디 탐색으로 방향을 좁히고 최하위 레이어에서 정밀 탐색을 수행하여 높은 정확도와 빠른 속도를 동시에 달성한다.
- `코사인 유사도 (Cosine Similarity)`: 두 벡터 사이의 각도를 기반으로 유사도를 측정하는 지표. 벡터의 크기를 무시하고 방향만 비교하므로 텍스트 임베딩 비교에 많이 활용된다.
- `RAG (Retrieval-Augmented Generation)`: LLM이 외부 문서를 실시간으로 검색해 컨텍스트로 활용하는 아키텍처. 벡터 DB에서 관련 문서를 검색 후 LLM의 입력에 포함시켜 환각을 줄이고 최신 정보를 반영한다.
- `pgvector`: PostgreSQL 확장으로 벡터 저장 및 유사도 검색(HNSW, IVFFlat)을 지원한다. 기존 PostgreSQL 인프라를 유지하면서 벡터 검색을 통합할 수 있다.
- `Pinecone`: 완전 관리형 서버리스 벡터 데이터베이스. 인프라 없이 빠른 도입이 가능하며 수십억 벡터 규모를 지원한다.
- `Milvus`: 대규모 분산 처리를 위한 오픈소스 벡터 데이터베이스. 수십억 개 이상의 벡터 처리에 강점이 있으며 Zilliz Cloud를 통해 관리형 서비스도 제공한다.
- `ANN (Approximate Nearest Neighbor)`: 완전 탐색 대신 근사적 방법으로 가장 가까운 이웃을 찾는 알고리즘 군. 약간의 정확도 손실을 허용하는 대신 100~1000배의 검색 속도 향상을 얻는다.
