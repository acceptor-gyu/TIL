# 세션과 쿠키 방식의 차이 이해하기

## 개요

웹 애플리케이션에서 HTTP는 기본적으로 Stateless(무상태) 프로토콜입니다. 서버는 이전 요청을 기억하지 않기 때문에, 로그인 상태 유지 같은 사용자 상태를 관리하려면 별도의 메커니즘이 필요합니다. 이를 위해 사용하는 대표적인 수단이 쿠키(Cookie)와 세션(Session)입니다. 쿠키와 세션은 종종 혼용되지만, 정확히는 역할과 저장 위치가 다릅니다.

---

## 상세 내용

### 쿠키(Cookie)란

쿠키는 서버가 HTTP 응답 헤더(`Set-Cookie`)를 통해 클라이언트 브라우저에 저장을 요청하는 작은 데이터 조각입니다. 이후 브라우저는 해당 도메인에 요청을 보낼 때마다 자동으로 `Cookie` 헤더에 담아 전송합니다.

```
HTTP/1.1 200 OK
Set-Cookie: user_id=42; Path=/; HttpOnly; Secure; SameSite=Lax
```

#### 쿠키의 특성

| 속성 | 설명 |
|------|------|
| `Expires` / `Max-Age` | 쿠키 만료 시간. 없으면 브라우저 닫을 때 삭제(세션 쿠키) |
| `Domain` | 쿠키를 전송할 도메인 범위 |
| `Path` | 쿠키를 전송할 URL 경로 범위 |
| `HttpOnly` | JavaScript에서 `document.cookie`로 접근 불가 (XSS 방어) |
| `Secure` | HTTPS 연결에서만 전송 |
| `SameSite` | 크로스 사이트 요청 시 쿠키 전송 정책 (CSRF 방어) |

#### SameSite 옵션 상세

- **Strict**: 동일 사이트 요청에서만 쿠키 전송. 타 사이트 링크를 클릭해도 쿠키 미전송
- **Lax**: 최상위 탐색(링크 클릭, GET)에는 전송. POST 등 cross-site 요청은 미전송. 기본값
- **None**: 크로스 사이트 요청에도 전송. 반드시 `Secure`와 함께 사용해야 함

### 세션(Session)이란

세션은 서버 측에서 사용자의 상태 정보를 저장하는 방식입니다. 서버는 사용자별 고유한 Session ID를 발급하고, 이 ID만 쿠키에 담아 클라이언트로 전달합니다. 실제 사용자 데이터는 서버 메모리나 DB에 저장됩니다.

```
로그인 요청
    │
    ▼
서버: 세션 생성 → session_id = "abc123" 저장 (서버 메모리/DB)
    │
    ▼
응답: Set-Cookie: JSESSIONID=abc123; HttpOnly; Secure
    │
    ▼
이후 요청마다: Cookie: JSESSIONID=abc123 전송
    │
    ▼
서버: session_id로 저장된 사용자 정보 조회 → 인증 완료
```

---

### 쿠키 기반 인증 vs 세션 기반 인증 비교

| 항목 | 쿠키 기반 | 세션 기반 |
|------|-----------|-----------|
| 데이터 저장 위치 | 클라이언트(브라우저) | 서버(메모리/DB) |
| 클라이언트 전달 정보 | 실제 데이터 또는 토큰 | Session ID만 |
| 서버 부하 | 낮음 (서버 조회 불필요) | 높음 (매 요청마다 세션 조회) |
| 확장성 | 높음 (Stateless 가능) | 낮음 (서버 간 세션 공유 필요) |
| 보안 | 클라이언트 노출 위험 (암호화 필요) | 서버 관리로 상대적으로 안전 |
| 로그아웃 처리 | 클라이언트 쿠키 삭제 | 서버 세션 무효화로 즉시 차단 가능 |
| 네트워크 부하 | 쿠키 크기 증가 가능 | Session ID만 전송 (소량) |

---

### 세션 기반 인증의 흐름

```
1. 클라이언트 → POST /login (ID, PW)
2. 서버 → DB에서 사용자 확인
3. 서버 → 세션 생성 (session_store["abc123"] = {userId: 42, role: "USER"})
4. 서버 → Set-Cookie: SESSION=abc123; HttpOnly; Secure
5. 클라이언트 → 이후 요청마다 Cookie: SESSION=abc123 자동 전송
6. 서버 → session_store에서 "abc123" 조회 → 사용자 정보 확인
7. 클라이언트 → POST /logout
8. 서버 → session_store["abc123"] 삭제 → 즉시 무효화
```

### 다중 서버 환경에서의 세션 문제

세션 기반 인증의 가장 큰 단점은 **수평 확장(Scale-out)** 시 발생합니다.

```
클라이언트
    │
    ▼
로드밸런서
    ├── 서버 A (session: abc123 있음) → 인증 성공
    └── 서버 B (session: abc123 없음) → 인증 실패!
```

#### 해결 방법

1. **Sticky Session(고정 세션)**: 같은 클라이언트 요청을 항상 같은 서버로 라우팅. 서버 장애 시 세션 소실 위험
2. **세션 스토리지 공유**: Redis 같은 외부 세션 저장소를 사용하여 모든 서버가 공유

```
클라이언트
    │
    ▼
로드밸런서
    ├── 서버 A ─┐
    └── 서버 B ─┤── Redis (세션 공유 저장소) → 모든 서버에서 세션 조회 가능
```

3. **토큰 기반 인증(JWT)**: 상태를 서버가 아닌 토큰 자체에 담아 Stateless로 전환

### 쿠키 보안 위협과 대응

#### XSS (Cross-Site Scripting)
공격자가 악성 스크립트를 심어 `document.cookie`로 쿠키를 탈취하는 공격

**대응**: `HttpOnly` 속성으로 JavaScript 접근 차단

#### CSRF (Cross-Site Request Forgery)
사용자가 악성 사이트를 방문했을 때, 그 사이트가 사용자 대신 인증된 요청을 보내는 공격

**대응**:
- `SameSite=Lax` 또는 `Strict` 설정
- CSRF 토큰 사용
- `Referer` / `Origin` 헤더 검증

#### 세션 하이재킹(Session Hijacking)
Session ID를 탈취하여 정상 사용자인 척 요청을 보내는 공격

**대응**:
- `Secure` 속성으로 HTTPS에서만 전송
- Session ID 재발급 (로그인 성공 시)
- 세션 만료 시간 설정

---

### 세션 vs 쿠키 vs JWT 요약

```
쿠키: HTTP 상태를 유지하기 위한 클라이언트 저장 메커니즘 (수단)
세션: 사용자 상태를 서버에서 관리하는 방식 (서버 측 전략)
JWT:  사용자 상태를 서버 저장 없이 토큰에 자체 포함하는 방식 (Stateless 전략)

세션 방식도 Session ID를 클라이언트에 전달하기 위해 쿠키를 활용한다.
```

#### 언제 어떤 방식을 선택할까?

- **세션 기반**: 단일 서버, 즉각적인 세션 무효화가 필요한 경우, 보안이 중요한 금융 서비스
- **JWT 기반**: 마이크로서비스, 모바일 API, 크로스 도메인 인증이 필요한 경우
- **쿠키 단독**: 간단한 사용자 설정, 장바구니처럼 보안 민감도가 낮은 정보

## 키워드

| 키워드 | 설명 |
|--------|------|
| **Cookie** | 서버가 클라이언트 브라우저에 저장하는 작은 데이터 조각. HTTP 요청마다 자동으로 전송됨 |
| **Session** | 서버 측에서 사용자 상태(인증 정보 등)를 저장하는 메커니즘. 클라이언트에는 Session ID만 전달 |
| **Stateless** | 서버가 이전 요청 상태를 보존하지 않는 HTTP의 기본 특성. 세션/쿠키로 상태를 보완 |
| **HttpOnly** | JavaScript에서 쿠키 접근을 차단하는 속성. XSS 공격으로부터 쿠키 보호 |
| **SameSite** | 크로스 사이트 요청 시 쿠키 전송 여부를 제어하는 속성. CSRF 방어에 활용 |
| **Secure** | HTTPS 연결에서만 쿠키를 전송하는 속성. 네트워크 도청으로부터 보호 |
| **XSS (Cross-Site Scripting)** | 악성 스크립트를 삽입해 쿠키·세션을 탈취하는 공격. HttpOnly로 방어 |
| **CSRF (Cross-Site Request Forgery)** | 인증된 사용자 권한을 도용해 위조 요청을 보내는 공격. SameSite·CSRF 토큰으로 방어 |
| **Session Hijacking** | 세션 ID를 탈취해 정상 사용자로 위장하는 공격 |
| **Sticky Session** | 로드밸런서가 동일 클라이언트 요청을 항상 같은 서버로 라우팅하는 방식. 세션 공유 문제의 해결책 중 하나 |
