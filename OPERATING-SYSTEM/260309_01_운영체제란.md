# 운영체제란

## 개요
운영체제(Operating System)의 정의, 역할, 구조에 대해 학습한다. 컴퓨터 시스템에서 하드웨어와 소프트웨어 사이의 중재자로서 운영체제가 어떤 기능을 수행하는지 전반적으로 살펴본다.

## 상세 내용

### 1. 운영체제의 정의와 목적

#### 운영체제란 무엇인가

운영체제(OS, Operating System)는 컴퓨터 하드웨어와 사용자 사이에서 중재자 역할을 하는 시스템 소프트웨어이다. 운영체제는 사용자가 실행하는 응용 프로그램이 올바르게 동작하도록 지원하고, 필요한 하드웨어 자원(CPU, 메모리, 저장장치, 입출력 장치)을 효율적으로 할당·관리한다.

운영체제는 "항상 실행 중인 프로그램"이며, 일반적으로 **커널(Kernel)** 이라고 불린다. 커널을 제외한 나머지는 시스템 프로그램이거나 응용 프로그램으로 분류된다.

```
+------------------------------------------+
|          사용자 응용 프로그램             |
|  (웹 브라우저, 게임, 텍스트 에디터 등)   |
+------------------------------------------+
|          운영체제 (OS / 커널)             |
|  (자원 관리, 프로세스/메모리/IO 관리 등)  |
+------------------------------------------+
|          하드웨어                         |
|  (CPU, 메모리, 디스크, 네트워크 카드 등) |
+------------------------------------------+
```

#### 운영체제가 필요한 이유

운영체제가 없다면 각 응용 프로그램이 하드웨어에 직접 접근해야 한다. 이 경우 다음과 같은 문제가 발생한다.

- **자원 충돌**: 여러 프로그램이 동시에 같은 하드웨어 자원에 접근할 때 충돌이 발생한다.
- **보안 취약**: 악의적인 프로그램이 다른 프로그램의 메모리나 장치에 무단 접근할 수 있다.
- **복잡성 증가**: 개발자가 하드웨어 세부 사항을 모두 알아야 하므로 개발 난이도가 크게 높아진다.

운영체제는 응용 프로그램이 자원에 접근하려 할 때 반드시 운영체제를 통해서만 접근하도록 강제함으로써 자원을 보호하고 공평하게 분배한다.

#### 사용자 관점과 시스템 관점에서의 역할

| 관점 | 역할 |
|------|------|
| 사용자 관점 | 편리한 인터페이스 제공, 복잡한 하드웨어 세부 사항 추상화 |
| 시스템 관점 | 하드웨어 자원의 효율적 할당 및 관리, 자원 중재자 |

### 2. 운영체제의 처리 방식

운영체제는 작업을 처리하는 방식에 따라 여러 유형으로 분류된다. 각 방식은 서로 다른 목적과 환경에 최적화되어 있다.

#### 일괄 처리 시스템 (Batch Processing System)

사용자의 요청을 즉시 처리하지 않고, 일정량의 작업을 모아서 한꺼번에 순차적으로 처리하는 방식이다. 초기 컴퓨터 시스템에서 주로 사용되었다.

```
작업 큐:
[ Job A ] → [ Job B ] → [ Job C ] → ...
   ↓
CPU가 Job A 완료 후 → Job B 시작 → Job B 완료 후 → Job C 시작
```

- **장점**: CPU 유휴 시간을 줄여 처리량(Throughput)이 높다
- **단점**: 사용자와 상호작용이 불가능하며, 하나의 작업이 끝날 때까지 다음 작업은 대기해야 한다
- **예시**: 급여 계산, 은행 정산, 대규모 데이터 분석 등 정기적 대량 처리 작업

#### 시분할 시스템 (Time-Sharing System)

CPU 시간을 짧은 단위(타임 슬라이스)로 분할하여 여러 사용자 또는 프로세스에 번갈아 할당하는 방식이다. 각 사용자는 마치 자신이 컴퓨터를 독점적으로 사용하는 것처럼 느낀다.

```
타임 슬라이스: 10ms

시간 →  |  P1  |  P2  |  P3  |  P1  |  P2  |  P3  | ...
         0    10    20    30    40    50    60

→ 각 프로세스가 빠르게 교대하며 실행되어 동시에 실행되는 것처럼 보임
```

- **장점**: 빠른 응답 시간, 다중 사용자 지원
- **단점**: 컨텍스트 스위칭 오버헤드 발생, 타임 슬라이스 크기 설정이 성능에 영향
- **예시**: Unix, Linux 등 범용 운영체제

#### 실시간 시스템 (Real-Time System)

주어진 시간 안에 반드시 작업을 완료해야 하는 시스템이다. 처리 결과의 정확성뿐만 아니라 **마감 시한(Deadline)** 을 지키는 것이 핵심이다.

| 구분 | 경성 실시간 (Hard Real-Time) | 연성 실시간 (Soft Real-Time) |
|------|------------------------------|------------------------------|
| 마감 시한 | 절대적. 위반 시 시스템 실패로 간주 | 가능한 준수하되, 위반해도 성능 저하 수준 |
| 결과 | 마감 초과 시 결과가 무의미해짐 | 마감 초과 시 서비스 품질 저하 |
| 예시 | 항공기 제어, 의료 장비, 원자력 제어, ABS 브레이크 | 멀티미디어 스트리밍, 온라인 게임 |

```
경성 실시간 예시 (ABS 브레이크):
  센서 입력 → 제동 판단 → 브레이크 작동
  └── 전체 과정이 반드시 수 ms 이내에 완료되어야 함
      → 초과 시 사고 발생 가능

연성 실시간 예시 (영상 스트리밍):
  프레임 디코딩 → 화면 출력 (목표: 16.6ms / 60fps)
  └── 간헐적 초과 시 프레임 드랍 발생 (시스템 실패는 아님)
```

#### 분산 시스템 (Distributed System)

네트워크로 연결된 여러 대의 독립적인 컴퓨터가 하나의 시스템처럼 동작하는 방식이다. 각 노드는 자신의 운영체제를 가지고 있으며, 분산 운영체제 또는 미들웨어가 노드 간 협력을 조율한다.

```
+----------+     네트워크     +----------+     네트워크     +----------+
| Node A    |<------------> | Node B    |<------------>| Node C    |
| (OS + App)|               | (OS + App)|              | (OS + App)|
+----------+                +----------+                +----------+
         ↑                      ↑                            ↑
         └──────── 분산 미들웨어 / 분산 OS ──────────────────────┘
                   (하나의 시스템처럼 추상화)
```

- **장점**: 높은 가용성(한 노드 장애 시에도 시스템 유지), 확장성, 자원 공유
- **단점**: 네트워크 지연, 부분 장애 처리의 복잡성, 데이터 일관성 유지 어려움
- **예시**: Hadoop, Kubernetes, 클라우드 컴퓨팅 환경

#### 처리 방식 비교

| 방식 | 목적 | 응답 시간 | 사용자 상호작용 |
|------|------|-----------|-----------------|
| 일괄 처리 | 대량 작업의 효율적 처리 | 느림 (분~시간) | 없음 |
| 시분할 | 다중 사용자의 동시 사용 | 빠름 (ms~초) | 있음 |
| 실시간 | 마감 시한 내 정확한 처리 | 매우 빠름 (us~ms) | 제한적 |
| 분산 | 자원 공유 및 고가용성 | 네트워크 의존 | 있음 |

### 3. 운영체제의 구조

#### 커널(Kernel)의 개념과 역할

커널은 운영체제의 핵심 구성 요소로, 하드웨어와 소프트웨어 사이에서 직접 상호작용한다. 커널이 수행하는 주요 역할은 다음과 같다.

- **프로세스 관리**: 프로세스 생성, 스케줄링, 종료
- **메모리 관리**: 메모리 할당 및 회수, 가상 메모리 관리
- **파일 시스템 관리**: 파일 읽기/쓰기, 디렉토리 관리
- **장치 드라이버 관리**: 하드웨어 장치와의 통신
- **시스템 콜 처리**: 사용자 프로그램의 커널 서비스 요청 처리

#### 시스템 콜(System Call)

시스템 콜은 사용자 프로그램이 운영체제의 커널 서비스를 요청하는 인터페이스이다. 사용자 모드에서 직접 하드웨어에 접근할 수 없으므로, 반드시 시스템 콜을 통해 커널에 요청해야 한다.

```
사용자 프로그램               커널
      |                    |
      |  시스템 콜 요청       |
      |---read(fd, buf)--->|
      |                    |  커널 모드에서 실행
      |                    |  (하드웨어 I/O 수행)
      |   결과 반환          |
      |<-------------------|
      |  사용자 모드 복귀      |
```

대표적인 시스템 콜 유형은 다음과 같다.

| 유형 | 예시 |
|------|------|
| 프로세스 제어 | `fork()`, `exec()`, `exit()`, `wait()` |
| 파일 관리 | `open()`, `read()`, `write()`, `close()` |
| 장치 관리 | `ioctl()`, `read()`, `write()` |
| 정보 유지 | `getpid()`, `alarm()`, `sleep()` |
| 통신 | `pipe()`, `socket()`, `send()`, `recv()` |

#### 사용자 모드(User Mode)와 커널 모드(Kernel Mode)

이중 모드(Dual Mode)란 CPU가 명령어를 실행하는 모드를 사용자 모드와 커널 모드로 구분하는 방식이다. 이를 통해 운영체제는 사용자 프로그램이 하드웨어에 직접 접근하거나 중요한 커널 데이터를 손상시키는 것을 방지한다.

```
[ 사용자 모드 ]              [ 커널 모드 ]
- 제한된 명령어만 실행       - 모든 명령어 실행 가능
- 하드웨어 직접 접근 불가    - 하드웨어 직접 접근 가능
- 응용 프로그램 실행 공간    - 커널 코드 실행 공간

  사용자 모드 <-- 시스템 콜 인터럽트 --> 커널 모드
```

CPU는 모드 비트(Mode Bit)를 통해 현재 모드를 구분한다. `0` 이면 커널 모드, `1` 이면 사용자 모드이다.

#### 모놀리식 커널 vs 마이크로 커널

| 구분 | 모놀리식 커널 | 마이크로 커널 |
|------|--------------|--------------|
| 구조 | 운영체제 서비스 전부가 커널 공간에서 실행 | 핵심 기능만 커널, 나머지는 사용자 공간 |
| 성능 | 서비스 간 통신 오버헤드 없어 빠름 | IPC 오버헤드로 인해 상대적으로 느릴 수 있음 |
| 안정성 | 커널 내 버그가 전체 시스템에 영향 | 서비스 격리로 안정성 높음 |
| 예시 | Linux, 전통적인 Unix | Mach, L4, MINIX 3 |

### 4. 프로세스 관리

#### 프로세스의 생명주기(Life Cycle)와 상태 전이

프로세스(Process)는 실행 중인 프로그램의 인스턴스이다. 프로그램은 디스크에 저장된 정적인 파일이지만, 프로세스는 메모리에 올라가 실행 중인 동적인 개체이다.

프로세스는 다음 5가지 상태를 가진다.

```
                  스케줄러 선택
       생성 --------> 준비(Ready) ---------> 실행(Running)
       (New)           ^    ^                 /    |
                       |    |    인터럽트      /     |
                       |     \<------------/      |
                       |                          | I/O 또는
                       |      I/O 완료             | 이벤트 대기
                       |          |               |
                        \         v               v
                         \     대기(Waiting)    종료(Terminated)
```

| 상태 | 설명 |
|------|------|
| New (생성) | 프로세스가 생성 중인 상태 |
| Ready (준비) | CPU 할당을 기다리는 상태 |
| Running (실행) | CPU를 점유하여 명령어를 실행 중인 상태 |
| Waiting (대기) | I/O 완료 등 특정 이벤트를 기다리는 상태 |
| Terminated (종료) | 프로세스 실행이 완료된 상태 |

#### PCB(Process Control Block)

PCB는 운영체제가 각 프로세스를 관리하기 위해 유지하는 데이터 구조이다. 프로세스에 관한 모든 정보를 담고 있으며, 컨텍스트 스위칭 시 현재 상태를 저장하고 복원하는 데 사용된다.

```
+---------------------------+
|    Process Control Block  |
+---------------------------+
| 프로세스 ID (PID)          |
| 프로세스 상태              |
| 프로그램 카운터 (PC)       |
| CPU 레지스터 정보          |
| CPU 스케줄링 정보          |
| 메모리 관리 정보           |
| 입출력 상태 정보           |
| 계정 정보 (CPU 사용 시간)  |
+---------------------------+
```

#### 컨텍스트 스위칭(Context Switching)

컨텍스트 스위칭은 CPU가 현재 실행 중인 프로세스를 멈추고 다른 프로세스로 전환하는 작업이다. 멀티태스킹 환경에서 여러 프로세스가 CPU를 나눠 쓰기 위한 핵심 메커니즘이다.

```
프로세스 A 실행          프로세스 B 실행
     |                        |
     |         인터럽트/타임슬라이스 만료
     v                        ^
  [PCB_A에 상태 저장]            |
     |                        |
     v                        |
  [PCB_B에서 상태 복원] ------+
```

컨텍스트 스위칭 자체는 순수 오버헤드(overhead)이다. 스위칭이 일어나는 동안 CPU는 유용한 작업을 수행하지 못한다. 따라서 컨텍스트 스위칭 횟수와 소요 시간을 최소화하는 것이 성능에 중요하다.

컨텍스트 스위칭을 유발하는 주요 원인은 다음과 같다.

- 타임 슬라이스(시간 할당량) 만료
- I/O 요청으로 인한 대기 상태 전환
- 더 높은 우선순위 프로세스의 등장
- 하드웨어 인터럽트 발생

### 5. CPU 스케줄링

#### 스케줄링의 목적과 기준

CPU 스케줄링은 준비 큐(Ready Queue)에 있는 프로세스 중 어떤 프로세스에 CPU를 할당할지 결정하는 것이다. 좋은 스케줄링 알고리즘은 다음 기준을 고려한다.

| 기준 | 설명 |
|------|------|
| CPU 이용률 | CPU를 최대한 바쁘게 유지 |
| 처리량(Throughput) | 단위 시간당 완료된 프로세스 수 최대화 |
| 반환 시간(Turnaround Time) | 프로세스 제출부터 완료까지의 시간 최소화 |
| 대기 시간(Waiting Time) | 준비 큐에서 대기하는 시간 최소화 |
| 응답 시간(Response Time) | 요청 후 첫 응답까지의 시간 최소화 |

#### 선점형(Preemptive) vs 비선점형(Non-preemptive) 스케줄링

| 구분 | 선점형 | 비선점형 |
|------|--------|----------|
| 개념 | 실행 중인 프로세스를 강제로 CPU에서 빼앗을 수 있음 | 프로세스가 자발적으로 CPU를 반납할 때까지 계속 실행 |
| 장점 | 응답성 높음, 실시간 시스템에 적합 | 구현 단순, 컨텍스트 스위칭 오버헤드 낮음 |
| 단점 | 컨텍스트 스위칭 오버헤드 | 짧은 작업이 긴 작업에 밀려 대기(Convoy Effect) |
| 예시 | Round Robin, SRTF | FCFS, SJF (비선점) |

#### 대표 스케줄링 알고리즘

**FCFS (First-Come, First-Served)**

도착 순서대로 CPU를 할당하는 가장 단순한 알고리즘이다.

```
도착 순서: P1(24ms) -> P2(3ms) -> P3(3ms)

간트 차트:
| P1                    | P2  | P3  |
0                      24   27   30

P1 대기시간: 0ms
P2 대기시간: 24ms
P3 대기시간: 27ms
평균 대기시간: (0 + 24 + 27) / 3 = 17ms
```

단점: 긴 프로세스 뒤에 짧은 프로세스가 오래 기다리는 **호위 효과(Convoy Effect)** 가 발생한다.

**SJF (Shortest Job First)**

버스트 시간이 가장 짧은 프로세스에 먼저 CPU를 할당한다. 평균 대기 시간을 최소화하는 최적 알고리즘이지만, 다음 CPU 버스트 시간을 미리 알 수 없다는 한계가 있다.

```
도착 순서: P1(6ms), P2(8ms), P3(7ms), P4(3ms) (모두 동시 도착)

간트 차트:
| P4  | P1     | P3       | P2           |
0    3        9         16             24

평균 대기시간: (3 + 16 + 9 + 0) / 4 = 7ms (FCFS보다 훨씬 효율적)
```

**Round Robin (RR)**

각 프로세스에 동일한 시간 할당량(Time Quantum, 타임 슬라이스)을 순환 방식으로 할당하는 선점형 알고리즘이다. 시분할 시스템에 가장 적합하다.

```
타임 퀀텀: 4ms
프로세스: P1(24ms), P2(3ms), P3(3ms)

간트 차트:
| P1 | P2 | P3 | P1 | P1 | P1 | P1 | P1 |
0   4   7  10  14  18  22  26  30

- 타임 퀀텀이 너무 크면 → FCFS와 동일해짐
- 타임 퀀텀이 너무 작으면 → 컨텍스트 스위칭 오버헤드 증가
```

**Priority Scheduling (우선순위 스케줄링)**

각 프로세스에 우선순위를 부여하고, 우선순위가 높은 프로세스에 먼저 CPU를 할당한다. 낮은 우선순위 프로세스가 무한정 대기하는 **기아(Starvation)** 문제가 발생할 수 있으며, 이를 해결하기 위해 **노화(Aging)** 기법을 사용한다.

### 6. 메모리 관리

#### 물리 메모리와 가상 메모리

물리 메모리(Physical Memory)는 실제 RAM이다. 가상 메모리(Virtual Memory)는 실제 물리 메모리보다 더 많은 메모리를 사용할 수 있도록 하드 디스크의 일부를 메모리처럼 사용하는 기술이다.

가상 메모리의 핵심 개념은 **요구 페이징(Demand Paging)** 이다. 프로세스의 모든 내용을 메모리에 올리지 않고, 실제로 필요한 페이지만 메모리에 올린다. 필요한 페이지가 메모리에 없을 때 **페이지 폴트(Page Fault)** 가 발생하고, 운영체제가 해당 페이지를 디스크에서 메모리로 가져온다.

```
가상 주소 공간 (각 프로세스)          물리 메모리
+------------------+              +------------------+
| 페이지 0 (코드)  |----매핑------>| 프레임 3         |
| 페이지 1 (데이터) |----매핑------>| 프레임 7         |
| 페이지 2 (스택)  |---(디스크에)   | 프레임 1         |
| 페이지 3 (힙)   |----매핑------>| ...              |
+------------------+              +------------------+
```

#### 페이징(Paging)과 세그멘테이션(Segmentation)

**페이징(Paging)**

물리 메모리를 고정된 크기의 **프레임(Frame)** 으로, 논리 메모리를 같은 크기의 **페이지(Page)** 로 나눈다. 페이지 테이블을 통해 논리 주소를 물리 주소로 변환한다.

```
논리 주소: [페이지 번호(p) | 페이지 오프셋(d)]
                |
          페이지 테이블 참조
                |
물리 주소: [프레임 번호(f) | 페이지 오프셋(d)]
```

- 장점: 외부 단편화(External Fragmentation) 없음
- 단점: 내부 단편화(Internal Fragmentation) 발생 가능, 페이지 테이블 유지 비용

**세그멘테이션(Segmentation)**

프로그램을 논리적 단위(세그먼트: 코드, 데이터, 스택, 힙 등)로 나눈다. 각 세그먼트는 가변 크기를 가진다.

| 구분 | 페이징 | 세그멘테이션 |
|------|--------|-------------|
| 분할 단위 | 고정 크기 | 가변 크기 (논리 단위) |
| 단편화 | 내부 단편화 | 외부 단편화 |
| 사용자 가시성 | 사용자에게 투명 | 사용자가 세그먼트 인식 |

#### 페이지 교체 알고리즘

물리 메모리가 가득 찼을 때 어떤 페이지를 디스크로 내보낼지 결정하는 알고리즘이다.

**FIFO (First-In, First-Out)**

메모리에 가장 먼저 올라온 페이지를 교체한다. 구현이 단순하지만, 자주 사용되는 오래된 페이지도 교체될 수 있다. 또한 프레임 수를 늘렸는데 페이지 폴트가 오히려 증가하는 **벨라디의 모순(Belady's Anomaly)** 이 발생할 수 있다.

**LRU (Least Recently Used)**

가장 오랫동안 사용되지 않은 페이지를 교체한다. 최근에 사용된 페이지는 앞으로도 사용될 가능성이 높다는 지역성(Locality) 원리에 기반한다. 실제 운영체제(Linux 등)에서 변형된 형태로 널리 사용된다.

**LFU (Least Frequently Used)**

참조 횟수가 가장 적은 페이지를 교체한다. 최근에 올라온 페이지가 참조 횟수가 낮아 불필요하게 교체될 수 있다는 단점이 있다.

```
알고리즘 비교 (페이지 폴트 수 기준):
OPT (최적) < LRU ≈ LFU < FIFO
```

### 7. 동기화와 교착 상태

#### 임계 영역(Critical Section)

임계 영역이란 공유 자원에 접근하는 코드 구역이다. 여러 프로세스/스레드가 동시에 임계 영역에 진입하면 **경쟁 조건(Race Condition)** 이 발생하여 데이터 불일치가 생긴다.

임계 영역 문제를 해결하려면 다음 세 가지 조건을 충족해야 한다.

| 조건 | 설명 |
|------|------|
| 상호 배제(Mutual Exclusion) | 한 번에 하나의 프로세스만 임계 영역에 진입 |
| 진행(Progress) | 임계 영역 외부의 프로세스가 진입 결정에 관여하지 않음 |
| 한정 대기(Bounded Waiting) | 임계 영역 진입 요청 후 무한정 기다리지 않음 |

#### 뮤텍스(Mutex)와 세마포어(Semaphore)

**뮤텍스(Mutex)**

Mutual Exclusion의 약자로, 한 번에 하나의 스레드만 공유 자원에 접근할 수 있도록 하는 잠금(Lock) 메커니즘이다. 뮤텍스를 잠근 스레드만 잠금을 해제할 수 있다는 소유권(Ownership) 개념이 있다.

```
// 의사 코드(Pseudocode)
mutex.lock()         // 잠금 획득 (다른 스레드는 대기)
  임계 영역 실행
mutex.unlock()       // 잠금 해제
```

**세마포어(Semaphore)**

정수형 변수로 공유 자원의 개수를 나타내며, `wait(P)` 와 `signal(V)` 두 가지 원자적 연산으로 제어한다.

- **이진 세마포어(Binary Semaphore)**: 값이 0 또는 1, 뮤텍스와 유사
- **카운팅 세마포어(Counting Semaphore)**: 값이 0 이상, 여러 개의 자원 관리에 사용

```
// 의사 코드
wait(S):
    while S <= 0 do nothing  // 자원이 없으면 대기
    S = S - 1                 // 자원 획득

signal(S):
    S = S + 1                 // 자원 반납
```

| 구분 | 뮤텍스 | 세마포어 |
|------|--------|----------|
| 소유권 | 잠근 스레드만 해제 가능 | 소유권 개념 없음 |
| 값 범위 | 0 또는 1 (이진) | 0 이상의 정수 |
| 용도 | 단일 자원 상호 배제 | 다중 자원 접근 제어, 동기화 신호 |

#### 교착 상태(Deadlock)

교착 상태란 두 개 이상의 프로세스가 서로 상대방이 보유한 자원을 기다리며 영구적으로 대기하는 상태이다.

**교착 상태 발생 4가지 필요 조건** (Coffman 조건, 모두 동시에 충족되어야 함)

| 조건 | 설명 |
|------|------|
| 상호 배제(Mutual Exclusion) | 자원은 한 번에 하나의 프로세스만 사용 가능 |
| 점유와 대기(Hold & Wait) | 자원을 보유한 채 다른 자원을 기다림 |
| 비선점(No Preemption) | 다른 프로세스의 자원을 강제로 빼앗을 수 없음 |
| 순환 대기(Circular Wait) | 프로세스들이 순환 형태로 서로의 자원을 대기 |

```
교착 상태 예시:
P1: 자원 A 보유 → 자원 B 요청
P2: 자원 B 보유 → 자원 A 요청
→ P1과 P2가 서로를 기다리며 무한 대기
```

**교착 상태 해결 전략**

| 전략 | 설명 |
|------|------|
| 예방(Prevention) | 4가지 조건 중 하나 이상을 원천적으로 제거 |
| 회피(Avoidance) | 자원 할당 시 안전 상태인지 확인 (은행원 알고리즘) |
| 탐지 및 회복(Detection & Recovery) | 교착 상태 발생 허용 후, 탐지하여 프로세스 종료/자원 선점으로 회복 |
| 무시(Ignorance) | 교착 상태 발생 빈도가 낮다면 무시 (대부분의 범용 OS 채택) |

### 8. 파일 시스템

#### 파일 시스템의 역할

파일 시스템은 운영체제가 디스크와 같은 저장 장치에 데이터를 저장하고 검색하기 위해 사용하는 방법 및 자료 구조의 집합이다. 파일 시스템은 다음 기능을 제공한다.

- 파일 생성, 읽기, 쓰기, 삭제
- 디렉토리(폴더) 구조 관리
- 파일 접근 권한 및 보안 관리
- 디스크 공간 할당 및 회수

대표적인 파일 시스템: ext4 (Linux), NTFS (Windows), APFS (macOS)

#### 디렉토리 구조와 inode

**inode (Index Node)**

유닉스 계열 파일 시스템에서 파일이나 디렉토리에 대한 메타데이터를 저장하는 데이터 구조이다. 파일명은 inode에 저장되지 않으며, 디렉토리가 파일명과 inode 번호를 매핑한다.

```
inode 구조:
+---------------------------+
|  파일 유형 및 권한        |
|  소유자 UID, GID          |
|  파일 크기                |
|  생성/수정/접근 시간      |
|  하드 링크 수             |
|  데이터 블록 포인터       |
|  (직접, 간접, 이중 간접)  |
+---------------------------+

디렉토리 구조:
+------------------+--------+
| 파일명           | inode# |
+------------------+--------+
| "hello.txt"      |  1234  |
| "documents"      |  5678  |
+------------------+--------+
```

**디렉토리 구조 유형**

| 유형 | 설명 |
|------|------|
| 1단계 (단일) | 모든 파일이 하나의 디렉토리에 존재 |
| 2단계 | 사용자별 디렉토리 존재 |
| 트리 구조 | 현재 대부분의 OS에서 사용, 계층적 구조 |
| 비순환 그래프 | 파일/디렉토리 공유 가능 (하드 링크, 심볼릭 링크) |

#### 디스크 스케줄링

디스크 스케줄링은 디스크 I/O 요청의 처리 순서를 결정하여 헤드의 이동 시간을 최소화하는 것이다.

| 알고리즘 | 방식 | 특징 |
|----------|------|------|
| FCFS | 요청 순서대로 처리 | 공평하지만 비효율적 |
| SSTF | 현재 헤드 위치에서 가장 가까운 요청 처리 | 처리량 높지만 기아 가능 |
| SCAN (엘리베이터) | 헤드가 한 방향으로 이동하며 처리, 끝에서 방향 전환 | 균등한 대기 시간 |
| C-SCAN | SCAN과 유사하나 한쪽 방향으로만 처리 후 처음으로 복귀 | 균일한 대기 시간 보장 |

### 9. 입출력(I/O) 관리

#### I/O 처리 방식

**폴링(Polling)**

CPU가 주기적으로 I/O 장치의 상태를 직접 확인하는 방식이다. 구현은 단순하지만, CPU가 계속 상태를 확인하는 데 시간을 낭비하는 **바쁜 대기(Busy Waiting)** 문제가 있다.

```
while (device.status != DONE) {  // CPU가 반복 확인
    // 아무것도 하지 않음 (바쁜 대기)
}
process(data);
```

**인터럽트(Interrupt)**

I/O 장치가 작업 완료 시 CPU에 인터럽트 신호를 보내는 방식이다. CPU는 인터럽트가 올 때까지 다른 작업을 수행할 수 있어 효율적이다.

```
CPU: 다른 프로세스 실행 중
   ...
I/O 장치: "작업 완료!" → 인터럽트 신호 발생
CPU: 현재 작업 중단 → 인터럽트 처리 루틴 실행 → 인터럽트 처리 완료 → 이전 작업 재개
```

**DMA (Direct Memory Access)**

CPU의 개입 없이 I/O 장치가 메모리에 직접 데이터를 전송하는 방식이다. 대용량 데이터 전송에 매우 효율적이다.

```
[전통적인 방식]
장치 → CPU → 메모리   (CPU가 모든 전송을 중재)

[DMA 방식]
장치 → DMA 컨트롤러 → 메모리   (CPU는 완료 시에만 인터럽트)
CPU는 전송 동안 다른 작업 수행 가능
```

| 방식 | CPU 개입 | 적합한 상황 |
|------|----------|-------------|
| 폴링 | 높음 (바쁜 대기) | 간단한 저속 장치 |
| 인터럽트 | 낮음 | 일반적인 I/O |
| DMA | 매우 낮음 | 대용량 데이터 전송 (디스크, 네트워크) |

#### 버퍼링(Buffering)과 캐싱(Caching)

**버퍼링(Buffering)**

데이터를 한 장치에서 다른 장치로 전송하는 동안 임시로 데이터를 저장하는 메모리 영역(버퍼)을 사용하는 기법이다. 생산자와 소비자 간의 속도 차이를 완화한다.

- 예: 네트워크에서 수신한 데이터를 커널 버퍼에 임시 저장 후 애플리케이션에 전달

**캐싱(Caching)**

자주 접근하는 데이터를 더 빠른 저장 장치에 복사해두는 기법이다. 캐시에 데이터가 있으면 느린 원본 장치에 접근하지 않아도 된다.

```
속도 (빠름 → 느림):
CPU 레지스터 → L1/L2/L3 캐시 → 메인 메모리(RAM) → SSD → HDD
```

| 구분 | 버퍼링 | 캐싱 |
|------|--------|------|
| 목적 | 속도 차이 완화 | 접근 시간 단축 |
| 데이터 | 새로운 데이터의 임시 저장 | 자주 쓰는 데이터의 복사본 저장 |
| 위치 | 주로 메인 메모리 | 더 빠른 저장 장치 |

## 핵심 정리
- 운영체제는 하드웨어 자원을 효율적으로 관리하고 사용자에게 편리한 인터페이스를 제공하는 시스템 소프트웨어이다
- 커널은 운영체제의 핵심으로, 시스템 콜을 통해 사용자 프로그램과 하드웨어를 중재한다
- 프로세스는 실행 중인 프로그램의 인스턴스이며, PCB에 의해 관리되고 컨텍스트 스위칭으로 CPU를 공유한다
- CPU 스케줄링 알고리즘(FCFS, SJF, RR, Priority)은 각각 장단점이 있으며, 시스템 목적에 맞는 알고리즘을 선택해야 한다
- 가상 메모리는 요구 페이징 기법으로 물리 메모리보다 더 많은 메모리를 사용할 수 있게 한다
- 교착 상태는 상호 배제, 점유와 대기, 비선점, 순환 대기 4가지 조건이 동시에 충족될 때 발생한다
- 파일 시스템은 inode를 통해 파일 메타데이터를 관리하며, 트리 구조 디렉토리로 파일을 조직한다
- I/O 관리는 폴링, 인터럽트, DMA 방식으로 처리하며, 버퍼링과 캐싱으로 성능을 최적화한다

## 키워드

- `일괄 처리(Batch Processing)`: 작업을 모아서 한꺼번에 순차 처리하는 방식. 사용자 상호작용 없이 대량 작업을 효율적으로 처리
- `시분할(Time-Sharing)`: CPU 시간을 짧은 타임 슬라이스로 분할하여 여러 프로세스에 번갈아 할당하는 방식. 다중 사용자 환경에 적합
- `실시간 시스템(Real-Time System)`: 정해진 마감 시한(Deadline) 내에 반드시 작업을 완료해야 하는 시스템. 경성/연성으로 구분
- `운영체제(OS)`: 하드웨어와 응용 프로그램 사이의 중재자 역할을 하는 시스템 소프트웨어
- `커널(Kernel)`: 운영체제의 핵심 구성 요소로 자원 관리, 프로세스 제어, 시스템 콜 처리를 담당
- `시스템 콜(System Call)`: 사용자 프로그램이 커널 서비스를 요청하는 인터페이스 (사용자 모드 → 커널 모드 전환)
- `프로세스(Process)`: 실행 중인 프로그램의 인스턴스. 코드, 데이터, 스택, 힙으로 구성
- `컨텍스트 스위칭(Context Switching)`: CPU가 한 프로세스에서 다른 프로세스로 전환할 때 현재 상태를 PCB에 저장하고 새 프로세스 상태를 복원하는 작업
- `CPU 스케줄링`: Ready Queue의 프로세스 중 CPU를 할당할 프로세스를 결정하는 운영체제의 작업
- `가상 메모리(Virtual Memory)`: 물리 메모리보다 큰 주소 공간을 제공하는 기술로, 필요한 페이지만 물리 메모리에 올리는 요구 페이징 기법 사용
- `페이징(Paging)`: 논리 메모리와 물리 메모리를 고정 크기 단위(페이지/프레임)로 분할하여 관리하는 메모리 관리 기법
- `교착 상태(Deadlock)`: 프로세스들이 서로의 자원을 기다리며 영구적으로 대기하는 상태. 상호 배제, 점유와 대기, 비선점, 순환 대기 4조건 동시 충족 시 발생
- `인터럽트(Interrupt)`: 하드웨어나 소프트웨어가 CPU에 긴급 처리를 요청하는 신호. I/O 완료, 오류, 타이머 만료 등의 상황에서 발생
