# Datadog 활용하기

## 개요
Datadog은 클라우드 기반 모니터링 및 분석 플랫폼으로, 인프라 모니터링, APM, 로그 관리, 알림 등을 통합적으로 제공한다. 서비스 운영에서 Datadog을 효과적으로 활용하는 방법을 학습한다.

## 상세 내용

### 1. Datadog이란

Datadog의 핵심 철학은 **Observability(관측 가능성)** 이다. 시스템의 내부 상태를 외부 출력(Metrics, Logs, Traces)만으로 파악할 수 있는 능력을 의미한다.

**SaaS 기반 모니터링 플랫폼의 장점**
- 별도 인프라 운영 없이 즉시 사용 가능
- 수백 개의 서비스 통합(Integration) 기본 제공
- 데이터 저장, 쿼리 엔진, 시각화, 알림을 하나의 플랫폼에서 제공

**Datadog vs 다른 솔루션 비교**

| 항목 | Datadog | Prometheus + Grafana | ELK Stack |
|---|---|---|---|
| 형태 | SaaS (유료) | 오픈소스 (자체 운영) | 오픈소스 (자체 운영) |
| 강점 | 올인원, 빠른 셋업 | 메트릭 중심, Kubernetes 친화적 | 로그 중심, 강력한 검색 |
| APM | 기본 제공 | 별도 구성 필요 (Tempo 등) | 별도 구성 필요 |
| 비용 | 사용량 기반 과금 | 인프라 비용만 | 인프라 비용만 |
| 통합 관리 | 단일 플랫폼 | 여러 도구 조합 필요 | 여러 도구 조합 필요 |

---

### 2. Datadog의 3가지 핵심 기둥

Observability의 세 가지 핵심 요소(Three Pillars)를 Datadog은 단일 플랫폼에서 모두 지원한다.

**Metrics (지표)**
- 시간에 따른 수치 데이터 (CPU 사용률, 응답 시간, 요청 수 등)
- 시스템의 현재 상태와 추세를 파악하는 데 사용

**Traces (분산 추적, APM)**
- 하나의 요청이 여러 서비스를 거치는 전체 흐름을 추적
- 병목 지점과 오류 발생 위치를 정확히 식별

**Logs (로그)**
- 서비스에서 발생한 이벤트의 상세 기록
- 메트릭과 트레이스로 발견한 문제의 원인을 구체적으로 파악

### 3. Datadog Agent 설정

**Agent 아키텍처와 동작 원리**

Datadog Agent는 호스트에서 실행되는 경량 소프트웨어로, 다음 세 가지 주요 컴포넌트로 구성된다.

- **Collector**: 시스템 메트릭과 통합 데이터를 수집
- **DogStatsD**: UDP(기본 8125 포트)를 통해 커스텀 메트릭을 수신하여 집계
- **Forwarder**: 수집된 데이터를 SSL로 암호화하여 Datadog 서버로 전송

**Docker 환경에서의 Agent 배포**

```yaml
# docker-compose.yml
services:
  datadog-agent:
    image: gcr.io/datadoghq/agent:latest
    environment:
      - DD_API_KEY=${DD_API_KEY}
      - DD_SITE=datadoghq.com
      - DD_APM_ENABLED=true
      - DD_LOGS_ENABLED=true
      - DD_LOGS_CONFIG_CONTAINER_COLLECT_ALL=true
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock:ro
      - /proc/:/host/proc/:ro
      - /sys/fs/cgroup/:/host/sys/fs/cgroup:ro
    ports:
      - "8125:8125/udp"   # DogStatsD
      - "8126:8126"       # APM Trace
```

**Spring Boot 애플리케이션과 연동 (dd-java-agent)**

dd-java-agent는 바이트코드 조작을 통해 코드 수정 없이 자동으로 트레이스를 수집한다.

```bash
# 1. Java Agent 다운로드
wget -O dd-java-agent.jar 'https://dtdg.co/latest-java-tracer'
```

```bash
# 2. JVM 옵션으로 실행 (-javaagent는 반드시 -jar 앞에 위치)
java \
  -javaagent:/path/to/dd-java-agent.jar \
  -Ddd.service=my-spring-app \
  -Ddd.env=production \
  -Ddd.version=1.0.0 \
  -Ddd.logs.injection=true \
  -Ddd.profiling.enabled=true \
  -Ddd.trace.sample.rate=1.0 \
  -jar my-spring-app.jar
```

```yaml
# application.yml - Spring Boot 환경 변수 설정
spring:
  application:
    name: my-spring-app

# Datadog 관련 환경 변수는 JVM 옵션 또는 환경 변수로 주입
# DD_SERVICE, DD_ENV, DD_VERSION, DD_AGENT_HOST
```

> **중요**: dd-java-agent.jar는 classpath에 추가하면 안 된다. 반드시 `-javaagent` JVM 옵션으로만 사용해야 한다.

### 4. APM (Application Performance Monitoring)

**분산 추적(Distributed Tracing)의 개념**

마이크로서비스 환경에서 하나의 사용자 요청은 여러 서비스를 순차적으로 또는 병렬로 거친다. 분산 추적은 이 요청의 전체 여정을 단일 흐름으로 시각화한다.

**Trace, Span, Service의 관계**

```
Trace (하나의 요청 전체)
├── Span: API Gateway (10ms)
│   ├── Span: User Service (30ms)
│   │   └── Span: DB Query - SELECT (5ms)
│   └── Span: Order Service (50ms)
│       ├── Span: DB Query - INSERT (8ms)
│       └── Span: Payment Service gRPC (35ms)
```

- **Trace**: 하나의 요청이 시스템을 통과하는 전체 경로. 고유한 Trace ID로 식별
- **Span**: Trace 내의 개별 작업 단위 (서비스 호출, DB 쿼리, HTTP 요청 등). 시작 시간과 소요 시간을 포함
- **Service**: 비즈니스 기능을 수행하는 독립적인 프로세스 그룹

**서비스 맵(Service Map)**

서비스 맵은 서비스 간 의존 관계를 자동으로 시각화한다. 요청 경로, 오류율, 평균 응답 시간을 한눈에 파악할 수 있으며, 마이크로서비스 아키텍처에서 장애 전파 경로를 추적하는 데 유용하다.

**느린 쿼리 및 병목 구간 식별**

APM의 Flame Graph(화염 그래프)로 각 Span의 소요 시간을 시각화하여 병목 구간을 빠르게 특정할 수 있다.

### 5. Dashboard & Metrics

**주요 인프라 메트릭**

- CPU: `system.cpu.user`, `system.cpu.system`
- Memory: `system.mem.used`, `system.mem.pct_usable`
- Disk: `system.disk.read_bytes`, `system.disk.write_bytes`
- Network: `system.net.bytes_rcvd`, `system.net.bytes_sent`

**애플리케이션 커스텀 메트릭 전송 (DogStatsD)**

DogStatsD는 UDP를 사용하므로 네트워크 지연 없이 비동기로 메트릭을 전송한다.

```java
// build.gradle
implementation 'com.datadoghq:java-dogstatsd-client:4.4.3'
```

```java
// Spring Boot - DogStatsD 커스텀 메트릭 예시
@Component
public class OrderMetrics {

    private final StatsDClient statsd;

    public OrderMetrics() {
        // Datadog Agent가 동일 호스트에 있으면 localhost:8125
        this.statsd = new NonBlockingStatsDClientBuilder()
            .prefix("myapp")
            .hostname("localhost")
            .port(8125)
            .build();
    }

    public void recordOrderCreated(String region) {
        // COUNT: 주문 생성 횟수
        statsd.incrementCounter("order.created", "region:" + region);
    }

    public void recordOrderProcessingTime(long milliseconds) {
        // HISTOGRAM: 주문 처리 시간 분포
        statsd.histogram("order.processing_time", milliseconds);
    }

    public void recordActiveUsers(int count) {
        // GAUGE: 현재 활성 사용자 수
        statsd.gauge("users.active", count);
    }
}
```

**SLO(Service Level Objective) 설정**

SLO는 서비스 품질 목표를 수치로 정의하고 추적한다.

- **Metric-based SLO**: 메트릭 비율로 정의 (예: 전체 요청 중 성공 응답 비율 99.9%)
- **Monitor-based SLO**: 기존 Monitor의 상태(UP/DOWN)를 기반으로 정의
- 7일, 30일, 90일 단위의 오류 예산(Error Budget)을 자동으로 계산

### 6. 알림(Monitors & Alerts) 설정

**Monitor 종류**

| 종류 | 설명 | 예시 |
|---|---|---|
| Metric Monitor | 메트릭 임계값 기반 | CPU > 90% |
| APM Monitor | 트레이스 오류율, 지연 | p99 응답 시간 > 2s |
| Log Monitor | 특정 로그 패턴 감지 | ERROR 로그 급증 |
| Composite Monitor | 여러 Monitor 조합 | CPU 높음 AND 메모리 높음 |

**임계값 기반 알림 vs 이상 탐지(Anomaly Detection)**

- **임계값 기반**: 고정된 값을 초과하면 알림. 단순하지만 정상 패턴의 변동을 반영하지 못함
- **이상 탐지**: 과거 패턴(시간대별, 요일별 패턴)을 학습하여 평소와 다른 이상 징후를 감지. 트래픽이 평소 화요일 오전 대비 급격히 낮을 경우 알림

**알림 피로(Alert Fatigue)를 줄이는 전략**

- **다운타임(Downtime)** 설정으로 배포 시간대 알림 억제
- **복합 조건(Composite Monitor)** 으로 단일 조건 오탐 방지
- **재알림 간격** 조정으로 반복 알림 최소화
- 알림 우선순위를 P1~P5로 분류하여 심각도 구분

### 7. 로그 관리 (Log Management)

**로그 수집 파이프라인 구성**

```yaml
# datadog.yaml (Agent 설정)
logs_enabled: true

# Spring Boot 로그 수집 설정
logs:
  - type: file
    path: /var/log/myapp/*.log
    service: my-spring-app
    source: java
```

**로그 파싱과 패싯(Facet) 설정**

Grok Parser를 사용하여 비정형 로그를 구조화된 속성으로 파싱한다. 파싱된 속성 중 검색/필터링에 자주 사용하는 것을 **Facet**으로 등록하면 Log Explorer에서 빠르게 분류할 수 있다.

**로그와 Trace 연결 (Log-Trace Correlation)**

`-Ddd.logs.injection=true` 옵션을 사용하면 dd-java-agent가 자동으로 로그에 `dd.trace_id`, `dd.span_id`를 주입한다. 이를 통해 특정 요청에서 발생한 로그를 트레이스와 연결하여 탐색할 수 있다.

```json
// 로그-트레이스 연결이 적용된 로그 예시
{
  "timestamp": "2026-03-13T10:00:00Z",
  "level": "ERROR",
  "message": "주문 처리 실패: 재고 부족",
  "dd.trace_id": "7890123456789012345",
  "dd.span_id": "1234567890123456",
  "dd.service": "order-service",
  "dd.env": "production"
}
```

**로그 기반 메트릭 생성**

로그에서 메트릭을 생성하면 로그 보관 비용을 줄이면서 장기 추세를 메트릭으로 저장할 수 있다. 예: `ERROR` 레벨 로그 발생 횟수를 메트릭으로 집계.

### 8. 실무 활용 사례

**장애 발생 시 Datadog을 활용한 원인 분석 흐름**

```
1. 알림 수신 (Monitor 임계값 초과 또는 Anomaly 감지)
        ↓
2. Dashboard에서 전체 메트릭 확인 (응답 시간, 오류율 급증 확인)
        ↓
3. APM Service Map에서 장애 영향 범위 파악
        ↓
4. Flame Graph로 병목 Span 특정
        ↓
5. 해당 Trace에서 연결된 로그 조회 (Log-Trace Correlation)
        ↓
6. 로그에서 구체적인 오류 원인 확인 (SQL 오류, 외부 API 타임아웃 등)
```

**DB 커넥션 풀 모니터링**

Spring Boot + HikariCP 환경에서는 Datadog이 자동으로 커넥션 풀 메트릭을 수집한다.
- `hikaricp.connections.active`: 현재 사용 중인 커넥션 수
- `hikaricp.connections.pending`: 대기 중인 커넥션 요청 수
- 커넥션 풀 고갈 시 Composite Monitor로 즉시 감지

**API 응답 시간 추적 및 병목 개선**

APM의 p50, p75, p95, p99 백분위수 응답 시간 추적으로 소수의 느린 요청이 전체 서비스에 미치는 영향을 파악한다. 특히 p99 이상 구간의 Trace를 분석하여 캐시 미스, N+1 쿼리, 외부 API 지연 등을 식별한다.

### 9. 신입 서버 개발자의 현실적인 Datadog 활용법

입사 초기에는 Datadog의 모든 기능을 사용할 필요가 없다. 일상 업무에서 바로 쓸 수 있는 기능부터 단계적으로 익히는 것이 효과적이다.

**1단계: 내가 만든 API가 정상 동작하는지 확인하기**

기능 개발 후 배포했을 때, 가장 먼저 해야 할 일은 내 API가 운영 환경에서 잘 동작하는지 확인하는 것이다.

```
APM > Services > 내 서비스 선택 > Resources 탭
```

- **Resources 탭**에서 내가 개발한 엔드포인트(예: `GET /api/v1/orders`)를 찾는다
- 해당 엔드포인트의 평균 응답 시간, 오류율, 초당 요청 수를 확인한다
- 배포 전후로 응답 시간이 급격히 변했거나 오류율이 올라갔다면 문제가 있는 것이다

> **팁**: 배포 직후 10~30분간 해당 엔드포인트의 메트릭을 관찰하는 습관을 들이면 장애를 조기에 발견할 수 있다.

**2단계: 내 코드에서 느린 쿼리 찾기**

코드 리뷰에서 "이 쿼리 괜찮아?" 라는 피드백을 받았을 때, 실제 운영 데이터로 증명할 수 있다.

```
APM > Traces > 내 엔드포인트 필터링 > 느린 요청 클릭 > Flame Graph
```

- Flame Graph에서 각 Span의 소요 시간을 확인한다
- DB 쿼리 Span을 클릭하면 **실제 실행된 SQL과 소요 시간**이 표시된다
- `duration:>1s`로 필터링하면 1초 이상 걸리는 느린 요청만 모아볼 수 있다

```
예시: Flame Graph에서 발견한 문제
──────────────────────────────────────
Controller (120ms)
├── Service (115ms)
│   ├── SELECT * FROM orders WHERE user_id = ? (2ms)
│   ├── SELECT * FROM order_items WHERE order_id = ? (3ms)  ← 반복!
│   ├── SELECT * FROM order_items WHERE order_id = ? (2ms)  ← N+1 쿼리!
│   ├── SELECT * FROM order_items WHERE order_id = ? (3ms)
│   └── ... (50번 반복, 총 100ms)
```

이런 패턴을 발견하면 `JOIN` 또는 `IN` 쿼리로 개선할 수 있다는 근거가 된다.

**3단계: 로그에서 에러 원인 추적하기**

운영 환경에서 500 에러가 발생했다는 알림을 받았을 때, 서버에 직접 SSH 접속하지 않고 Datadog에서 바로 원인을 찾을 수 있다.

```
Logs > 검색: service:my-spring-app status:error
```

- 에러 로그를 클릭하면 스택 트레이스 전체를 확인할 수 있다
- 해당 로그에 Trace ID가 주입되어 있으므로 `View Trace` 버튼으로 APM 트레이스로 바로 이동 가능하다
- 트레이스에서 어떤 요청이, 어떤 서비스를 거치다가, 어떤 지점에서 에러가 발생했는지 전체 맥락을 파악할 수 있다

**4단계: 배포 전후 비교로 성능 회귀 감지하

**

내가 리팩토링한 코드가 성능에 영향을 주지 않았는지 확인할 때 유용하다.

```
APM > Services > 내 서비스 > 시간 범위를 배포 전후로 설정
```

- 배포 시점을 기준으로 좌우를 비교한다
- 응답 시간 그래프에서 배포 후 p50, p99가 올라갔는지 확인한다
- Datadog의 **Deployment Tracking** 기능을 사용하면 배포 이벤트가 그래프에 수직선으로 표시되어 전후 비교가 더 직관적이다

```bash
# 배포 시 Datadog에 이벤트 전송 (CI/CD 파이프라인에 추가)
curl -X POST "https://api.datadoghq.com/api/v1/events" \
  -H "DD-API-KEY: ${DD_API_KEY}" \
  -d @- << EOF
{
  "title": "my-spring-app 배포",
  "text": "v1.2.3 배포 완료",
  "tags": ["service:my-spring-app", "env:production"]
}
EOF
```

**5단계: 팀 온콜 대응 시 활용하기**

온콜 담당이 되었을 때, 알림이 오면 당황하지 않고 다음 순서로 대응한다.

```
1. 알림 내용 확인 → 어떤 Monitor가 트리거되었는지 파악
2. 해당 Monitor의 Dashboard로 이동 → 메트릭 추이 확인
3. 영향받는 서비스의 APM 확인 → 오류율, 응답 시간 변화 확인
4. 에러 Trace 클릭 → Flame Graph에서 실패 지점 확인
5. 연결된 로그 확인 → 구체적 에러 메시지와 스택 트레이스 확인
6. 필요 시 관련 인프라 메트릭 확인 → CPU, 메모리, 디스크, 커넥션 풀
```

> **핵심은 "넓은 범위에서 좁은 범위로" 좁혀가는 것이다.** Dashboard(전체 현황) → APM(서비스 레벨) → Trace(요청 레벨) → Log(코드 레벨) 순서로 파고든다.

## 핵심 정리
- Datadog은 Metrics, Traces, Logs를 하나의 플랫폼에서 통합 관리하여 문제 발생 시 원인을 빠르게 추적할 수 있는 올인원 Observability 플랫폼이다
- dd-java-agent를 `-javaagent` JVM 옵션으로 적용하면 코드 수정 없이 Spring Boot 애플리케이션의 분산 추적이 활성화된다
- Log-Trace Correlation은 `-Ddd.logs.injection=true` 설정 하나로 로그에 Trace ID를 자동 주입하여 특정 요청의 로그를 트레이스와 연결하는 기능이다
- Anomaly Detection Monitor는 과거 패턴을 학습하여 임계값 지정 없이도 비정상 징후를 감지하므로, 트래픽 패턴이 복잡한 서비스에서 임계값 기반 알림보다 오탐을 크게 줄일 수 있다
- SLO를 통해 서비스 품질 목표를 수치로 정의하고 오류 예산(Error Budget)을 관리하면, 장애 허용 범위를 팀이 공유하고 의사결정의 근거로 삼을 수 있다

## 키워드
- `Datadog`: Metrics, Traces, Logs를 통합 제공하는 클라우드 기반 SaaS 모니터링 플랫폼. 수백 개의 서비스 통합과 AI 기반 분석 기능을 제공한다.
- `Observability`: 시스템 외부 출력(메트릭, 로그, 트레이스)만으로 내부 상태를 파악할 수 있는 능력. 단순 모니터링을 넘어 "왜" 문제가 발생했는지 추적하는 것이 목표다.
- `APM`: Application Performance Monitoring의 약자. 애플리케이션의 요청 흐름, 오류율, 지연 시간을 코드 레벨까지 추적한다.
- `Distributed Tracing`: 마이크로서비스 환경에서 여러 서비스에 걸친 요청의 전체 경로를 단일 흐름으로 추적하는 기술. Trace ID로 서비스 간 요청을 연결한다.
- `DogStatsD`: Datadog Agent에 내장된 메트릭 집계 서버. UDP(8125 포트)로 커스텀 메트릭을 수신하여 Datadog으로 전송한다. COUNT, GAUGE, HISTOGRAM, TIMER 타입을 지원한다.
- `Service Map`: APM에서 서비스 간 호출 관계를 자동으로 시각화하는 기능. 각 노드에 오류율과 응답 시간이 표시되어 장애 전파 경로를 파악하는 데 유용하다.
- `Log-Trace Correlation`: 로그에 Trace ID와 Span ID를 주입하여 특정 요청에서 발생한 로그를 APM 트레이스와 직접 연결하는 기능. dd-java-agent의 `dd.logs.injection=true` 옵션으로 자동 활성화된다.
- `Anomaly Detection`: 과거 데이터의 계절성(시간대, 요일 패턴)을 학습하여 고정 임계값 없이 비정상적인 패턴을 감지하는 Monitor 유형. 오탐율을 낮추는 데 효과적이다.
- `SLO`: Service Level Objective의 약자. 서비스 품질 목표를 수치로 정의(예: 가용성 99.9%)하고 오류 예산을 관리한다. 메트릭 기반과 Monitor 기반 두 가지 방식을 지원한다.
- `Datadog Agent`: 호스트에 설치되어 메트릭, 로그, 트레이스를 수집하여 Datadog 서버로 전송하는 경량 소프트웨어. Collector, DogStatsD, Forwarder 세 가지 컴포넌트로 구성된다.
