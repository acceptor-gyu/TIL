# 로드밸런싱이란

## 개요
로드밸런싱(Load Balancing)은 다수의 서버에 네트워크 트래픽이나 요청을 분산시켜 특정 서버에 부하가 집중되지 않도록 하는 기술이다. 서비스의 가용성, 확장성, 안정성을 확보하기 위해 필수적으로 도입되는 인프라 구성 요소다.

## 상세 내용

### 1. 로드밸런서의 역할과 동작 원리

로드밸런서는 클라이언트와 백엔드 서버 사이에 위치하는 중간 계층이다. 클라이언트는 로드밸런서의 단일 IP(Virtual IP)로 요청을 보내고, 로드밸런서는 설정된 알고리즘에 따라 가용한 백엔드 서버 중 하나를 선택하여 요청을 전달한다.

```
          클라이언트 요청
               │
               ▼
     ┌──────────────────┐
     │    로드밸런서       │  ← Virtual IP (VIP)
     │  (Load Balancer) │
     └──────────────────┘
          │    │    │
          ▼    ▼    ▼
       [S1]  [S2]  [S3]   ← 백엔드 서버 풀
```

**핵심 역할**

- **트래픽 분산**: 요청을 여러 서버에 나누어 단일 서버의 과부하를 방지한다.
- **헬스체크**: 서버 상태를 주기적으로 확인하고 비정상 서버를 풀에서 제외한다.
- **세션 유지**: 같은 클라이언트의 요청이 동일 서버로 라우팅되도록 보장할 수 있다.
- **SSL 처리**: SSL/TLS 암복호화를 대신 수행하여 백엔드 서버 부담을 줄인다.
- **고가용성**: 로드밸런서 자체를 이중화하여 단일 장애 지점(SPOF)을 제거한다.

### 2. L4 로드밸런싱 vs L7 로드밸런싱

OSI 7계층 모델에서 어느 계층의 정보를 기반으로 분산 결정을 내리느냐에 따라 L4와 L7로 구분된다.

#### L4 로드밸런싱 (전송 계층)

L4 로드밸런서는 IP 주소와 TCP/UDP 포트 정보만을 보고 트래픽을 분산한다. 패킷의 실제 내용(payload)은 검사하지 않으므로 처리 속도가 매우 빠르다.

```
클라이언트 → 로드밸런서(IP:Port 확인) → 백엔드 서버
              └── IP: 10.0.0.X
              └── Port: 80 or 443
```

**특징**
- 패킷 내용을 열어보지 않으므로 처리 속도가 빠르고 지연이 낮다.
- HTTP/HTTPS뿐만 아니라 임의의 TCP/UDP 트래픽도 처리 가능하다.
- URL 기반 라우팅, 쿠키 기반 분산 등 애플리케이션 수준의 기능은 제공하지 않는다.
- AWS에서는 NLB(Network Load Balancer)가 L4에 해당한다.

#### L7 로드밸런싱 (애플리케이션 계층)

L7 로드밸런서는 HTTP 헤더, URL 경로, 쿠키, 요청 본문 등 애플리케이션 계층의 정보를 분석하여 라우팅 결정을 내린다.

```
클라이언트 → 로드밸런서(HTTP 내용 분석) → 백엔드 서버
              └── /api/* → API 서버 풀
              └── /static/* → CDN 또는 정적 서버 풀
              └── Host: admin.example.com → 관리자 서버 풀
```

**특징**
- URL 경로, HTTP 헤더, 쿠키, 메서드 등 다양한 기준으로 라우팅이 가능하다.
- SSL Termination, WAF 통합, 헤더 조작 등 부가 기능을 제공한다.
- 패킷을 파싱하고 검사하므로 L4 대비 약간의 지연이 발생한다.
- AWS에서는 ALB(Application Load Balancer)가 L7에 해당한다.

#### 비교 요약

| 항목 | L4 로드밸런싱 | L7 로드밸런싱 |
|------|--------------|--------------|
| 동작 계층 | 전송 계층 (Transport) | 애플리케이션 계층 (Application) |
| 분산 기준 | IP 주소, TCP/UDP 포트 | URL, HTTP 헤더, 쿠키 |
| 처리 속도 | 빠름 (패킷 내용 미검사) | 상대적으로 느림 |
| 지원 프로토콜 | TCP, UDP 모두 지원 | HTTP, HTTPS 위주 |
| 콘텐츠 기반 라우팅 | 불가 | 가능 |
| SSL Termination | 제한적 | 지원 |
| 대표 솔루션 | AWS NLB, HAProxy (TCP) | AWS ALB, Nginx, HAProxy (HTTP) |

### 3. 로드밸런싱 알고리즘

#### Round Robin (라운드 로빈)

요청을 백엔드 서버 목록에 순서대로 순환하며 분배하는 가장 기본적인 방식이다. 모든 서버의 사양이 동일하고 요청 처리 시간이 유사할 때 적합하다.

```
요청 1 → Server A
요청 2 → Server B
요청 3 → Server C
요청 4 → Server A  ← 다시 처음으로
요청 5 → Server B
...
```

- 구현이 단순하고 예측 가능하다.
- 서버 간 성능 차이나 현재 부하를 고려하지 않는다.
- Nginx 기본 로드밸런싱 방식이다.

```nginx
upstream myapp {
    # round-robin이 기본값이므로 별도 지시자 불필요
    server srv1.example.com;
    server srv2.example.com;
    server srv3.example.com;
}
```

#### Weighted Round Robin (가중치 라운드 로빈)

각 서버에 가중치(weight)를 부여하여, 가중치 비율에 따라 요청을 분배한다. 서버 간 성능 차이가 있을 때 유용하다.

```
Server A (weight=3) → 5개 요청 중 3개
Server B (weight=1) → 5개 요청 중 1개
Server C (weight=1) → 5개 요청 중 1개
```

```nginx
upstream myapp {
    server srv1.example.com weight=3;
    server srv2.example.com weight=1;
    server srv3.example.com weight=1;
}
```

#### Least Connections (최소 연결)

현재 활성 연결(Active Connection) 수가 가장 적은 서버로 요청을 보내는 방식이다. 요청 처리 시간이 일정하지 않은 서비스에 적합하다.

```
Server A: 연결 10개
Server B: 연결 3개  ← 다음 요청은 이 서버로
Server C: 연결 7개
```

```nginx
upstream myapp {
    least_conn;
    server srv1.example.com;
    server srv2.example.com;
    server srv3.example.com;
}
```

#### IP Hash (IP 해시)

클라이언트의 IP 주소를 해시하여 항상 동일한 서버로 라우팅한다. 별도의 세션 클러스터링 없이도 세션 유지가 가능하다.

```
Client IP: 192.168.1.10 → hash() → Server B (항상 고정)
Client IP: 10.0.0.25   → hash() → Server A (항상 고정)
```

```nginx
upstream myapp {
    ip_hash;
    server srv1.example.com;
    server srv2.example.com;
    server srv3.example.com;
}
```

- 서버가 추가되거나 제거되면 기존 매핑이 변경될 수 있다.
- 특정 IP 대역의 요청이 많으면 특정 서버에 부하가 집중될 수 있다.

#### Least Response Time (최소 응답 시간)

활성 연결 수와 서버 응답 시간을 동시에 고려하여 가장 빠르게 응답하는 서버로 요청을 분배한다. Nginx Plus, HAProxy 등에서 지원한다.

```
Server A: 연결 5개, 응답시간 200ms
Server B: 연결 3개, 응답시간 50ms  ← 다음 요청은 이 서버로
Server C: 연결 2개, 응답시간 150ms
```

- 가장 정교한 알고리즘으로 성능 최적화 효과가 크다.
- 응답 시간 측정을 위한 추가 오버헤드가 발생한다.

#### 알고리즘 선택 가이드

| 상황 | 권장 알고리즘 |
|------|-------------|
| 서버 사양 동일, 짧은 처리 시간 | Round Robin |
| 서버 사양 차이가 있을 때 | Weighted Round Robin |
| 요청 처리 시간이 불규칙할 때 | Least Connections |
| 세션 유지가 필요할 때 | IP Hash |
| 최적의 성능이 필요할 때 | Least Response Time |

### 4. 헬스체크 (Health Check)

헬스체크는 로드밸런서가 백엔드 서버의 상태를 모니터링하여 비정상 서버를 자동으로 트래픽 풀에서 제외하는 메커니즘이다.

#### Active Health Check (능동적 헬스체크)

로드밸런서가 주기적으로 백엔드 서버에 직접 헬스체크 요청(probe)을 보내 상태를 확인한다. 실제 사용자 요청과 무관하게 서버 상태를 선제적으로 파악할 수 있다.

```
로드밸런서 ──[GET /health]──→ Server A  ← 200 OK → 정상
로드밸런서 ──[GET /health]──→ Server B  ← 응답 없음 → 비정상 → 풀에서 제외
로드밸런서 ──[GET /health]──→ Server C  ← 200 OK → 정상
```

- NGINX Plus, HAProxy에서 지원한다.
- 실패한 서버를 사용자 요청 전에 미리 감지할 수 있다.
- 헬스체크 엔드포인트(/health, /ping 등)를 별도로 구현해야 한다.

```nginx
# Nginx Plus Active Health Check 예시
upstream myapp {
    server srv1.example.com;
    server srv2.example.com;
}

server {
    location / {
        proxy_pass http://myapp;
        health_check interval=10 fails=3 passes=2 uri=/health;
    }
}
```

#### Passive Health Check (수동적 헬스체크)

별도의 프로브를 보내지 않고 실제 사용자 요청에 대한 서버 응답을 모니터링하여 서버 상태를 판단한다. Nginx 오픈소스 버전의 기본 방식이다.

```
클라이언트 → Server A → 응답 실패 (max_fails=3 초과)
              → Server A를 fail_timeout 동안 비활성화
              → fail_timeout 경과 후 자동으로 재시도
```

```nginx
upstream myapp {
    server srv1.example.com max_fails=3 fail_timeout=30s;
    server srv2.example.com max_fails=3 fail_timeout=30s;
    server srv3.example.com max_fails=3 fail_timeout=30s;
}
```

| 파라미터 | 기본값 | 설명 |
|---------|-------|------|
| `max_fails` | 1 | 서버를 비활성화하기까지 허용되는 최대 연속 실패 횟수 |
| `fail_timeout` | 10s | 비활성화 상태를 유지하는 시간. 이 시간이 지나면 재시도 |

#### Active vs Passive 비교

| 항목 | Active Health Check | Passive Health Check |
|------|--------------------|--------------------|
| 동작 방식 | 로드밸런서가 주기적으로 프로브 전송 | 실제 요청 결과를 모니터링 |
| 장애 감지 속도 | 빠름 (사용자 요청 전 감지) | 느림 (실제 실패 후 감지) |
| 사용자 영향 | 없음 | 실패한 요청이 사용자에게 노출될 수 있음 |
| 추가 트래픽 | 헬스체크 요청만큼 발생 | 없음 |
| 지원 | NGINX Plus, HAProxy | NGINX 오픈소스, HAProxy |

### 5. 고가용성 (High Availability) 구성

로드밸런서 자체가 단일 장애 지점(SPOF, Single Point of Failure)이 되지 않도록 이중화 구성이 필수적이다.

#### Active-Standby 구성

하나의 로드밸런서(Active)가 모든 트래픽을 처리하고, 나머지 하나(Standby)는 대기 상태를 유지한다. Active가 장애를 일으키면 Standby가 즉시 역할을 넘겨받는다.

```
              ┌──────────────┐
클라이언트 →     │ LB-1 (Active)│  ← 모든 트래픽 처리
              └──────────────┘
                     │ 장애 발생 시 Failover
              ┌──────────────┐
              │LB-2 (Standby)│  ← 대기 상태
              └──────────────┘
```

- 구성이 단순하고 관리가 용이하다.
- Standby 서버의 리소스가 평소에 유휴 상태로 낭비된다.
- Failover 시 짧은 순단(Brief Interruption)이 발생할 수 있다.

#### Active-Active 구성

두 대 이상의 로드밸런서가 모두 트래픽을 처리한다. DNS 라운드 로빈이나 상위 라우터가 로드밸런서 간 트래픽을 분배한다.

```
              ┌──────────────┐
         ┌──→ │ LB-1 (Active)│ → 백엔드 서버
클라이언트  │    └──────────────┘
         └──→ ┌──────────────┐
              │ LB-2 (Active)│ → 백엔드 서버
              └──────────────┘
```

- 리소스를 최대한 활용할 수 있고 처리 용량이 증가한다.
- 한 로드밸런서가 장애를 일으켜도 나머지가 계속 서비스를 유지한다.
- 세션 동기화 등 구성이 더 복잡하다.

#### VRRP (Virtual Router Redundancy Protocol)

VRRP는 여러 라우터나 로드밸런서를 하나의 가상 IP(Virtual IP, VIP)로 묶어 고가용성을 제공하는 표준 프로토콜이다(RFC 3768). Keepalived 등의 소프트웨어를 이용하여 구현한다.

```
              Virtual IP: 10.0.0.1
              ┌────────────────────┐
              │      VRRP Group    │
              │  ┌──────────────┐  │
              │  │ LB-1 (Master)│  │ ← VRRP 광고 전송 (1초마다)
              │  └──────────────┘  │
              │  ┌──────────────┐  │
              │  │ LB-2 (Backup)│  │ ← 광고 미수신 시 Master로 승격
              │  └──────────────┘  │
              └────────────────────┘
```

**동작 원리**

1. Master 로드밸런서가 VRRP Advertisement 패킷을 주기적으로 전송한다.
2. Backup은 Advertisement를 수신하며 Master 상태를 감시한다.
3. 일정 시간(기본 3초) 동안 Advertisement가 수신되지 않으면 Backup이 Master로 승격된다.
4. Master 승격 시 Gratuitous ARP를 브로드캐스트하여 네트워크에 VIP 매핑 변경을 알린다.
5. 클라이언트는 동일한 Virtual IP로 계속 접근하므로 Failover가 투명하게 이루어진다.

### 6. 세션 유지 (Session Persistence / Sticky Session)

Sticky Session은 같은 클라이언트의 요청이 항상 동일한 백엔드 서버로 라우팅되도록 보장하는 메커니즘이다.

#### 동작 원리

로드밸런서가 최초 요청 시 클라이언트에게 세션 식별 쿠키를 발급하고, 이후 요청에서 해당 쿠키를 읽어 동일 서버로 라우팅한다.

```
1. 첫 요청: 클라이언트 → LB → Server A
              LB가 쿠키 발급: Set-Cookie: SERVERID=A

2. 이후 요청: 클라이언트 → LB (Cookie: SERVERID=A 확인) → Server A (고정)
3. 이후 요청: 클라이언트 → LB (Cookie: SERVERID=A 확인) → Server A (고정)
```

#### Sticky Session의 한계

- **수평 확장 제약**: 특정 서버에 세션이 집중되어 로드밸런싱 효과가 감소한다.
- **서버 장애 시 세션 손실**: 고정된 서버가 다운되면 해당 서버의 모든 세션이 유실된다.
- **서버 추가/제거 어려움**: 서버를 제거할 때 해당 서버의 세션이 완료될 때까지 기다려야 한다.

#### 세션 클러스터링과의 비교

| 항목 | Sticky Session | 세션 클러스터링 (Redis 등) |
|------|---------------|------------------------|
| 구현 복잡도 | 낮음 | 높음 |
| 확장성 | 낮음 | 높음 |
| 서버 장애 시 | 세션 손실 | 세션 유지 |
| 로드밸런싱 효율 | 저하될 수 있음 | 최적 유지 |
| 추가 인프라 | 불필요 | Redis/Memcached 필요 |

#### Nginx Sticky Session 설정

```nginx
upstream myapp {
    ip_hash;  # IP 기반 세션 유지 (간단한 방법)
    server srv1.example.com;
    server srv2.example.com;
}
```

### 7. SSL Termination

SSL Termination(또는 SSL Offloading)은 클라이언트와 로드밸런서 간의 SSL/TLS 암복호화를 로드밸런서에서 처리하고, 로드밸런서에서 백엔드 서버로는 평문 HTTP로 통신하는 방식이다.

#### 동작 방식

```
클라이언트 ──[HTTPS/TLS]──→ 로드밸런서 ──[HTTP]──→ 백엔드 서버
                              (SSL 인증서 보유)
                              (암복호화 수행)
```

1. 클라이언트가 HTTPS로 로드밸런서에 연결한다.
2. 로드밸런서가 SSL 핸드셰이크를 수행하고 트래픽을 복호화한다.
3. 복호화된 HTTP 요청을 백엔드 서버에 전달한다.
4. 백엔드 서버의 응답을 다시 암호화하여 클라이언트에 전달한다.

#### SSL Termination의 장점

- **백엔드 서버 부담 감소**: RSA-2048 핸드셰이크는 약 10ms의 CPU를 소모한다. 이를 로드밸런서에서 전담하면 백엔드 서버는 비즈니스 로직에만 집중할 수 있다.
- **인증서 중앙 관리**: SSL 인증서를 로드밸런서 한 곳에서만 관리하면 되므로 갱신과 교체가 단순해진다.
- **HTTP 기능 활용**: 복호화된 요청을 검사할 수 있어 L7 라우팅, WAF, 헤더 조작 등이 가능해진다.

#### 보안 고려사항

SSL Termination 이후 로드밸런서와 백엔드 서버 간의 내부 네트워크 구간은 평문으로 통신한다. 이 구간의 보안을 위해 다음을 고려해야 한다.

- 내부 네트워크 망분리(Private Subnet) 구성
- 로드밸런서와 백엔드 서버 간에도 TLS를 적용하는 **SSL Bridging** 방식 사용
- 내부 트래픽 암호화가 규정(PCI-DSS, HIPAA 등)상 필수인 경우 SSL Passthrough 고려

#### SSL 처리 방식 비교

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| SSL Termination | LB에서 복호화 후 HTTP로 전달 | 성능, 중앙 관리 | 내부 구간 평문 전송 |
| SSL Passthrough | TLS 트래픽을 그대로 통과시킴 | 종단간 암호화 | L7 기능 불가 |
| SSL Bridging | LB에서 복호화 후 재암호화하여 전달 | 종단간 암호화 + L7 기능 | CPU 부담 증가 |

```nginx
# Nginx SSL Termination 설정 예시
server {
    listen 443 ssl;
    ssl_certificate     /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;

    location / {
        proxy_pass http://myapp;  # 백엔드로는 HTTP 전달
        proxy_set_header X-Forwarded-Proto https;  # 원본 프로토콜 전달
    }
}
```

## 핵심 정리
- 로드밸런싱은 트래픽을 여러 서버에 분산시켜 서비스의 가용성과 확장성을 확보하는 핵심 기술이다
- L4는 전송 계층에서 IP/Port 기반으로 빠르게 분산하고, L7은 애플리케이션 계층에서 요청 내용을 분석하여 정밀하게 분산한다
- 로드밸런싱 알고리즘은 서비스 특성에 맞게 선택해야 하며, Round Robin이 가장 기본적이다
- 헬스체크를 통해 비정상 서버를 자동으로 감지하고 트래픽에서 제외하여 서비스 안정성을 보장한다
- 로드밸런서 자체가 SPOF가 되지 않도록 VRRP 등을 활용한 이중화 구성이 필수적이다
- Sticky Session은 세션 유지 문제를 해결하지만 수평 확장성을 저해하므로, Redis 기반 세션 클러스터링이 보다 확장 가능한 대안이다
- SSL Termination은 백엔드 서버의 암복호화 부담을 로드밸런서로 집중시켜 성능을 향상시키지만, 내부 구간 보안에 유의해야 한다

## 키워드
- `로드밸런싱`: 네트워크 트래픽을 여러 서버에 분산시켜 부하를 균등하게 나누는 기술
- `L4`: OSI 전송 계층. IP 주소와 포트 기반으로 트래픽을 분산하는 로드밸런싱
- `L7`: OSI 애플리케이션 계층. HTTP 헤더, URL, 쿠키 등 요청 내용을 분석하여 분산하는 로드밸런싱
- `Round Robin`: 요청을 서버 목록에 순서대로 순환 분배하는 가장 기본적인 알고리즘
- `Least Connections`: 현재 활성 연결 수가 가장 적은 서버로 요청을 보내는 알고리즘
- `헬스체크`: 로드밸런서가 백엔드 서버의 상태를 주기적으로 확인하여 비정상 서버를 자동으로 트래픽 풀에서 제외하는 메커니즘
- `고가용성`: 시스템 장애가 발생해도 서비스가 중단되지 않고 지속적으로 운영되는 특성. 로드밸런서 이중화로 달성
- `Sticky Session`: 같은 클라이언트의 요청을 항상 동일한 백엔드 서버로 라우팅하여 세션을 유지하는 방식
- `SSL Termination`: 로드밸런서가 클라이언트와의 SSL/TLS 암복호화를 대신 수행하여 백엔드 서버 부담을 줄이는 방식
- `Nginx`: 고성능 오픈소스 웹 서버이자 L7 역방향 프록시/로드밸런서
