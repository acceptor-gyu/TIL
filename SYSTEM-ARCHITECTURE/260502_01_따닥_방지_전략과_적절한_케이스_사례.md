# 따닥 방지 전략과 적절한 케이스 사례

## 개요
사용자가 버튼을 빠르게 두 번 클릭하거나, 네트워크 타임아웃 후 자동 재시도가 발생하거나, 브라우저 새로고침으로 동일한 폼을 다시 제출할 때 같은 요청이 서버에 중복으로 도달한다. 이를 통칭하여 "따닥" 문제라고 부른다. 이 문서에서는 중복 요청이 발생하는 원인을 분석하고, 각 상황에 적합한 방지 전략과 케이스별 선택 기준을 정리한다.

## 상세 내용

### 1. 따닥(중복 요청) 문제란

중복 요청은 크게 세 가지 경로에서 발생한다.

**사용자 실수**

사용자가 응답이 느리다고 느껴 버튼을 반복 클릭하거나, 결제 완료 페이지에서 브라우저 뒤로가기 후 재제출한다. 모바일 환경에서는 터치 이벤트가 두 번 발생하는 경우도 있다.

**네트워크 재시도**

HTTP 클라이언트가 타임아웃을 감지하면 자동으로 동일한 요청을 재전송한다. 서버는 첫 번째 요청을 이미 처리했을 수도 있고, 처리 중일 수도 있고, 실제로 실패했을 수도 있다. 클라이언트는 이를 알 수 없다.

**분산 시스템의 메시지 재전달**

Kafka, SQS 같은 메시지 큐는 최소 한 번 전달(at-least-once delivery)을 보장한다. 컨슈머가 처리 후 커밋을 못하면 동일 메시지가 재전달되어 처리 로직이 두 번 실행된다.

중복 요청이 시스템에 미치는 영향은 심각도에 따라 다르다. 결제 중복 처리는 이중 청구로 이어지고, 재고 차감이 두 번 발생하면 재고 불일치가 생긴다. 포인트 적립이 두 번 되면 손실이 발생하며, 게시글 중복 생성은 사용자 경험을 저하시킨다.

### 2. 클라이언트 측 방지 전략

**버튼 비활성화 (Disable after click)**

버튼을 클릭하는 순간 비활성화하고 로딩 상태를 표시한다. 가장 간단하고 사용자 경험도 좋다.

```javascript
const button = document.getElementById('submit-btn');
button.addEventListener('click', async () => {
  button.disabled = true;
  try {
    await submitOrder();
  } finally {
    button.disabled = false; // 실패 시 재시도 가능하도록 복원
  }
});
```

**Debounce / Throttle 적용**

Debounce는 마지막 이벤트 발생 후 일정 시간이 지나야 실행하며, Throttle은 일정 시간 내 이벤트를 한 번만 실행한다. 클릭이 연속으로 발생해도 실제 요청은 한 번만 전송된다.

```javascript
// Debounce 예시 (500ms 이내 연속 클릭 무시)
const debouncedSubmit = debounce(submitOrder, 500);
button.addEventListener('click', debouncedSubmit);
```

**클라이언트 측 방지만으로 충분하지 않은 이유**

클라이언트 측 방어는 정상적인 브라우저 사용 시나리오에서만 동작한다. JavaScript를 우회한 직접 API 호출, 네트워크 레벨의 자동 재시도, 모바일 앱의 백그라운드 재시도는 클라이언트 측에서 막을 수 없다. 따라서 서버 측 방어와 반드시 함께 사용해야 한다.

### 3. 서버 측 방지 전략

**PRG (Post-Redirect-Get) 패턴**

POST 요청 처리 후 즉시 GET으로 리다이렉트한다. 브라우저 새로고침이 발생해도 GET 요청만 재실행되므로 폼 재제출이 일어나지 않는다.

```
[클라이언트] POST /orders
[서버] 주문 생성 → 302 Redirect → /orders/{id}
[클라이언트] GET /orders/{id}
[브라우저 새로고침] GET /orders/{id} (재제출 없음)
```

**요청별 고유 토큰 발급 및 검증 (Submission Token)**

폼 렌더링 시 서버가 일회용 토큰을 발급하고 숨겨진 필드에 포함시킨다. 서버는 제출 시 토큰을 검증하고 즉시 소비(무효화)한다. 같은 토큰으로 두 번 제출하면 두 번째 요청은 거부된다.

```
[서버] GET /order-form → 폼 + token=abc123 발급 (세션 또는 DB에 저장)
[클라이언트] POST /orders body: {token: "abc123", ...}
[서버] token 검증 및 소비 → 처리
[클라이언트] POST /orders body: {token: "abc123", ...} (재제출)
[서버] token 이미 소비됨 → 409 Conflict
```

**멱등성 키 (Idempotency Key) 패턴**

클라이언트가 요청마다 고유한 UUID를 생성하여 헤더에 포함시킨다. 서버는 키를 기준으로 처음 요청인지 재시도인지를 판단한다. 처음 요청이면 처리하고 결과를 키와 함께 저장한다. 재시도 요청이면 저장된 결과를 그대로 반환한다.

```java
// Spring Boot + Redis 예시
@PostMapping("/payments")
public ResponseEntity<PaymentResponse> createPayment(
    @RequestHeader("Idempotency-Key") String idempotencyKey,
    @RequestBody PaymentRequest request
) {
    // Redis에서 기존 처리 결과 조회
    String cached = redisTemplate.opsForValue().get("idem:" + idempotencyKey);
    if (cached != null) {
        return ResponseEntity.ok(objectMapper.readValue(cached, PaymentResponse.class));
    }

    // 처음 요청 처리
    PaymentResponse response = paymentService.process(request);

    // 결과 저장 (24시간 TTL)
    redisTemplate.opsForValue().set("idem:" + idempotencyKey,
        objectMapper.writeValueAsString(response), Duration.ofHours(24));

    return ResponseEntity.ok(response);
}
```

Stripe, PayPal 등 결제 게이트웨이가 표준으로 사용하는 방식이다. TTL은 일반적으로 24시간~7일 사이로 설정한다.

**Redis 기반 중복 요청 필터링 (TTL 활용)**

요청의 특징(사용자 ID + 요청 내용의 해시)으로 키를 만들고 Redis에 저장한다. 일정 시간 내 동일 키가 들어오면 중복으로 판단한다.

```java
String dedupKey = "dedup:" + userId + ":" + DigestUtils.md5Hex(requestBody);
Boolean isNew = redisTemplate.opsForValue()
    .setIfAbsent(dedupKey, "1", Duration.ofSeconds(5));

if (!isNew) {
    throw new DuplicateRequestException("중복 요청입니다");
}
```

**데이터베이스 Unique Constraint 활용**

DB 레벨에서 유니크 인덱스를 설정하여 중복 삽입 자체를 차단한다. 모든 애플리케이션 레벨 방어가 뚫리더라도 최후의 안전망 역할을 한다.

```sql
-- 멱등성 키를 DB에 저장하는 경우
CREATE TABLE idempotency_records (
    idempotency_key VARCHAR(64) PRIMARY KEY,
    response_body   TEXT,
    created_at      DATETIME,
    expires_at      DATETIME
);

-- 결합 유니크 인덱스로 비즈니스 레벨 중복 차단
CREATE UNIQUE INDEX ux_order_user_product_window
ON orders (user_id, product_id, DATE_FORMAT(created_at, '%Y%m%d%H'));
```

**분산 락 (Distributed Lock) 적용**

Redis `SET NX`로 특정 자원에 대한 잠금을 획득한 요청만 처리한다. 동시에 여러 요청이 들어올 때 한 요청만 처리되도록 보장한다.

```java
// Redis SET NX (원자적 잠금)
Boolean acquired = redisTemplate.opsForValue()
    .setIfAbsent("lock:order:" + userId, "locked", Duration.ofSeconds(10));

if (!acquired) {
    throw new DuplicateRequestException("처리 중인 요청이 있습니다");
}

try {
    orderService.create(request);
} finally {
    redisTemplate.delete("lock:order:" + userId);
}
```

### 4. API Gateway / 인프라 레벨 방지

**Rate Limiting과 따닥 방지의 차이**

Rate Limiting은 단위 시간 내 요청 횟수를 제한하는 것으로, 특정 사용자의 과도한 요청을 막는 데 목적이 있다. 따닥 방지는 동일한 논리적 작업이 두 번 처리되지 않도록 하는 것이다. Rate Limiting이 트래픽 제어라면, 따닥 방지는 데이터 정합성 보호다. 둘은 상호 보완적으로 함께 사용하는 경우가 많다.

**API Gateway에서의 중복 요청 필터링**

AWS API Gateway는 멱등성 토큰을 지원하며, Kong과 같은 API Gateway에서는 Request Transformer 플러그인으로 요청을 식별하고 중복 처리를 차단하는 커스텀 플러그인을 적용할 수 있다.

인프라 레벨 방지는 애플리케이션 코드 변경 없이 적용할 수 있다는 장점이 있지만, 비즈니스 로직을 알 수 없어 세밀한 제어가 어렵다. 따라서 인프라 레벨은 1차 방어선으로, 애플리케이션 레벨을 최종 방어선으로 사용하는 다층 구조가 권장된다.

### 5. 케이스별 적절한 전략 선택

| 케이스 | 권장 전략 | 이유 |
|---|---|---|
| 결제 / 송금 | 멱등성 키 + DB Unique Constraint | 금액 오류가 치명적이며 네트워크 재시도 잦음 |
| 좋아요 / 투표 | Redis 중복 체크 또는 DB Unique Constraint | 빠른 처리 필요, 중복 삽입 시 무시 |
| 폼 제출 (게시글 작성 등) | 클라이언트 Disable + PRG 패턴 + 서버 토큰 검증 | 브라우저 새로고침 방지, 단순 케이스 |
| 주문 생성 | 멱등성 키 + 분산 락 | 재시도 시나리오 빈번, 정확히 한 번 처리 필요 |
| 채팅 메시지 | 클라이언트 Dedup + 메시지 ID 기반 서버 필터링 | 지연 허용, 중복 메시지 사용자 경험 방지 |
| 메시지 큐 컨슈머 | 멱등성 키 + 분산 락 | at-least-once 환경에서 처리 보장 |
| 포인트 적립 / 차감 | 분산 락 + DB Unique Constraint | 순서 보장과 정확한 잔액 유지 필요 |

### 6. 멱등성 설계 원칙

**HTTP 메서드별 멱등성 특성**

| HTTP 메서드 | 멱등성 | 안전성 | 설명 |
|---|---|---|---|
| GET | O | O | 조회만 수행, 반복해도 동일한 결과 |
| PUT | O | X | 전체 리소스 교체, 같은 데이터로 반복해도 결과 동일 |
| DELETE | O | X | 이미 삭제된 리소스 재삭제는 404이지만 상태 변경 없음 |
| POST | X | X | 실행마다 새 리소스 생성 가능, 기본적으로 멱등하지 않음 |
| PATCH | X | X | 부분 수정, 누적 연산 시 결과가 달라질 수 있음 |

POST와 PATCH는 기본적으로 멱등하지 않기 때문에 중복 요청 방지가 필요한 경우 애플리케이션에서 명시적으로 멱등성을 구현해야 한다.

**멱등성 보장을 위한 응답 캐싱**

멱등성 키로 요청을 식별한 후 동일 키의 재시도 요청에는 저장된 원본 응답을 그대로 반환한다. 이때 HTTP 상태 코드와 헤더까지 원본과 동일하게 반환하는 것이 중요하다. 클라이언트가 재시도인지 원본 응답인지 구분할 필요가 없어야 한다.

**멱등성 키의 생명주기 관리**

멱등성 키는 재시도가 발생할 수 있는 기간보다 충분히 길게 유지해야 한다. 일반적인 기준은 다음과 같다.

- API 요청: 24시간
- 메시지 큐 컨슈머: 7일 (큐 보존 기간과 동일)
- 결제 처리: 30일 (분쟁 처리 기간 고려)

TTL이 만료된 키로 같은 요청이 오면 새로운 요청으로 처리한다.

**동시성 처리 (Race Condition 방지)**

같은 키로 요청이 동시에 들어오면 첫 요청 처리 완료 전에 두 번째 요청이 들어올 수 있다. 이를 막으려면 키 저장을 원자적으로 처리해야 한다.

```java
// IN_PROGRESS 상태로 원자적 저장
Boolean acquired = redisTemplate.opsForValue()
    .setIfAbsent("idem:" + key, "IN_PROGRESS", Duration.ofMinutes(5));

if (!acquired) {
    String status = redisTemplate.opsForValue().get("idem:" + key);
    if ("IN_PROGRESS".equals(status)) {
        // 503 Service Unavailable 또는 재시도 안내
        throw new RequestInProgressException("요청 처리 중입니다. 잠시 후 재시도하세요");
    }
    return getCachedResult(key); // COMPLETED 상태라면 저장된 결과 반환
}

try {
    PaymentResponse response = paymentService.process(request);
    redisTemplate.opsForValue().set("idem:" + key,
        objectMapper.writeValueAsString(response), Duration.ofHours(24));
    return ResponseEntity.ok(response);
} catch (Exception e) {
    redisTemplate.delete("idem:" + key); // 처리 실패 시 키 제거하여 재시도 허용
    throw e;
}
```

## 핵심 정리
- 따닥(중복 요청)은 사용자 실수, 네트워크 재시도, 메시지 재전달 세 가지 경로에서 발생한다
- 클라이언트 비활성화와 PRG 패턴은 사용자 실수에 의한 중복을 방지하지만 서버 측 재시도에는 무력하다
- 멱등성 키는 가장 범용적인 서버 측 방어 수단으로, 결제 / 주문처럼 정확히 한 번 처리가 중요한 케이스에 적합하다
- 분산 락은 동시성을 제어할 때 유효하지만 처리 시간이 길면 데드락 위험이 있으며 TTL을 반드시 설정해야 한다
- DB 유니크 제약 조건은 모든 방어를 뚫었을 때의 최후 안전망이다
- POST와 PATCH는 기본적으로 멱등하지 않으므로 중복 방지가 필요한 경우 애플리케이션에서 명시적으로 멱등성을 구현해야 한다
- 멱등성 키와 DB 제약 조건을 함께 사용하는 다층 방어가 결제처럼 치명적 케이스의 기본 설계다

## 키워드

**따닥 (Double Submit)**: 사용자의 빠른 이중 클릭 또는 네트워크 재시도로 인해 동일한 요청이 서버에 두 번 이상 도달하는 현상

**멱등성 (Idempotency)**: 동일한 연산을 여러 번 수행해도 결과가 첫 번째와 동일한 성질. GET, PUT, DELETE는 멱등적이지만 POST는 기본적으로 멱등적이지 않다

**Idempotency Key (멱등성 키)**: 클라이언트가 각 논리적 요청에 부여하는 고유 식별자(UUID). 서버는 이 키를 기준으로 재시도 여부를 판단하고 저장된 결과를 반환한다

**Debounce / Throttle**: Debounce는 마지막 이벤트 후 일정 시간이 지나야 실행하는 기법, Throttle은 일정 시간 내 한 번만 실행하는 기법. 클라이언트 측 중복 요청 방지에 활용한다

**분산 락 (Distributed Lock)**: Redis나 DB를 이용하여 분산 환경에서 특정 자원에 대한 단일 접근을 보장하는 메커니즘. `SET NX`(SET if Not eXists) 명령어로 원자적으로 잠금을 획득한다

**중복 요청 필터링 (Deduplication)**: 이미 처리된 요청과 동일한 요청을 식별하여 재처리를 차단하는 것. 멱등성이 결과의 동일성을 보장하는 개념이라면, Deduplication은 중복 실행 자체를 억제하는 구현 기법이다

**Rate Limiting**: 단위 시간 내 요청 횟수를 제한하는 기법. 과도한 트래픽을 막는 것이 목적으로, 데이터 정합성 보호를 목적으로 하는 따닥 방지와는 다르다

**Redis TTL (Time To Live)**: Redis 키에 만료 시간을 설정하는 기능. 멱등성 키나 분산 락에 TTL을 설정하여 처리 실패 시 자동으로 만료되도록 하여 데드락을 방지한다

**Unique Constraint (유니크 제약 조건)**: 데이터베이스 레벨에서 특정 컬럼 조합의 중복 삽입을 차단하는 제약. 애플리케이션 레벨 방어가 모두 실패했을 때의 최후 안전망이다

**Optimistic UI**: 서버 응답을 기다리지 않고 성공 상태를 미리 표시하는 UX 패턴. 사용자 경험을 향상시키지만, 서버 처리 실패 시 롤백 처리가 필요하다
