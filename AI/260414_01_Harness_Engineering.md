# 하네스 엔지니어링 (Harness Engineering)

## 개요

하네스 엔지니어링(Harness Engineering)은 AI 에이전트에서 모델 자체를 제외한 모든 외부 시스템과 설정을 설계하는 엔지니어링 분야이다. `Agent = Model + Harness` 공식에서 하네스에 해당하는 부분을 체계적으로 구성하여 에이전트의 신뢰성, 비용 효율성, 안전성을 높이는 것이 목표다.

2025년이 "에이전트의 해"였다면, 2026년은 "하네스의 해"로 불린다. 에이전트를 만드는 것 자체는 쉬워졌지만, 이를 프로덕션에서 안정적으로 운영하는 것이 핵심 과제로 떠올랐기 때문이다.

> "The model is probably fine. It's just a skill issue." — 성능 문제의 원인은 모델이 아닌 하네스 설정인 경우가 대부분이다.

## 상세 내용

### 하네스 엔지니어링이란

Mitchell Hashimoto의 정의: "에이전트가 실수를 할 때마다 그 실수를 다시 반복하지 않도록 해결책을 엔지니어링하는 것."

LangChain의 정의: "모델의 본질적으로 불안정한 지능을 우리가 관심 있는 작업에 맞게 조율하는 시스템."

하네스 엔지니어링은 다음 네 가지를 체계적으로 설계한다.

| 역할 | 설명 |
|---|---|
| **제약 (Constrain)** | 에이전트가 할 수 있는 것과 없는 것을 경계로 규정 |
| **안내 (Inform)** | 에이전트가 무엇을 해야 하는지 컨텍스트로 알려줌 |
| **검증 (Verify)** | 에이전트가 올바르게 수행했는지 확인 |
| **교정 (Correct)** | 잘못됐을 때 자동으로 피드백하여 수정 유도 |

### 프롬프트 엔지니어링 vs 하네스 엔지니어링

| 구분 | 프롬프트 엔지니어링 | 하네스 엔지니어링 |
|---|---|---|
| **초점** | 모델에게 "무엇을 할지" 잘 전달 | 모델이 "어떻게 실행할지" 환경 구축 |
| **범위** | 프롬프트 텍스트 | 도구, 메모리, 검증, 안전 장치 전체 |
| **성과** | 단일 응답 품질 향상 | 멀티 스텝 태스크 신뢰성 향상 |
| **지속성** | 세션별 적용 | 시스템 수준에서 지속 적용 |

### 가이드(피드포워드)와 센서(피드백)

하네스의 제어 구조는 두 축으로 나뉜다.

**가이드 (Guide / Feedforward)**
에이전트가 행동하기 전에 올바른 방향으로 사전 조향하는 요소.
- AGENTS.md / CLAUDE.md 파일: 에이전트에 자동 주입되는 규칙과 지침
- 시스템 프롬프트: 역할, 제약, 출력 형식 명시
- 구조화된 작업 템플릿: 범위, 파일 경로, 승인 기준, 테스트 요구사항 포함

**센서 (Sensor / Feedback)**
에이전트가 행동한 후 문제를 감지하고 자동 수정하는 요소.
- 타입 체크, 린터, 테스트 실행 결과를 에이전트에 피드백
- AI 기반 코드 리뷰: 의미적 정확성 검사
- 루프 탐지 미들웨어: 반복 패턴 감지 후 접근 방식 재검토 요청

### LangChain의 하네스 엔지니어링 성과

LangChain은 하네스만 변경하고 모델(gpt-5.2-codex)은 고정한 채로 Terminal Bench 2.0에서 **52.8점에서 66.5점으로 13.7포인트** 향상시켰다. 개선에 사용된 기법은 다음과 같다.

**1. 자체 검증 루프 (Self-Verification Loop)**

```
계획 → 구축 → 검증 → 수정
 ▲                    │
 └────────────────────┘
```

에이전트가 코드를 작성한 직후 테스트를 실행해 스스로 검증하도록 프롬프트에 명시적으로 포함. 모델은 첫 번째 그럴싸한 해결책에 편향되는 경향이 있어 이를 보완한다.

**2. 환경 맥락 주입 (Environment Context Injection)**
- 디렉토리 구조 자동 매핑 후 에이전트에 주입
- 사용 가능한 도구 탐색 결과 주입
- 시간 제약, 작업 범위 명시

**3. 루프 탐지 미들웨어 (Loop Detection)**
같은 파일을 반복 수정하는 "악순환 패턴"을 감지해 에이전트에게 접근 방식 재검토를 요청.

**4. 추론 샌드위치 패턴 (Reasoning Sandwich)**
```
고추론 (계획 단계)
    │
저추론 (코드 실행 단계)
    │
고추론 (검증 단계)
```
계획과 검증에 높은 추론 예산을, 실제 실행에 낮은 추론 예산을 할당하여 비용을 절감하면서 품질을 유지.

### 하네스 엔지니어링의 핵심 영역

#### 도구 설계 (Tool Design)

효과적인 도구는 다음 원칙을 따른다.
- 도구 설명(description)을 정확하고 구체적으로 작성: 에이전트는 설명을 보고 도구를 선택하기 때문에 품질이 직접 성능에 영향을 준다
- 오류 메시지에 수정 방법을 포함: `"파일을 찾을 수 없습니다: {path}. 먼저 ls 도구로 디렉토리를 확인하세요."`
- 도구 수를 최소화: 불필요한 도구는 에이전트를 "Dumb Zone"으로 유도

**AGENTS.md / CLAUDE.md**
저장소 최상단에 위치하여 에이전트의 시스템 프롬프트에 자동 주입되는 설정 파일.
- 60줄 이하로 간결하게 유지
- LLM이 자동 생성하지 말고 사람이 직접 작성
- 코딩 규칙, 도구 사용 방법, 금지 행동 등 명시

**MCP 서버 (Model Context Protocol)**
에이전트의 도구를 확장하는 프로토콜. 도구 설명이 시스템 프롬프트에 주입되므로 신뢰할 수 있는 서버만 연결하고, 필요한 도구만 활성화해야 한다.

#### 컨텍스트 관리 전략

**컨텍스트 윈도우 최적화**
- 성공 메시지는 억제하고 실패만 상세히 표시 (컨텍스트 낭비 방지)
- 전체 테스트 스위트가 아닌 관련 테스트만 실행
- 장기 작업 결과는 파일시스템에 오프로딩 후 요약만 컨텍스트에 유지

**정보의 선택적 노출**
현재 단계에서 필요한 정보만 컨텍스트에 포함한다. 전체 코드베이스가 아닌 변경 대상 파일과 관련 패턴만 주입.

#### 에이전트 루프 최적화

ReAct 패턴(Reason → Act → Observe)의 실제 구현 시 고려사항.

```
현재 상태 수신
      │
      ▼
  [추론: 다음 행동 결정]   ← LLM (고추론 예산)
      │
      ▼
  [도구 실행]             ← Harness Tool Router (저추론)
      │
      ▼
  [결과 검증]             ← Back-Pressure (타입체크, 테스트)
      │
      ▼
  목표 달성? ──Yes──▶ 검증 루프 실행 ──통과──▶ 종료
      │
      No
      │
      ▼
    (반복, 최대 N회 후 루프 탐지)
```

**무한 루프 방지**
- 최대 반복 횟수 하드리밋 설정
- 동일 파일 N회 이상 수정 시 루프 탐지 트리거
- 루프 감지 시 지금까지 시도한 접근 방식과 실패 이유를 컨텍스트에 주입

#### 메모리 아키텍처 설계

```
┌──────────────────────────────┐
│  컨텍스트 윈도우 (단기)          │  ← 현재 작업 중 활성 정보
│  파일시스템 / DB (중기)          │  ← 작업 진행 상태, 중간 결과
│  벡터 DB / 장기 저장소 (장기)    │  ← 과거 작업, 학습된 패턴
└──────────────────────────────┘
```

파일시스템이 실용적인 장기 상태 관리의 기초로 활용된다. 벡터 DB보다 단순하고 에이전트가 직접 읽고 쓸 수 있어 선호된다.

**Skills를 통한 점진적 공개**
에이전트가 필요할 때만 특정 지식/도구에 접근하도록 제한. SKILL.md 파일 기반으로 동작하며 컨텍스트 오염을 방지한다.

**Sub-Agents를 통한 컨텍스트 격리**
별도의 격리된 컨텍스트 윈도우에서 개별 작업을 수행. "Context Firewall" 역할로 중간 결과물이 부모 스레드를 오염시키지 않는다.

#### 안전성 및 가드레일

**Hooks (생명주기 제어)**
에이전트 실행 특정 시점에 자동으로 실행되는 제어 장치.
- PreToolUse: 위험한 도구 호출을 실행 전에 차단
- PostToolUse: 실행 결과를 검증하고 오류를 에이전트에 피드백
- Stop: 특정 조건에서 강제 종료

**Back-Pressure (검증 메커니즘)**
타입체크, 빌드, 테스트, 커버리지 확인을 통한 검증. 에이전트가 오류를 인식하고 수정하도록 출력을 구조화한다.

### 하네스 규제 3가지 범주

Martin Fowler의 분류에 따르면 하네스는 다음 세 가지 유형으로 나뉜다.

**유지보수성 하네스 (Maintainability Harness)**
코드 품질 관리. 가장 성숙한 분야.
- 린터, 포매터, 정적 분석 도구
- 코드 복잡도 측정, 중복 탐지

**아키텍처 피트니스 (Architecture Fitness)**
성능·설계 요구사항 검증.
- 의존성 규칙 검사 (ArchUnit 등)
- 패키지 구조 강제

**동작 하네스 (Behavioral Harness)**
기능적 정확성 검증. 아직 미성숙한 단계.
- 에이전트 출력의 의미적 정확성 검사
- LLM 기반 코드 리뷰

### 하네스 엔지니어링의 평가와 측정

**에이전트 성능 벤치마킹**
- Terminal Bench, SWE-bench 등 표준 벤치마크 활용
- 하네스 변경 전후 점수 비교로 개선 효과 정량화

**관측 가능성(Observability) 구축**
LangSmith의 Trace Analyzer를 활용해 에이전트 실패 패턴을 자동 분석. 실패 트레이스를 모아 공통 원인을 찾아내고 하네스를 개선한다.

**비용 대비 효과 분석**
추론 샌드위치 패턴 등을 적용해 토큰 사용량 대비 성능 비율을 측정. 고추론이 필요한 단계와 그렇지 않은 단계를 구분한다.

### 하네스 설계 원칙

**실패 기반 개선**
에이전트가 실수할 때마다 해당 실수를 방지하는 하네스 요소를 추가한다. 사전에 모든 것을 최적화하려 하지 않고, 실제 실패 후에만 설정을 개선한다.

**간결성 우선**
좋은 AGENTS.md와 pre-commit hook은 복잡한 미들웨어보다 효과적이다. 도구와 지시사항을 최소한으로 유지한다.

**컨텍스트 효율성**
성공 메시지는 억제하고 실패만 상세히 에이전트에게 전달한다. 불필요한 정보는 컨텍스트를 낭비하여 후속 판단의 품질을 떨어뜨린다.

**모델 독립성**
하네스는 특정 모델에 종속되지 않도록 설계한다. 모델 교체 시 하네스는 재사용 가능해야 한다.

### 인간의 역할

하네스는 인간 개발자의 경험과 지식을 명시화하려는 시도이다. 하지만 다음은 여전히 인간이 담당해야 한다.
- 조직의 맥락과 역사적 배경
- 윤리적 판단과 트레이드오프
- 의도된 목표와 비즈니스 의미
- 아직 코드화되지 않은 암묵적 지식

## 핵심 정리

- 하네스 엔지니어링은 `Agent = Model + Harness`에서 하네스를 체계적으로 설계하는 분야이다
- 가이드(피드포워드)와 센서(피드백) 두 축으로 에이전트를 제어한다
- LangChain은 하네스만 개선하여 모델 고정 상태에서 벤치마크 점수를 13.7포인트 향상시켰다
- AGENTS.md, MCP, Skills, Sub-Agents, Hooks, Back-Pressure가 핵심 하네스 구성 요소다
- 실패 발생 시에만 설정을 개선하고, 도구와 지시사항은 최소한으로 유지한다
- 성공은 무음으로, 실패만 상세히 출력하여 컨텍스트를 절약한다
- 추론 샌드위치 패턴으로 계획/검증에 고추론, 실행에 저추론을 배분해 비용을 최적화한다

## 키워드

### Harness Engineering (하네스 엔지니어링)
AI 에이전트에서 모델을 제외한 모든 외부 시스템(프롬프트, 도구, 설정, 검증 로직 등)을 체계적으로 설계하는 엔지니어링 분야. 에이전트의 신뢰성과 생산성을 높이는 핵심 역할을 한다.

### Context Engineering (컨텍스트 엔지니어링)
LLM의 제한된 컨텍스트 윈도우를 효율적으로 활용하기 위해 정보의 압축, 선택적 노출, 오프로딩 등을 설계하는 기술. 하네스 엔지니어링의 핵심 하위 분야이다.

### Feedforward / Feedback (피드포워드 / 피드백)
하네스의 두 가지 제어 방향. 피드포워드(가이드)는 에이전트가 행동하기 전에 방향을 사전 조향하고, 피드백(센서)는 행동 후 결과를 감지하여 자동 수정한다.

### Self-Verification Loop (자체 검증 루프)
에이전트가 작업 완료 후 스스로 테스트를 실행하여 결과를 검증하고 수정하는 사이클. 모델의 "첫 번째 그럴싸한 해결책 편향"을 보완한다.

### Reasoning Sandwich (추론 샌드위치)
계획과 검증 단계에는 높은 추론 예산을, 실행 단계에는 낮은 추론 예산을 할당하는 패턴. 품질과 비용을 동시에 최적화한다.

### Tool Design (도구 설계)
에이전트가 사용할 도구의 인터페이스, 설명, 입출력 형식, 오류 처리를 설계하는 영역. 도구 설명의 품질과 오류 메시지의 구체성이 에이전트 성능에 직접 영향을 미친다.

### Back-Pressure (백 프레셔)
에이전트의 출력을 검증하는 메커니즘 (타입체크, 빌드, 테스트 등). 성공은 무음으로 처리하고 실패만 상세히 에이전트에게 피드백하여 컨텍스트 효율성을 높인다.

### Context Firewall (컨텍스트 방화벽)
Sub-Agent를 활용해 중간 작업의 결과물이 부모 에이전트의 컨텍스트를 오염시키지 않도록 격리하는 패턴.

### Loop Detection (루프 탐지)
에이전트가 같은 작업을 반복하는 "악순환 패턴"을 감지하여 접근 방식 재검토를 유도하는 미들웨어.

### Guardrails (가드레일)
에이전트의 행동 범위를 제한하고 안전성을 확보하기 위한 제약 조건과 정책. Hooks를 통해 실행 시점에 위험한 도구 호출을 차단하거나 출력을 필터링한다.

### Durable Execution (내구성 실행)
에이전트 실행 중 장애가 발생해도 중단 지점부터 재개할 수 있는 실행 모델. Temporal, LangGraph 등이 이를 지원한다.

### Dumb Zone (덤 존)
에이전트에 너무 많은 도구나 지시사항이 주입되었을 때 판단력이 저하되는 상태. 도구 수를 최소한으로 유지해야 하는 이유이다.

### Agent Observability (에이전트 관측 가능성)
에이전트의 각 실행 단계를 추적, 로깅, 시각화하여 디버깅과 성능 최적화를 가능하게 하는 체계. LangSmith, Datadog 등이 대표 도구이다.

### Orchestrator-Worker Pattern (오케스트레이터-워커 패턴)
상위 에이전트가 작업을 분할하여 전문화된 하위 에이전트들에게 위임하는 멀티 에이전트 설계 패턴. 복잡한 태스크를 병렬화하고 컨텍스트를 격리하는 데 효과적이다.
