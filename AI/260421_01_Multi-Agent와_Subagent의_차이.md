# Multi-Agent와 Subagent의 차이

## 개요

Claude Code에는 작업을 병렬화하고 위임하는 두 가지 핵심 메커니즘이 있다: Subagent와 Multi-Agent(Agent Teams). 두 개념은 겉보기에 유사해 보이지만 통신 구조, 조율 방식, 토큰 비용, 적합한 사용 시나리오에서 근본적인 차이를 가진다. 이 차이를 명확히 이해해야 상황에 맞는 아키텍처를 선택할 수 있다.

## 상세 내용

### 1. 핵심 아키텍처 차이

**Subagent (서브에이전트)**

Subagent는 메인 세션(오케스트레이터) 내에서 호출되는 독립 실행 단위다. 메인 에이전트가 작업을 분해하고 서브에이전트에게 위임하며, 서브에이전트는 작업을 완료한 뒤 결과만 메인 에이전트에게 반환한다. 서브에이전트 간의 직접 통신은 없으며, 모든 정보는 메인 에이전트를 통해서만 흐른다.

```
메인 에이전트 (오케스트레이터)
    │
    ├── 서브에이전트 A → [결과 반환] → 메인 에이전트
    ├── 서브에이전트 B → [결과 반환] → 메인 에이전트
    └── 서브에이전트 C → [결과 반환] → 메인 에이전트
```

**Multi-Agent / Agent Teams (에이전트 팀)**

Agent Teams는 복수의 Claude Code 인스턴스가 팀으로 협업하는 구조다. 팀 리드(Team Lead)가 전체 작업을 조율하고, 각 팀원(Teammate)은 독립적인 Claude Code 세션으로 동작한다. 팀원끼리 공유 태스크 목록(Shared Task List)과 메일박스(Mailbox) 시스템을 통해 직접 통신할 수 있다.

```
팀 리드 (Team Lead)
    │
    ├── 팀원 A ←→ 팀원 B  (직접 메시지 교환)
    │       ↕
    └── 공유 태스크 목록 (Shared Task List)
```

### 2. 주요 비교 기준

**자율성 수준 (Autonomy Level)**

Subagent는 위임된 목표와 제한된 자율성을 가진다. 메인 에이전트가 명확히 정의한 태스크를 수행하고 결과를 반환하는 것이 전부다. 반면 Agent Teams의 팀원은 공유 태스크 목록에서 자율적으로 태스크를 클레임하고, 다른 팀원과 직접 소통하며 방향을 조율할 수 있다.

**통신 방향성 (Communication Directionality)**

Subagent는 단방향 제어 흐름을 따른다. 메인 에이전트 → 서브에이전트 → 결과 반환의 구조다. Agent Teams는 양방향 및 수평 통신이 가능하다. 팀원 A가 팀원 B에게 직접 메시지를 보내거나, 팀 전체에 브로드캐스트할 수 있다.

**의사결정 권한 (Decision Authority)**

Subagent 구조에서는 메인 에이전트가 모든 조율 책임을 가진다. Agent Teams에서는 팀원이 자율적으로 태스크를 선택하고 서로의 작업에 피드백을 제공하므로, 의사결정이 분산된다.

**실패 처리 방식 (Failure Handling)**

Subagent 실패 시 메인 에이전트가 오류를 감지하고 재시도나 대안을 결정한다. Agent Teams에서는 팀원이 스스로 다른 팀원에게 도움을 요청하거나, 리드가 막힌 팀원을 감지하고 태스크를 재배정할 수 있다.

**상태 공유 범위 (State Sharing Scope)**

Subagent는 메인 에이전트와 파일 시스템을 통해서만 상태를 간접 공유한다. Agent Teams는 공유 태스크 목록을 통해 전체 팀의 진행 상황을 실시간으로 파악하고, 메일박스로 구조화된 메시지를 교환한다.

### 3. 주요 비교표

| 항목 | Subagent | Agent Teams |
|------|----------|-------------|
| **컨텍스트** | 독립 컨텍스트 윈도우; 결과만 메인으로 반환 | 완전 독립 Claude Code 세션 |
| **에이전트 간 통신** | 메인 에이전트를 통해서만 가능 | 팀원끼리 직접 메시지 교환 가능 |
| **조율 방식** | 메인 에이전트가 모든 작업 관리 | 공유 태스크 목록으로 자율 조율 |
| **토큰 비용** | 4~7x (단일 세션 대비) | ~15x (단일 세션 대비) |
| **적합한 용도** | 결과만 중요한 집중 작업 | 토론과 협업이 필요한 복잡한 작업 |
| **활성화 조건** | 기본 활성화 | 실험적 기능, 별도 활성화 필요 |
| **중첩 생성** | 서브에이전트 내 서브에이전트 불가 | 팀원 내 팀 생성 불가 |

### 4. Subagent 실제 적용 사례

**코딩 에이전트의 도구 호출 패턴**

대용량 출력을 생성하는 작업에서 서브에이전트의 컨텍스트 격리가 핵심적인 역할을 한다. 전체 테스트 스위트 실행, 수천 줄의 로그 분석, 방대한 코드베이스 탐색 등은 메인 세션의 컨텍스트 윈도우를 빠르게 소진시킨다. 서브에이전트가 이런 작업을 격리된 컨텍스트에서 수행하고 요약만 반환하면, 메인 세션이 핵심 작업에 집중할 수 있다.

**병렬 조사 패턴**

```text
인증, 데이터베이스, API 모듈을 각각 별도의 서브에이전트로 병렬 조사해줘
```

각 서브에이전트가 독립 컨텍스트에서 다른 모듈을 동시에 탐색한다. 단, 각 서브에이전트의 결과가 메인 컨텍스트로 반환되므로, 결과가 상세할수록 메인 컨텍스트를 많이 소비한다.

**파이프라인 내 전문 작업 위임**

커스텀 서브에이전트를 `.claude/agents/` 디렉토리에 정의하면, 동일한 역할의 워커를 여러 프로젝트에서 재사용할 수 있다. `code-reviewer`, `security-auditor`, `test-runner` 등 역할 특화 서브에이전트를 미리 정의해두면 필요할 때 자동으로 위임된다.

**제약사항**: 서브에이전트는 다른 서브에이전트를 생성할 수 없다. 중첩 위임이 필요하다면 메인 세션에서 순차적으로 호출하거나 Skills를 활용해야 한다.

### 5. Agent Teams 실제 적용 사례

**토론/합의 시스템**

단일 에이전트는 처음 발견한 그럴듯한 이론에 고착되는 경향(Anchoring Bias)이 있다. Agent Teams에서 경쟁 가설을 가진 팀원들이 서로를 반증하려 하면, 살아남는 이론이 실제 원인일 가능성이 훨씬 높아진다.

```text
사용자가 첫 메시지 후 앱이 종료된다고 보고했습니다.
5명의 에이전트 팀원을 생성해 서로 다른 가설을 조사하고,
서로의 이론을 반증하려 시도하며 과학적 토론 방식으로 진행해주세요.
합의된 결론을 findings 문서에 업데이트해주세요.
```

**분산 문제 해결: 독립 모듈 병렬 개발**

팀원 각자가 파일 충돌 없이 독립적인 모듈을 소유하고 동시에 작업할 수 있다. 프론트엔드, 백엔드, 테스트를 각각 다른 팀원이 담당하면서 공유 태스크 목록을 통해 진행 상황을 조율한다.

**Agent Teams 활성화 방법**

```json
// settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

### 6. 토큰 비용과 선택 기준

| 방식 | 토큰 배율 (단일 세션 대비) | 적합한 상황 |
|------|--------------------------|------------|
| 단일 세션 | 1x | 소규모 작업, 빈번한 상호작용 |
| Subagent | 4~7x | 집중적 독립 작업, 컨텍스트 절감 필요 |
| Agent Teams | ~15x | 에이전트 간 협업, 경쟁 가설 검증 |

Anthropic의 연구 결과에 따르면, Claude Opus 4를 리드로 Claude Sonnet 4를 서브에이전트로 사용한 멀티 에이전트 시스템이 단일 Claude Opus 4 대비 복잡한 리서치 태스크에서 90.2% 더 나은 성능을 보였다. 토큰 사용량이 성능 분산의 약 80%를 설명한다는 점에서, 토큰 투자 대비 효과를 신중히 계산해야 한다.

### 7. 의사결정 흐름

```
작업이 에이전트 간 직접 통신이나 토론을 필요로 하는가?
    │
    ├── Yes → Agent Teams 사용
    │          (경쟁 가설 검증, 독립 모듈 병렬 개발, 리뷰/합의)
    │
    └── No
         │
         작업이 빠르고 집중된 실행으로 결과만 필요한가?
         │
         ├── Yes → Subagent 사용
         │          (대용량 로그 분석, 독립 병렬 조사, 도구 제한 작업)
         │
         └── No → 단일 세션 유지
                   (빈번한 상호작용, 단계 간 컨텍스트 공유, 소규모 작업)
```

## 핵심 정리

- Subagent는 메인 에이전트를 통해서만 소통하며 결과를 반환하는 계층적 단방향 구조다
- Agent Teams는 팀원 간 직접 통신과 공유 태스크 목록을 통해 자율 조율하는 수평 협업 구조다
- Subagent는 4~7x, Agent Teams는 ~15x 토큰을 소비하므로 작업 가치에 비례한 구조 선택이 필요하다
- 서브에이전트는 다른 서브에이전트를 생성할 수 없으며, Agent Teams의 팀원도 중첩 팀을 생성할 수 없다
- 단순히 결과가 필요하다면 Subagent, 에이전트 간 논쟁/검증/협업이 필요하다면 Agent Teams가 적합하다
- Agent Teams는 현재 실험적 기능으로, 세션 재개 및 태스크 조율에 알려진 제한 사항이 있다

## 키워드

### Multi-Agent System (멀티 에이전트 시스템)
복수의 에이전트가 역할을 분담하여 협업하는 아키텍처. Claude Code에서는 Agent Teams 형태로 구현되며, 팀원 간 직접 통신과 공유 태스크 목록을 통해 자율적으로 조율한다. 단일 에이전트보다 복잡하고 비용이 높지만, 에이전트 간 토론과 검증이 필요한 작업에서 품질을 크게 높인다.

### Subagent (서브에이전트)
메인 세션(오케스트레이터)으로부터 특정 작업을 위임받아 독립적인 컨텍스트 윈도우에서 실행되는 에이전트. 작업 완료 후 결과 요약만 메인 에이전트에게 반환하며, 에이전트 간 직접 통신은 불가하다. 커스텀 서브에이전트는 `.claude/agents/` 디렉토리에 YAML 프론트매터로 정의한다.

### Agent Teams (에이전트 팀)
복수의 Claude Code 인스턴스가 팀으로 협업하는 구조. 팀 리드가 작업을 조율하고, 팀원 각각은 독립 세션으로 동작하며 공유 태스크 목록과 메일박스를 통해 직접 소통한다. `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` 환경 변수로 활성화하는 실험적 기능이다.

### Orchestrator-Worker Pattern (오케스트레이터-워커 패턴)
중앙 오케스트레이터가 작업을 동적으로 분해하고 워커에게 위임하는 아키텍처 패턴. Subagent 구조가 이 패턴을 따른다. 모든 조율 책임이 오케스트레이터에 집중되므로 워커 간 독립성이 높다.

### Peer-to-Peer Communication (에이전트 간 동등 통신)
Agent Teams에서 팀원들이 팀 리드를 거치지 않고 직접 소통하는 방식. 메일박스(Mailbox) 시스템을 통해 특정 팀원에게 메시지를 보내거나, 전체 팀에 브로드캐스트할 수 있다. Subagent 구조와 달리 수평적 협업이 가능하다.

### Shared Task List (공유 태스크 목록)
Agent Teams에서 팀원 간 작업을 조율하는 핵심 메커니즘. 팀원이 태스크를 자율적으로 클레임(Claim)하고 완료 처리한다. 파일 락(File Lock)으로 동시 클레임 시 경쟁 조건을 방지하며, 태스크 의존성도 자동으로 관리된다.

### Context Isolation (컨텍스트 격리)
각 에이전트가 독립적인 컨텍스트 윈도우를 가지며, 불필요한 중간 과정이 메인 세션을 오염시키지 않도록 하는 원칙. Subagent와 Agent Teams 모두 이를 통해 메인 세션의 토큰 효율을 유지하고 컨텍스트 윈도우 한계 도달을 늦춘다.

### Model Tiering (모델 티어링)
팀 리드에는 고성능 모델(Opus), 팀원이나 서브에이전트에는 중간 모델(Sonnet)을 배정하여 비용 효율성과 품질을 동시에 확보하는 전략. Anthropic 연구에서 Opus 리드 + Sonnet 서브에이전트 조합이 단일 Opus 모델 대비 90.2% 높은 성능을 기록했다.

### Anchoring Bias (고착 편향)
단일 에이전트가 처음 발견한 그럴듯한 설명에 고착되어 이후 탐색을 멈추는 경향. Agent Teams에서 경쟁 가설을 가진 여러 에이전트가 서로를 반증하려 함으로써 이 편향을 극복하고 더 정확한 원인 분석이 가능해진다.
