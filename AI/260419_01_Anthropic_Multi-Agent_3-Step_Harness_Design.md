# Anthropic Multi-Agent 3-Step Harness Design

## 개요
Anthropic이 제안하는 멀티 에이전트 시스템에서의 3단계 하네스 설계 패턴을 학습한다. 단일 에이전트가 아닌 복수의 에이전트가 협업할 때 하네스를 어떻게 구성해야 신뢰성과 품질을 확보할 수 있는지, Anthropic의 공식 가이드라인과 실전 설계 원칙을 정리한다.

## 상세 내용

### 1. Anthropic의 Multi-Agent 아키텍처 개요

Anthropic은 LLM 단독 호출을 넘어, 여러 에이전트가 역할을 분담하여 복잡한 작업을 처리하는 멀티 에이전트 아키텍처를 권장한다. 단일 에이전트와 멀티 에이전트 사이에는 명확한 트레이드오프가 존재한다.

**단일 에이전트 vs 멀티 에이전트 트레이드오프**

| 항목 | 단일 에이전트 | 멀티 에이전트 |
|------|-------------|-------------|
| 비용 | 낮음 | 높음 |
| 속도 | 빠름 | 느림 |
| 컨텍스트 관리 | 단순 | 복잡 |
| 작업 품질 | 단순 작업에 충분 | 복잡한 장기 작업에 우수 |
| 자기 평가 편향 | 높음 | 독립 평가로 완화 가능 |

실제 게임 빌딩 프로젝트에서 단일 에이전트(20분, $9)는 핵심 기능이 작동하지 않았으나, 완전한 하네스 기반 멀티 에이전트(6시간, $200)는 실제로 플레이 가능한 결과물을 만들어냈다. 복잡도와 품질 요건에 따라 아키텍처 선택이 달라진다.

**Anthropic이 정의하는 5가지 기본 조합 패턴**
- **Prompt Chaining**: 각 단계가 이전 단계 결과를 기반으로 순차 처리
- **Routing**: 작업 유형에 따라 적절한 에이전트나 모델로 분기
- **Parallelization**: 독립적인 서브태스크를 병렬 실행
- **Orchestrator-Workers**: 오케스트레이터가 동적으로 태스크를 분해하고 워커에게 위임
- **Evaluator-Optimizer**: 피드백 루프를 통한 반복적 품질 개선

### 2. 3-Step Harness Design 패턴

Anthropic이 장기 실행 코딩 작업을 위해 실제 구현한 하네스는 **Planner - Generator - Evaluator** 3단계로 구성된다. GAN(생성 적대 네트워크)에서 영감을 받은 구조이며, 각 에이전트는 명확히 분리된 역할을 수행한다.

**Step 1: Planner (계획 단계)**

Planner 에이전트는 짧은 사용자 프롬프트(1~4문장)를 전체 제품 스펙으로 확장하는 역할을 담당한다.

- 제품 컨텍스트와 고수준 기술 설계에 집중
- 세부 구현이 아닌 전달 가능한 결과물 정의에 제약
- AI 기능을 자연스럽게 통합할 기회 탐색
- 스프린트 단위로 작업을 분해하고 에이전트에 할당
- "스프린트 계약(Sprint Contract)" 문서 작성: 각 스프린트의 범위와 검증 기준 사전 합의

**Step 2: Generator (실행 단계)**

Generator 에이전트는 Planner가 정의한 스펙에 따라 실제 구현을 수행한다.

- 스프린트 단위로 하나의 기능씩 구현 (한 번에 모든 것을 구현하려 하지 않음)
- 각 스프린트 완료 후 자체 검토 후 Evaluator로 핸드오프
- Git을 상태 복구 메커니즘으로 활용 (설명적인 커밋 메시지로 진행 상황 기록)
- 컨텍스트 윈도우 고갈 전에 진행 상황 문서화 (예: `claude-progress.txt`)

**Step 3: Evaluator (검증 단계)**

Evaluator 에이전트는 Generator의 결과물을 독립적으로 검증한다.

- Playwright MCP 등 브라우저 자동화 도구로 실행 중인 애플리케이션 직접 테스트
- UI 기능, API 엔드포인트, 데이터베이스 상태를 코드가 아닌 실제 사용자 관점에서 검증
- "스프린트 계약" 기준으로 성공 여부 판정
- 구체적인 버그와 개선 사항을 피드백으로 Generator에 전달
- 자동 재시도 횟수 제한 (Stop Condition 적용)

**독립 평가의 중요성**

에이전트는 자신이 만든 결과물을 평가할 때 과도하게 긍정적으로 평가하는 자기 편향(Self-Evaluation Bias)이 있다. 독립된 Evaluator를 분리하는 것만으로도 이를 크게 완화할 수 있으며, Evaluator 프롬프트 튜닝이 Generator의 자기비판 프롬프트보다 훨씬 실질적인 효과를 낸다.

### 3. 에이전트 간 하네스 설계 원칙

**컨텍스트 격리와 공유의 균형**

에이전트 간 컨텍스트 전달은 파일 기반 핸드오프 아티팩트(Handoff Artifact)로 구현한다. 한 에이전트가 구조화된 파일을 작성하면, 다음 에이전트가 이를 읽고 작업을 이어간다. 전체 컨텍스트를 그대로 넘기는 대신, 다음 에이전트가 작업을 시작하는 데 필요한 최소한의 구조화된 정보만 전달한다.

- 기능 목록을 JSON 형식으로 관리하여 완료/미완료 상태 추적
- `claude-progress.txt` 파일로 세션 간 진행 상황 유지
- `init.sh` 스크립트로 새 에이전트 세션 시작 시 환경 초기화 비용 제거

**역할 제약 (Role Constraint)**

각 에이전트의 역할과 책임 범위를 명확히 정의한다. 툴 정의 자체가 행동 가드레일로 기능하며, 도구 설계 시 사용 예시, 엣지 케이스, 입력 형식 요건, 다른 도구와의 경계를 명확히 문서화해야 한다.

**Stop Condition과 Handoff 프로토콜**

무한 재시도 루프를 방지하기 위해 반드시 최대 반복 횟수 등의 종료 조건을 설정한다. Evaluator가 n회 이상 실패 판정을 내리면 사람에게 에스컬레이션하거나 작업을 중단하는 구조가 필요하다.

**Context Anxiety 문제 대응**

모델이 컨텍스트 윈도우 한계에 근접할수록 작업을 조기에 완료 처리하려는 경향(Context Anxiety)이 있다. 이를 해결하기 위한 전략은 다음과 같다:
- 컨텍스트 리셋: 윈도우 초기화 후 새 에이전트 세션 시작 (구 모델에서 필수)
- Opus 4.5 이상에서는 이 동작이 대부분 해결되어 연속 세션 운영 가능

### 4. 실전 적용: Claude Code의 서브에이전트 구조

Claude Code는 메인 에이전트와 서브에이전트의 관계를 실제로 구현한 환경이다.

**메인 에이전트 - 서브에이전트 관계**

메인 에이전트(오케스트레이터)는 사용자의 요청을 분석하고 전략을 수립한 뒤 서브에이전트를 생성하여 세부 작업을 위임한다. 서브에이전트는 독립적인 컨텍스트 창을 가지며, 결과를 메인 에이전트에게 반환한다.

**파일 기반 상태 전달 패턴**

에이전트 간 상태 전달의 실용적인 방법으로 `/tmp` 파일 활용이 일반적이다. 예를 들어, 1단계 에이전트가 생성한 파일명을 `/tmp/til-created-file.txt`에 저장하고, 2단계 에이전트가 이를 읽어 이어서 작업한다. 이 패턴은:

- 에이전트 간 직접 통신 없이 상태 전달 가능
- 에이전트 실패 시 중간 상태 복구 가능
- 각 에이전트의 독립성 보장

**모델 티어링 전략 (Model Tiering)**

작업 복잡도에 따라 모델을 달리 배정하여 비용 효율성과 품질을 동시에 확보한다.

- 계획(Planner) 단계: 고성능 모델(예: Opus) - 전략적 사고와 스펙 작성
- 실행(Generator) 단계: 중간 모델(예: Sonnet) - 반복적인 코드 생성
- 검증(Evaluator) 단계: 중간 또는 전용 모델 - 품질 기준 적용
- 라우팅(Routing) 패턴: 쉬운 질문은 Haiku, 어려운 질문은 Sonnet으로 분기

### 5. 하네스 설계 시 흔한 실수와 해결책

**과도한 컨텍스트 전달로 인한 토큰 낭비**

이전 에이전트의 전체 컨텍스트를 그대로 넘기면 토큰 비용이 기하급수적으로 증가한다. 해결책은 핵심 정보만 구조화된 핸드오프 아티팩트로 요약하여 전달하는 것이다. "다음 에이전트가 작업을 시작하는 데 필요한 최소한의 정보"를 원칙으로 삼아야 한다.

**에이전트 간 책임 경계 모호**

각 에이전트가 서로의 역할을 침범하거나 중복 작업을 수행하면 하네스 전체의 비용이 증가하고 일관성이 깨진다. 툴 정의와 시스템 프롬프트에서 역할 경계를 명확히 문서화해야 한다.

**검증 단계 누락으로 인한 품질 저하**

에이전트는 자신의 결과물을 실행하지 않고 완료로 표시하는 경향이 있다. Evaluator가 실제 브라우저 자동화(Playwright, Puppeteer 등)를 통해 사용자 관점에서 검증하는 단계를 반드시 포함해야 한다.

**재시도 루프의 무한 반복 방지**

Evaluator가 계속 실패를 반환하면 Generator-Evaluator 루프가 무한히 반복될 수 있다. 최대 반복 횟수를 설정하고, 한계 도달 시 사람에게 에스컬레이션하거나 작업 범위를 줄이는 Stop Condition이 필요하다.

**하네스 구성 요소의 주기적 재평가**

하네스의 각 구성 요소는 "모델이 스스로 할 수 없는 것"에 대한 가정을 인코딩한다. 모델 성능이 향상되면 일부 스캐폴딩이 불필요해질 수 있다. Anthropic도 Opus 4.6 출시 이후 스프린트 구조를 제거하고 Evaluator를 조건부로 적용하는 방향으로 하네스를 단순화했다. 주기적으로 각 구성 요소가 여전히 필요한지 재평가해야 한다.

## 핵심 정리
- 멀티 에이전트 하네스는 Planner-Generator-Evaluator 3단계로 구성하여 역할을 명확히 분리
- 독립적인 Evaluator는 자기 평가 편향을 제거하고 시스템 전체 품질을 결정하는 핵심 요소
- 에이전트 간 상태 전달은 파일 기반 핸드오프 아티팩트로 최소한의 구조화된 정보만 전달
- 모델 티어링(Opus + Sonnet 조합)으로 비용 효율성과 품질을 동시에 확보
- Stop Condition 없는 하네스는 무한 루프와 비용 폭발의 위험이 있음
- 모델 성능 향상에 따라 하네스 복잡성을 주기적으로 재평가하고 단순화해야 함

## 키워드

### Multi-Agent System (멀티 에이전트 시스템)
단일 LLM 호출이 아닌, 여러 에이전트가 역할을 분담하여 협업하는 아키텍처. 복잡한 장기 실행 작업에서 단일 에이전트보다 높은 품질의 결과물을 만들어내지만, 비용과 복잡성이 증가한다.

### Harness Design (하네스 설계)
멀티 에이전트 시스템에서 에이전트의 실행, 컨텍스트 관리, 핸드오프, 검증을 통제하는 외부 스캐폴딩 구조. 하네스의 품질이 멀티 에이전트 시스템 전체의 신뢰성을 결정한다.

### Orchestrator-Worker Pattern (오케스트레이터-워커 패턴)
중앙 오케스트레이터 LLM이 작업을 동적으로 분해하고 워커 에이전트에게 위임하는 패턴. 서브태스크를 사전에 예측할 수 없는 복잡한 작업에 적합하다.

### Planning-Execution-Verification (계획-실행-검증)
Anthropic의 3단계 하네스 설계의 핵심 흐름. Planner가 작업을 정의하고, Generator가 구현하며, Evaluator가 독립적으로 검증하는 구조로 품질을 보장한다.

### Context Isolation (컨텍스트 격리)
각 에이전트가 독립적인 컨텍스트 창을 가지며, 에이전트 간 전달 정보를 최소화하는 원칙. 토큰 비용 절감과 에이전트 간 오염 방지를 위해 필수적이다.

### Agent Handoff (에이전트 핸드오프)
한 에이전트의 결과를 다음 에이전트에게 전달하는 메커니즘. 파일 기반 핸드오프 아티팩트(구조화된 JSON, 진행 상황 문서 등)를 통해 구현하며, 전체 컨텍스트 대신 최소한의 구조화된 정보만 전달한다.

### Model Tiering (모델 티어링)
작업 복잡도에 따라 다른 성능과 비용의 모델을 배정하는 전략. Planner에는 고성능 모델, Generator/Evaluator에는 중간 모델을 사용하여 비용 효율성과 품질을 동시에 확보한다.

### Sub-Agent (서브 에이전트)
오케스트레이터로부터 특정 작업을 위임받아 독립적인 컨텍스트 창에서 실행되는 에이전트. 결과를 오케스트레이터에게 반환하며, Claude Code에서는 Task 도구를 통해 생성된다.

### Stop Condition (종료 조건)
에이전트 루프가 무한히 반복되는 것을 방지하기 위한 최대 반복 횟수 또는 시간 제한. 한계 도달 시 사람에게 에스컬레이션하거나 작업을 종료하는 구조가 반드시 포함되어야 한다.

### Role Constraint (역할 제약)
각 에이전트의 행동 범위를 제한하여 책임 경계를 명확히 하는 원칙. 툴 정의와 시스템 프롬프트에서 역할 경계를 문서화하며, 에이전트 간 역할 중복과 책임 모호성을 방지한다.
