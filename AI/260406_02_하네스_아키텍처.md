# 하네스 아키텍처 (Harness Architecture)

## 개요
AI 시스템에서 하네스(Harness)란 LLM 또는 AI 모델을 감싸고, 입력 전처리 / 출력 후처리 / 도구 호출 / 오류 복구 등을 담당하는 오케스트레이션 계층을 말한다. 에이전트 프레임워크의 핵심 구성 요소이며, 모델 자체와 실행 환경 사이의 중간 계층 역할을 한다.

`Agent = Model + Harness` 공식으로 요약할 수 있으며, 하네스는 모델 자체를 제외한 "모든 코드, 설정, 실행 로직"을 포함한다. 모델의 지능을 실제 업무에 유용하게 만드는 시스템이다.

## 상세 내용

### 하네스의 정의와 역할
- AI 모델을 "테스트 하네스"처럼 감싸는 래퍼 계층
- 프롬프트 구성, 도구 디스패치, 메모리 관리, 안전 장치 등을 통합 관리
- 모델이 기본적으로 할 수 없는 작업(상태 유지, 코드 실행, 실시간 지식 접근)을 하네스 수준에서 해결
- 동일한 모델이라도 잘 설계된 하네스와 함께 동작하면 성능이 크게 향상됨

### 하네스가 등장한 이유
기존 LLM은 다음과 같은 한계를 가졌다.
- **제한된 메모리**: 고정된 컨텍스트 윈도우, 세션 간 지속 기억 불가
- **외부 도구 부재**: 웹 검색, 코드 실행 등 독립적 행동 불가
- **구조화되지 않은 워크플로우**: 다단계 계획 및 검증 메커니즘 부재
- **세션 불연속성**: 별도 상호작용에서 진행 상황 유지 불가

이를 해결하기 위해 하네스 계층이 모델 외부에서 이 모든 역할을 담당하게 되었다.

### 하네스의 핵심 구성 요소

| 구성 요소 | 역할 |
|---|---|
| **Prompt Builder** | 시스템 프롬프트 + 사용자 입력 + 컨텍스트 조합 |
| **Tool Router** | 모델의 도구 호출 요청을 실제 도구로 라우팅 |
| **Memory / State Manager** | 대화 히스토리, 작업 상태 유지 |
| **Safety Layer** | 출력 필터링, 권한 제어, 위험 행동 차단 |
| **Retry / Error Handler** | 실패 시 재시도 전략, 폴백 처리 |
| **Sandbox / Execution Env** | 안전하게 격리된 코드 실행 환경 |
| **Context Manager** | 컨텍스트 압축, 오프로딩, 선택적 노출 |

### 메모리의 3계층 구조
하네스에서 메모리는 세 계층으로 나뉘어 관리된다.

```
┌──────────────────────────────┐
│    컨텍스트 윈도우 (RAM)         │  ← 현재 작업 중 활성 정보
│    세션 상태 (파일시스템)         │  ← 작업 진행 상태, 중간 결과
│    장기 메모리 (DB / 벡터 DB)    │  ← 과거 작업, 학습된 지식
└──────────────────────────────┘
```

파일시스템이 장기 상태 관리의 기초로 활용되며, 벡터 데이터베이스보다 전통적인 파일 시스템 기반 접근이 실용적으로 채택되는 추세이다.

### 에이전트 루프와 하네스의 관계
에이전트 루프(ReAct 패턴)의 각 단계를 하네스가 제어한다.

```
현재 상태 수신
      │
      ▼
  [추론: 다음 행동 결정]   ← LLM
      │
      ▼
  [도구 실행]             ← Harness (Tool Router)
      │
      ▼
  [결과 관찰]             ← Harness (Memory/State Manager)
      │
      ▼
  목표 달성? ──Yes──▶ 종료
      │
      No
      │
      ▼
    (반복)
```

복잡한 작업의 경우 **오케스트레이터-워커 패턴**으로 확장되어, 상위 에이전트가 전문화된 하위 에이전트들에게 작업을 위임한다.

### "100번째 도구 호출 문제"
복잡한 멀티 스텝 태스크에서 수십~수백 번의 도구 호출이 발생하면 컨텍스트 윈도우가 가득 차 핵심 정보가 소실된다. 이를 **컨텍스트 부패(Context Degradation)** 라고 한다.

하네스는 이 문제를 다음 방법으로 해결한다.
- 컨텍스트 압축 (오래된 내용 요약)
- 도구 출력 오프로딩 (파일시스템에 저장 후 필요 시 참조)
- 점진적 정보 공개 (현재 단계에 필요한 정보만 컨텍스트에 포함)

### 대표적인 하네스 구현체
- **LangChain DeepAgents**: 계획 도구, 파일시스템 백엔드, 서브에이전트 오케스트레이션을 포함한 배터리 포함형 하네스 (2025년 말 출시)
- **LangGraph**: 그래프 기반 아키텍처로 흐름, 재시도, 오류 처리를 세밀하게 제어 (현재 LangChain 프로덕션 권장)
- **Claude Code의 에이전트 루프**: 파일 읽기/쓰기, 도구 실행, 서브에이전트 위임을 포함하는 하네스
- **AutoGPT / CrewAI의 태스크 러너**
- **OpenAI Agents SDK**

### 하네스의 3계층 아키텍처 (Harrison Chase 분류)
```
┌─────────────────────────────────┐
│   오피니언 인프라 (Harness)         │  ← DeepAgents, Claude Code 등
├─────────────────────────────────┤
│   실행 엔진 (Runtime)             │  ← LangGraph, Temporal, Trigger.dev
├─────────────────────────────────┤
│   빌딩 블록 (Framework)           │  ← LangChain, LlamaIndex 등
└─────────────────────────────────┘
```

### 하네스 설계 시 고려사항
- **모델 독립성**: 하네스는 특정 모델에 종속되지 않아야 함
- **관측 가능성(Observability)**: 각 단계의 로깅과 트레이싱 (LangSmith, Datadog 등)
- **확장성**: 새로운 도구나 정책을 쉽게 추가할 수 있는 구조
- **비용 제어**: 토큰 사용량 추적 및 제한
- **Durable Execution**: 장애 발생 시 중단 지점부터 재개하는 내구성 있는 실행
- **피드백 루프**: 자체 검증 루프로 출력 품질을 모델 재학습 없이 향상

### 2026년 하네스 트렌드
- 멀티 에이전트 워크플로우가 2025년 6~10월 사이 327% 성장 (Databricks 보고서)
- "모델 지능" 경쟁에서 "하네스 인프라" 경쟁으로 전환
- MCP(Model Context Protocol)와 하네스 통합이 표준화 추세
- 관측 가능성(Observability) 도구의 중요성 급증

## 핵심 정리
- 하네스는 AI 모델과 실행 환경 사이의 오케스트레이션 계층이다
- `Agent = Model + Harness` 로 에이전트를 정의할 수 있다
- 프롬프트 구성, 도구 라우팅, 상태 관리, 안전 장치를 통합 관리한다
- 에이전트 루프(ReAct: 추론-행동-관찰)의 실행 주체가 하네스이다
- 컨텍스트 부패 문제를 해결하는 컨텍스트 엔지니어링이 핵심 역할이다
- 모델 독립적이고 관측 가능하며 내구성 있는 설계가 중요하다

## 키워드

### Harness Architecture (하네스 아키텍처)
LLM 모델을 감싸는 오케스트레이션 래퍼 계층. 모델 자체를 제외한 모든 실행 인프라(프롬프트 빌딩, 도구 라우팅, 메모리, 안전 장치 등)를 포함하며, `Agent = Model + Harness`로 정의된다.

### Agent Loop (에이전트 루프)
에이전트가 목표를 달성할 때까지 반복하는 사이클. 일반적으로 **ReAct 패턴**(Reason → Act → Observe)을 따르며, 하네스가 루프의 제어권을 갖는다.

### Orchestration Layer (오케스트레이션 계층)
여러 컴포넌트(LLM, 도구, 메모리, 서브에이전트 등)의 실행 순서와 데이터 흐름을 조율하는 계층. LangGraph, Temporal 등이 대표적인 오케스트레이션 엔진이다.

### Tool Routing (도구 라우팅)
모델이 생성한 도구 호출 요청을 실제 도구(API, DB, 코드 실행 환경 등)로 매핑하여 실행하는 메커니즘.

### Prompt Builder (프롬프트 빌더)
시스템 프롬프트, 사용자 입력, 컨텍스트 히스토리, 도구 명세 등을 조합하여 최종 프롬프트를 구성하는 컴포넌트.

### State Management (상태 관리)
에이전트가 멀티 턴 대화나 장기 작업을 수행할 때 진행 상태, 중간 결과, 대화 히스토리를 유지하는 메커니즘. 파일시스템, DB, 인메모리 캐시 등 다양한 백엔드를 활용한다.

### Safety Layer (안전 계층)
모델 출력의 유해성 필터링, 권한 제어, 위험한 도구 호출 차단 등을 담당하는 보안 계층.

### Observability (관측 가능성)
에이전트의 각 실행 단계(LLM 호출, 도구 호출, 상태 변경 등)를 로깅하고 트레이싱하는 능력. LangSmith, Datadog, OpenTelemetry 등이 대표 도구이며, 에이전트 디버깅과 성능 개선의 기반이 된다.

### LLM Wrapper (LLM 래퍼)
LLM API 호출을 추상화하여 모델 교체, 재시도, 토큰 사용량 추적 등을 일관되게 처리하는 컴포넌트.

### Agent Framework (에이전트 프레임워크)
에이전트를 구축하기 위한 빌딩 블록을 제공하는 라이브러리/프레임워크. LangChain, LlamaIndex, AutoGen 등이 있으며, 하네스 계층의 기반이 된다.
