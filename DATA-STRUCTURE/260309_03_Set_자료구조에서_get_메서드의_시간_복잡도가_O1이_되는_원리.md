# Set 자료구조에서 get() 메서드의 시간 복잡도가 O(1)이 되는 원리

## 개요
Set(HashSet) 자료구조에서 요소를 조회할 때 O(1)의 시간 복잡도를 달성하는 내부 동작 원리를 학습한다.

## 상세 내용

### 1. Hash Function의 역할

#### Hash Function이란
Hash Function(해시 함수)은 임의의 길이를 가진 데이터를 고정된 길이의 정수값(해시값)으로 변환하는 함수다. 좋은 해시 함수는 다음 조건을 만족해야 한다.

- **결정성**: 동일한 입력에 대해 항상 동일한 출력을 반환
- **균일한 분포**: 해시값이 가능한 범위에 고르게 분포
- **빠른 연산**: 해시값 계산 자체가 O(1)에 수행

#### 객체의 hashCode()를 통한 해시값 생성
Java에서는 모든 객체가 `Object` 클래스로부터 `hashCode()` 메서드를 상속받는다. 이 메서드가 해시 함수의 역할을 수행한다.

```java
// String의 hashCode() 예시
String key = "hello";
int hash = key.hashCode(); // 예: 99162322
```

Java의 `HashMap`은 `hashCode()`가 반환한 값에 추가적인 보조 해시 함수를 적용한다. 이는 품질이 낮은 `hashCode()` 구현으로 인한 충돌을 완화하기 위해서다.

```java
// Java 8 HashMap 내부의 보조 해시 함수
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

상위 16비트를 하위 16비트와 XOR하여 해시값 분포를 개선한다.

#### 해시값을 배열 인덱스로 변환하는 과정 (modular 연산)
해시값은 일반적으로 매우 큰 정수이므로, 배열의 유효한 인덱스로 변환하는 과정이 필요하다.

```java
// 배열 크기(n)가 항상 2의 거듭제곱이기 때문에 비트 AND 연산으로 modular 처리
int index = (n - 1) & hash;
// 이는 hash % n 과 동일하지만, 2의 거듭제곱일 때 비트 연산이 훨씬 빠름
```

예를 들어 배열 크기가 16(= 2^4)이면 `(16 - 1) & hash = 0b1111 & hash`로, 해시값의 하위 4비트만 인덱스로 사용한다.

### 2. 내부 구조: 배열 기반 버킷

#### HashSet의 내부 구조 (HashMap 기반)
`HashSet`은 내부적으로 `HashMap`을 사용한다. 요소를 `HashMap`의 **키(Key)**로 저장하고, 값(Value)에는 고정된 더미 객체(`PRESENT`)를 넣는다.

```java
// HashSet 내부 구현 (JDK 소스 요약)
public class HashSet<E> extends AbstractSet<E> {
    private transient HashMap<E, Object> map;
    private static final Object PRESENT = new Object(); // 더미 값

    public boolean add(E e) {
        return map.put(e, PRESENT) == null;
    }

    public boolean contains(Object o) {
        return map.containsKey(o);
    }
}
```

따라서 `HashSet`의 성능은 곧 `HashMap`의 성능이다.

#### 버킷(Bucket) 배열과 인덱스 매핑
`HashMap`의 핵심은 `Node<K,V>[] table`이라는 배열이다. 각 배열 슬롯을 **버킷(Bucket)**이라고 부른다.

```
table 배열:
index 0: [ Node(key="apple") ]
index 1: null
index 2: [ Node(key="banana") -> Node(key="cherry") ]  // 충돌 발생
index 3: null
...
index 15: [ Node(key="mango") ]
```

요소 삽입/조회 시 해시값으로 버킷 인덱스를 즉시 계산하므로, 배열 전체를 탐색할 필요가 없다.

#### 배열 인덱스 접근이 O(1)인 이유 (메모리 주소 직접 계산)
배열은 **연속된 메모리 공간**을 사용하기 때문에, 인덱스만 알면 메모리 주소를 직접 계산할 수 있다.

```
메모리 주소 = 배열 시작 주소 + (인덱스 × 요소 크기)
```

이 계산은 덧셈과 곱셈으로만 이루어지므로 O(1)이다. 즉, 인덱스가 0이든 100만이든 동일한 연산 횟수로 접근한다.

### 3. Hash Collision (해시 충돌)

#### 해시 충돌이 발생하는 원인
해시 충돌은 서로 다른 두 키가 같은 배열 인덱스에 매핑될 때 발생한다.

- **비둘기집 원리**: 키의 가능한 경우의 수(무한)가 배열 크기(유한)보다 크므로 충돌은 필연적으로 발생
- **나쁜 hashCode() 구현**: 해시값이 고르게 분포되지 않을 경우 특정 버킷에 집중

```java
// 극단적으로 나쁜 hashCode() 예시
@Override
public int hashCode() {
    return 1; // 모든 객체가 같은 버킷에 몰림 -> O(n) 성능
}
```

#### Separate Chaining 방식 (LinkedList → Red-Black Tree)
Java `HashMap`은 **Separate Chaining** 방식으로 충돌을 처리한다. 같은 버킷에 여러 항목이 들어오면 연결하여 저장한다.

**Java 7 이하**: 단순 LinkedList로 연결
```
bucket[2]: Node("banana") -> Node("cherry") -> Node("date") -> null
```

**Java 8 이상**: 일정 임계값 초과 시 Red-Black Tree로 전환
```
bucket[2]: TreeNode 구조 (균형 이진 탐색 트리)
```

#### Java 8 이후 Treeify 임계값 (TREEIFY_THRESHOLD = 8)

| 상수명 | 값 | 설명 |
|---|---|---|
| `TREEIFY_THRESHOLD` | 8 | 버킷의 노드 수가 이 값 초과 시 LinkedList → Red-Black Tree 전환 |
| `UNTREEIFY_THRESHOLD` | 6 | 노드 수가 이 값 이하로 줄면 Red-Black Tree → LinkedList 전환 |
| `MIN_TREEIFY_CAPACITY` | 64 | Treeify가 일어나려면 전체 테이블 용량이 이 값 이상이어야 함 |

임계값이 6과 8로 다른 이유는 **hysteresis(이력 현상)** 때문이다. 노드 추가/삭제가 반복될 때 8 근처에서 자료구조가 계속 변환되는 것을 방지한다.

#### 충돌 시 시간 복잡도 변화

| 상황 | 시간 복잡도 |
|---|---|
| 충돌 없음 (이상적) | O(1) |
| Separate Chaining + LinkedList (최악) | O(n) |
| Separate Chaining + Red-Black Tree (Java 8+) | O(log n) |

### 4. equals()와 hashCode()의 관계

#### hashCode() 계약: equals()가 true이면 hashCode()도 같아야 한다
Java 명세(`Object.hashCode()` Javadoc)는 다음 계약을 정의한다.

> "If two objects are equal according to the equals(Object) method, then calling the hashCode method on each of the two objects must produce the same integer result."

이 계약이 위반되면 `HashMap`/`HashSet`이 올바르게 동작하지 않는다.

```java
// 계약 위반 예시
class BrokenKey {
    int id;

    @Override
    public boolean equals(Object o) {
        return this.id == ((BrokenKey) o).id; // equals 재정의
        // hashCode는 재정의하지 않음 (기본값: 메모리 주소 기반)
    }
}

BrokenKey k1 = new BrokenKey(1);
BrokenKey k2 = new BrokenKey(1);

Set<BrokenKey> set = new HashSet<>();
set.add(k1);
set.contains(k2); // false! k1.equals(k2)는 true지만 hashCode가 달라 다른 버킷을 찾음
```

#### hashCode()가 같아도 equals()가 다를 수 있다 (충돌)
이것이 바로 해시 충돌이다. 동일한 해시값을 가지더라도 실제로는 다른 객체일 수 있으므로, **버킷 내에서 equals()로 최종 동등성을 확인**한다.

```
조회 과정:
1. hashCode()로 버킷 인덱스 계산 -> O(1)
2. 해당 버킷의 노드들을 순회하며 equals()로 동등성 확인 -> O(1) (충돌 없을 때)
```

#### hashCode()와 equals()를 함께 재정의해야 하는 이유
두 메서드는 항상 쌍으로 재정의해야 한다.

```java
// 올바른 재정의 예시
class Member {
    Long id;
    String name;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Member)) return false;
        Member m = (Member) o;
        return Objects.equals(id, m.id);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id); // equals에서 사용하는 필드와 동일하게
    }
}
```

**규칙 요약**:
- `equals()` true → `hashCode()` 반드시 동일 (필수)
- `hashCode()` 동일 → `equals()` true일 수도, 아닐 수도 있음 (해시 충돌)
- `hashCode()` 다름 → `equals()` 반드시 false

### 5. Load Factor와 Rehashing

#### Load Factor (기본값 0.75)의 의미
Load Factor는 해시 테이블이 얼마나 채워졌을 때 크기를 확장할지 결정하는 임계 비율이다.

```
threshold = capacity × load factor
기본값: 16 × 0.75 = 12
```

즉, 16개의 버킷을 가진 `HashMap`에 12번째 항목이 추가되면 Rehashing이 발생한다.

- **Load Factor가 낮을수록**: 빈 버킷이 많아 충돌 감소 → 조회 빠름, 메모리 낭비
- **Load Factor가 높을수록**: 충돌 증가 → 조회 느림, 메모리 효율적

0.75는 시간과 공간의 트레이드오프에서 실험적으로 검증된 최적값이다.

#### 임계값 초과 시 Rehashing (배열 크기 2배 확장)
Rehashing은 다음 단계로 진행된다.

```
1. 새 배열(크기 = 기존 × 2) 생성
2. 기존 모든 항목에 대해 새 배열에서의 인덱스 재계산
3. 재계산된 인덱스에 항목 재배치

예시:
- 기존: capacity=16, threshold=12
- Rehashing 후: capacity=32, threshold=24
```

배열 크기가 항상 2의 거듭제곱이기 때문에 인덱스 재계산 시 비트 연산(`(n-1) & hash`)을 사용할 수 있어 효율적이다.

#### Rehashing이 amortized O(1)을 보장하는 원리
Rehashing 자체는 O(n) 작업이다. 하지만 Rehashing이 발생하기 전까지 최소 `n/2`번의 삽입이 보장되므로, 각 삽입의 **분할 상환 비용(amortized cost)**은 O(1)이다.

```
분석:
- n개 항목 삽입 시 Rehashing 횟수: log₂(n)번
- 각 Rehashing의 비용: O(capacity) = O(n)
- 전체 비용: O(n log n)이 아니라...

실제로는:
- 마지막 Rehashing 비용: O(n)
- 그 전 Rehashing 비용: O(n/2)
- 그 전전: O(n/4) ...
- 합계: O(n + n/2 + n/4 + ...) = O(2n) = O(n)
- 삽입 1회당 평균: O(n) / n = O(1) ✓
```

### 6. Java HashSet vs 다른 언어의 Set 구현 비교

#### Java HashSet (HashMap 기반)

| 항목 | 내용 |
|---|---|
| 충돌 처리 | Separate Chaining (LinkedList + Red-Black Tree) |
| 기본 용량 | 16 |
| Load Factor | 0.75 |
| 조회 성능 | 평균 O(1), 최악 O(log n) (Java 8+) |
| 특징 | `HashMap<E, PRESENT>`를 그대로 활용 |

```java
HashSet<String> set = new HashSet<>();
set.add("hello");
set.contains("hello"); // true, O(1)
```

#### Python set (Open Addressing)
CPython의 `set`은 **Open Addressing** 방식을 사용한다. 충돌 발생 시 다른 빈 슬롯을 탐사(probe)하여 항목을 저장한다. Chaining처럼 링크드 리스트를 만들지 않고, 배열 내 다른 위치를 찾는다.

| 항목 | 내용 |
|---|---|
| 충돌 처리 | Open Addressing (Linear Probing + Random Probing 조합) |
| 초기 용량 | 8 (내부 smalltable) |
| Load Factor | 약 2/3 (기본) |
| 조회 성능 | 평균 O(1) |
| 특징 | 캐시 친화적 (배열 연속 접근), 삭제 시 tombstone 마킹 |

```python
s = set()
s.add("hello")
"hello" in s  # True, O(1)
```

**Open Addressing vs Separate Chaining**:
- Open Addressing: 메모리 연속성 덕분에 CPU 캐시 효율이 높음, 하지만 삭제가 복잡(tombstone 필요)
- Separate Chaining: 삭제가 단순, 하지만 포인터 추적으로 캐시 비효율

#### C++ unordered_set (Separate Chaining)

| 항목 | 내용 |
|---|---|
| 충돌 처리 | Separate Chaining (주로 LinkedList) |
| Load Factor | 기본 1.0 |
| 조회 성능 | 평균 O(1), 최악 O(n) |
| 특징 | C++ 표준은 충돌 처리 방식을 명시하지 않으나, Bucket API 설계상 Chaining이 자연스러움 |

```cpp
std::unordered_set<std::string> s;
s.insert("hello");
s.count("hello"); // 1, O(1)
```

#### 언어별 비교 요약

| 언어 | 자료구조 | 충돌 처리 | Load Factor | 최악 복잡도 |
|---|---|---|---|---|
| Java | `HashSet` | Separate Chaining (→ Red-Black Tree) | 0.75 | O(log n) |
| Python | `set` | Open Addressing | ~0.67 | O(n) |
| C++ | `unordered_set` | Separate Chaining | 1.0 | O(n) |

## 핵심 정리
- `HashSet`은 내부적으로 `HashMap`을 사용하며, 해시 함수를 통해 배열 인덱스에 직접 접근하므로 O(1)
- 배열의 인덱스 접근이 O(1)인 이유는 연속된 메모리 주소를 직접 계산할 수 있기 때문
- 해시 충돌이 발생하면 최악의 경우 O(n)이지만, Java 8 이후 Red-Black Tree로 O(log n)까지 개선
- `hashCode()`와 `equals()`의 올바른 재정의가 O(1) 성능의 전제 조건
- Rehashing은 O(n)이지만 분할 상환 분석으로 삽입 1회당 amortized O(1) 보장

## 키워드

| 키워드 | 설명 |
|---|---|
| `HashSet` | `Set` 인터페이스를 구현하며 내부적으로 `HashMap`을 사용하는 컬렉션. 중복 불허, 순서 미보장 |
| `HashMap` | 키-값 쌍을 해시 테이블 기반으로 저장하는 자료구조. `HashSet`의 실제 저장소 역할 |
| `Hash Function` | 임의 데이터를 고정 길이 정수(해시값)로 변환하는 함수. 배열 인덱스 결정에 사용 |
| `hashCode()` | Java `Object`의 메서드. 객체를 해시값(int)으로 변환. `HashMap` 버킷 결정에 사용 |
| `equals()` | 두 객체의 논리적 동등성을 비교하는 메서드. 같은 버킷 내 최종 동등성 검증에 사용 |
| `Hash Collision` | 서로 다른 키가 동일한 해시값(버킷 인덱스)에 매핑되는 현상 |
| `Separate Chaining` | 해시 충돌 시 같은 버킷에 LinkedList나 Tree로 연결하여 저장하는 충돌 해결 방식 |
| `Load Factor` | 해시 테이블의 현재 항목 수 / 버킷 수 비율. 임계값 초과 시 Rehashing 트리거 |
| `Rehashing` | Load Factor 임계값 초과 시 배열 크기를 2배로 늘리고 모든 항목을 재배치하는 작업 |
| `Red-Black Tree` | 자가 균형 이진 탐색 트리. Java 8부터 버킷의 항목이 8개 초과 시 LinkedList 대신 사용하여 O(n) → O(log n) 개선 |
