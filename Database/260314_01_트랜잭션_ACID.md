# 트랜잭션 ACID

## 개요
ACID는 데이터베이스 트랜잭션이 안전하게 수행되기 위해 보장해야 하는 4가지 속성(Atomicity, Consistency, Isolation, Durability)이다. 각 속성의 의미와 RDBMS가 이를 구현하는 메커니즘을 학습한다.

## 상세 내용

### 1. 트랜잭션이란
트랜잭션(Transaction)은 하나의 논리적 작업 단위를 구성하는 연산의 집합이다. 여러 SQL 문을 묶어 하나의 단위로 처리하며, 중간에 실패하면 전체를 취소할 수 있다.

**트랜잭션의 상태**
- **Active**: 트랜잭션이 실행 중인 상태
- **Partially Committed**: 마지막 연산까지 실행했지만 아직 커밋되지 않은 상태
- **Committed**: 트랜잭션이 성공적으로 완료되어 변경이 영구 반영된 상태
- **Failed**: 트랜잭션 실행 중 오류가 발생한 상태
- **Aborted**: 롤백이 완료되어 트랜잭션 이전 상태로 복원된 상태

**COMMIT과 ROLLBACK**
- `COMMIT`: 트랜잭션의 모든 변경 사항을 영구적으로 저장한다.
- `ROLLBACK`: 트랜잭션의 모든 변경 사항을 취소하고 이전 상태로 되돌린다.

```sql
START TRANSACTION;
UPDATE accounts SET balance = balance - 10000 WHERE id = 1;
UPDATE accounts SET balance = balance + 10000 WHERE id = 2;
COMMIT; -- 또는 오류 시 ROLLBACK;
```

---

### 2. Atomicity (원자성)
트랜잭션 내의 모든 연산은 전부 성공하거나 전부 실패해야 한다. 일부만 성공하는 중간 상태는 존재하지 않는다.

**InnoDB의 원자성 보장: Undo Log**

InnoDB는 데이터를 수정하기 전에 **변경 이전 값(Before Image)** 을 Undo Log에 기록한다. 트랜잭션이 실패하거나 `ROLLBACK`이 호출되면 Undo Log를 역순으로 읽어 원래 상태로 복원한다.

```
[변경 전] 잔액: 50,000원
[Undo Log 기록] id=1, balance=50,000
[데이터 변경] 잔액: 40,000원
[실패 발생 → ROLLBACK] Undo Log 적용 → 잔액: 50,000원 복원
```

**Savepoint를 활용한 부분 롤백**

```sql
START TRANSACTION;
INSERT INTO orders (id, item) VALUES (1, 'A');
SAVEPOINT sp1;
INSERT INTO orders (id, item) VALUES (2, 'B');
ROLLBACK TO SAVEPOINT sp1; -- 'B' 삽입만 취소, 'A'는 유지
COMMIT;
```

---

### 3. Consistency (일관성)
트랜잭션이 실행되기 전후로 데이터베이스는 항상 일관된 상태를 유지해야 한다. 미리 정의된 규칙(제약 조건 등)을 위반하는 트랜잭션은 허용되지 않는다.

**데이터베이스 레벨 일관성 (제약 조건)**
- `PRIMARY KEY`: 중복 및 NULL 값 방지
- `FOREIGN KEY`: 참조 무결성 보장 (존재하지 않는 부모 레코드 참조 불가)
- `UNIQUE`: 컬럼 값의 유일성 보장
- `NOT NULL`: NULL 값 삽입 방지
- `CHECK`: 특정 조건을 만족하는 값만 허용 (MySQL 8.0.16+)

제약 조건을 위반하면 InnoDB가 해당 트랜잭션을 자동으로 롤백한다.

```sql
-- FOREIGN KEY 위반 → 자동 롤백
INSERT INTO orders (user_id, item) VALUES (9999, 'A');
-- ERROR 1452: Cannot add or update a child row: a foreign key constraint fails
```

**애플리케이션 레벨 일관성**

비즈니스 규칙(예: "잔액은 0 이상이어야 한다")은 DB 제약 조건만으로 완전히 표현하기 어렵다. 이 경우 애플리케이션 코드에서 검증 로직을 추가해야 한다.

### 4. Isolation (격리성)
동시에 실행되는 트랜잭션들은 서로 영향을 주지 않아야 한다. 각 트랜잭션은 다른 트랜잭션이 없는 것처럼 실행되어야 한다.

**격리성이 없을 때 발생하는 문제**

| 이상 현상 | 설명 |
|---|---|
| **Dirty Read** | 아직 커밋되지 않은 다른 트랜잭션의 변경 데이터를 읽음 |
| **Non-Repeatable Read** | 같은 트랜잭션 내에서 같은 행을 두 번 읽었을 때 값이 다름 (중간에 다른 트랜잭션이 수정·커밋) |
| **Phantom Read** | 같은 트랜잭션 내에서 같은 조건으로 조회했을 때 결과 행 수가 달라짐 (중간에 다른 트랜잭션이 삽입·삭제·커밋) |

**트랜잭션 격리 수준 4단계**

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|:---:|:---:|:---:|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED | 방지 | 발생 | 발생 |
| REPEATABLE READ (InnoDB 기본) | 방지 | 방지 | 부분 방지 |
| SERIALIZABLE | 방지 | 방지 | 방지 |

```sql
-- 격리 수준 확인
SELECT @@transaction_isolation;

-- 격리 수준 변경 (세션 단위)
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

**InnoDB의 격리성 구현: MVCC + Lock**

InnoDB는 **MVCC(Multi-Version Concurrency Control)** 를 통해 읽기 연산에서 잠금 없이 격리성을 구현한다. 각 행에 변경 이력(버전)을 관리하여, 트랜잭션 시작 시점의 스냅샷을 기준으로 데이터를 읽는다.

- **REPEATABLE READ**: 트랜잭션 시작 시점의 스냅샷을 유지하여 같은 쿼리는 항상 같은 결과 반환
- **READ COMMITTED**: 쿼리 실행 시점마다 최신 커밋 스냅샷 사용

Phantom Read는 MVCC만으로 완전히 해결되지 않으므로, InnoDB는 **Gap Lock**(인덱스 레코드 사이 공간 잠금)과 **Next-Key Lock**을 추가로 활용한다.

### 5. Durability (지속성)
커밋된 트랜잭션의 결과는 시스템 장애(전원 차단, 서버 크래시 등)가 발생해도 영구적으로 보존되어야 한다.

**InnoDB의 지속성 보장: Write-Ahead Logging (WAL)**

InnoDB는 **WAL 원칙**에 따라 데이터를 실제 파일에 쓰기 전에 먼저 Redo Log에 기록한다. 커밋 시 Redo Log가 디스크에 flush되므로, 크래시 이후에도 Redo Log를 재생(replay)하여 커밋된 변경 사항을 복원할 수 있다.

```
[쓰기 순서]
1. 변경 사항 → Redo Log Buffer
2. COMMIT → Redo Log Buffer를 디스크에 flush (fsync)
3. 이후 Buffer Pool의 Dirty Page가 실제 데이터 파일에 비동기 기록
```

**`innodb_flush_log_at_trx_commit` 설정**

| 값 | 동작 | 내구성 | 성능 |
|---|---|:---:|:---:|
| `0` | 약 1초마다 로그 버퍼를 OS 버퍼에 쓰고 flush | 낮음 | 높음 |
| `1` (기본) | 매 커밋 시 디스크에 flush (fsync) | 높음 | 낮음 |
| `2` | 매 커밋 시 OS 버퍼에 쓰되, flush는 약 1초마다 | 중간 | 중간 |

완전한 ACID 보장을 위해서는 기본값 `1`을 사용해야 한다.

**Doublewrite Buffer**

InnoDB는 데이터 페이지를 실제 데이터 파일에 쓰기 전 Doublewrite Buffer에 먼저 기록한다. 쓰기 도중 크래시가 발생해도 Doublewrite Buffer에서 완전한 페이지를 복구할 수 있다(Torn Page 방지).

**Crash Recovery 과정**

1. 서버 재시작 시 InnoDB가 Redo Log를 스캔
2. 마지막 체크포인트 이후의 변경 사항을 데이터 파일에 재적용
3. 커밋되지 않은 트랜잭션은 Undo Log를 이용해 롤백

---

### 6. ACID 속성 간의 관계와 Trade-off

**Isolation과 성능의 Trade-off**

격리 수준이 높을수록 동시성이 낮아지고 성능이 저하된다. 실무에서는 대부분 InnoDB 기본값인 `REPEATABLE READ`를 사용하며, 읽기 성능이 중요한 경우 `READ COMMITTED`를 선택하기도 한다.

**CAP 정리와 ACID**

ACID는 단일 노드(또는 강한 일관성 보장 클러스터) 데이터베이스를 위한 속성이다. 분산 시스템에서는 CAP 정리에 의해 일관성(C), 가용성(A), 분할 내성(P) 중 두 가지만 동시에 보장 가능하다.

**BASE와의 비교**

NoSQL 등 분산 시스템에서는 ACID 대신 BASE(Basically Available, Soft state, Eventually consistent) 모델을 채택하는 경우가 많다. 강한 일관성 대신 높은 가용성과 확장성을 우선한다.

| | ACID | BASE |
|---|---|---|
| 일관성 | 강한 일관성 | 결과적 일관성 |
| 가용성 | 트랜잭션 완료까지 대기 | 항상 응답 |
| 주 사용처 | RDBMS (MySQL, PostgreSQL) | NoSQL (Cassandra, DynamoDB) |

## 핵심 정리
- InnoDB는 **Undo Log**로 원자성(Atomicity)을 보장하고, **Redo Log(WAL)** 로 지속성(Durability)을 보장한다.
- 격리성(Isolation)은 **MVCC**와 **Lock**(Gap Lock, Next-Key Lock)을 조합하여 구현하며, 기본 격리 수준은 `REPEATABLE READ`이다.
- `innodb_flush_log_at_trx_commit=1`(기본값)은 매 커밋마다 Redo Log를 디스크에 flush하여 완전한 지속성을 보장하지만, 성능 비용이 있다.
- Crash Recovery 시 InnoDB는 Redo Log를 재생(replay)하여 커밋된 트랜잭션을 복원하고, Undo Log로 미완료 트랜잭션을 롤백한다.
- 격리 수준이 높을수록 이상 현상(Dirty Read, Phantom Read 등)을 더 많이 방지하지만 동시성과 성능은 낮아진다.

## 키워드
- `ACID`: 데이터베이스 트랜잭션의 신뢰성을 보장하는 4가지 속성(Atomicity, Consistency, Isolation, Durability)의 약자
- `Atomicity`: 트랜잭션 내 모든 연산이 전부 성공하거나 전부 실패해야 하는 원자성 속성
- `Consistency`: 트랜잭션 전후로 데이터베이스가 항상 제약 조건을 만족하는 일관된 상태를 유지해야 하는 속성
- `Isolation`: 동시에 실행되는 트랜잭션들이 서로 영향을 주지 않도록 독립적으로 실행되어야 하는 속성
- `Durability`: 커밋된 트랜잭션의 결과는 시스템 장애 이후에도 영구적으로 보존되어야 하는 속성
- `Undo Log`: 트랜잭션 롤백 및 MVCC를 위해 데이터 변경 이전 값(Before Image)을 저장하는 InnoDB 내부 로그
- `Redo Log`: WAL 원칙에 따라 변경 사항을 먼저 기록해두어 크래시 복구 시 커밋된 데이터를 재적용하는 InnoDB 내부 로그
- `MVCC`: 행의 여러 버전을 관리하여 읽기 연산 시 잠금 없이 트랜잭션 격리를 구현하는 Multi-Version Concurrency Control 기법
- `WAL`: 데이터 파일 변경 전 로그를 먼저 기록하는 Write-Ahead Logging 원칙으로, InnoDB Redo Log의 근간
- `Crash Recovery`: 서버 재시작 시 Redo Log 재생으로 커밋된 변경 복원, Undo Log로 미완료 트랜잭션 롤백하는 InnoDB 복구 과정
