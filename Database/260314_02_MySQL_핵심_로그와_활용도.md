# MySQL 핵심 로그와 활용도

## 개요
MySQL은 다양한 로그를 통해 데이터 복구, 복제, 성능 분석, 장애 추적 등을 지원한다. 각 로그의 역할과 실무에서의 활용 방법을 학습한다.

## 상세 내용

### 1. InnoDB 엔진 로그

#### Redo Log (리두 로그)

Redo Log는 **WAL(Write-Ahead Logging)** 기반으로 데이터의 지속성(Durability)을 보장하는 디스크 기반 데이터 구조다. 커밋된 트랜잭션 변경사항을 실제 데이터 파일에 쓰기 전에 먼저 Redo Log에 기록한다.

**구조와 순환 기록 방식**

MySQL 8.0.30 이후, Redo Log 파일은 `#innodb_redo` 디렉토리 아래 `#ib_redo_N` 형태로 관리된다. InnoDB는 총 32개의 파일을 유지하며, 각 파일 크기는 `innodb_redo_log_capacity / 32` 이다. 파일을 순환(circular) 방식으로 재사용한다.

```sql
-- Redo Log 파일 상태 확인
SELECT FILE_NAME, START_LSN, END_LSN
FROM performance_schema.innodb_redo_log_files;

-- Redo Log 용량 설정 (런타임 변경 가능)
SET GLOBAL innodb_redo_log_capacity = 8589934592;  -- 8GB

-- Redo Log 상태 모니터링
SHOW STATUS LIKE 'Innodb_redo_log%';
```

주요 상태 변수:
- `Innodb_redo_log_checkpoint_lsn`: 현재 Checkpoint LSN
- `Innodb_redo_log_current_lsn`: 현재 LSN
- `Innodb_redo_log_flushed_to_disk_lsn`: 디스크에 플러시된 LSN

**Checkpoint와 Redo Log의 관계**

Checkpoint는 버퍼 풀의 더티 페이지(dirty page)가 실제 데이터 파일에 쓰인 시점의 LSN을 의미한다. Checkpoint LSN 이전 Redo Log는 Crash Recovery 시 불필요하므로 덮어쓸 수 있다.

- Redo Log 사용량 < 설정값 → 더티 페이지를 느긋하게 플러시
- Redo Log 사용량 > 설정값 → 더티 페이지를 적극적으로 플러시 (Checkpoint 진행)

**Crash Recovery에서의 역할**

서버가 비정상 종료 후 재시작되면, InnoDB는 최신 Checkpoint LSN을 찾아 그 이후의 Redo Log 레코드를 모두 재실행(redo)한다. 이후 미완료 트랜잭션은 Undo Log를 이용해 롤백한다.

---

#### Undo Log (언두 로그)

Undo Log는 트랜잭션이 변경한 데이터의 **이전 상태(old version)**를 기록한다. 두 가지 핵심 역할을 한다.

1. **트랜잭션 롤백**: `ROLLBACK` 시 Undo Log의 이전 값을 이용해 원상 복구
2. **MVCC 지원**: 다른 트랜잭션이 일관된 스냅샷 읽기를 수행할 수 있도록 이전 버전 데이터 제공

**Undo Tablespace 구조**

| 종류 | 대상 | 특징 |
|------|------|------|
| Undo Tablespace | 일반 테이블 트랜잭션 | Redo 로깅됨 (크래시 복구 필요) |
| Global Temporary Tablespace | 임시 테이블 트랜잭션 | Redo 로깅 안됨 (성능 최적화) |

```sql
-- Undo Log 및 트랜잭션 상태 확인
SHOW ENGINE INNODB STATUS;

-- Rollback Segment 설정
-- innodb_rollback_segments = 128  (기본값, my.cnf)
```

**Undo Log와 장기 실행 트랜잭션**

오래 실행되는 트랜잭션이 있으면, 해당 트랜잭션의 Read View보다 오래된 Undo Log를 Purge할 수 없다. 트랜잭션이 길어질수록 Undo Tablespace가 지속적으로 증가해 성능 저하 및 디스크 고갈로 이어질 수 있다.

**Purge 프로세스**

커밋된 트랜잭션의 Undo Log는 즉시 삭제되지 않는다. 백그라운드 Purge 스레드가 주기적으로 "더 이상 어떤 트랜잭션도 참조하지 않는" Undo Log를 정리해 공간을 해제한다.

```sql
-- Purge 스레드 설정
-- innodb_purge_threads = 4          (기본값)
-- innodb_purge_batch_size = 300     (기본값)
```

### 2. MySQL 서버 로그

#### Binary Log (binlog)

Binary Log(이하 binlog)는 MySQL 서버 레벨에서 관리하는 로그로, 데이터베이스의 모든 **변경 이벤트**를 기록한다. SELECT·SHOW 같은 조회문은 기록되지 않는다.

**Binlog의 두 가지 핵심 역할**

1. **복제(Replication)**: Source 서버의 변경 사항을 Replica 서버로 전달
2. **PITR(Point-in-Time Recovery)**: 백업 시점 이후의 변경 사항을 재실행해 특정 시점으로 복구

**Binlog 포맷 3가지**

| 포맷 | 기록 방식 | 장점 | 단점 |
|------|----------|------|------|
| STATEMENT | 실행된 SQL 문 그대로 기록 | 로그 크기 작음 | `NOW()`, `RAND()` 등 비결정적 함수에서 복제 불일치 위험 |
| ROW | 변경된 각 행의 이전/이후 상태 기록 | 복제 정확성 높음, 안전 | 대량 UPDATE 시 로그 크기 매우 커짐 |
| MIXED | 상황에 따라 STATEMENT/ROW 자동 선택 | 크기와 안전성의 균형 | 예측이 어려울 수 있음 |

프로덕션 환경에서는 일반적으로 **ROW** 포맷을 권장한다.

```sql
-- 현재 binlog 포맷 확인
SHOW VARIABLES LIKE 'binlog_format';

-- 포맷 변경
SET GLOBAL binlog_format = 'ROW';

-- binlog 파일 목록 확인
SHOW BINARY LOGS;

-- 현재 binlog 위치 확인
SHOW MASTER STATUS;

-- binlog 내용 조회
SHOW BINLOG EVENTS IN 'mysql-bin.000001';

-- 오래된 binlog 삭제
PURGE BINARY LOGS BEFORE DATE_SUB(NOW(), INTERVAL 7 DAY);
```

**mysqlbinlog 도구 활용**

```bash
# binlog를 SQL 형태로 출력
mysqlbinlog mysql-bin.000001

# 특정 시간 범위만 추출 (PITR 활용)
mysqlbinlog --start-datetime='2026-03-14 09:00:00' \
            --stop-datetime='2026-03-14 11:30:00' \
            mysql-bin.000001 > recovery.sql

# MySQL에 적용
mysql < recovery.sql
```

**Binlog와 Redo Log의 2PC(Two-Phase Commit)**

InnoDB의 Redo Log와 MySQL 서버의 Binlog는 서로 다른 계층에서 관리된다. 트랜잭션 COMMIT 시 두 로그의 일관성을 보장하기 위해 **내부 XA 2PC**를 사용한다.

```
[Prepare 단계]
  1. InnoDB Redo Log에 Prepare 상태로 기록 + 플러시

[Commit 단계]
  2. Binlog에 커밋 이벤트 기록 + 플러시
  3. InnoDB Redo Log에 Commit 상태로 기록
```

Crash Recovery 시:
- Redo Log: Prepare / Binlog: 기록 없음 → **롤백**
- Redo Log: Prepare / Binlog: 기록 있음 → **커밋** (복제 서버에 이미 전파됐을 수 있으므로)

---

#### General Log

모든 클라이언트 연결 및 수신된 SQL 문을 순서대로 기록한다. 서버가 처리하기 전에 기록되므로, 실제로 실행에 성공했는지와 무관하게 로그에 남는다.

**사용 시 주의사항**

- **기본값: 비활성화** (성능 영향이 크므로)
- 모든 쿼리를 기록하므로 고트래픽 환경에서는 I/O 부하와 디스크 사용량이 급격히 증가
- 비밀번호는 자동으로 `<secret>`으로 가려짐
- 프로덕션에서는 일시적인 디버깅·감사(Audit) 용도로만 사용

```sql
-- 활성화 (런타임)
SET GLOBAL general_log = ON;
SET GLOBAL general_log_file = '/var/log/mysql/general.log';

-- 비활성화 (진단 완료 후 즉시)
SET GLOBAL general_log = OFF;

-- 세션 단위 비활성화 (자신의 쿼리만 제외)
SET SESSION sql_log_off = ON;

-- 로그 테이블로 출력 (파일 대신 mysql.general_log 테이블 사용)
SET GLOBAL log_output = 'TABLE';
SELECT * FROM mysql.general_log ORDER BY event_time DESC LIMIT 10;
```

#### Slow Query Log

`long_query_time` 초를 초과하는 쿼리와 `min_examined_row_limit` 행 이상 검사한 쿼리를 기록한다. 성능 튜닝의 출발점으로 가장 자주 활용되는 로그다.

```sql
-- Slow Query Log 활성화
SET GLOBAL slow_query_log = ON;
SET GLOBAL slow_query_log_file = '/var/log/mysql/slow.log';

-- 2초 이상 소요된 쿼리 기록 (기본값: 10초)
SET GLOBAL long_query_time = 2;

-- 인덱스를 사용하지 않는 쿼리도 기록
SET GLOBAL log_queries_not_using_indexes = ON;

-- 분당 기록 건수 제한 (로그 폭발 방지)
SET GLOBAL log_throttle_queries_not_using_indexes = 100;

-- 추가 통계 정보 기록
SET GLOBAL log_slow_extra = ON;
```

**mysqldumpslow를 활용한 분석**

```bash
# 실행 횟수 많은 쿼리 상위 10개
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 평균 실행 시간 기준 정렬
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 평균 검사 행 수 기준 정렬
mysqldumpslow -s r -t 10 /var/log/mysql/slow.log
```

---

#### Error Log

MySQL 서버의 시작/종료, 오류, 경고, 진단 메시지를 기록한다. 장애 발생 시 가장 먼저 확인해야 할 로그다.

**주요 확인 메시지**

- `[ERROR]`: 서버가 처리할 수 없는 오류 (즉시 조치 필요)
- `[Warning]`: 잠재적 문제 (예: 잘못된 설정값)
- `InnoDB: Starting crash recovery`: 비정상 종료 후 복구 시작
- `Crash recovery finished`: 복구 완료
- `Too many connections`: 연결 초과

```sql
-- Error Log 경로 확인
SHOW VARIABLES LIKE 'log_error';
```

### 3. Relay Log (릴레이 로그)

Relay Log는 **복제(Replication) 구성에서 Replica 서버에만 존재**하는 로그다. Source 서버의 Binlog를 복사해 로컬에 임시 저장하고, SQL 스레드가 이를 읽어 실행한다.

**복제 데이터 흐름**

```
Source 서버
  └── Binlog 기록
        └── [I/O Thread] → Replica로 전송
              └── Relay Log (Replica 로컬 저장)
                    └── [SQL Thread] → Relay Log 읽어 실행
                          └── Replica 데이터 반영
```

```sql
-- Relay Log 상태 확인 (Replica에서 실행)
SHOW REPLICA STATUS\G

-- 주요 확인 항목
-- Relay_Log_File: 현재 읽고 있는 Relay Log 파일
-- Relay_Log_Pos: 현재 위치
-- Seconds_Behind_Source: 복제 지연 시간 (초)
```

복제가 완료되면 Relay Log는 자동으로 삭제된다. `relay_log_purge = ON`이 기본값이다.

### 4. 각 로그 간의 관계와 데이터 흐름

**DML 실행 시 로그 기록 순서**

```
1. 트랜잭션 시작
2. 변경 데이터를 InnoDB Buffer Pool(더티 페이지)에 기록
3. Undo Log 기록 (이전 값 보존 → MVCC, 롤백 지원)
4. Redo Log에 변경 내용 기록 (WAL)

--- COMMIT ---
5. Redo Log Prepare (XA 2PC 1단계)
6. Binlog에 커밋 이벤트 기록 + 플러시
7. Redo Log Commit (XA 2PC 2단계)
8. 클라이언트에 커밋 완료 응답

(비동기) Buffer Pool의 더티 페이지를 데이터 파일에 기록
```

**로그별 목적 분류**

| 로그 | 주요 목적 | 관리 주체 |
|------|----------|----------|
| Redo Log | 크래시 복구, 지속성 보장 | InnoDB 엔진 |
| Undo Log | 롤백, MVCC | InnoDB 엔진 |
| Binary Log | 복제, PITR | MySQL 서버 |
| Relay Log | 복제 중간 저장 | MySQL 서버 (Replica) |
| Slow Query Log | 성능 분석 | MySQL 서버 |
| General Log | 감사, 디버깅 | MySQL 서버 |
| Error Log | 장애 추적 | MySQL 서버 |

### 5. 실무에서의 로그 활용

**장애 복구: Redo Log + Binlog를 활용한 PITR**

```bash
# 1. 전체 백업에서 복원
mysqldump --all-databases --single-transaction > full_backup.sql
mysql < full_backup.sql

# 2. 백업 이후 특정 시점까지의 Binlog 적용
mysqlbinlog --start-datetime='2026-03-14 00:00:00' \
            --stop-datetime='2026-03-14 10:30:00' \
            mysql-bin.000001 mysql-bin.000002 | mysql
```

**복제 모니터링: Binlog Position 기반 vs GTID 기반**

```sql
-- Position 기반 확인 (Source)
SHOW MASTER STATUS;

-- GTID 기반 확인 (더 안정적, MySQL 5.6+)
SHOW VARIABLES LIKE 'gtid_mode';
SHOW VARIABLES LIKE 'gtid_executed';
```

**성능 튜닝: Slow Query Log 기반 쿼리 최적화 흐름**

```
1. Slow Query Log 활성화 (long_query_time = 1~2초)
2. mysqldumpslow로 문제 쿼리 식별
3. EXPLAIN으로 실행 계획 분석
4. 인덱스 추가 또는 쿼리 리팩토링
5. 개선 전후 실행 시간 비교
```

**로그 관리: 크기 제어와 로테이션**

```sql
-- Binlog 보관 기간 설정 (기본값: 30일)
SET GLOBAL binlog_expire_logs_seconds = 604800;  -- 7일

-- Binlog 수동 삭제
PURGE BINARY LOGS BEFORE DATE_SUB(NOW(), INTERVAL 7 DAY);

-- 로그 파일 교체 (새 파일 시작)
FLUSH BINARY LOGS;
FLUSH SLOW LOGS;
FLUSH ERROR LOGS;
```

## 핵심 정리
- Redo Log는 WAL 방식으로 Crash Recovery를 보장하고, Undo Log는 MVCC와 롤백을 지원한다. 두 로그 모두 InnoDB 엔진이 관리한다.
- Binlog는 MySQL 서버 레벨에서 관리되며, 복제(Replication)와 PITR의 핵심이다. Redo Log와는 내부 XA 2PC로 일관성을 유지한다.
- 트랜잭션 COMMIT 시 Redo Log Prepare → Binlog 기록 → Redo Log Commit 순서를 따른다. 이 순서 덕분에 크래시 후 Binlog 유무로 커밋/롤백 여부를 결정할 수 있다.
- 장기 실행 트랜잭션은 Undo Log Purge를 막아 Undo Tablespace를 비대하게 만든다. 실무에서 트랜잭션을 짧게 유지해야 하는 중요한 이유 중 하나다.
- General Log는 성능 영향이 매우 크므로 프로덕션에서는 기본 비활성화 상태를 유지하고, Slow Query Log를 주요 성능 분석 도구로 활용한다.

## 키워드
- `Redo Log`: WAL 방식으로 변경 내용을 먼저 기록해 Crash Recovery 시 미완료 트랜잭션을 재실행할 수 있게 하는 InnoDB 로그
- `Undo Log`: 트랜잭션의 이전 데이터를 저장해 롤백과 MVCC(일관된 스냅샷 읽기)를 지원하는 InnoDB 로그
- `Binary Log`: 데이터 변경 이벤트를 기록하는 MySQL 서버 로그로, 복제(Replication)와 PITR의 핵심
- `Slow Query Log`: `long_query_time`을 초과하는 쿼리를 기록해 성능 병목 쿼리를 식별하는 로그
- `General Log`: 모든 SQL 문과 연결을 기록하는 로그로, 성능 영향이 크므로 디버깅·감사 목적으로만 일시 사용
- `Error Log`: 서버 시작/종료, 오류, 경고를 기록하며 장애 발생 시 가장 먼저 확인해야 할 로그
- `Relay Log`: 복제 구성에서 Replica가 Source의 Binlog를 수신해 임시 저장하는 로그
- `WAL`: Write-Ahead Logging. 데이터 파일 변경 전 먼저 로그에 기록하는 방식으로, Redo Log의 기반 원리
- `PITR`: Point-in-Time Recovery. 전체 백업 + Binlog 재실행으로 특정 시점의 데이터 상태로 복구하는 기법
- `Crash Recovery`: 비정상 종료 후 재시작 시 Redo Log를 재실행하고 미완료 트랜잭션을 Undo Log로 롤백해 데이터 일관성을 복원하는 과정
