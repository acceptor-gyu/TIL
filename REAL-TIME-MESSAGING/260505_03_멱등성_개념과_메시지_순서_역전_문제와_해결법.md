# 멱등성 개념과 메시지 순서 역전 문제와 해결법

## 개요
- 실시간 메시징 시스템에서 멱등성(Idempotency)의 정의와 중요성
- 메시지 순서 역전(Out-of-Order)이 발생하는 원인과 영향
- 멱등성과 순서 보장이 분산 메시징 시스템에서 함께 다뤄져야 하는 이유

## 상세 내용

### 1. 멱등성(Idempotency) 개념

멱등성이란 동일한 연산을 여러 번 수행해도 결과가 같아지는 성질이다.

수학적으로는 `f(f(x)) = f(x)` 가 성립하는 함수를 멱등 함수라 부르며, 분산 시스템에서는 "동일한 요청을 N번 보내도 1번 보낸 것과 서버 상태가 동일해야 한다"는 개념으로 확장된다.

**HTTP 메서드 멱등성 비교**

| 메서드 | 멱등성 | 설명                                |
| ------ | ------ | ----------------------------------- |
| GET    | O      | 조회 요청이므로 서버 상태 불변      |
| PUT    | O      | 동일 상태로 덮어쓰기                |
| DELETE | O      | 두 번 삭제해도 결과는 "삭제됨"     |
| POST   | X      | 매 호출마다 새 리소스 생성 가능     |
| PATCH  | X      | 부분 업데이트이므로 누적될 수 있음  |

**실시간 메시징에서 멱등성이 필요한 상황**

- At-Least-Once 전송 보장: 메시지 유실을 방지하기 위해 재전송하면, 동일 메시지가 2회 이상 도달할 수 있다.
- 네트워크 단절 후 재연결: WebSocket 재연결 시 서버가 이전 메시지를 다시 푸시한다.
- Consumer 오프셋 커밋 실패: 처리는 완료됐지만 Kafka 오프셋을 커밋하지 못하면 동일 메시지를 재처리한다.
- 클라이언트 타임아웃 후 재시도: 결제·주문 등에서 응답을 받지 못해 동일 요청을 재전송한다.

**자연 멱등성 vs 설계 멱등성**

- 자연 멱등성: 사용자 이름/전화번호 업데이트처럼 같은 값을 두 번 써도 결과가 동일한 경우
- 설계 멱등성: 결제·차감처럼 자연적으로 멱등하지 않은 연산에 고유 키를 도입해 중복을 탐지하는 경우

### 2. 메시지 중복 발생 시나리오

**시나리오 1 - ACK 유실로 인한 Producer 재전송**

```
Producer → Broker: 메시지 전송
Broker → Producer: ACK (네트워크 유실)
Producer: ACK를 받지 못했으므로 동일 메시지 재전송
Broker: 동일 메시지 2회 저장
```

**시나리오 2 - Consumer 오프셋 커밋 실패**

```
Kafka → Consumer: 메시지 전달
Consumer: 비즈니스 로직 처리 완료
Consumer → Kafka: 오프셋 커밋 (장애 발생, 실패)
Consumer 재시작: 커밋되지 않은 오프셋부터 다시 소비 → 중복 처리
```

**시나리오 3 - WebSocket 재연결 시 재수신**

```
Server → Client: 메시지 100번 전송
Client: 네트워크 단절로 ACK 미응답
Server: 재연결 후 100번 메시지 재전송
Client: 100번 메시지 2회 렌더링
```

**시나리오 4 - SQS Visibility Timeout 초과**

```
Consumer A: 메시지 수신, 처리 중
Visibility Timeout 초과 → 메시지 다시 큐에 노출
Consumer B: 동일 메시지 수신 → 중복 처리
```

### 3. 멱등성 보장 전략

**전략 1 - Idempotency Key (멱등 키)**

요청마다 비즈니스 의미가 담긴 고유 식별자를 부여하고, 서버에서 처리 여부를 기록한다.

```
클라이언트: 요청 헤더에 Idempotency-Key: {UUID v4} 포함
서버: 동일 키가 이미 처리됐으면 이전 응답을 그대로 반환
```

멱등 키는 Producer가 임의로 생성하는 것이 아니라 비즈니스 맥락을 담아야 한다.
- 결제: `userId:orderId:amount:timestamp`
- 채팅: `roomId:senderId:clientMessageId`

**전략 2 - DB Unique Constraint**

`idempotency_key` 컬럼에 Unique Index를 걸어 중복 INSERT 시 DB 레벨에서 거부한다.

```sql
CREATE TABLE processed_messages (
    idempotency_key VARCHAR(255) PRIMARY KEY,
    result          TEXT,
    created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

처리 결과를 저장하고, 중복 요청 시 저장된 결과를 반환한다.

**전략 3 - Redis SETNX 기반 Dedup 캐시**

DB 조회 없이 빠르게 중복 여부를 판단할 때 사용한다.

```
SETNX dedup:{messageId} "processed" EX 86400
→ 반환값이 1이면 최초 처리, 0이면 이미 처리된 메시지
```

TTL 설계가 중요하다.
- API 요청: 24시간
- 큐 메시지: 7일 (재처리 가능성이 있는 최대 시간 기준)
- 영구 저장 시 스토리지 비용이 급증하므로 비즈니스 요건에 따라 TTL 결정

**전략 4 - Kafka Producer 멱등성 옵션**

Kafka 0.11.0.0부터 Producer 수준에서 멱등성을 지원한다.

```properties
enable.idempotence=true
acks=all
max.in.flight.requests.per.connection=5
retries=5
```

내부적으로 Producer ID(PID)와 시퀀스 번호를 부여해, 브로커가 중복 메시지를 감지하고 버린다. 단일 파티션 내에서만 순서와 멱등성을 보장한다.

**전략 5 - 클라이언트 측 멱등 처리 (UI 레벨)**

서버 보장 외에도 프론트엔드에서 메시지 ID 기반 중복 렌더링을 방지한다.

```javascript
const renderedMessageIds = new Set();

function renderMessage(msg) {
    if (renderedMessageIds.has(msg.id)) return; // 중복 스킵
    renderedMessageIds.add(msg.id);
    // 실제 렌더링
}
```

### 4. 메시지 순서 역전 문제

**순서 역전이 발생하는 원인**

1. 멀티 파티션 / 멀티 샤드 큐: Kafka 파티션이 여럿이면 파티션별 독립 큐이므로 파티션 간 순서는 보장되지 않는다.
2. 네트워크 지연과 재전송: 재전송된 메시지가 나중에 도착하면 순서가 역전된다.
3. 비동기 병렬 처리: Consumer가 멀티 스레드로 처리하면 처리 완료 순서가 수신 순서와 달라진다.
4. Consumer 여러 인스턴스 분산: 같은 파티션을 여러 인스턴스가 처리하면(잘못된 설계) 순서가 뒤섞인다.

**순서 역전이 치명적인 도메인 사례**

| 도메인 | 순서 역전 영향 |
|--------|--------------|
| 채팅 메시지 | 답장이 원본보다 먼저 표시됨 |
| 주문 상태 변경 | "배송 완료"가 "결제 완료"보다 먼저 처리됨 |
| 실시간 거래 | 매도 주문이 매수 주문보다 먼저 처리되어 오류 발생 |
| 게임 이벤트 | 캐릭터 사망 이벤트가 이동 이벤트보다 먼저 적용됨 |

### 5. 메시지 순서 보장 전략

**전략 1 - 단일 파티션 활용**

하나의 파티션에 모든 메시지를 넣으면 Kafka가 순서를 보장한다. 그러나 처리량의 병목이 된다.

```
적합: 소규모 트래픽 + 글로벌 순서가 반드시 필요한 경우
부적합: 대규모 트래픽 + 높은 처리량이 필요한 경우
```

**전략 2 - Partition Key를 활용한 도메인 단위 순서 보장**

전체 글로벌 순서 대신 도메인 단위의 순서만 보장한다. 이것이 현실적인 최선이다.

```
Kafka Producer: partitionKey = userId (또는 roomId, orderId)
→ 동일 userId의 메시지는 항상 같은 파티션으로 라우팅
→ 해당 파티션 내에서 순서 보장
→ 서로 다른 userId는 병렬 처리 가능
```

**전략 3 - 시퀀스 번호(Sequence Number) 부여 후 Consumer 측 재정렬**

Producer가 메시지에 시퀀스 번호를 부여하고, Consumer가 순서에 맞게 버퍼링 후 처리한다.

```
Producer: msg.seq = atomicCounter.incrementAndGet()

Consumer 버퍼:
- seq=1 도착 → 처리
- seq=3 도착 → 버퍼에 보관 (seq=2 미도착)
- seq=2 도착 → 2 처리 후 3도 처리
```

지연 vs 정확성 trade-off가 있으므로 타임아웃 정책을 함께 정의해야 한다. 타임아웃 초과 시 seq=2를 건너뛰고 seq=3을 처리할지, 아니면 오류로 처리할지 비즈니스 요건에 따라 결정한다.

**전략 4 - Lamport Timestamp (논리 시계)**

물리 시계 동기화 없이 이벤트 간 인과 관계(happened-before)를 추적하는 논리 시계다.

```
규칙:
1. 이벤트 발생 시 자신의 카운터 +1
2. 메시지 전송 시 현재 카운터 포함
3. 메시지 수신 시: counter = max(내 카운터, 메시지 카운터) + 1
```

한계: 동시 발생 이벤트의 선후 관계는 판단하지 못한다. 동일 타임스탬프가 발생할 수 있다.

**전략 5 - Vector Clock (벡터 시계)**

Lamport Timestamp의 한계를 극복하여 인과 관계와 동시성을 모두 감지한다.

```
노드 A, B, C가 있을 때 각 노드는 [A의 카운터, B의 카운터, C의 카운터] 형태의 벡터 유지

노드 A가 이벤트 발생: [1, 0, 0]
노드 A가 B에게 메시지 전송: B의 벡터 = [1, 1, 0] (A의 카운터 흡수)
노드 C가 독립적으로 이벤트 발생: [0, 0, 1] → 동시 발생 감지 가능
```

실무 적용: 채팅 앱에서 오프라인 메시지 동기화 시 conflict 감지, 분산 DB CRDT(Conflict-Free Replicated Data Types)에서 활용.

**전략 6 - WebSocket 환경에서의 클라이언트 측 시퀀스 검증**

서버가 메시지에 시퀀스 번호를 부여하고, 클라이언트가 누락된 시퀀스를 감지하면 서버에 재요청한다.

```javascript
let expectedSeq = 1;

socket.on('message', (msg) => {
    if (msg.seq !== expectedSeq) {
        // 누락된 메시지 요청
        socket.emit('request_missed', { from: expectedSeq, to: msg.seq - 1 });
        return;
    }
    process(msg);
    expectedSeq++;
});
```

### 6. 멱등성 + 순서 보장의 통합 설계

**Outbox Pattern (트랜잭셔널 메시징)**

DB 업데이트와 메시지 발행을 원자적으로 처리하기 위한 패턴이다.

```
문제: DB 커밋은 성공했지만 Kafka 발행 실패 → 불일치 발생
해결:

1. DB 트랜잭션 안에서 비즈니스 데이터 + outbox 테이블에 메시지 동시 저장
2. 별도 프로세스(또는 CDC)가 outbox 테이블을 읽어 Kafka에 발행
3. 발행 성공 시 outbox 레코드 삭제/마킹

→ DB와 메시지 브로커의 일관성 보장 (Eventual Consistency)
→ outbox에서 Kafka로 발행 시 At-Least-Once이므로 Consumer의 멱등성 처리 필수
```

CDC(Change Data Capture) 도구인 Debezium을 사용하면 DB 바이너리 로그를 읽어 outbox polling 없이 자동 발행할 수 있다.

**Kafka Exactly-Once Semantics (EOS)**

Kafka 트랜잭션을 사용하면 Kafka 클러스터 내에서 Exactly-Once를 보장한다.

```
전제 조건:
- Producer: enable.idempotence=true + transactional.id 설정
- Consumer: isolation.level=read_committed

동작 원리:
1. Producer가 트랜잭션 시작
2. 여러 파티션에 메시지 발행
3. Consumer 오프셋도 같은 트랜잭션으로 커밋
4. 트랜잭션 완료 → Consumer는 커밋된 메시지만 읽음
```

주의: EOS는 Kafka 클러스터 경계 안에서만 보장된다. 외부 DB나 API 호출이 포함되면 별도 멱등성 설계가 필요하다.

**Consumer Idempotent Processor 패턴**

멱등 키 저장과 실제 처리를 동일 트랜잭션에서 수행해야 한다.

```java
@Transactional
public void process(Message msg) {
    // 이미 처리된 메시지면 스킵
    if (processedMessageRepo.existsById(msg.getIdempotencyKey())) {
        return;
    }

    // 실제 비즈니스 로직 처리
    businessService.handle(msg);

    // 처리 완료 기록 (같은 트랜잭션)
    processedMessageRepo.save(new ProcessedMessage(msg.getIdempotencyKey()));
}
```

멱등 키 저장과 비즈니스 처리가 다른 트랜잭션이면 장애 시 둘 중 하나만 성공할 수 있으므로 반드시 같은 트랜잭션 안에 묶어야 한다.

## 핵심 정리
- 멱등성은 At-Least-Once 전송을 Exactly-Once 결과처럼 만드는 핵심 장치다
- 순서 보장은 파티셔닝 단위와 시퀀스 번호의 조합으로 풀어가는 문제다
- 분산 환경에서 글로벌 순서 보장은 사실상 불가능하며, "비즈니스 도메인 단위의 순서 보장"이 현실적이다
- 멱등 키는 비즈니스 의미가 담긴 키여야 하며, Producer가 임의로 생성한 키는 재시도 시 다른 키가 될 수 있다

## 기술적 한계와 보완 전략
- 단일 파티션은 순서를 보장하지만 처리량 병목이 됨 → Partition Key로 도메인 단위 분산
- 글로벌 순서 보장은 사실상 불가능 → 도메인 단위로 순서 범위 축소
- 멱등 키 저장소(DB/Redis)의 TTL 설계 필수: 영구 저장 시 스토리지 비용 급증
- Consumer 재정렬 버퍼는 메모리 압박과 지연을 유발 → 타임아웃 정책과 함께 운영
- Exactly-Once는 Kafka 클러스터 경계 안에서만 보장됨 → 외부 시스템 호출 시 별도 멱등성 필요
- Outbox Pattern에서 CDC(Debezium)를 활용하면 polling 오버헤드 없이 안정적 발행 가능

## 키워드

### Idempotency (멱등성)
동일한 연산을 여러 번 수행해도 결과가 같아지는 성질. 분산 시스템에서 At-Least-Once 전송으로 인한 중복 메시지를 안전하게 처리하기 위해 필수적으로 설계해야 하는 속성이다.

### Idempotency Key (멱등 키)
요청의 고유성을 식별하기 위한 키. UUID v4를 단순 사용하기보다 비즈니스 맥락(userId + orderId 등)을 조합해 의미 있는 키로 설계해야 한다. 서버는 동일 키가 존재하면 이전 응답을 그대로 반환한다.

### Exactly-Once Semantics (EOS)
메시지가 정확히 한 번만 처리되는 것을 보장하는 전달 방식. Kafka에서는 idempotent producer와 트랜잭션 API를 함께 사용해 클러스터 내에서 EOS를 달성한다.

### At-Least-Once Delivery
메시지 유실은 없지만 중복 전달이 발생할 수 있는 전달 보장 방식. 대부분의 메시징 시스템 기본값이며, 수신 측 멱등성 처리와 함께 사용해야 안전하다.

### Out-of-Order Message (메시지 순서 역전)
멀티 파티션 환경, 네트워크 지연, 비동기 병렬 처리 등으로 메시지가 발송 순서와 다른 순서로 소비자에게 도달하는 현상. 채팅, 주문, 거래 등의 도메인에서 치명적인 결과를 초래할 수 있다.

### Partition Key
Kafka에서 동일 키를 가진 메시지를 같은 파티션으로 라우팅하기 위해 사용하는 키. userId, roomId, orderId 등 도메인 식별자를 사용하면 해당 단위의 순서를 보장하면서 병렬 처리를 유지할 수 있다.

### Sequence Number (시퀀스 번호)
Producer가 메시지에 부여하는 단조 증가 번호. Consumer가 시퀀스 번호를 기준으로 누락 감지 및 재정렬에 활용하며, Kafka 멱등 Producer도 내부적으로 시퀀스 번호를 사용한다.

### Outbox Pattern
DB 업데이트와 메시지 발행의 원자성을 보장하는 패턴. 비즈니스 데이터와 메시지를 같은 DB 트랜잭션으로 저장하고, 별도 프로세스(또는 CDC)가 메시지를 브로커로 발행한다. 이중 쓰기(dual-write) 문제를 해결한다.

### Lamport Timestamp (람포트 타임스탬프)
물리 시계 없이 분산 이벤트 간 순서(happened-before)를 추적하는 논리 시계 알고리즘. 단, 동시 발생 이벤트를 구별하지 못하는 한계가 있다.

### Dedup Cache (Redis SETNX)
Redis의 SETNX 명령으로 멱등 키 존재 여부를 원자적으로 확인하고 기록하는 중복 제거 캐시. TTL을 반드시 설정해야 하며, 처리 결과 저장과 SETNX를 같은 트랜잭션으로 묶어야 정합성이 보장된다.
