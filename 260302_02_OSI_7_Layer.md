# OSI 7 Layer

## 개요
네트워크 통신을 7개의 계층으로 나누어 설명하는 OSI(Open Systems Interconnection) 참조 모델에 대해 학습한다.

OSI 모델은 1984년 국제표준화기구(ISO)가 제정한 네트워크 통신 표준 참조 모델이다. 서로 다른 시스템 간의 통신을 위해 통신 과정을 7개의 계층으로 나누어 각 계층이 독립적으로 기능하도록 설계되었다. 실제 인터넷은 TCP/IP 모델을 기반으로 동작하지만, OSI 모델은 네트워크 개념 이해와 트러블슈팅의 기준 모델로 널리 활용된다.

## 각 계층 역할과 프로토콜

| 계층 | 이름 | PDU | 주요 프로토콜 | 대표 장비 |
|------|------|-----|-------------|---------|
| Layer 7 | 응용 계층 (Application) | 데이터(Data) | HTTP, FTP, SMTP, DNS, Telnet | - |
| Layer 6 | 표현 계층 (Presentation) | 데이터(Data) | SSL/TLS, JPEG, MPEG, ASCII | - |
| Layer 5 | 세션 계층 (Session) | 데이터(Data) | SSH, RPC, NetBIOS, SQL | - |
| Layer 4 | 전송 계층 (Transport) | 세그먼트(Segment) | TCP, UDP | L4 스위치 |
| Layer 3 | 네트워크 계층 (Network) | 패킷(Packet) | IP, ICMP, ARP, RIP, OSPF, BGP | 라우터, L3 스위치 |
| Layer 2 | 데이터 링크 계층 (Data Link) | 프레임(Frame) | Ethernet, PPP, HDLC, Wi-Fi(802.11) | 브리지, L2 스위치 |
| Layer 1 | 물리 계층 (Physical) | 비트(Bit) | RS-232, X.21 | 허브, 리피터, NIC |

---

### Layer 7 - 응용 계층 (Application)

사용자와 직접 상호작용하는 계층으로, 네트워크 서비스를 애플리케이션에 제공한다. 이메일, 웹 브라우저, FTP 클라이언트 등 사용자가 실제로 사용하는 프로그램이 이 계층에서 동작한다.

- **역할**: 응용 프로그램과 네트워크 서비스 간 인터페이스 제공, 리소스 공유, 파일 접근
- **PDU**: 데이터(Data) 또는 메시지(Message)
- **주요 프로토콜**

| 프로토콜 | 포트 | 설명 |
|---------|------|------|
| HTTP | 80 | 웹 페이지 전송 |
| HTTPS | 443 | 암호화된 웹 페이지 전송 |
| FTP | 20/21 | 파일 전송 |
| SMTP | 25 | 이메일 전송 |
| DNS | 53 | 도메인 이름 해석 |
| SSH | 22 | 보안 원격 접속 |

---

### Layer 6 - 표현 계층 (Presentation)

데이터의 형식 변환, 암호화, 압축을 담당한다. 서로 다른 시스템 간에 데이터 표현 방식이 달라도 통신이 가능하도록 중간에서 변환 역할을 한다.

- **역할**: 데이터 인코딩/디코딩, 암호화/복호화, 압축/해제
- **PDU**: 데이터(Data)
- **주요 기능**
  - **인코딩 변환**: ASCII ↔ EBCDIC 등 문자 코드 변환
  - **암호화**: SSL/TLS를 통한 데이터 암호화
  - **압축**: 데이터 전송량 최소화를 위한 압축 (JPEG, MPEG, ZIP 등)

> 실무에서는 표현 계층과 세션 계층의 역할이 애플리케이션 계층에 통합되는 경우가 많아 TCP/IP 모델에서는 별도 계층으로 분리하지 않는다.

---

### Layer 5 - 세션 계층 (Session)

두 시스템 간의 통신 세션(연결)을 설정, 유지, 종료하는 역할을 담당한다.

- **역할**: 논리적 연결 생성 및 제어, 세션 복구 및 체크포인트 관리
- **PDU**: 데이터(Data)
- **주요 기능**
  - **세션 수립/해제**: 통신 시작 및 종료 관리
  - **대화 제어**: 전이중(Full-Duplex), 반이중(Half-Duplex) 통신 방식 결정
  - **동기화**: 체크포인트(Checkpoint)를 이용한 장애 발생 시 세션 복구
- **주요 프로토콜**: SSH, RPC, NetBIOS

---

### Layer 4 - 전송 계층 (Transport)

종단 간(end-to-end) 신뢰성 있는 데이터 전송을 담당한다. 포트 번호를 사용해 프로세스를 식별하며, 데이터의 분할과 재조립을 수행한다.

- **역할**: 종단 간 신뢰성 있는 데이터 전송, 오류 제어, 흐름 제어, 혼잡 제어
- **PDU**: 세그먼트(Segment)
- **대표 장비**: L4 스위치 (포트 번호 기반 로드밸런싱)

**TCP vs UDP 비교**

| 구분 | TCP | UDP |
|------|-----|-----|
| 연결 방식 | 연결 지향(Connection-Oriented) | 비연결(Connectionless) |
| 신뢰성 | 높음 (순서 보장, 재전송) | 낮음 (순서 미보장, 재전송 없음) |
| 속도 | 상대적으로 느림 | 빠름 |
| 흐름 제어 | 있음 | 없음 |
| 혼잡 제어 | 있음 | 없음 |
| 사용 사례 | HTTP, FTP, 이메일 | DNS, 스트리밍, VoIP, 게임 |

---

### Layer 3 - 네트워크 계층 (Network)

논리적 주소(IP 주소)를 기반으로 패킷의 최적 경로를 결정하고 라우팅하는 역할을 한다.

- **역할**: 논리적 주소 지정(IP), 라우팅, 패킷 전달 경로 결정
- **PDU**: 패킷(Packet)
- **대표 장비**: 라우터, L3 스위치
- **주요 프로토콜**

| 프로토콜 | 설명 |
|---------|------|
| IP (IPv4/IPv6) | 논리적 주소 지정 및 패킷 라우팅 |
| ICMP | 오류 메시지 및 제어 메시지 전달 (ping에서 사용) |
| ARP | IP 주소를 MAC 주소로 변환 |
| OSPF | 링크 상태 기반 내부 라우팅 프로토콜 |
| BGP | 인터넷 경계 라우팅 프로토콜 |

---

### Layer 2 - 데이터 링크 계층 (Data Link)

물리적 주소(MAC 주소)를 사용하여 인접 노드 간의 신뢰성 있는 데이터 전송을 담당한다. 오류 감지 및 흐름 제어를 수행한다.

- **역할**: 물리적 주소 지정(MAC), 프레임 전송, 오류 감지, 흐름 제어
- **PDU**: 프레임(Frame)
- **대표 장비**: 브리지, L2 스위치
- **두 개의 서브 계층**
  - **LLC(Logical Link Control)**: 상위 계층과의 인터페이스 제공
  - **MAC(Media Access Control)**: 물리 매체에 대한 접근 제어, MAC 주소 관리
- **프레임 구조 (Ethernet 기준)**

```
+----------------+----------------+--------+--------+---------+
| 목적지 MAC(6B) | 출발지 MAC(6B) | 타입(2B) | 데이터  | FCS(4B) |
+----------------+----------------+--------+--------+---------+
```

  - **FCS(Frame Check Sequence)**: CRC를 이용한 오류 감지용 트레일러

---

### Layer 1 - 물리 계층 (Physical)

비트 단위의 데이터를 전기 신호, 광 신호, 전자기파 등 물리적 신호로 변환하여 전송한다.

- **역할**: 비트 스트림의 물리적 전송, 신호 변환
- **PDU**: 비트(Bit)
- **대표 장비**: 허브(Hub), 리피터(Repeater), NIC(Network Interface Card)
- **전송 매체**: 동축 케이블, 광케이블, UTP 케이블, 무선(Wi-Fi)
- **주요 특성**: 데이터의 내용을 해석하지 않고 단순히 전기적/물리적 신호만 전달

---

## 데이터 캡슐화 (Encapsulation)

캡슐화는 상위 계층에서 하위 계층으로 데이터가 전달될 때, 각 계층이 자신의 헤더(및 트레일러)를 붙여 나가는 과정이다. 수신 측에서는 반대로 하위 계층부터 헤더를 제거하는 역캡슐화(Decapsulation)가 이루어진다.

### 계층별 PDU 명칭

```
송신 측 (캡슐화)                                      수신 측 (역캡슐화)
┌──────────────────────────────────┐               ┌──────────────────────────────────┐
│ Layer 7 Application              │               │ Layer 7 Application              │
│ ┌────────────────────────────┐   │               │ ┌────────────────────────────┐   │
│ │           DATA             │   │               │ │           DATA             │   │
│ └────────────────────────────┘   │               │ └────────────────────────────┘   │
│              ↓                   │               │              ↑                   │
│ Layer 4 Transport                │               │ Layer 4 Transport                │
│ ┌──────┬─────────────────────┐   │               │ ┌──────┬─────────────────────┐   │
│ │TCP/H │        DATA         │   │               │ │TCP/H │        DATA         │   │
│ └──────┴─────────────────────┘   │               │ └──────┴─────────────────────┘   │
│   [Segment]      ↓               │               │   [Segment]      ↑               │
│ Layer 3 Network                  │               │ Layer 3 Network                  │
│ ┌─────┬──────┬──────────────┐    │               │ ┌─────┬──────┬──────────────┐    │
│ │ IP/H│TCP/H │    DATA      │    │               │ │ IP/H│TCP/H │    DATA      │    │
│ └─────┴──────┴──────────────┘    │               │ └─────┴──────┴──────────────┘    │
│   [Packet]         ↓             │               │   [Packet]         ↑             │
│ Layer 2 Data Link                │               │ Layer 2 Data Link                │
│ ┌────┬─────┬──────┬─────┬─────┐  │               │ ┌────┬─────┬──────┬─────┬─────┐  │
│ │ETH/H│IP/H│TCP/H │DATA │FCS  │  │               │ │ETH/H│IP/H│TCP/H │DATA │FCS  │  │
│ └────┴─────┴──────┴─────┴─────┘  │               │ └────┴─────┴──────┴─────┴─────┘  │
│   [Frame]           ↓            │               │   [Frame]           ↑            │
│ Layer 1 Physical                 │               │ Layer 1 Physical                 │
│         1010110100110101...      │  ─────────▶   │         1010110100110101...      │
└──────────────────────────────────┘               └──────────────────────────────────┘
```

### 캡슐화 단계 요약

| 단계 | 계층 | 동작 | 결과 PDU |
|------|------|------|---------|
| 1 | 응용/표현/세션 | 데이터 생성 | 데이터(Data) |
| 2 | 전송 | TCP/UDP 헤더 추가 (포트 번호, 시퀀스 번호 등) | 세그먼트(Segment) |
| 3 | 네트워크 | IP 헤더 추가 (출발지/목적지 IP 주소) | 패킷(Packet) |
| 4 | 데이터 링크 | Ethernet 헤더(MAC 주소) + FCS 트레일러 추가 | 프레임(Frame) |
| 5 | 물리 | 비트 신호로 변환 후 전송 | 비트(Bit) |

## OSI vs TCP/IP 모델 비교

| OSI 7계층 | TCP/IP 4계층 | 설명 |
|-----------|-------------|------|
| Layer 7 - 응용 계층 | 응용 계층 (Application) | HTTP, FTP, DNS, SMTP 등 |
| Layer 6 - 표현 계층 | 응용 계층 (Application) | 인코딩, 암호화, 압축 |
| Layer 5 - 세션 계층 | 응용 계층 (Application) | 세션 수립/유지/종료 |
| Layer 4 - 전송 계층 | 전송 계층 (Transport) | TCP, UDP |
| Layer 3 - 네트워크 계층 | 인터넷 계층 (Internet) | IP, ICMP, ARP |
| Layer 2 - 데이터 링크 계층 | 네트워크 액세스 계층 (Network Access) | Ethernet, Wi-Fi |
| Layer 1 - 물리 계층 | 네트워크 액세스 계층 (Network Access) | 케이블, 신호 |

```
OSI 모델              TCP/IP 모델
┌─────────────────┐   ┌─────────────────┐
│ 7. 응용 계층    │   │                 │
├─────────────────┤   │  응용 계층      │
│ 6. 표현 계층    │   │  (Application)  │
├─────────────────┤   │                 │
│ 5. 세션 계층    │   │                 │
├─────────────────┼───┼─────────────────┤
│ 4. 전송 계층    │   │  전송 계층      │
│                 │   │  (Transport)    │
├─────────────────┼───┼─────────────────┤
│ 3. 네트워크계층 │   │  인터넷 계층    │
│                 │   │  (Internet)     │
├─────────────────┼───┼─────────────────┤
│ 2. 데이터링크   │   │  네트워크       │
├─────────────────┤   │  액세스 계층   │
│ 1. 물리 계층    │   │  (Network Acc.) │
└─────────────────┘   └─────────────────┘
```

### TCP/IP를 실무에서 더 많이 사용하는 이유

- **단순성**: 7개 계층 → 4개 계층으로 단순화되어 실제 구현이 용이
- **실용성**: 인터넷을 실제로 운영하기 위해 설계된 모델 (OSI는 이론적 참조 모델)
- **선점 효과**: 인터넷이 TCP/IP 기반으로 먼저 발전하여 표준으로 자리잡음
- **유연성**: 계층 간 경계가 느슨하여 프로토콜 추가/변경이 용이

## 계층별 장비와 프로토콜 매핑

```
Layer 7 (Application)   ┌─────────────┐
Layer 6 (Presentation)  │   L7 스위치  │  HTTP 헤더, URL, 쿠키 기반 스위칭
Layer 5 (Session)       └─────────────┘
─────────────────────────────────────────
Layer 4 (Transport)     ┌─────────────┐
                        │   L4 스위치  │  IP + 포트(TCP/UDP) 기반 로드밸런싱
                        └─────────────┘
─────────────────────────────────────────
Layer 3 (Network)       ┌─────────────┐
                        │  라우터      │  IP 주소 기반 패킷 라우팅
                        │  L3 스위치   │  IP 기반 스위칭 + 라우팅
                        └─────────────┘
─────────────────────────────────────────
Layer 2 (Data Link)     ┌─────────────┐
                        │  L2 스위치   │  MAC 주소 기반 프레임 전달
                        │  브리지      │  두 네트워크 세그먼트 연결
                        └─────────────┘
─────────────────────────────────────────
Layer 1 (Physical)      ┌─────────────┐
                        │  허브        │  신호 증폭 및 전체 포트 브로드캐스트
                        │  리피터      │  신호 재생 및 증폭
                        │  NIC         │  네트워크 카드, 물리 신호 변환
                        └─────────────┘
```

### L4 스위치 vs L7 스위치

| 구분 | L4 스위치 | L7 스위치 |
|------|---------|---------|
| 동작 계층 | 전송 계층 (Layer 4) | 응용 계층 (Layer 7) |
| 분류 기준 | IP 주소 + TCP/UDP 포트 번호 | URL, HTTP 헤더, 쿠키, 페이로드 내용 |
| 로드밸런싱 | 포트 기반 (단순) | 콘텐츠 기반 (정교함) |
| 보안 기능 | 기본적인 방화벽 | WAF(웹 방화벽), DDoS 방어 |
| 처리 속도 | 빠름 | 상대적으로 느림 (페이로드 분석 필요) |
| 가격 | 낮음 | 높음 |
| 사용 사례 | 서버 팜 로드밸런싱 | API 게이트웨이, CDN, WAF |

## 트러블슈팅 시 계층별 접근법

네트워크 장애 발생 시 **하위 계층(Layer 1)부터 상위 계층(Layer 7) 순서**로 원인을 추적하는 것이 효율적이다.

### 계층별 진단 도구 및 방법

```
Layer 7 (Application)
  - 진단 도구: curl, wget, 브라우저 개발자 도구
  - 확인 사항: HTTP 응답 코드, API 응답, 애플리케이션 로그
  - 명령어 예시: curl -v https://example.com

Layer 4 (Transport)
  - 진단 도구: netstat, ss, telnet, nmap
  - 확인 사항: 포트 오픈 여부, TCP 연결 상태
  - 명령어 예시: netstat -an | grep ESTABLISHED
               telnet example.com 80

Layer 3 (Network)
  - 진단 도구: ping, traceroute (tracert), ip route
  - 확인 사항: IP 연결성, 라우팅 경로, 패킷 손실
  - 명령어 예시: ping 8.8.8.8
               traceroute example.com
               ip route show

Layer 2 (Data Link)
  - 진단 도구: arp, ip neighbor
  - 확인 사항: ARP 테이블, MAC 주소 충돌
  - 명령어 예시: arp -a
               ip neighbor show

Layer 1 (Physical)
  - 진단 도구: ip link show, ethtool
  - 확인 사항: 케이블 연결, 인터페이스 UP/DOWN 상태, 신호 세기
  - 명령어 예시: ip link show
               ethtool eth0
```

### 트러블슈팅 시나리오 예시

**시나리오**: 웹 서버에 접속이 안 되는 상황

```
Step 1. 물리 계층 확인 (Layer 1)
  → ip link show eth0
  → 결과: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> ...
  → 판단: 물리 계층 정상

Step 2. 네트워크 계층 확인 (Layer 3)
  → ping 8.8.8.8
  → 결과: 64 bytes from 8.8.8.8: icmp_seq=1 ttl=117 time=5.23 ms
  → 판단: 외부 IP 통신 정상

Step 3. DNS 확인 (Layer 7 일부)
  → nslookup example.com
  → 결과: Non-existent domain
  → 판단: DNS 해석 실패 → DNS 서버 설정 확인 필요

  → /etc/resolv.conf 또는 DNS 서버 설정 점검
  → 문제 원인: nameserver가 잘못된 IP로 설정됨
```

**시나리오**: 특정 포트로 연결이 안 되는 상황

```
Step 1. ping으로 IP 연결 확인 (Layer 3)
  → ping target-server
  → 결과: 정상 응답

Step 2. telnet으로 포트 연결 확인 (Layer 4)
  → telnet target-server 8080
  → 결과: Connection refused
  → 판단: 포트 8080 닫혀있음

Step 3. 서버에서 포트 리스닝 확인
  → ss -tlnp | grep 8080
  → 결과: 리스닝 없음
  → 판단: 애플리케이션이 기동되지 않았거나 방화벽에 의해 차단됨
```

## 핵심 정리

- **OSI 모델은 7계층**: 물리(1) → 데이터링크(2) → 네트워크(3) → 전송(4) → 세션(5) → 표현(6) → 응용(7)
- **PDU 명칭**: 물리=비트, 데이터링크=프레임, 네트워크=패킷, 전송=세그먼트, 상위 3계층=데이터
- **캡슐화**: 송신 측에서 상위→하위 계층으로 내려가며 각 계층이 헤더를 추가하는 과정
- **역캡슐화**: 수신 측에서 하위→상위 계층으로 올라가며 각 계층이 헤더를 제거하는 과정
- **TCP/IP 4계층**: OSI 7계층을 실용적으로 단순화한 모델 (응용/전송/인터넷/네트워크 액세스)
- **L4 스위치**: IP + 포트 기반 로드밸런싱, L7 스위치: URL/헤더/쿠키 기반 콘텐츠 스위칭
- **트러블슈팅**: Layer 1부터 Layer 7 방향으로 진단 도구를 활용하여 단계적으로 원인 추적
- **라우터는 Layer 3 장비**: IP 주소 기반 라우팅 / 스위치는 Layer 2 장비: MAC 주소 기반 전달

## 키워드

`OSI` - 국제표준화기구(ISO)가 제정한 개방형 시스템 상호연결 참조 모델. 네트워크 통신을 7계층으로 분리하여 정의한 표준

`TCP/IP` - 인터넷 실제 구현에 사용되는 4계층 프로토콜 스택 (응용/전송/인터넷/네트워크 액세스)

`캡슐화` - 송신 측에서 상위 계층에서 하위 계층으로 데이터를 전달할 때 각 계층이 헤더(및 트레일러)를 추가하는 과정

`PDU` - Protocol Data Unit. 각 계층에서 다루는 데이터 단위 (비트/프레임/패킷/세그먼트/데이터)

`라우팅` - 네트워크 계층(Layer 3)에서 IP 주소를 기반으로 패킷의 최적 전달 경로를 결정하는 과정

`MAC` - Media Access Control. 데이터 링크 계층에서 사용하는 48비트의 물리적 하드웨어 주소

`포트` - 전송 계층(Layer 4)에서 프로세스를 식별하는 16비트 번호 (0~65535). TCP/UDP 헤더에 포함됨

`프로토콜 스택` - 네트워크 통신을 위해 계층적으로 쌓인 프로토콜의 집합. 각 계층의 프로토콜이 협력하여 데이터를 전달

`L4 스위치` - 전송 계층(Layer 4)에서 동작하는 스위치. IP 주소와 TCP/UDP 포트 번호를 기반으로 트래픽을 로드밸런싱

`L7 스위치` - 응용 계층(Layer 7)에서 동작하는 스위치. URL, HTTP 헤더, 쿠키, 페이로드 내용까지 분석하여 정교한 트래픽 라우팅 수행
