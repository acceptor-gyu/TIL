# Claude Code에서 메인 세션과 서브 에이전트가 동작하는 세션의 차이와 호출 및 동작 원리

## 개요

Claude Code는 메인 세션(Main Session)과 서브 에이전트 세션(Sub-Agent Session)이라는 두 가지 실행 컨텍스트를 가진다. 이 두 세션은 권한, 도구 접근, 컨텍스트 윈도우, 라이프사이클 등에서 근본적인 차이가 있으며, 이를 이해해야 효과적인 자동화 파이프라인을 설계할 수 있다.

## 상세 내용

### 메인 세션(Main Session)의 특성

메인 세션은 사용자와 직접 상호작용하는 최상위 실행 단위다.

- **전체 컨텍스트 윈도우 사용**: 대화 전체 내용, 읽은 파일, 실행 결과가 누적된다
- **모든 도구에 대한 접근 권한 보유**: 설정한 권한 범위 내에서 Read, Write, Edit, Bash, Grep, Glob, MCP 도구 등을 사용할 수 있다
- **세션 상태 지속**: Working Directory, 환경 변수 등 세션 상태가 유지된다
- **사용자 권한 승인 흐름**: 파일 쓰기, Bash 명령 등 시스템 수정 작업에 대해 사용자 확인을 요청한다
- **서브 에이전트 오케스트레이션**: Agent 도구를 통해 서브 에이전트를 호출하고 결과를 조합하는 역할을 한다

### 서브 에이전트 세션(Sub-Agent Session)의 특성

서브 에이전트는 메인 세션에서 호출되는 독립적인 실행 단위다.

- **격리된 컨텍스트 윈도우**: 자체 컨텍스트 윈도우를 가지며, 메인 세션의 대화 히스토리를 공유하지 않는다
- **커스텀 시스템 프롬프트**: 서브 에이전트 정의의 마크다운 본문이 시스템 프롬프트가 된다. 기본 Claude Code 시스템 프롬프트가 아닌 이 프롬프트만 받는다
- **도구 접근 제한 가능**: `tools` (허용 목록) 또는 `disallowedTools` (거부 목록)로 특정 도구만 허용하거나 차단할 수 있다
- **독립적인 모델 선택**: `model` 필드로 메인 세션과 다른 모델(예: Haiku, Sonnet, Opus)을 사용할 수 있다
- **작업 완료 후 결과만 반환**: 중간 과정은 서브 에이전트 컨텍스트에 남고, 최종 결과 요약만 메인 세션으로 돌아온다
- **서브 에이전트는 다른 서브 에이전트를 생성할 수 없다**: 중첩 위임이 필요하다면 메인 세션에서 순차적으로 호출하거나 Skills를 활용해야 한다

### 호출 원리와 메커니즘

서브 에이전트는 세 가지 방식으로 호출된다.

**1. 자동 위임 (Automatic Delegation)**

Claude가 요청 내용과 서브 에이전트의 `description` 필드를 비교해 적합한 서브 에이전트에 자동으로 위임한다.

```text
테스트 스위트를 실행하고 실패한 테스트와 에러 메시지만 보고해줘
```

위 요청이 들어오면 Claude는 테스트 실행에 적합한 서브 에이전트를 자동으로 선택한다. description에 "proactively"를 포함하면 Claude가 더 적극적으로 해당 서브 에이전트를 활용한다.

**2. @-mention을 통한 명시적 호출**

```text
@"code-reviewer (agent)" 최근 변경된 인증 로직을 검토해줘
```

특정 서브 에이전트를 반드시 실행하고 싶을 때 사용한다. Claude가 전달받은 메시지를 바탕으로 서브 에이전트에 넘길 프롬프트를 구성한다.

**3. --agent 플래그로 세션 전체를 서브 에이전트로 실행**

```bash
claude --agent code-reviewer
```

메인 스레드 자체가 해당 서브 에이전트의 시스템 프롬프트, 도구 제한, 모델을 그대로 사용한다. 프로젝트 설정에서 기본값으로 지정할 수도 있다.

```json
{
  "agent": "code-reviewer"
}
```

### 컨텍스트 윈도우 관리 차이

| 항목 | 메인 세션 | 서브 에이전트 세션 |
| --- | --- | --- |
| 컨텍스트 공유 | 대화 전체 누적 | 독립적으로 시작 |
| 탐색 결과 | 메인 컨텍스트에 남음 | 서브 에이전트 컨텍스트에만 존재 |
| 반환 내용 | - | 결과 요약만 메인 세션으로 반환 |
| 자동 압축 | 약 95% 용량에서 트리거 | 동일한 로직으로 독립 압축 |
| 트랜스크립트 저장 | `~/.claude/projects/.../` | `~/.claude/projects/.../{sessionId}/subagents/agent-{agentId}.jsonl` |

서브 에이전트의 컨텍스트 격리가 특히 유용한 상황은 다음과 같다.

- 테스트 전체 실행, 대용량 로그 분석처럼 출력이 방대한 작업
- 독립적인 병렬 조사 (인증, DB, API 모듈을 각각 다른 서브 에이전트가 동시에 탐색)
- 메인 세션의 컨텍스트를 오염시키지 않아야 하는 탐색 작업

### 권한 모델과 보안

서브 에이전트는 메인 세션의 권한 컨텍스트를 기본으로 상속하되, `permissionMode`를 통해 독립적으로 재정의할 수 있다.

| permissionMode | 동작 |
| --- | --- |
| `default` | 표준 권한 확인 및 프롬프트 표시 |
| `acceptEdits` | 파일 편집 자동 허용 |
| `dontAsk` | 권한 프롬프트 자동 거부 (명시적으로 허용된 도구는 동작) |
| `bypassPermissions` | 권한 프롬프트 생략 (주의 필요) |
| `plan` | 읽기 전용 탐색 모드 |

단, 부모 세션이 `bypassPermissions`를 사용하는 경우, 이 설정이 최우선이며 서브 에이전트에서 재정의할 수 없다. 부모 세션이 auto 모드인 경우, 서브 에이전트는 auto 모드를 상속하고 `permissionMode` 설정은 무시된다.

`bypassPermissions`는 `.git`, `.claude`, `.vscode`, `.idea` 디렉토리에 대한 쓰기는 여전히 확인을 요청하며, `.claude/commands`, `.claude/agents`, `.claude/skills`는 예외다.

### 도구 접근 제어 (allowedTools / disallowedTools)

**허용 목록 방식 (tools 필드)**

명시된 도구만 사용 가능하다.

```yaml
---
name: safe-researcher
description: 읽기 전용 탐색 에이전트
tools: Read, Grep, Glob, Bash
---
```

**거부 목록 방식 (disallowedTools 필드)**

메인 세션의 모든 도구를 상속하되, 명시된 도구만 차단한다.

```yaml
---
name: no-writes
description: 파일 쓰기 불가 에이전트
disallowedTools: Write, Edit
---
```

두 필드가 동시에 설정된 경우, `disallowedTools`가 먼저 적용된 후 `tools`가 남은 풀에서 해결된다. 양쪽에 모두 있는 도구는 제거된다.

특정 서브 에이전트를 차단하려면 `settings.json`의 `deny` 배열에 `Agent(서브에이전트명)` 형식으로 추가한다.

```json
{
  "permissions": {
    "deny": ["Agent(Explore)", "Agent(my-custom-agent)"]
  }
}
```

### 상태 공유와 격리

메인 세션과 서브 에이전트 세션 사이에 메모리 상태는 공유되지 않는다. 상태를 공유해야 한다면 파일 시스템을 통해 간접적으로 이루어진다.

**Bash 도구의 cwd 리셋 동작**은 중요한 주의 사항이다. Bash 도구의 호출 간에 현재 작업 디렉토리(cwd)가 리셋된다. 따라서 스크립트 내에서 절대 경로를 사용하거나, 각 Bash 호출에서 명시적으로 디렉토리를 지정해야 한다.

**서브 에이전트 트랜스크립트**는 메인 세션과 독립적으로 저장된다. 메인 세션이 압축(compact)되어도 서브 에이전트 트랜스크립트는 영향을 받지 않는다. 세션을 재개하면 이전 서브 에이전트를 resume할 수도 있다.

```text
// 서브 에이전트 resume 예시
code-reviewer 서브 에이전트로 인증 모듈을 검토해줘
[에이전트 완료]

이전 코드 리뷰를 계속해서 인가(authorization) 로직도 분석해줘
[Claude가 이전 컨텍스트 전체를 가진 채로 서브 에이전트를 resume]
```

### 내장 서브 에이전트 (Built-in Subagents)

Claude Code는 기본 제공 서브 에이전트를 포함한다.

| 서브 에이전트 | 모델 | 도구 | 용도 |
| --- | --- | --- | --- |
| Explore | Haiku (빠른 응답) | 읽기 전용 | 코드베이스 탐색 및 파일 검색 |
| Plan | 메인 세션 상속 | 읽기 전용 | Plan 모드에서 컨텍스트 수집 |
| General-purpose | 메인 세션 상속 | 전체 도구 | 복잡한 다단계 작업 |
| Bash | 메인 세션 상속 | Bash | 별도 컨텍스트에서 터미널 명령 실행 |

Explore와 Plan은 읽기 전용 도구만 허용되어 파일 시스템을 수정할 수 없다. Plan 서브 에이전트는 서브 에이전트가 다른 서브 에이전트를 생성할 수 없다는 제약에서도 작동하도록 설계된 것이다.

### 실무 활용 시 설계 패턴

**메인 세션을 사용해야 하는 경우**

- 빈번한 상호작용이나 반복적인 개선이 필요한 작업
- 계획 → 구현 → 테스트처럼 단계 간 컨텍스트를 공유해야 하는 경우
- 작고 명확한 범위의 변경 작업
- 지연이 중요한 경우 (서브 에이전트는 새 컨텍스트로 시작하므로 준비 시간 필요)

**서브 에이전트를 사용해야 하는 경우**

- 방대한 출력을 생성하는 작업 (테스트 실행, 로그 분석 등)
- 특정 도구 제한이나 권한 설정이 필요한 경우
- 결과 요약만 메인 컨텍스트에 필요한 자기 완결적 작업
- 독립적인 병렬 조사 (각 서브 에이전트가 동시에 다른 영역을 탐색)

**서브 에이전트 체이닝 패턴**

```text
code-reviewer 서브 에이전트로 성능 문제를 찾고,
그 결과를 optimizer 서브 에이전트에 전달해서 수정해줘
```

각 서브 에이전트가 작업을 완료하고 결과를 메인 세션의 Claude에게 반환하면, Claude가 다음 서브 에이전트에 관련 컨텍스트를 전달한다.

**병렬 실행 패턴**

```text
인증, 데이터베이스, API 모듈을 각각 별도의 서브 에이전트로 병렬 조사해줘
```

단, 서브 에이전트가 완료되면 결과가 메인 세션으로 반환되므로, 각 서브 에이전트의 결과가 상세할수록 메인 컨텍스트를 많이 소비한다는 점을 고려해야 한다. 지속적인 병렬 작업이나 컨텍스트 윈도우를 초과하는 작업은 Agent Teams를 고려한다.

## 핵심 정리

- 메인 세션은 사용자와 직접 상호작용하며 전체 도구와 컨텍스트에 접근 가능한 최상위 실행 단위이다
- 서브 에이전트는 독립된 컨텍스트 윈도우에서 실행되며, 작업 완료 후 결과만 반환하는 격리된 실행 단위이다
- 서브 에이전트의 컨텍스트 격리는 토큰 효율성을 높이고, 메인 세션의 컨텍스트 오염을 방지한다
- 권한 모델에서 서브 에이전트는 메인 세션의 설정을 상속받되, `tools`, `disallowedTools`, `permissionMode`로 독립적으로 제어할 수 있다
- 세션 간 상태 공유는 파일 시스템을 통해 간접적으로 이루어지며, 메모리 상태는 공유되지 않는다
- 서브 에이전트는 다른 서브 에이전트를 생성할 수 없다. 중첩 위임이 필요하면 메인 세션에서 순차 호출하거나 Skills를 활용한다
- Bash 도구는 각 호출 간 cwd가 리셋되므로 절대 경로 사용이 안전하다
