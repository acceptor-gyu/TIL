# 제한 시간이 있는 게임에서 중도 종료한 유저의 게임 데이터 관리하기

## 개요
한 세트에 10개의 문제가 있고, 각 문제별 제한 시간이 존재하는 게임에서 유저가 중간에 게임을 종료했을 때 시도한 게임 데이터를 어떻게 관리할 것인가에 대한 설계 기록

---

## 문제 상황 정의

- 게임 한 세트 = 10문제, 각 문제별 개별 제한 시간 존재
- 유저가 5번 문제까지 풀고 앱을 종료하거나 이탈하는 경우
- 풀지 않은 나머지 5문제의 상태 처리 방식

이탈 유형은 크게 두 가지로 나뉜다.

| 이탈 유형 | 설명 | 감지 방법 |
|---|---|---|
| 명시적 종료 | 유저가 직접 종료 버튼을 누름 | 클라이언트 이벤트 전송 |
| 암묵적 이탈 | 앱 강제 종료, 네트워크 단절, 화면 이탈 | 하트비트 누락, 소켓 끊김 |

두 경우를 구분하는 것이 중요한데, 명시적 종료는 즉시 세션 상태를 `ABANDONED`로 전환할 수 있지만 암묵적 이탈은 서버 측에서 감지하기까지 시간이 걸리기 때문이다.

---

## 게임 세션 상태 설계

세션은 생성부터 종료까지 명확한 상태 머신(State Machine)으로 관리하는 것이 핵심이다. 상태가 불명확하면 점수 집계, 재시도 방지, 통계 산출 모두 어려워진다.

### 세션 상태 정의

```
IN_PROGRESS  →  COMPLETED   (모든 문제 응답 완료)
IN_PROGRESS  →  ABANDONED   (유저 명시적 종료 or 하트비트 타임아웃)
IN_PROGRESS  →  TIMEOUT     (세션 전체 유효 시간 초과)
```

| 상태 | 설명 |
|---|---|
| `IN_PROGRESS` | 게임이 진행 중인 상태. 문제 응답을 받을 수 있음 |
| `COMPLETED` | 10문제를 모두 응답(정답/오답/시간초과 포함)하여 정상 종료 |
| `ABANDONED` | 유저가 중간에 이탈하여 미완료로 종료 |
| `TIMEOUT` | 세션 전체에 부여된 유효 시간(예: 세션 생성 후 30분)이 지나 만료 |

### ABANDONED vs TIMEOUT 구분 기준

`ABANDONED`와 `TIMEOUT`을 굳이 구분하는 이유는 이후 통계 분석과 사용자 경험 개선에 활용하기 위해서다.

- `ABANDONED`: 유저의 **의도적 또는 환경적 이탈**. 특정 문제에서 이탈률이 높다면 해당 문제의 난이도 또는 UX 문제를 의심할 수 있다.
- `TIMEOUT`: 세션 자체의 **유효 기간 만료**. 유저가 게임 도중 장시간 자리를 비우거나 앱을 백그라운드로 내린 채 방치한 경우.

두 상태를 동일하게 처리하면 통계가 오염된다. 예를 들어 "3번 문제에서 이탈률 40%"라는 지표는 ABANDONED 기준으로만 집계해야 의미 있다.

### 세션 생성 및 종료 시점

- **생성 시점**: 유저가 게임 시작 버튼을 누른 순간 (클라이언트 → 서버 API 호출). 이 시점에 세션 ID와 문제 세트가 서버에 저장된다.
- **종료 시점**: COMPLETED / ABANDONED / TIMEOUT 중 하나의 상태로 전환될 때.

세션이 `IN_PROGRESS` 상태인 채로 중복 생성되는 것을 방지하기 위해, 세션 생성 API에서 해당 유저에게 이미 `IN_PROGRESS` 세션이 존재하는지 체크하는 로직이 필요하다. (이는 동시성 문제 섹션에서 상세히 다룬다)

## 문제별 데이터 저장 전략

### 전략 1: 문제를 풀 때마다 서버에 즉시 저장

유저가 각 문제에 응답하는 순간 즉시 서버 API를 호출해 `GameAnswer`를 저장한다.

**장점**
- 클라이언트가 갑자기 종료되어도 이미 응답한 문제의 데이터는 손실 없이 보존된다.
- 서버가 각 문제 응답 시점에 유효성 검증(제한 시간 초과 여부, 세션 상태 등)을 수행할 수 있다.
- 중도 종료 시점까지의 데이터를 완전하게 수집할 수 있어 통계 품질이 높다.

**단점**
- 문제 수만큼 API 호출이 발생한다 (10문제면 최대 10번).
- 네트워크 지연이 빈번하면 UX에 영향을 줄 수 있다.
- 서버 부하가 증가한다.

### 전략 2: 클라이언트 로컬 저장 후 세션 종료 시 일괄 전송

클라이언트에서 응답 데이터를 로컬(메모리 또는 로컬 스토리지)에 쌓아두다가 게임이 완료되면 한 번의 API 호출로 전송한다.

**장점**
- API 호출 횟수가 최소화된다.
- 네트워크 환경이 불안정해도 게임 진행 자체에는 영향이 없다.

**단점**
- 유저가 중도 이탈하면 로컬에 저장된 데이터가 서버로 전송되지 않아 **데이터 유실** 발생.
- 앱 강제 종료(OOM Kill 등)의 경우 메모리에 있는 데이터까지 사라진다.
- 클라이언트 측 데이터 조작(변조) 가능성이 높아진다.

### 전략 선택

중도 종료 유저의 데이터를 보존하는 것이 핵심 요구사항이라면 **전략 1(즉시 저장)** 이 유일한 선택지다.

단, API 호출 부하를 줄이기 위해 **하이브리드 방식**을 고려할 수 있다.

- 문제 응답 시: 클라이언트에서 임시 저장
- 하트비트 전송 시 (예: 5초마다): 아직 서버에 전송되지 않은 응답을 함께 포함하여 전송 (piggyback)
- 세션 종료 시: 최종 미전송 데이터 일괄 전송

이 방식은 API 호출 횟수를 줄이면서도 데이터 유실 위험을 최소화할 수 있다.

## 중도 종료 감지 방법

### 1. 클라이언트 하트비트 (Heartbeat)

클라이언트가 주기적으로 (예: 10초마다) 서버에 "나 아직 살아있어" 신호를 보내고, 서버가 일정 시간(예: 30초) 동안 하트비트를 받지 못하면 세션을 `ABANDONED`로 전환하는 방식이다.

```kotlin
// 클라이언트 (Android/Kotlin)
fun startHeartbeat(sessionId: String) {
    heartbeatJob = CoroutineScope(Dispatchers.IO).launch {
        while (isActive) {
            delay(10_000L) // 10초마다
            runCatching {
                gameApi.heartbeat(sessionId)
            }
        }
    }
}

fun stopHeartbeat() {
    heartbeatJob?.cancel()
}
```

```kotlin
// 서버 (Spring/Kotlin): Redis를 이용한 하트비트 TTL 관리
@Service
class HeartbeatService(
    private val redisTemplate: StringRedisTemplate,
    private val gameSessionRepository: GameSessionRepository
) {
    fun refreshHeartbeat(sessionId: String) {
        // 30초 TTL 설정. TTL이 만료되면 스케줄러가 세션을 ABANDONED 처리
        redisTemplate.opsForValue().set(
            "heartbeat:$sessionId", "alive",
            Duration.ofSeconds(30)
        )
    }
}
```

서버에서 하트비트 TTL 만료를 감지하는 방법은 두 가지다.

- **Redis Keyspace Notification**: Redis에서 키가 만료될 때 이벤트를 발행하고, 서버가 이를 구독하여 세션 상태를 변경.
- **스케줄러 폴링**: 주기적으로 `IN_PROGRESS` 상태인 세션 중 마지막 하트비트 시간이 임계값을 초과한 것을 조회하여 처리.

Redis Keyspace Notification이 반응성은 좋지만, 설정 복잡도와 이벤트 유실 가능성을 고려하면 소규모에서는 스케줄러 폴링이 더 안정적이다.

### 2. WebSocket 연결 끊김 감지

게임이 WebSocket 기반이라면, 소켓 연결이 끊기는 시점에 즉시 감지할 수 있다.

```kotlin
// Spring WebSocket Handler
override fun afterConnectionClosed(session: WebSocketSession, status: CloseStatus) {
    val sessionId = session.attributes["gameSessionId"] as? String ?: return
    gameSessionService.markAsAbandoned(sessionId, reason = "WEBSOCKET_DISCONNECT")
}
```

단, 네트워크 일시적 단절(터널 통과, Wi-Fi 전환 등)에도 반응하므로, 즉시 `ABANDONED` 처리하지 않고 **재연결 유예 시간(Grace Period)**을 두는 것이 좋다. 예를 들어 연결이 끊기면 30초 유예를 주고, 그 안에 재연결하면 세션을 유지한다.

### 3. 앱 라이프사이클 이벤트 활용

모바일 앱에서는 `onPause`, `onStop` 이벤트를 활용해 백그라운드 전환 시점을 감지할 수 있다.

```kotlin
// Android Activity / Fragment
override fun onStop() {
    super.onStop()
    // 백그라운드 전환 시 하트비트 중단 및 서버에 일시정지 알림
    viewModel.pauseGame(sessionId)
}

override fun onDestroy() {
    super.onDestroy()
    // 명시적 종료
    viewModel.abandonGame(sessionId)
}
```

다만 `onDestroy`는 항상 호출된다고 보장할 수 없다 (프로세스 강제 종료 시). 따라서 라이프사이클 이벤트에만 의존하지 않고, **하트비트 + 라이프사이클 이벤트**를 함께 사용하는 이중 안전장치가 필요하다.

## 미완료 문제 처리 정책

유저가 5번 문제까지 풀고 이탈했을 때, 6~10번 문제를 어떻게 처리할 것인가는 서비스의 성격에 따라 달라진다.

### 정책 A: 미시도 문제를 오답으로 처리

```sql
-- 세션이 ABANDONED로 전환될 때 미응답 문제를 일괄 오답 처리
UPDATE game_answer
SET status = 'WRONG', answered_at = NOW()
WHERE session_id = :sessionId
  AND status = 'PENDING';
```

**적합한 경우**: 경쟁 랭킹 시스템에서 이탈이 전략적으로 남용될 수 있는 경우 (예: 틀릴 것 같으면 종료). 이탈에 불이익을 주어 완주를 유도하는 효과가 있다.

**문제점**: 네트워크 장애처럼 비자발적 이탈도 오답으로 처리된다. 유저 불만이 생길 수 있다.

### 정책 B: 미시도 문제를 별도 상태(NOT_ATTEMPTED)로 유지

```sql
-- game_answer 테이블의 status 값
-- CORRECT, WRONG, TIMEOUT(개별 문제 시간초과), NOT_ATTEMPTED
```

미응답 문제에 `NOT_ATTEMPTED` 상태를 부여하고, 점수 계산 시 이를 0점으로 환산한다.

**적합한 경우**: 학습 앱, 비경쟁 퀴즈처럼 유저 경험이 우선인 경우. 통계적으로도 "응답하지 않은 문제"와 "틀린 문제"를 구분하여 분석할 수 있다.

**점수 계산 예시**:

```kotlin
data class SessionResult(
    val totalQuestions: Int,       // 10
    val attemptedQuestions: Int,   // 5
    val correctCount: Int,         // 3
    val wrongCount: Int,           // 2
    val notAttemptedCount: Int,    // 5
    val score: Int                 // correctCount * pointPerQuestion
)
```

### 정책 C: 세션 완료 여부에 따라 점수 자체를 무효화

중도 종료된 세션은 랭킹/기록에 반영하지 않는다. 완료된 세션만 집계에 사용하고, 중도 종료 세션은 학습용 데이터로만 보존.

**이 프로젝트의 선택**: 정책 B (NOT_ATTEMPTED)를 채택했다. 학습을 지원하는 앱의 특성상 강제 오답 처리는 부적절하고, 데이터 분석 측면에서도 더 풍부한 인사이트를 얻을 수 있다. 단, `COMPLETED` 세션만 공식 랭킹에 반영하여 중도 이탈 남용을 방지했다.

## 데이터 모델 설계

### GameSession 엔티티

```kotlin
@Entity
@Table(name = "game_session")
class GameSession(

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long = 0,

    @Column(nullable = false)
    val userId: Long,

    @Column(nullable = false)
    val questionSetId: Long,  // 어떤 문제 세트인지

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    var status: GameSessionStatus = GameSessionStatus.IN_PROGRESS,

    @Column(nullable = false)
    val startedAt: LocalDateTime = LocalDateTime.now(),

    var completedAt: LocalDateTime? = null,

    var lastHeartbeatAt: LocalDateTime? = null,

    @Column(nullable = false)
    val sessionToken: String = UUID.randomUUID().toString(),  // 클라이언트 식별용

    @Version
    val version: Long = 0  // 낙관적 락
)

enum class GameSessionStatus {
    IN_PROGRESS, COMPLETED, ABANDONED, TIMEOUT
}
```

### GameAnswer 엔티티

```kotlin
@Entity
@Table(
    name = "game_answer",
    uniqueConstraints = [UniqueConstraint(columnNames = ["session_id", "question_id"])]
)
class GameAnswer(

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    val id: Long = 0,

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "session_id", nullable = false)
    val session: GameSession,

    @Column(nullable = false)
    val questionId: Long,

    @Column(nullable = false)
    val questionOrder: Int,  // 세트 내 문제 순서 (1~10)

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    var status: AnswerStatus = AnswerStatus.NOT_ATTEMPTED,

    var selectedOptionId: Long? = null,  // 유저가 선택한 보기 ID

    var answeredAt: LocalDateTime? = null,

    var timeTakenMs: Long? = null,  // 응답까지 걸린 시간 (ms)

    @Column(nullable = false)
    val idempotencyKey: String  // 중복 요청 방지용 클라이언트 생성 UUID
)

enum class AnswerStatus {
    NOT_ATTEMPTED, CORRECT, WRONG, TIMEOUT  // 개별 문제 시간 초과
}
```

## 재접속 및 이어하기 정책

### 이어하기 허용 시

재접속 시 `IN_PROGRESS` 세션이 존재하면 해당 세션을 이어서 진행한다. 이 경우 남은 시간 계산이 중요하다.

```kotlin
fun getRemainingTime(questionAnswer: GameAnswer, limitSeconds: Int): Long {
    val elapsedMs = questionAnswer.timeTakenMs ?: 0L
    val limitMs = limitSeconds * 1000L
    return maxOf(0L, limitMs - elapsedMs)
}
```

단, **이어하기 허용 기간**을 제한해야 한다. 예를 들어 세션 시작 후 30분이 지나면 이어하기 불가로 설정한다. 그렇지 않으면 유저가 쉬운 답을 찾아볼 시간을 벌 수 있다.

```kotlin
fun canResume(session: GameSession): Boolean {
    val elapsed = Duration.between(session.startedAt, LocalDateTime.now())
    return session.status == GameSessionStatus.IN_PROGRESS
        && elapsed.toMinutes() < 30  // 30분 이내만 이어하기 허용
}
```

### 이어하기 불허 시

재접속 시 기존 `IN_PROGRESS` 세션을 즉시 `ABANDONED` 처리하고 새 세션을 생성하게 할 수 있다. 이 방식은 단순하지만 유저 경험이 나쁠 수 있다.

### 이 프로젝트의 선택

각 문제에 **개별 제한 시간**이 있고, 세션 중 이탈 시간 동안 유저가 답을 찾아볼 가능성이 있기 때문에 이어하기를 불허하기로 결정했다. 재접속 시 기존 세션을 `ABANDONED`로 변경하고 새 세션을 시작한다. 단, `ABANDONED` 세션의 응답 데이터는 삭제하지 않고 보존한다.

## 동시성 및 데이터 정합성

### 중복 세션 생성 방지

같은 유저가 빠르게 두 번 게임 시작 버튼을 누르거나, 클라이언트 버그로 중복 요청이 오는 경우를 방지해야 한다.

**방법 1: DB 유니크 인덱스 + 예외 처리**

```sql
-- user_id에 대해 IN_PROGRESS 세션이 이미 존재하면 INSERT 실패
-- (Partial Unique Index - MySQL에서는 Filtered Index가 없으므로 애플리케이션 레벨 체크 필요)
```

MySQL은 조건부 유니크 인덱스를 지원하지 않으므로 애플리케이션 레벨에서 처리한다.

```kotlin
@Transactional
fun createSession(userId: Long, questionSetId: Long): GameSession {
    val existing = gameSessionRepository.findByUserIdAndStatus(
        userId, GameSessionStatus.IN_PROGRESS
    )
    if (existing != null) {
        throw GameSessionAlreadyExistsException("이미 진행 중인 세션이 있습니다.")
    }
    return gameSessionRepository.save(GameSession(userId = userId, questionSetId = questionSetId))
}
```

**방법 2: Redis 분산 락 활용**

동시 요청이 많은 경우 DB 조회만으로는 Race Condition이 발생할 수 있다. Redis 분산 락으로 세션 생성 구간을 보호한다.

```kotlin
fun createSessionWithLock(userId: Long, questionSetId: Long): GameSession {
    val lockKey = "session:create:lock:$userId"
    return redisLockService.withLock(lockKey, timeout = 3.seconds) {
        createSession(userId, questionSetId)
    }
}
```

### 답변 제출 멱등성 보장

네트워크 지연으로 클라이언트가 같은 답변을 두 번 전송하는 경우를 처리해야 한다.

클라이언트는 각 답변 제출 요청에 고유한 `idempotencyKey` (UUID v4)를 생성하여 포함한다. 서버는 이 키가 이미 처리된 요청인지 확인한다.

```kotlin
@Transactional
fun submitAnswer(request: SubmitAnswerRequest): SubmitAnswerResponse {
    // 멱등성 키로 기존 처리 결과 조회
    val existing = gameAnswerRepository.findByIdempotencyKey(request.idempotencyKey)
    if (existing != null) {
        // 이미 처리된 요청 → 동일한 응답 반환
        return SubmitAnswerResponse.from(existing)
    }

    val session = gameSessionRepository.findById(request.sessionId)
        ?: throw SessionNotFoundException()

    // 세션 상태 검증
    if (session.status != GameSessionStatus.IN_PROGRESS) {
        throw SessionNotActiveException()
    }

    // 서버 측 제한 시간 검증
    val questionLimit = questionService.getTimeLimitMs(request.questionId)
    if (request.timeTakenMs > questionLimit) {
        throw QuestionTimeExceededException()
    }

    val answer = GameAnswer(
        session = session,
        questionId = request.questionId,
        questionOrder = request.questionOrder,
        status = if (request.isCorrect) AnswerStatus.CORRECT else AnswerStatus.WRONG,
        selectedOptionId = request.selectedOptionId,
        answeredAt = LocalDateTime.now(),
        timeTakenMs = request.timeTakenMs,
        idempotencyKey = request.idempotencyKey
    )

    return SubmitAnswerResponse.from(gameAnswerRepository.save(answer))
}
```

### 서버 타임아웃과 클라이언트 마지막 전송 충돌 처리

스케줄러가 세션을 `ABANDONED`로 변경하는 시점과 클라이언트의 마지막 답변 전송이 동시에 발생하는 Race Condition이 존재한다.

이를 해결하기 위해 `GameSession` 엔티티에 `@Version`을 적용한 **낙관적 락(Optimistic Lock)** 을 사용한다.

```kotlin
// 세션 상태 변경 시 버전 충돌 감지
@Transactional
fun markAsAbandoned(sessionId: Long) {
    val session = gameSessionRepository.findByIdWithLock(sessionId)
        ?: return

    if (session.status != GameSessionStatus.IN_PROGRESS) {
        return  // 이미 다른 트랜잭션에서 상태가 변경된 경우 무시
    }

    session.status = GameSessionStatus.ABANDONED
    session.completedAt = LocalDateTime.now()
    // @Version으로 인해 다른 트랜잭션이 먼저 변경했다면 OptimisticLockException 발생
}
```

`OptimisticLockException`이 발생하면 재시도하거나, 이미 처리된 것으로 간주하고 무시한다. 답변 제출 API에서는 세션 상태가 `IN_PROGRESS`인지 확인하는 검증 로직이 있으므로, 세션이 `ABANDONED`로 전환된 직후 들어온 답변은 자연스럽게 거부된다.

## 핵심 정리

- **세션 상태는 명확한 상태 머신으로 관리한다.** `IN_PROGRESS`, `COMPLETED`, `ABANDONED`, `TIMEOUT`을 구분하면 통계 분석, 재접속 정책, 데이터 정합성 처리가 모두 명확해진다.
- **중도 종료 감지는 하트비트 + 라이프사이클 이벤트 이중 구조로 구성한다.** 라이프사이클 이벤트만 믿으면 강제 종료 시 감지하지 못하고, 하트비트만 믿으면 타임아웃까지 시간이 걸린다.
- **미완료 문제는 NOT_ATTEMPTED 상태로 보존한다.** 오답으로 강제 처리하면 데이터 의미가 희석되고 유저 불만이 생긴다. 상태를 구분해 보존하면 분석 품질이 높아진다.
- **답변 제출 API는 멱등성 키로 중복 요청을 방어한다.** 네트워크 재시도로 인한 이중 저장을 `idempotencyKey` + DB 유니크 제약으로 막는다.
- **세션 생성에는 분산 락을 고려한다.** 동시 요청이 적다면 애플리케이션 레벨 중복 체크로 충분하지만, 트래픽이 높다면 Redis 분산 락으로 Race Condition을 원천 차단한다.
- **서버가 항상 타임 오브 트러스트(Time of Truth)다.** 클라이언트에서 제한 시간을 넘겨 전송한 답변은 서버에서 반드시 검증하여 거부한다. 클라이언트 시계는 조작 가능하다.

## 키워드

- `게임 세션 관리`: 게임 한 판의 시작부터 종료까지 상태를 추적하고 관리하는 개념. 세션 ID, 상태, 생성/종료 시각 등을 서버에서 중앙 관리한다.
- `중도 종료`: 유저가 게임을 완료하지 않고 이탈하는 것. 명시적(버튼) 또는 암묵적(앱 강제 종료, 네트워크 단절) 이탈로 구분된다.
- `세션 타임아웃`: 일정 시간 동안 활동이 없거나 유효 기간이 만료되어 세션이 자동으로 종료되는 메커니즘.
- `하트비트`: 클라이언트가 주기적으로 서버에 생존 신호를 전송하는 패턴. 서버는 하트비트가 끊기면 연결 이상 또는 세션 이탈로 판단한다.
- `멱등성`: 동일한 요청을 여러 번 보내도 결과가 한 번 보낸 것과 동일하게 유지되는 성질. `idempotency-key`(UUID)로 구현하며, 네트워크 재시도로 인한 중복 처리를 방지한다.
- `게임 데이터 저장 전략`: 문제 응답을 즉시 저장할지, 로컬에 쌓다가 일괄 전송할지에 대한 설계 결정. 데이터 보존성과 서버 부하 사이의 트레이드오프가 존재한다.
- `NOT_ATTEMPTED 상태`: 유저가 시도하지 않은 문제의 상태값. 오답(WRONG)과 구분함으로써 분석 데이터의 의미를 정확하게 유지한다.
- `WebSocket`: 클라이언트-서버 간 양방향 지속 연결 프로토콜. 연결 끊김 이벤트를 통해 실시간으로 세션 이탈을 감지할 수 있다.
- `세션 상태 머신`: 세션이 가질 수 있는 상태와 상태 간 전환 규칙을 명확히 정의한 모델. 유효하지 않은 상태 전환(예: COMPLETED → IN_PROGRESS)을 방지한다.
- `데이터 정합성`: 분산 환경에서 동시 요청이 발생할 때도 데이터가 일관된 상태를 유지하는 것. 낙관적 락, 분산 락, 유니크 제약을 조합하여 보장한다.
