# 5,000만 건 테이블 쿼리 튜닝으로 41초에서 2.6초로 성능 개선의 여정

## 문제 인식 — 서버 로그와 DB 로그 분석

API Server Log에서 DB Connection 부족 에러가 발생했고, Slow Query Log를 함께 보니 AWS Lambda에서 실행하는 데이터 분석용 쿼리가 문제라는 것을 특정할 수 있었다. 해당 쿼리가 약 **2분** 동안 DB connection을 점유하며 일부 요청이 연결을 획득하지 못했고, 이 쿼리에서 조회하는 테이블은 1시간마다 약 8,000건씩 쌓이던 테이블로 문제를 인지한 시점에는 **5,000만 건** 이상의 데이터가 누적되어 있었다.

---

## 해결 과정

### 1. 로컬 재현 환경 구축

운영 DB에 직접 손대기 전, Go로 운영과 동일한 구조의 CSV 데이터를 5,000만 건 생성하고 `docker cp` + `LOAD DATA`로 로컬 MySQL에 삽입해 동일한 환경을 재현했다.

```bash
# CSV 파일을 Docker 컨테이너로 복사
docker cp ./coin_candles.csv mysql-container:/tmp/coin_candles.csv

# MySQL에서 대량 데이터 삽입
LOAD DATA INFILE '/tmp/coin_candles.csv'
INTO TABLE coin_candles
FIELDS TERMINATED BY ','
LINES TERMINATED BY '\n';
```

`LOAD DATA`는 row-by-row INSERT와 달리 파싱·인덱스 갱신을 배치로 처리하기 때문에 대용량 초기 적재에 적합하다. 이 과정 없이 운영에 바로 DDL을 적용하면 예측하지 못한 부하나 데이터 손실이 발생할 수 있어, 검증 환경 구축을 최우선으로 삼았다.

---

### 2. 월별 RANGE 파티셔닝 설계

테이블이 시간 단위로 균등하게 쌓이는 특성을 보고 `timestamp` 컬럼 기준 월별 파티셔닝을 선택했다. 데이터가 균등하게 분산될수록 파티션 간 핫스팟이 생기지 않고 Partition Pruning의 효과가 일정하게 유지된다.

MySQL 제약으로 파티셔닝 키가 반드시 PRIMARY KEY에 포함되어야 해서 `PRIMARY KEY (id)`를 `PRIMARY KEY (id, timestamp)`로 변경했다. 이는 단순한 컬럼 추가가 아니라 클러스터드 인덱스 구조 자체가 바뀌는 작업으로, 데이터 물리 정렬 순서에 영향을 미친다는 점도 함께 고려했다.

**기존 테이블 DDL**

```sql
CREATE TABLE coin_candles (
    id bigint AUTO_INCREMENT PRIMARY KEY,
    ticker varchar(255),
    timestamp bigint,
    open double,
    high double,
    low double,
    close double,
    volume double,
    source varchar(255) NULL,
    unit varchar(255) NULL
);
```

**파티셔닝 적용 테이블 DDL**

```sql
CREATE TABLE partitioned_coin_candles (
    id BIGINT AUTO_INCREMENT,
    timestamp BIGINT NOT NULL,
    ticker VARCHAR(255),
    open DOUBLE,
    high DOUBLE,
    low DOUBLE,
    close DOUBLE,
    volume DOUBLE,
    source VARCHAR(255) NULL,
    unit VARCHAR(255) NULL,
    PRIMARY KEY (id, timestamp)
)
PARTITION BY RANGE (timestamp) (
    PARTITION p202401 VALUES LESS THAN (1706713200),
    PARTITION p202402 VALUES LESS THAN (1709218800),
    PARTITION p202403 VALUES LESS THAN (1711897200),
    PARTITION p202404 VALUES LESS THAN (1714489200),
    PARTITION p202405 VALUES LESS THAN (1717167600),
    PARTITION p202406 VALUES LESS THAN (1719759600),
    PARTITION p202407 VALUES LESS THAN (1722438000),
    PARTITION p202408 VALUES LESS THAN (1725116400),
    PARTITION p202409 VALUES LESS THAN (1727708400),
    PARTITION p202410 VALUES LESS THAN (1730386800),
    PARTITION p202411 VALUES LESS THAN (1732978800),
    PARTITION p202412 VALUES LESS THAN (1735657200),
    PARTITION pmax    VALUES LESS THAN MAXVALUE
);
```

### 3. 파티션 생명주기 자동화 설계

`pmax` 파티션은 안전망 역할을 하지만 임시방편에 불과하다. 새 파티션을 추가하지 않으면 모든 데이터가 `pmax`에 몰려 Partition Pruning이 무의미해진다. 이를 막기 위해 매월 말 다음 달 파티션을 자동으로 추가하고, 보존 기간이 지난 파티션은 `DROP PARTITION`으로 제거하는 스크립트를 함께 설계했다.

```sql
-- 새 파티션 추가 (pmax를 분할)
ALTER TABLE partitioned_coin_candles
REORGANIZE PARTITION pmax INTO (
    PARTITION p202501 VALUES LESS THAN (1738335600),
    PARTITION pmax    VALUES LESS THAN MAXVALUE
);

-- 오래된 파티션 삭제
ALTER TABLE partitioned_coin_candles DROP PARTITION p202401;
```

`DROP PARTITION`은 파티션 단위로 파일을 제거하기 때문에 수천만 건의 `DELETE` 문을 실행하는 것보다 I/O 부하가 훨씬 적고 트랜잭션 로그도 발생하지 않는다.

---

### 4. 데이터 보존 정책 정의

오래된 데이터를 무작정 삭제하는 것이 아니라, "얼마나 오래된 데이터까지 실제로 필요한가"를 먼저 정의했다. 알고리즘이 참조하는 최대 시간 범위를 코드 레벨에서 분석해 보존 기간을 결정하고, 그 이전 파티션은 자동 삭제되도록 정책을 설계했다.

1시간에 8,000건이면 연간 약 7,000만 건이 추가되는 구조이므로, 보존 정책 없이는 파티셔닝 효과가 시간이 지날수록 희석된다.

### 5. 파티셔닝 적용 및 성능 측정

로컬 환경에서 `EXPLAIN ANALYZE`로 Partition Pruning이 실제로 동작하는지 확인했다. `WHERE timestamp BETWEEN ...` 조건이 있을 때 실행 계획에서 전체 파티션이 아닌 해당 월 파티션만 스캔되는 것을 눈으로 검증했다.

```sql
EXPLAIN ANALYZE
SELECT * FROM partitioned_coin_candles
WHERE timestamp BETWEEN 1704034800 AND 1704466800;
```

### 6. Covering Index 적용

파티셔닝으로 만족하지 않고 실제 알고리즘 코드를 분석했더니, `SELECT *`로 조회하지만 `timestamp`와 `ticker` 두 컬럼만 사용하고 있었다.

`(timestamp, ticker)` 복합 인덱스를 추가하면 클러스터드 인덱스(데이터 페이지) 접근 없이 세컨더리 인덱스의 리프 노드에서 바로 결과를 반환할 수 있는 **Covering Index**를 구성할 수 있다고 판단했다.

```sql
CREATE TABLE partitioned_coin_candles_covered (
    id BIGINT NOT NULL AUTO_INCREMENT,
    timestamp BIGINT NOT NULL,
    ticker VARCHAR(255) DEFAULT NULL,
    open DOUBLE DEFAULT NULL,
    high DOUBLE DEFAULT NULL,
    low DOUBLE DEFAULT NULL,
    close DOUBLE DEFAULT NULL,
    volume DOUBLE DEFAULT NULL,
    source VARCHAR(255) DEFAULT NULL,
    unit VARCHAR(255) DEFAULT NULL,
    PRIMARY KEY (timestamp, id),
    KEY idx_id (id),
    KEY idx_timestamp_ticker (timestamp, ticker)
)
PARTITION BY RANGE (timestamp) (
    PARTITION p202401 VALUES LESS THAN (1706713200) ENGINE = InnoDB,
    ...
    PARTITION pmax VALUES LESS THAN MAXVALUE ENGINE = InnoDB
);
```

주목할 변경점은 두 가지다.

1. **PRIMARY KEY를 `(timestamp, id)`로 변경** — `timestamp`를 PK의 선두 컬럼으로 두어 파티션 내에서 timestamp 기준 정렬이 유지되도록 했다.
2. **`idx_timestamp_ticker` 복합 인덱스 추가** — 실제 쿼리에서 사용하는 `timestamp`와 `ticker`만으로 구성된 Covering Index다.

#### Covering Index가 큰 성능 차이를 만드는 이유

파티셔닝만 적용했을 때(5.3s)와 Covering Index까지 적용했을 때(2.6s)의 차이는 약 2배다. 단순히 "인덱스를 탔다"는 것만으로는 이 차이를 설명하기 어렵다. 근본적인 원인은 **I/O 접근 방식의 변화**에 있다.

**1. Row Lookup(Random I/O) 제거**

파티셔닝만 적용된 `SELECT *` 쿼리의 실행 경로:

1. Partition Pruning으로 해당 월 파티션을 선택
2. 파티션 내 클러스터드 인덱스(데이터 페이지)를 풀 스캔
3. 모든 컬럼 데이터를 읽어 반환

Covering Index를 적용한 `SELECT timestamp, ticker` 쿼리의 실행 경로:

1. Partition Pruning으로 해당 월 파티션을 선택
2. `idx_timestamp_ticker` 세컨더리 인덱스만 스캔
3. **데이터 페이지에 접근하지 않고** 인덱스 리프 노드에서 바로 결과 반환

즉, 클러스터드 인덱스(데이터 페이지)로의 **Row Lookup이 완전히 사라진다**. Row Lookup은 인덱스의 정렬 순서와 데이터의 물리적 위치가 달라 Random I/O를 유발하는 대표적인 원인이다.

**2. 읽어야 할 페이지 수의 차이**

InnoDB는 데이터를 16KB 페이지 단위로 관리한다. 이 테이블의 한 행에는 약 **800B 이상**의 데이터가 담긴다.

- **데이터 페이지**: 한 페이지에 약 20행 저장 (16KB / 800B)
- **인덱스 페이지** (`timestamp` + `ticker`만 포함): 한 페이지에 약 **50~60행** 저장 (16KB / ~270B)

100만 건 기준으로 읽어야 할 페이지 수:

| 접근 방식 | 읽어야 할 페이지 수 |
|-----------|---------------------|
| 데이터 페이지 풀 스캔 (`SELECT *`) | ~50,000 페이지 |
| 인덱스 페이지만 스캔 (Covering Index) | ~18,000 페이지 |

읽어야 할 페이지 수 자체가 **약 2.8배 적다**.

**3. Buffer Pool 효율 극대화**

인덱스 페이지는 데이터 페이지보다 크기가 작으므로, 동일한 Buffer Pool 공간에 **더 많은 인덱스 페이지를 캐싱**할 수 있다. 이는 Buffer Pool 히트율을 높여 디스크 I/O 빈도를 추가로 감소시킨다.

> Covering Index의 성능 개선은 단순히 "인덱스를 사용해서"가 아니라, **Row Lookup Random I/O 제거 + 읽어야 할 페이지 수 감소 + Buffer Pool 캐싱 효율 증가**라는 세 가지 요인이 복합적으로 작용한 결과다.

### 7. 쓰기 성능 트레이드오프 검토

인덱스 추가는 읽기 성능을 높이지만 쓰기 성능은 저하된다. 이 테이블은 1시간마다 8,000건 INSERT가 발생하므로, `idx_timestamp_ticker` 인덱스를 유지하는 쓰기 비용이 지속적으로 발생한다.

다만 InnoDB의 **Change Buffer** 덕분에 실제 오버헤드는 미미하다. Change Buffer는 세컨더리 인덱스 업데이트를 즉시 디스크에 반영하지 않고 버퍼에 모았다가, 해당 인덱스 페이지가 Buffer Pool에 로드될 때 일괄 적용한다.

100건 배치 INSERT 기준 추정:

| 항목 | 수치 |
|------|------|
| 쓰기 부하 | 8,000건/시간 ≈ 2.2건/초 |
| 인덱스 항목 크기 | timestamp(8B) + ticker(~20B) + PK 포인터(16B) ≈ 44B |
| 100건 인덱스 데이터 | 4.4KB (16KB 페이지 1개 미만) |

| 접근 방식 | 예상 소요 시간 |
|-----------|----------------|
| 인덱스 없음 | ~2~5ms |
| `idx_timestamp_ticker` 있음 | ~2.5~6ms |
| **차이** | **약 0.5~1ms** |

배치 INSERT는 커밋이 한 번이므로 redo log fsync도 한 번이다. 인덱스가 추가되어도 fsync 횟수는 변하지 않고, 추가 비용은 Buffer Pool 내 메모리 수정 수준에 불과하다.

쓰기 부하가 초당 2.2건 수준으로 낮아, 인덱스 추가로 인한 쓰기 성능 저하가 서비스에 영향을 줄 가능성이 없다고 판단하고 적용을 결정했다.

## 실무에서 추가로 고려했어야 할 것들

### 1. 운영 DB 무중단 마이그레이션 전략

로컬에서는 `ALTER TABLE`을 자유롭게 실행했지만, 운영 DB의 5,000만 건 테이블에 파티셔닝 DDL을 적용하면 InnoDB는 **테이블 풀 리빌드**를 수행한다. 수십 분에서 수 시간 동안 테이블 락 또는 심각한 I/O 부하가 발생할 수 있어 서비스 중단으로 이어질 수 있다.

실무에서는 다음 중 하나의 전략이 필요하다.

- **`pt-online-schema-change`** — 원본 테이블을 복사해 새 테이블에 변경 사항을 적용하면서, 그 동안의 DML을 트리거로 동기화한 뒤 테이블을 교체한다.
- **`gh-ost`** — 트리거 대신 binlog를 읽어 변경 사항을 반영하기 때문에 원본 테이블 부하가 더 적다.
- **Blue-Green 전략** — 새 테이블을 병행 운영하다 트래픽을 전환한다.

### 2. Lambda 동시 실행 제어

쿼리 하나의 실행 시간이 2분이더라도, Lambda는 동시에 여러 인스턴스가 실행될 수 있다. 동시 실행 인스턴스가 3개라면 DB connection 3개가 동시에 2분씩 점유되고, 이는 파티셔닝으로 쿼리 시간을 줄이더라도 구조적으로 반복될 수 있는 문제다.

- **Reserved Concurrency** — Lambda의 동시 실행 수를 제한해 connection 점유를 제어한다.
- **SQS 직렬화** — SQS를 앞에 두어 Lambda 실행을 직렬로 처리한다.

### 3. 모니터링과 사전 감지 체계

이번 장애는 서비스가 완전히 멈춘 뒤에야 인지했다. 다음 항목에 대한 알림이 있었다면 connection 고갈이 되기 전에 대응할 수 있었다.

- **Slow Query Log 임계값 설정** — 일정 시간 이상 실행되는 쿼리를 자동으로 감지
- **DB connection 사용률 알림** — 사용률이 임계치를 넘으면 알림 발송
- **쿼리 실행 시간 대시보드** — 주기적으로 실행되는 Lambda 쿼리의 실행 시간 추이 모니터링

장애를 "해결"하는 것도 중요하지만, "예방"하는 체계를 갖추는 것이 더 근본적인 해결책이다.
