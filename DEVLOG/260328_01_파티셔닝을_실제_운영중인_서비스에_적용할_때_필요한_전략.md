# 파티셔닝을 실제 운영중인 서비스에 적용할 때 필요한 전략

## 개요
데이터베이스 파티셔닝(Database Partitioning)을 실제 운영 중인 서비스에 도입할 때 고려해야 할 전략과 주의사항을 정리한다. 단순한 개념 학습이 아닌, 운영 환경에서 마주치는 실무적인 의사결정 포인트에 초점을 맞춘다.

## 상세 내용

### 1. 파티셔닝 도입 전 판단 기준

파티셔닝은 만능 해결책이 아니다. 섣불리 도입하면 오히려 복잡도만 늘고 성능은 개선되지 않는 경우도 있다.

**파티셔닝이 유효한 시점**

- 단일 테이블 크기가 수십 GB ~ TB 수준에 도달했을 때
- 특정 기간(최근 N일) 또는 특정 범위 데이터만 주로 조회하는 패턴이 명확할 때
- 오래된 데이터를 주기적으로 삭제해야 하는데, `DELETE` 문이 너무 느릴 때
- 인덱스 추가만으로는 쿼리 성능을 더 이상 개선할 수 없을 때

**파티셔닝 vs 샤딩 vs 아카이빙 비교**

| 전략 | 분산 단위 | 주목적 | 애플리케이션 변경 |
|---|---|---|---|
| 파티셔닝 | 단일 DB 서버 내 | 쿼리 성능 향상, 데이터 관리 편의 | 최소 (투명하게 동작) |
| 샤딩 | 여러 DB 서버 | 수평 확장 (Write 처리량) | 필수 (라우팅 로직 필요) |
| 아카이빙 | 별도 저장소로 이전 | 운영 테이블 경량화 | 필요 (아카이브 조회 로직) |

- **파티셔닝**: 단일 서버에서 처리 가능한 데이터량이지만 테이블 크기가 쿼리 성능에 영향을 주고 있을 때. 시계열 데이터나 로그 테이블이 대표적인 적용 대상.
- **샤딩**: 단일 서버의 Write 한계에 도달했을 때. 구현 복잡도가 매우 높아 파티셔닝과 아카이빙으로 해결되지 않을 때 최후 수단으로 고려.
- **아카이빙**: 오래된 데이터를 별도 저장소(cold storage, S3, 별도 DB 등)로 이전하여 운영 테이블을 가볍게 유지. 파티셔닝과 병행하면 DROP PARTITION으로 간단하게 처리 가능.

### 2. 파티셔닝 키 선정 전략

파티셔닝 키 선정은 가장 중요한 의사결정이다. 잘못 선정하면 파티셔닝을 했는데 파티션 프루닝이 전혀 되지 않아 전체 파티션을 풀스캔하는 상황이 된다.

**Range / List / Hash / Key 파티셔닝 실무 시나리오**

- **RANGE 파티셔닝**: 날짜 기반의 시계열 데이터에 가장 많이 사용된다. 로그 테이블, 주문 내역, 이벤트 이력 등이 대표적이다. 오래된 파티션을 통째로 DROP해서 데이터를 빠르게 삭제할 수 있는 것이 핵심 장점이다.

```sql
CREATE TABLE orders (
    id BIGINT NOT NULL,
    user_id BIGINT NOT NULL,
    created_at DATETIME NOT NULL,
    status VARCHAR(20) NOT NULL
)
PARTITION BY RANGE (TO_DAYS(created_at)) (
    PARTITION p202501 VALUES LESS THAN (TO_DAYS('2025-02-01')),
    PARTITION p202502 VALUES LESS THAN (TO_DAYS('2025-03-01')),
    PARTITION p202503 VALUES LESS THAN (TO_DAYS('2025-04-01')),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);
```

- **LIST 파티셔닝**: 지역 코드, 국가 코드, 상태값(enum) 등 값의 집합이 명확하게 정해져 있을 때 사용한다. 예를 들어 국가별 데이터를 물리적으로 분리하는 다국어 서비스에서 활용할 수 있다.

```sql
CREATE TABLE user_activity (
    id BIGINT NOT NULL,
    region_code TINYINT NOT NULL,
    activity_type VARCHAR(50)
)
PARTITION BY LIST (region_code) (
    PARTITION p_kr VALUES IN (1),
    PARTITION p_us VALUES IN (2),
    PARTITION p_jp VALUES IN (3),
    PARTITION p_others VALUES IN (4, 5, 6, 7)
);
```

- **HASH 파티셔닝**: 특정 컬럼으로 데이터를 균등하게 분산시키고 싶을 때 사용한다. 프루닝 효과는 제한적이지만 데이터가 편중되지 않는 장점이 있다. 단, 정수형 컬럼 또는 정수를 반환하는 표현식만 사용 가능하다.

- **KEY 파티셔닝**: HASH와 유사하지만 MySQL이 내부 해시 함수를 제공하므로 정수 외에 DATE, VARCHAR 등 다양한 타입도 파티셔닝 키로 사용할 수 있다.

**파티셔닝 키와 WHERE 조건의 관계**

파티셔닝의 핵심 성능 이점은 파티션 프루닝(Partition Pruning)이다. 프루닝은 WHERE 조건이 파티셔닝 키와 직접 비교될 때만 동작한다.

```sql
-- 프루닝 동작 (created_at이 파티셔닝 키)
SELECT * FROM orders WHERE created_at >= '2025-03-01' AND created_at < '2025-04-01';

-- 프루닝 미동작 (user_id는 파티셔닝 키가 아님)
SELECT * FROM orders WHERE user_id = 12345;
```

두 번째 쿼리처럼 파티셔닝 키가 WHERE 조건에 포함되지 않으면 MySQL은 모든 파티션을 풀스캔하게 되어 오히려 일반 테이블보다 느릴 수 있다.

**복합 파티셔닝(Sub-partitioning)이 필요한 경우**

RANGE 또는 LIST 파티셔닝을 기준 파티셔닝으로 하고, 그 안에서 HASH 또는 KEY로 서브파티셔닝을 구성할 수 있다. 단일 파티션 내 데이터가 여전히 많아 성능 문제가 있거나, 파티션별로 파일 수준의 물리적 분산이 필요한 경우에 사용한다. 단, 파티션 수가 급격히 증가하므로 신중하게 결정해야 한다.

```sql
CREATE TABLE logs (
    log_id BIGINT NOT NULL,
    log_date DATE NOT NULL,
    user_id INT NOT NULL
)
PARTITION BY RANGE (YEAR(log_date))
SUBPARTITION BY HASH (user_id) SUBPARTITIONS 4 (
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p2025 VALUES LESS THAN (2026)
);
```

### 3. 기존 운영 테이블에 파티셔닝 적용하기

이미 운영 중인 테이블에 파티셔닝을 적용하는 것은 신규 테이블에 적용하는 것보다 훨씬 까다롭다. MySQL에서 일반 테이블을 파티셔닝 테이블로 변환하는 `ALTER TABLE ... PARTITION BY ...` 구문은 기본적으로 테이블 전체를 재구성(table rebuild)하기 때문에 대형 테이블에서는 상당한 시간과 락이 발생한다.

**무중단 마이그레이션 전략 비교**

| 방법 | 원리 | 특징 | 적합한 상황 |
|---|---|---|---|
| MySQL Online DDL | `ALTER TABLE ... ALGORITHM=INPLACE` | MDL 락 존재, 일부 DDL은 COPY 강제 | 소규모 테이블 또는 중단 허용 가능 시 |
| pt-online-schema-change | 트리거 기반 섀도 테이블 | 트리거 오버헤드, Percona Toolkit 필요 | 트리거 사용이 가능한 환경 |
| gh-ost | 바이너리 로그 기반 섀도 테이블 | 트리거 없음, 동적 스로틀링, 일시정지 가능 | 대형 테이블, 고트래픽 운영 환경 |

**gh-ost를 통한 무중단 마이그레이션 흐름**

gh-ost는 GitHub이 개발한 트리거 없는(Triggerless) MySQL 온라인 스키마 변경 도구다. 트리거 대신 바이너리 로그 스트림을 구독하여 원본 테이블의 변경사항을 섀도 테이블에 반영한다.

1. `_테이블명_gho` (섀도 테이블): 새 스키마로 생성된 빈 테이블. 원본 데이터를 청크 단위로 복사하면서 동시에 binlog를 통해 변경사항도 반영한다.
2. `_테이블명_ghc` (변경 로그 테이블): 마이그레이션 상태와 로그를 저장한다.
3. 복사 완료 후 원자적 테이블 교체(`RENAME TABLE`)로 전환.

```bash
gh-ost \
  --user="root" --password="pass" \
  --host="replica.host" \
  --database="mydb" --table="orders" \
  --alter="PARTITION BY RANGE (TO_DAYS(created_at)) (...)" \
  --execute
```

gh-ost는 복제본(Replica)에서 binlog를 읽어오기 때문에 마스터 부하를 최소화할 수 있으며, 마이그레이션 도중 스로틀링을 동적으로 조절하거나 일시정지도 가능하다.

**대용량 테이블 전환 시 락 이슈**

MySQL Online DDL에서 파티셔닝 변환은 `ALGORITHM=COPY`로 처리되는 경우가 많다. 이 경우 DML 작업을 허용하더라도 초기와 완료 시점에 메타데이터 락(MDL, Metadata Lock)을 획득하기 위해 기존 트랜잭션이 모두 완료될 때까지 기다린다. 장시간 실행 중인 트랜잭션이나 유휴 트랜잭션이 있으면 Online DDL이 블로킹될 수 있다.

실무 대응 방안:
- DDL 실행 전 `SHOW PROCESSLIST`로 장기 실행 트랜잭션 확인 및 종료
- `innodb_lock_wait_timeout` 값을 적절히 설정하여 타임아웃 제어
- gh-ost 사용 시 `--max-load` 옵션으로 서버 부하에 따른 자동 스로틀링 적용

**데이터 정합성 검증 절차**

마이그레이션 후 반드시 아래 항목을 검증해야 한다.

```sql
-- 원본과 신규 테이블 행 수 비교
SELECT COUNT(*) FROM orders;               -- 원본
SELECT COUNT(*) FROM orders_partitioned;   -- 신규

-- 파티션별 데이터 분포 확인
SELECT PARTITION_NAME, TABLE_ROWS
FROM INFORMATION_SCHEMA.PARTITIONS
WHERE TABLE_NAME = 'orders_partitioned';

-- 체크섬 비교 (샘플링 방식)
CHECKSUM TABLE orders;
CHECKSUM TABLE orders_partitioned;
```

**롤백 계획**

gh-ost나 pt-osc는 원본 테이블을 삭제하지 않고 이름을 변경한 뒤 보관한다(`_테이블명_del`). 문제 발생 시 빠르게 원본 테이블로 되돌릴 수 있도록 테이블 교체 이전에 원본을 일정 기간 보존한다. `RENAME TABLE`은 원자적으로 동작하므로 교체 과정 자체에서의 데이터 손실은 없다.

### 4. 파티션 관리 운영 자동화

파티셔닝 도입보다 이후 운영이 더 중요하다. 특히 RANGE 파티셔닝을 날짜 기반으로 운영하면 매월 또는 매주 새 파티션을 추가해야 하며, 일정 기간이 지난 파티션은 삭제해야 한다. 이를 수동으로 관리하다 보면 반드시 누락이 발생한다.

**파티션 추가 자동화 (파티션 로테이션)**

MySQL Event Scheduler를 활용하여 주기적으로 새 파티션을 추가하는 Stored Procedure를 호출한다.

```sql
DELIMITER $$

CREATE PROCEDURE add_monthly_partition()
BEGIN
    DECLARE next_month DATE;
    DECLARE partition_name VARCHAR(20);
    DECLARE boundary_value VARCHAR(20);

    SET next_month = DATE_ADD(LAST_DAY(NOW()), INTERVAL 1 DAY);  -- 다음달 1일
    SET partition_name = CONCAT('p', DATE_FORMAT(next_month, '%Y%m'));
    SET boundary_value = DATE_FORMAT(DATE_ADD(next_month, INTERVAL 1 MONTH), '%Y-%m-01');

    -- MAXVALUE 파티션을 제거하고 새 파티션 + MAXVALUE 재추가
    SET @sql = CONCAT(
        'ALTER TABLE orders REORGANIZE PARTITION p_future INTO (',
        'PARTITION ', partition_name, ' VALUES LESS THAN (TO_DAYS(''', boundary_value, ''')),',
        'PARTITION p_future VALUES LESS THAN MAXVALUE)'
    );
    PREPARE stmt FROM @sql;
    EXECUTE stmt;
    DEALLOCATE PREPARE stmt;
END$$

DELIMITER ;

-- 매월 25일 23:00에 다음달 파티션 생성
CREATE EVENT evt_add_monthly_partition
ON SCHEDULE EVERY 1 MONTH
STARTS '2025-01-25 23:00:00'
DO CALL add_monthly_partition();
```

**오래된 데이터 파티션 DROP vs TRUNCATE 전략**

| 방법 | 속도 | 복구 가능 여부 | 용도 |
|---|---|---|---|
| `DROP PARTITION` | 매우 빠름 (파일 삭제) | 불가 | 완전 삭제가 목적일 때 |
| `TRUNCATE PARTITION` | 빠름 (데이터만 삭제) | 불가 | 파티션 구조는 유지하되 데이터만 비울 때 |
| `DELETE ... WHERE` | 느림 (행 단위 삭제) | 트랜잭션 롤백 가능 | 조건부 삭제가 필요할 때 |

오래된 파티션을 통째로 삭제하는 것이 파티셔닝의 핵심 이점 중 하나다. `DELETE WHERE created_at < '2024-01-01'`은 수백만 건의 행을 하나씩 삭제하지만, `ALTER TABLE orders DROP PARTITION p202401`은 파일 시스템 레벨에서 즉시 처리된다.

```sql
-- 특정 파티션 데이터만 비우기 (파티션 구조 유지)
ALTER TABLE orders TRUNCATE PARTITION p202401;

-- 파티션 자체를 완전 삭제
ALTER TABLE orders DROP PARTITION p202401;
```

**모니터링 항목**

```sql
-- 파티션별 데이터 분포 및 크기 모니터링
SELECT
    PARTITION_NAME,
    TABLE_ROWS,
    DATA_LENGTH / 1024 / 1024 AS data_mb,
    INDEX_LENGTH / 1024 / 1024 AS index_mb
FROM INFORMATION_SCHEMA.PARTITIONS
WHERE TABLE_SCHEMA = 'mydb' AND TABLE_NAME = 'orders'
ORDER BY PARTITION_ORDINAL_POSITION;
```

파티션 수가 너무 빠르게 늘어나거나 특정 파티션으로 데이터가 집중되는 현상(핫 파티션)을 정기적으로 확인해야 한다.

### 5. 파티셔닝 적용 후 쿼리 성능 관리

**파티션 프루닝(Partition Pruning) 동작 조건**

파티션 프루닝이란 옵티마이저가 WHERE 조건을 분석하여 해당 조건을 만족하는 행이 없는 파티션은 스캔에서 제외하는 최적화 기법이다. 프루닝이 제대로 동작하면 쿼리 속도가 파티션 수 배만큼 향상될 수 있다.

프루닝이 동작하는 조건:
- WHERE 절에 파티셔닝 키 컬럼이 직접 포함될 것
- 비교 연산자(`=`, `<`, `>`, `<=`, `>=`, `BETWEEN`, `IN`)를 사용할 것
- 파티셔닝 컬럼에 함수를 적용하지 않을 것 (단, `YEAR()`, `TO_DAYS()`, `TO_SECONDS()`는 MySQL이 최적화)

프루닝이 동작하지 않는 경우:
```sql
-- 함수 적용으로 프루닝 불가
SELECT * FROM orders WHERE DATE_FORMAT(created_at, '%Y-%m') = '2025-03';

-- 파티셔닝 키가 아닌 컬럼으로만 조회
SELECT * FROM orders WHERE status = 'PENDING';

-- HASH/KEY 파티셔닝에서 범위 조건의 범위가 파티션 수보다 클 때
-- (파티션이 8개인데 범위가 9개 이상의 값을 포함하는 경우)
```

**EXPLAIN을 통한 실행 계획 분석**

```sql
-- MySQL 5.7 이하
EXPLAIN PARTITIONS SELECT * FROM orders WHERE created_at >= '2025-03-01';

-- MySQL 8.0 이상 (PARTITIONS 키워드 없이 자동 표시)
EXPLAIN SELECT * FROM orders WHERE created_at >= '2025-03-01';
```

EXPLAIN 결과의 `partitions` 컬럼에서 실제로 접근하는 파티션 목록을 확인한다. 전체 파티션이 나열된다면 프루닝이 되지 않고 있다는 의미다.

```
+----+-------------+--------+------------+-------+
| id | select_type | table  | partitions | type  |
+----+-------------+--------+------------+-------+
|  1 | SIMPLE      | orders | p202503    | range |
+----+-------------+--------+------------+-------+
```

`partitions` 컬럼에 단 하나 또는 소수의 파티션만 표시되면 프루닝이 정상 동작하는 것이다.

**파티셔닝 적용 후 오히려 느려지는 케이스**

- 파티셔닝 키가 WHERE 조건에 없는 쿼리: 전체 파티션을 순차 스캔하여 오히려 더 느림
- 파티션 수가 너무 많은 경우: MySQL 내부적으로 파티션 메타데이터를 관리하는 오버헤드 발생
- 파티션을 넘나드는 집계 쿼리: `GROUP BY`, `ORDER BY` 시 각 파티션 결과를 합산하는 비용 발생
- 파티션 간 JOIN: 파티셔닝된 테이블과 일반 테이블 JOIN 시 파티션 수만큼 반복 접근 가능

**인덱스 전략: 로컬 인덱스**

MySQL의 파티셔닝은 로컬 인덱스(Local Index)만 지원한다. 즉, 인덱스는 각 파티션 내에서만 유효하고 파티션 전체를 아우르는 글로벌 인덱스(Global Index)는 존재하지 않는다. 이는 파티셔닝 키가 포함되지 않은 컬럼으로의 UNIQUE 제약이나 빠른 조회가 불가능함을 의미한다.

### 6. 파티셔닝과 함께 고려할 아키텍처 설계

**파티셔닝과 레플리케이션 조합 시 주의점**

파티셔닝은 레플리케이션과 독립적으로 동작한다. 마스터에서 파티셔닝된 테이블은 레플리카에도 동일한 파티셔닝 구조로 복제된다. 주의할 점은 파티션 추가/삭제 DDL(`ALTER TABLE ... ADD PARTITION`, `DROP PARTITION`)도 binlog를 통해 레플리카에 전파된다는 것이다. 따라서 마스터에서 파티션을 DROP할 때 레플리카에도 동일하게 적용되므로, 레플리카를 분석용 쿼리 전용으로 사용 중이라면 레플리케이션 지연 상황에서의 DDL 충돌을 주의해야 한다.

**파티셔닝된 테이블의 JOIN 성능 이슈**

파티셔닝된 테이블을 JOIN할 때 드라이빙 테이블의 각 행에 대해 파티션 프루닝이 동작하기는 하지만, 파티션 수가 많을수록 옵티마이저의 실행 계획 수립 비용이 증가한다. 가능하면 JOIN 조건에 파티셔닝 키가 포함되도록 쿼리를 작성하거나, JOIN 대상 테이블의 파티션 수를 적절히 제한해야 한다.

**ORM/애플리케이션 레벨에서의 파티션 인지 쿼리 작성**

ORM(JPA 등)은 파티셔닝을 투명하게 처리하므로 별도 설정이 필요하지 않다. 그러나 프루닝 동작 여부가 성능을 크게 좌우하기 때문에, ORM이 생성하는 쿼리의 WHERE 조건에 파티셔닝 키가 포함되는지 반드시 확인해야 한다.

```java
// 잘못된 예: created_at 조건 없이 user_id만으로 조회 (전체 파티션 풀스캔)
orderRepository.findByUserId(userId);

// 올바른 예: created_at 범위 조건 추가 (파티션 프루닝 동작)
orderRepository.findByUserIdAndCreatedAtBetween(userId, startDate, endDate);
```

**파티셔닝과 캐시 전략의 연계**

파티셔닝 적용 후에도 빈번하게 접근되는 최신 파티션(예: 이번 달, 이번 주)의 데이터는 Redis 등 캐시에 올려두면 DB 파티션 스캔 자체를 줄일 수 있다. 오래된 파티션은 캐시 우선순위를 낮추고 아카이빙 대상으로 관리하는 전략이 효과적이다.

### 7. 실무에서 자주 만나는 함정과 트러블슈팅

**Unique Key 제약조건과 파티셔닝 키의 관계**

MySQL 파티셔닝의 핵심 제약 중 하나: **테이블의 모든 유니크 키(Primary Key 포함)는 파티셔닝 표현식에 사용된 모든 컬럼을 포함해야 한다.**

```sql
-- 오류 발생: id가 PRIMARY KEY이지만 파티셔닝 키(created_at)를 포함하지 않음
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT,
    created_at DATETIME NOT NULL,
    PRIMARY KEY (id)
)
PARTITION BY RANGE (TO_DAYS(created_at)) (...);
-- ERROR 1503 (HY000): A PRIMARY KEY must include all columns in the table's partitioning function

-- 해결책: PRIMARY KEY에 파티셔닝 키를 포함
CREATE TABLE orders (
    id BIGINT NOT NULL,
    created_at DATETIME NOT NULL,
    PRIMARY KEY (id, created_at)
)
PARTITION BY RANGE (TO_DAYS(created_at)) (...);
```

`id`만으로는 전역 유일성을 보장할 수 없게 되어 (파티션 로컬 유니크만 보장) 애플리케이션에서 중복 확인 로직을 별도로 관리해야 하는 상황이 생긴다. 이를 해결하기 위해 `AUTO_INCREMENT` 대신 UUID나 Snowflake 기반의 분산 ID를 사용하는 방식도 고려할 수 있다.

**Foreign Key를 사용할 수 없는 제약**

파티셔닝된 InnoDB 테이블은 Foreign Key를 지원하지 않는다. 파티셔닝된 테이블은 다른 테이블의 FK 참조 대상이 될 수 없으며, 파티셔닝된 테이블도 다른 테이블을 FK로 참조할 수 없다.

기존에 FK를 사용하던 테이블에 파티셔닝을 적용하려면:
- FK 제약을 DROP하고 파티셔닝 적용
- 참조 무결성을 애플리케이션 레벨에서 직접 관리

```sql
-- FK가 있는 테이블에 파티셔닝 적용 시도 시 오류
-- ERROR 1506 (HY000): Foreign keys are not yet supported in conjunction with partitioning
```

**AUTO_INCREMENT 동작 차이**

파티셔닝된 테이블에서 `AUTO_INCREMENT`는 테이블 전체에서 단조 증가하지만, 파티션 키가 PRIMARY KEY에 포함되어야 하는 제약 때문에 복합 PK 구성이 강제된다. 이 경우 `AUTO_INCREMENT` 컬럼 단독으로 `UNIQUE` 제약을 줄 수 없어 전역 유일성이 테이블 레벨이 아닌 파티션 레벨로 한정될 수 있다. 대부분의 경우 `AUTO_INCREMENT` + 파티셔닝 키를 복합 PK로 구성하면 AUTO_INCREMENT 단독 유니크 특성은 유지된다.

**파티션 수가 과도하게 많아졌을 때의 성능 저하**

MySQL의 최대 파티션 수는 8,192개(서브파티션 포함)이지만, 실무에서는 수백 개만 넘어도 성능 저하 증상이 나타날 수 있다.

- 옵티마이저가 실행 계획을 수립할 때 모든 파티션의 메타데이터를 검사하는 오버헤드 발생
- `INFORMATION_SCHEMA.PARTITIONS` 조회 성능 저하
- `OPEN_FILES_LIMIT` 제한에 걸릴 수 있음 (파티션마다 파일이 생성됨)

해결 방법:
- 파티션 크기를 조정하여 파티션 수를 줄임 (월별 → 분기별 등)
- 오래된 파티션을 주기적으로 DROP 또는 아카이빙
- 파티션 병합(`ALTER TABLE ... REORGANIZE PARTITION`)으로 소형 파티션 통합

## 핵심 정리
- 파티셔닝은 도입보다 운영이 핵심이다. 파티션 키 선정, 자동화된 관리, 쿼리 최적화까지 종합적으로 설계해야 한다.
- 기존 운영 테이블에 적용할 때는 무중단 마이그레이션 전략(gh-ost 권장)과 롤백 계획이 반드시 수반되어야 한다.
- 파티션 프루닝이 제대로 동작하는지 `EXPLAIN`으로 반드시 확인하고, 프루닝이 되지 않는 쿼리는 오히려 성능이 저하될 수 있다.
- Unique Key와 Primary Key는 파티셔닝 키 컬럼을 반드시 포함해야 하고, Foreign Key는 지원되지 않으므로 사전에 애플리케이션 레벨 대응 방안을 마련해야 한다.
- 파티션 관리(추가/삭제)는 Event Scheduler와 Stored Procedure를 활용해 자동화하고, INFORMATION_SCHEMA.PARTITIONS로 정기 모니터링을 수행한다.

### 8. 시계열 데이터 테이블에 운영 중 파티셔닝 적용하기

1분/5분/15분마다 데이터가 지속적으로 쌓이는 모니터링 메트릭, 센서 데이터 등 시계열 테이블은 **신규 테이블 컷오버** 전략이 가장 현실적이다.

#### 왜 일반 ALTER TABLE은 안 되나

```sql
-- 이 명령은 테이블 전체를 재구성 → 수억 건이면 수십 분 ~ 수 시간 락
ALTER TABLE metrics_1min PARTITION BY RANGE (...);
```

1분마다 INSERT가 들어오는 테이블에 이 명령을 실행하면 그 시간 동안 **전체 쓰기가 중단**된다.

#### 파티션 단위 선택

보관 주기와 쿼리 패턴에 따라 파티션 단위를 다르게 가져간다.

| 테이블 | 데이터 밀도 | 권장 파티션 단위 | 이유 |
|--------|------------|-----------------|------|
| 1분 집계 | 높음 | **일별(Daily)** | 단기 보관, 빠른 로테이션 |
| 5분 집계 | 중간 | **주별(Weekly)** | 중기 보관 |
| 15분 집계 | 낮음 | **월별(Monthly)** | 장기 보관 |

> **주의**: 1분 데이터를 시간별로 파티션 나누면 하루 24개, 1년 8760개로 파티션 수가 폭증한다. 파티션이 1000개를 넘으면 메타데이터 조회 자체가 느려지므로 **일별이 최소 단위**가 안전하다.

#### 신규 테이블 컷오버 전략

시계열 데이터는 과거 수정이 없고 INSERT만 발생한다는 특성을 활용한다.

```
[현재]                    [전환 후]
metrics_1min (무파티션) → metrics_1min (파티션 적용)
                          metrics_1min_old (구 데이터 보관)
```

**Step 1. 파티션 적용된 신규 테이블 생성**

```sql
CREATE TABLE metrics_1min_new (
    id          BIGINT NOT NULL,
    metric_name VARCHAR(100) NOT NULL,
    value       DOUBLE NOT NULL,
    recorded_at DATETIME NOT NULL,
    PRIMARY KEY (id, recorded_at)   -- 파티션 키가 PK에 포함 필수
) PARTITION BY RANGE (TO_DAYS(recorded_at)) (
    PARTITION p20260327 VALUES LESS THAN (TO_DAYS('2026-03-28')),
    PARTITION p20260328 VALUES LESS THAN (TO_DAYS('2026-03-29')),
    PARTITION p_future  VALUES LESS THAN MAXVALUE  -- 자동화 실패 시 INSERT 오류 방지 안전망
);
```

**Step 2. 듀얼 라이트 (Dual Write)**

트리거 방식과 애플리케이션 방식 중 선택한다.

```sql
-- 트리거 방식 (애플리케이션 변경 없음, 단 INSERT 성능에 오버헤드 발생)
CREATE TRIGGER sync_to_new
AFTER INSERT ON metrics_1min
FOR EACH ROW
  INSERT INTO metrics_1min_new VALUES (NEW.id, NEW.metric_name, NEW.value, NEW.recorded_at);
```

```java
// 애플리케이션 방식 (트리거 오버헤드 없음, 코드 배포 필요)
metricsRepository.save(metric);        // 기존 테이블
metricsNewRepository.save(metric);     // 신규 파티션 테이블
```

> 1분마다 대량 INSERT가 발생하는 테이블은 트리거 오버헤드가 누적되므로 **애플리케이션 레벨 듀얼 라이트**가 더 안전하다.

**Step 3. 구 데이터 배치 마이그레이션 (서비스 중 진행)**

```sql
-- 날짜 단위로 조금씩 옮기기. 한 번에 하면 I/O 폭증
INSERT INTO metrics_1min_new
SELECT * FROM metrics_1min
WHERE recorded_at >= '2026-01-01' AND recorded_at < '2026-01-02';

-- 다음 날짜 범위로 반복 (배치 단위를 슬로우 쿼리 임계값 이하로 조절)
```

**Step 4. RENAME TABLE으로 컷오버 (수 초 락)**

구 데이터 마이그레이션이 완료되면 새벽 트래픽 저점에 수행한다.

```sql
-- 메타데이터 락만 걸리므로 실제 중단 시간은 밀리초 ~ 수 초
RENAME TABLE metrics_1min     TO metrics_1min_old,
             metrics_1min_new TO metrics_1min;
```

**Step 5. 듀얼 라이트 제거 + 구 테이블 정리**

```java
// 신규 테이블(파티션 적용)만 쓰도록 코드 배포
metricsRepository.save(metric);
```

구 테이블은 일정 기간 보관 후 DROP한다.

#### 전체 흐름 요약

```
Day 1:   신규 파티션 테이블 생성
Day 1~:  듀얼 라이트 시작 (두 테이블 동시 INSERT)
Day 1~:  구 데이터 배치 마이그레이션 (날짜 단위로 천천히)
Day N:   RENAME TABLE으로 컷오버 (수 초)
Day N+1: 듀얼 라이트 제거 배포
Day N+7: 구 테이블 DROP
```

#### INSERT 핫스팟은 오히려 장점

항상 최신 파티션에만 INSERT가 집중되지만, 이는 역으로 장점이다. InnoDB 버퍼 풀에서 최신 파티션 페이지가 메모리에 상주하여 디스크 I/O 없이 INSERT가 가능하다. 단, `innodb_buffer_pool_size`를 충분히 확보해야 한다.

#### 적용 순서

세 테이블을 동시에 전환하지 말고, **데이터 밀도가 낮은 것부터** 검증 후 순차 적용한다.

```
15분 테이블 → 검증 → 5분 테이블 → 검증 → 1분 테이블
```

## 키워드

- `Database Partitioning`: 단일 DB 서버 내에서 하나의 테이블을 여러 물리적 파티션으로 분할하여 저장하는 기법. 쿼리 성능 향상과 데이터 관리 편의성이 주목적이다.
- `Partition Pruning`: 옵티마이저가 WHERE 조건을 분석하여 해당 조건을 만족하는 행이 없는 파티션을 쿼리 실행 대상에서 제외하는 최적화 기법.
- `Range Partitioning`: 파티셔닝 키 컬럼의 값 범위(`LESS THAN`)를 기준으로 각 파티션에 행을 분배하는 방식. 날짜 기반 시계열 데이터에 가장 많이 사용된다.
- `파티션 키 선정`: 파티셔닝 적용 시 가장 중요한 의사결정. 주 쿼리 패턴의 WHERE 조건과 일치해야 파티션 프루닝이 동작하여 성능 이점을 얻을 수 있다.
- `무중단 마이그레이션`: 서비스를 중단하지 않고 대형 테이블의 스키마를 변경하는 전략. gh-ost나 pt-osc 같은 도구를 활용한다.
- `gh-ost`: GitHub이 개발한 트리거 없는(Triggerless) MySQL 온라인 스키마 변경 도구. 바이너리 로그를 구독하여 섀도 테이블에 변경사항을 반영하고, 원자적 테이블 교체로 마이그레이션을 완료한다.
- `pt-online-schema-change`: Percona Toolkit의 온라인 스키마 변경 도구. 트리거 기반으로 동작하며, 트리거 오버헤드가 있지만 오랫동안 검증된 도구다.
- `파티션 로테이션`: 날짜 기반 파티셔닝에서 새로운 파티션을 주기적으로 추가하고 오래된 파티션을 삭제하는 운영 작업. Event Scheduler와 Stored Procedure로 자동화한다.
- `EXPLAIN PARTITIONS`: 파티션 프루닝 동작 여부를 확인하는 실행 계획 분석 명령. MySQL 8.0에서는 `EXPLAIN SELECT ...`만으로 `partitions` 컬럼이 자동 표시된다.
- `데이터 아카이빙`: 오래된 데이터를 운영 테이블에서 분리하여 별도 저장소로 이전하는 전략. 파티셔닝과 병행하면 `DROP PARTITION`으로 대량 데이터를 빠르게 삭제할 수 있다.
